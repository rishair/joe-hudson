# QA: Coach web-app: chat E2E, persistence, modal, mobile, streaming

## Scope

Coach web-app (G-010), all user-facing surfaces shipped by E-039..E-046 plus the
E-052 progressive-streaming fix:

- E-045 first-visit welcome flow
- E-041 client-side SQLite conversation persistence
- E-042 wiki view + wikilink resolution
- E-043 v5b coach pipeline end-to-end (retrieval → walker → coach reply)
- E-052 progressive streaming (progress events during the retrieval wait)
- E-044 resource-attribution modal (split layout, switch, close)
- mobile rendering at phone viewport
- error states + security smoke (no secret leak, CORP/COEP headers)

## Test plan

Full pass per [[qa-frontend]]. Concrete scenarios with expected outcomes:

1. **Welcome (E-045)** — fresh visit auto-opens the native `<dialog>` with the four
   sections (What / How / Privacy / Get started), explicit unaffiliation + privacy
   language. "I got it →" dismisses + writes `localStorage.welcomeAcknowledged=true`.
   Reload does NOT reopen. `?` help icon reopens (button now reads "Close"). Escape closes.
2. **Persistence (E-041)** — OPFS `web-app.sqlite3` exists; migration log lines fire on
   first load; "+ New conversation" adds a sidebar entry; reload preserves conversations.
3. **Wiki (E-042)** — `/wiki` index renders 2376 entries across 13 categories;
   `/wiki/concepts/limiting-belief` renders title + substantial body + resolved wikilinks,
   0 broken; clicking a body wikilink navigates to the target page; `/wiki/<bad>` → 404.
4. **Coach E2E (E-043)** — one real turn ("I feel like an imposter at work…") yields a
   coherent Joe-style question (ask, not advise) with a per-message resource strip.
5. **Streaming (E-052)** — first progress event arrives <1s after Send; progress stages
   tick through (analyzing → walking → composing) during the retrieval wait; progress line
   hides once coach text arrives. (Prior pass baseline: 42s of silence before any text.)
6. **Modal (E-044)** — clicking the resource strip opens the split `<dialog>` (left = focused
   wiki entry body, right = resource list); clicking another resource swaps the left pane;
   Escape closes.
7. **Mobile** — 412×915 phone viewport; check horizontal overflow, sidebar behaviour,
   chat-input usability.
8. **Error states** — empty input disables Send; bad wiki page → 404; bad `/api/wiki/page`
   slug → JSON 404.
9. **Security smoke** — scan ALL client JS chunks for `sk-or-v1-` / `sk-ant-` / `hf_` /
   system-prompt fragments; verify every chunk + root carry `CORP: same-origin` +
   `COEP: require-corp`.

### 7. Mobile rendering — NEEDS-FIX (2 important)

Device emulation: Pixel-5 UA, 412×915, `isMobile`, `hasTouch`, DPR 2.

```
visualViewport (phone screen): 412 px
layoutViewport / docScrollWidth: 527 px   ← page is wider than the screen
horizontalOverflow:             true (527 > 412)
sidebar width:                  216 px (fixed, never collapses)
chat input width / left:        200 px wide, starts at x=232
```
On a 412px phone the page lays out to 527px, forcing a horizontal scrollbar, and the fixed
216px sidebar leaves the chat input at ~200px. Screenshot `q001-10-mobile-chat.png` shows the
sidebar consuming the left half and the chat column cramped against it. No hamburger / collapse
affordance exists. The wiki surface is fine at mobile (no overflow). This reproduces findings
6.1 + 6.2 from the 2026-05-24 prior pass — E-052 did not touch mobile, so they remain open.

## Issues found

| # | Severity | Feature | Description | Repro | Recommended fix |
|---|---|---|---|---|---|
| 7.1 | **important** | Mobile | Sidebar never collapses on mobile. Fixed 216px sidebar (`pageStyles.sidebar`, `display:flex`, no media query) leaves the chat input ~200px wide at 412px viewport. | Load `/` at 412×915; sidebar = 216px, chat input = 200px. | Add `@media (max-width:768px)` to hide the sidebar behind a hamburger toggle, or stack sidebar→chat vertically. |
| 7.2 | **important** | Mobile | Horizontal overflow: document lays out to 527px on a 412px screen → horizontal scrollbar while typing. Same fixed-sidebar root cause plus the chat column's `maxWidth:720` not adapting. | Load `/` at 412×915; `documentElement.scrollWidth=527 > innerWidth=412`. | Same as 7.1 — responsive shell. Fix once and both 7.1 and 7.2 resolve. |
| 5.1 | minor | Streaming/coach latency | E-052 fixed perceived silence (first byte 0.6s) but the real coach text still takes ~47s because the retrieval+walker pass is ~47s wall-clock on the 2376-entry corpus. | Send any concern prompt; progress ticks for ~47s before text. | Either a walker step/edge budget review (the seed-detector + ~8 walker steps dominate) or a docs note that 40-50s is expected. Carried over from prior pass finding 5.1. |

No `blocker` issues. No secret leaks. No console/page errors beyond the cosmetic favicon 404.

## Verdict

**needs-fix.** Per the [[qa-frontend]] severity rule, `passed` requires zero `blocker` AND zero
`important`. There are 0 blockers but 2 `important` mobile-layout findings (7.1 sidebar never
collapses, 7.2 horizontal overflow), so the verdict is `needs-fix`.

Caveat for the orchestrator: these two are NOT regressions and NOT new — they were already
documented as `important` in the 2026-05-24 prior pass and are out of scope for E-052 (which
only touched streaming). Desktop is fully shippable; every feature works end-to-end on desktop
with no blockers. If "mobile-responsive" is treated as a G-010 nice-to-have (the goal lists
"Mobile-responsive design" under Nice-to-haves, not Must-haves), the orchestrator may elect to
park 7.1/7.2 as backlog with explicit reason and accept the desktop build — but the QA verdict
as scoped stays `needs-fix` until a responsive shell lands and a re-run confirms 412px renders
without overflow.
