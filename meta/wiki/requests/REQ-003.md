# Cloudflare account, API token, and GitHub repo secrets for E-046 deploy

## What's needed

To complete [[E-046]] (deploy the web-app to Cloudflare with an automated GitHub→Cloudflare pipeline), the following must be provisioned by the user:

1. **Cloudflare account.** Free tier is sufficient for v1 (Workers Free includes 100k requests/day, 10ms CPU per request; we'll upgrade to Paid only if cost-cap or CPU profile demands it).
2. **`CLOUDFLARE_API_TOKEN`** — a scoped token created at https://dash.cloudflare.com/profile/api-tokens with these permissions:
   - `Account > Cloudflare Workers Scripts > Edit`
   - `Account > Account Settings > Read`
   - `User > User Details > Read`
   - `Account > Workers KV Storage > Edit` (only if R2/KV caching is added; not needed for v1)
   Save the token somewhere safe — it is shown only once.
3. **`CLOUDFLARE_ACCOUNT_ID`** — visible on the right sidebar of the Cloudflare dashboard home page.
4. **GitHub repo secrets** added under https://github.com/rishair/joe-hudson/settings/secrets/actions:
   - `CLOUDFLARE_API_TOKEN` (from step 2)
   - `CLOUDFLARE_ACCOUNT_ID` (from step 3)
   - `OPENROUTER_API_KEY` (same value as local `.env`)
5. **`gh auth login`** (optional — only needed if you want me to add the secrets via CLI rather than the GitHub UI). Skipping is fine — the dashboard works.

## Why it's needed

E-046's success criteria require:
- The web app being reachable at a public URL
- An automated push-to-main → rebuild → redeploy pipeline
- `OPENROUTER_API_KEY` stored as a Cloudflare secret (never in the repo or client bundle)

None of these are achievable without an actual Cloudflare account, an API token to authenticate the deploy, and GitHub-side secrets to feed the GitHub Action. These are credentials only the user can create.

## What's been prepared

Everything that can be done without credentials has been done in the same E-046 work cycle:

- Cloudflare deployment approach evaluated and decided (OpenNext adapter → Workers, not Pages, not direct Workers, not Vercel — see E-046 Analysis)
- Code changes for Cloudflare-compatibility staged on the `web-app/` tree (env-detected static-assets fallback for the wiki content reader, `wrangler.jsonc`, `open-next.config.ts`, build/deploy scripts in `package.json`)
- `.github/workflows/deploy.yml` written with the canonical `cloudflare/wrangler-action@v3` shape
- Wiki content (`content/wiki/`) routed through the ASSETS binding to bypass the 10 MiB Worker bundle limit (full content is 38 MiB across 2,379 files)
- `cloudflare-deploy` playbook drafted with the pattern, gotchas, and a "how to apply to a new project" recipe

Once the credentials land:
1. `gh secret set` (or dashboard) for the three GitHub secrets
2. `wrangler secret put OPENROUTER_API_KEY` (one-time, to set the runtime secret on the Worker)
3. `git push origin main` triggers the first deploy
4. Verify the 6 features (welcome, chat, wiki, persistence, modal, cost-cap) work on the public URL
5. Move E-046 from claimed → succeeded and tick the last G-010 exit criterion

## What's blocked without it

- E-046 cannot complete (no actual deploy)
- G-010 exit criterion #12 (the Cloudflare-deploy must-have) cannot be ticked
- G-010 cannot move from `active` to `completed`
- Any future project that wants to reuse the `cloudflare-deploy` playbook is blocked on the same kind of credential chain (but the playbook itself can land regardless — it documents the pattern, not the secret values)

## Notes

- Free tier vs Paid: Workers Free covers the expected v1 traffic profile (hobbyist usage). Workers Paid is $5/mo and adds a higher CPU-per-request budget, which we may want if a single streaming coach turn (graph-walk + Sonnet walker + Sonnet coach + post-check) overshoots the 10ms free-tier CPU window. Recommendation: start Free, watch the metrics, upgrade if 503s appear.
- The token in step 2 can be revoked at any time from the same dashboard; nothing this project does is "locked in" by minting it.
- The `OPENROUTER_API_KEY` set as a Cloudflare secret is stored encrypted; only the Worker's running process sees its plaintext.
