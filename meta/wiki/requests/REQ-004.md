# Cloudflare Tunnel + Access dashboard prerequisites and API token for G-012 PaaS

## What's needed

To complete [[G-012]] (Hetzner personal-PaaS with Cloudflare Tunnel + Access routing), the following must be provisioned by the user:

### 1. Verify `ris.hair` DNS is on Cloudflare nameservers

Dashboard check: DNS section for the `ris.hair` zone shows status "Active." If not, the user must change nameservers at the domain registrar — a one-time but irreversible-feeling step.

### 2. SSL coverage path for `*.play.ris.hair` — DECIDED 2026-05-24

**Decision: per-subdomain explicit cert.** No wildcard, no recurring cost. URLs stay clean as `<project>.play.ris.hair`. `paas register` is responsible for:

1. Creating the DNS A record for `<project>.play.ris.hair` (pointing at the tunnel — actually a CNAME to `<tunnel-uuid>.cfargotunnel.com` per cloudflared convention)
2. Cloudflare auto-issues a single-name cert for that hostname (free; Universal SSL covers explicit hostnames even at two levels)
3. Polling cert issuance to active (~10-30s typical) before considering the registration complete

Rejected alternatives:
- **Advanced Certificate Manager ($10/mo per zone)**: covers `*.play.ris.hair` wildcard but recurring cost not justified for personal-play scope
- **Flatten to `<project>-play.ris.hair`**: free wildcard works, but loses the visual sub-subdomain clarity and complicates future Cloudflare Pages graduation

### 3. Enable Zero Trust on the Cloudflare account

Cloudflare Access requires Zero Trust to be enabled. Free for up to 50 users — no credit card required for the free tier. One-time activation in the dashboard: Zero Trust > Settings.

### 4. Set up identity providers in Zero Trust

Dashboard: Zero Trust > Settings > Authentication > Login methods:

- Add **Google** as a login method (any Google account, no Workspace) — guided wizard, no separate Google API setup needed.
- Add **One-time PIN** as a fallback (no setup beyond the toggle).

### 5. Mint an API token (OR commit to dashboard-only setup)

**If preferring the agent-driven path** (recommended — lets `paas` register projects without dashboard clicks):

Dashboard: Profile > API Tokens > Create Token > Custom token. Required scopes:

- `Account > Cloudflare Tunnel > Edit` (create/manage tunnels)
- `Zone > DNS > Edit` (scoped to the `ris.hair` zone — for both `cloudflared tunnel route dns` AND `paas register` creating per-subdomain DNS records under the per-subdomain-cert decision above)
- `Zone > SSL and Certificates > Edit` (scoped to the `ris.hair` zone — needed so `paas register` can poll cert-issuance status after creating each new DNS record; without this scope the tunnel still works but cert-readiness can't be programmatically verified, only inferred from successful HTTPS handshake)
- `Account > Access: Apps and Policies > Edit` (if Access app creation via API is desired; can be skipped if step 6 is done via dashboard)

Save token securely — Cloudflare shows it only once. Add to the box's environment as `CLOUDFLARE_TUNNEL_TOKEN` (or any name; E-047 reads from `.env`).

**Note**: this is a distinct token from REQ-003's `CLOUDFLARE_API_TOKEN` (which is scoped to `Cloudflare Workers Scripts:Edit` for the G-010 web app deploy). The two tokens CAN be combined into one with all six scopes, but they're distinct concerns. Recommend two separate tokens for principle-of-least-privilege.

**If preferring dashboard-only setup**: skip the token; the user creates the tunnel and Access app in the dashboard, copies the tunnel UUID and credentials JSON to the box, and `paas` operates purely on local files (no API calls). Simpler for one-time setup, less flexible going forward.

### 6. Configure the Cloudflare Access application for `*.play.ris.hair`

Either via dashboard (per the recipe in R-019, "Cloudflare Access setup for `*.play.ris.hair`" section) or via API call (if token from step 5 has the Access scope).

Settings:
- Application name: `play.ris.hair PaaS` (or similar)
- Self-hosted application type
- Application domain: subdomain `*`, domain `play.ris.hair`, path empty
- Session duration: 24 hours (reasonable for "only me")
- Identity providers: Google + One-time PIN
- Policy: single Include rule, `Emails | <user-email>`, action Allow

## Why it's needed

Without these prerequisites, E-047 cannot run (no tunnel can be created, no Access policy will protect the apps, DNS won't route). G-012's exit criteria — "Cloudflare Tunnel is running as a `systemd` service that survives a Hetzner reboot" and "Cloudflare Access blocks unauthenticated requests to `*.play.ris.hair`" — depend on this setup being in place.

## Notes

- Distinct from [[REQ-003]] which provisions `CLOUDFLARE_API_TOKEN` for G-010's Cloudflare Workers deploy. Different token scopes, different products (Workers vs Tunnel + Access). Both can be resolved from the same Cloudflare account dashboard.
- The SSL decision is resolved: per-subdomain explicit cert. `paas register` creates the DNS record + polls cert issuance per registration. No recurring cost.
- The dashboard-only-setup path (step 5 fallback) is fine for v1 and avoids minting a third token. The agent-driven path becomes valuable if/when `paas register` should be able to create new tunnels (e.g., one tunnel per project group).
