# Workers deploy token on the ris.hair account, for moving the web app to joebot.ris.hair

## What's needed

A separate, least-privilege Cloudflare API token minted on the **ris.hair account** (`Windowasher@gmail.com`'s account, id `75e26858f8d36b8fad28ad311dd0f360`), with exactly:

- **Account > Workers Scripts > Edit** (the ris.hair account) — deploy the Worker
- **Account > Account Settings > Read** (the ris.hair account) — wrangler reads account
- **Zone > Workers Routes > Edit** (zone: ris.hair) — bind the `joebot.ris.hair` custom domain
- **Zone > DNS > Edit** (zone: ris.hair) — custom-domain DNS record

Save to `.env` as `CLOUDFLARE_JOEBOT_WORKERS_TOKEN`. This is distinct from `CLOUDFLARE_TUNNEL_TOKEN` (PaaS: Tunnel+Access+DNS+SSL) and from the old `CLOUDFLARE_API_TOKEN` (Workers on the rishair.com account, to be retired after the move).

## Why it's needed

User decided 2026-05-25 to consolidate the web app onto the ris.hair account (same account as the G-012 PaaS) and rename it to "joebot", served at the custom domain `joebot.ris.hair`. The existing `CLOUDFLARE_TUNNEL_TOKEN` is PaaS-scoped and has **no Workers permission** (verified denied against the API), so it cannot deploy the Worker. A dedicated Workers token keeps app-deploy and PaaS as separately-revocable credentials (user chose the separate-token path over broadening the PaaS token).

## What's blocked without it

- [[E-056]] (move web-app deploy to ris.hair account, rename joebot, custom domain) cannot run.
