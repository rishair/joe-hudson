### What was tested

- OPFS file presence after bootstrap (should create `web-app.sqlite3`).
- Initial conversation auto-creation on a fresh visit.
- Migration logs in console (`[migrations] applied 0001-create-conversations`, `[migrations] applied 0002-create-messages`).
- "+ New conversation" button creates a second entry in the sidebar.

### Result — Functional, two polish gaps

```

after +New:               ['New conversation', 'New conversation']
after reload:             ['New conversation', 'New conversation']     // persistence works

after delete:             ['New conversation']                          // delete works
```

Console showed both migration log lines on first load. Schema persists across reloads.

### Screenshots

- `/tmp/pw-qa/qa-screenshots/03-chat-multiple-convos.png` — sidebar with two conversations, both titled "New conversation".

### Issues found

| # | Severity | Description |
|---|----------|-------------|
| 2.1 | minor | All conversations are titled literally "New conversation" — there is no auto-titling from the first user message, so the sidebar list becomes ambiguous as soon as you have >1 chat. (Noted in `app/page.tsx` comments as "future polish", but for v1 mobile usability it lands as an UX gap.) |
| 2.2 | minor | Deletion uses `window.confirm(...)`. Native confirm dialogs look out-of-place on mobile and break the visual language of the rest of the app. |

### Recommended fix

- 2.1: After the first assistant turn completes, derive a 4-6 word title from the first user message (or first sentence of the coach reply) and `UPDATE` the conversations row. A simple substring-then-truncate is enough for v1.
- 2.2: Replace `window.confirm` with an inline confirmation (e.g., second click on `×` within 2s, or a small inline "Confirm delete?" affordance).

### What was tested

### Result

```
  h1 fontSize: 18px

  modal width 379 / height 778 (fits inside 412 viewport — OK)

  h1 fontSize: 30px
```

### Screenshots

- `/tmp/pw-qa/qa-screenshots/10-mobile-chat.png` — sidebar (216px: "Browse Wiki", "+ New conversation", convo list) consumes >half the viewport. The chat column on the right is cramped: bubbles wrap awkwardly, "5 sources" link is squashed against the message body, message bubble sits next to the strip button instead of below it.
- `/tmp/pw-qa/qa-screenshots/11-mobile-welcome.png` — modal looks fine at mobile, but the help icon `?` button sits under the modal stacking context (z-index issue not exercised explicitly, but visible in the screenshot).
- `/tmp/pw-qa/qa-screenshots/12-mobile-wiki.png` — wiki page renders fine at mobile; the body is 412 wide, h1 reads cleanly, paragraphs are readable.

### Issues found

| # | Severity | Description |
|---|----------|-------------|
| 6.1 | important | **Sidebar does not collapse on mobile.** The 240px-fixed sidebar in `app/page.tsx`'s `pageStyles.sidebar` is `display: flex` always-visible, with no media query. Result: at 412px viewport, the chat surface is reduced to ~190px wide. Real-world phone users cannot meaningfully use the chat input on mobile. |
| 6.2 | important | Horizontal overflow (`bodyWidth 527 > viewport 412`). Caused by the same fixed sidebar plus the chat column's `maxWidth: 720` not adapting to remaining space — user sees a horizontal scrollbar at the worst possible moment (typing into a chat). |
| 6.3 | minor | The resource strip ("◇5 sources›") wraps onto the same line as the message body; awkward at any width, especially mobile. Could be moved to its own line below the message text. |

### Recommended fix

- 6.1 / 6.2: Add a breakpoint (`@media (max-width: 768px)`) that either (a) hides the sidebar behind a hamburger toggle, or (b) stacks sidebar → chat vertically. The current inline-styles pattern in `app/page.tsx` makes this awkward — may be the right time to introduce a single CSS module for the chat shell or migrate the shell to Tailwind utility classes (chat would benefit; rest of app stays as-is for now).
- 6.3: Move the `ResourceStrip` into its own flex row below the message text in `app/page.tsx` (the strip is rendered as a sibling of the message text spans, but the parent `article` has `whiteSpace: 'pre-wrap'` which makes them sit inline).

## Cross-cutting observations

### Console hygiene

Across the entire QA session, the browser console accumulated only **4 errors** and **0 page errors**. All 4 errors were the same favicon 404 (one per page navigation). That is genuinely clean.

### Network failure noise

The raw network failure count (42) is misleading. Filtering out (a) Next.js RSC prefetch aborts (`ERR_ABORTED` on `?_rsc=…` URLs — these are deliberate client-router cancellations) and (b) the favicon 404 leaves **3 real failures** — all of which are 404s I deliberately requested while testing error paths.

### CORP / COEP

The CORP+COEP fix landed earlier today is verified live: every sampled chunk carries both headers correctly. Root HTML also carries `cross-origin-embedder-policy: require-corp` and `cross-origin-opener-policy: same-origin`.

### Cost per turn

Two real chat turns showed up in `wrangler tail` during this pass (one mine, one from a sibling agent). Both were in the $0.16–$0.17 range, above the noted expected band of $0.03–$0.10. The retrieval component is the dominant share (~70%). Either the band needs updating in docs or the walker's LLM calls need a budget review.
