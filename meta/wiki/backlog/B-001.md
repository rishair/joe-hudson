# Corpus directory/index structure experiments

## The idea

The compendium's `coach/` directory uses a flat-ish category structure (`concerns/`, `moves/`, `reads/`, etc., each a flat list of files). E-043's surprise — `coach/_index.md` outgrew Anthropic's 200K context cap because absorption tripled the file count — exposes that the current structure has scaling limits as a seed-index input. Worth exploring whether a different shape of directory/index trades better quality for token cost.

## Open questions worth fleshing out

- **Depth vs breadth**: deep nested categories (e.g., `concerns/relational/family/dad`) reduce flat fan-out and might let the seed-index be a tree of summaries instead of a flat list. Does this help retrieval quality, or just shift the same problem one level down?
- **Right token budget for the index**: the slimmed `buildSeedIndex` got the seed-detection prompt to ~60K tokens by restricting to seedable categories. Is there a per-call sweet spot (10K? 30K? 100K?) that maximizes seed-detection quality per dollar?
- **Per-category indexing strategies**: some categories (e.g., `reads/`) are dense and signature-question-heavy; others (e.g., `applications/`) are thin and contextual. Maybe different categories deserve different index representations (one-line aliases vs. one-paragraph summaries vs. full first-section text).
- **Are there structures we haven't tried at all** that would change the retrieval game — e.g., a learned embedding hierarchy, a manually-curated decision tree, a per-concern "if the client says X, look at Y" mapping?

## Why this is backlog (not a current goal)

The current retrieval (graph-walk + Sonnet walker, v5b) is working — eval mean 2.71 and the web-app coach pipeline (E-043) is shipped. This is "could it be better" not "is it broken." Worth exploring in a future session when the v1 product is live and we have a real signal about which retrieval failures hurt UX most.
