# Build web app exposing coach chat and wiki

## Outcome

A Next.js web app with two surfaces: (1) a multi-conversation chat interface where users talk to any of the G-009 coach models with full per-message resource attribution, and (2) a navigable wiki view of the `coach/` compendium. The chat and wiki share the same underlying ingested compendium files (one source of truth), so a "resource used" link in a chat message goes to the same wiki entry a user could navigate to manually. The app is locally-first (client-side SQLite for conversation history) with a stateless Next.js backend that owns the system prompt and the OpenRouter API key.

The output is a deployable Next.js app at `web-app/` (or similar) plus an ingestion script that mirrors and link-normalizes `coach/` into the app's local wiki directory.

## Correctness

### Must-haves

- **Stack**: Next.js (App Router) + TypeScript + Vercel AI SDK + OpenRouter as the LLM gateway. Per [[ai-web-app]] playbook.
- **Architecture**: Repository pattern (owns reads + writes for a domain) with constructor-injected composition per [[coding-architecture]] playbook. Zustand is a thin reactive layer over repositories; not a separate "Store" abstraction.
- **Chat surface**:
  - Multiple conversations per user (browser-scoped — no server-side user accounts in v1)
  - Each conversation has many messages; messages persist to **client-side SQLite** (e.g., wa-sqlite + OPFS, or sqlocal) — chosen empirically in R-015
  - Messages are stored at **full AI SDK fidelity**: text, tool calls, tool results, message parts, role, all of it. Symmetry on round-trip: client sends back exactly what AI SDK produced.
  - The **system prompt never reaches the client.** Lives in the Next.js route handler. Client sends messages; server prepends system prompt; server returns streamed response.
  - Chat is wired to the G-009 coach models (initially v5b Sonnet walker; future configs are pluggable). The route handler invokes the same retrieval pipeline as the eval-side coach.
- **Per-message resource attribution**:
  - Each AI message shows the resources (compendium files) used to produce it
  - Display is subtle (does not break conversational flow) — e.g., a small icon or chip count
  - Click → modal with split layout: **left** = full wiki entry for the focused resource, **right** = list of all resources used in that message
  - Clicking another resource in the right panel switches the left panel to that resource's wiki entry
  - "Back to chat" close action returns to the conversation
  - "Expand" action on the wiki entry opens it as a full page in the wiki view
- **Wiki surface**:
  - Navigable view of the ingested `coach/` content — every file is a viewable page
  - Wikilinks within entries (`[[some-file]]`, `[[category/some-file]]`) resolve and are clickable
  - Direct linking via URL slug (e.g., `/wiki/concepts/limiting-belief`)
- **Ingestion script** at `web-app/scripts/ingest-coach.ts` (or similar):
  - Copies `coach/` → `web-app/content/wiki/` (or similar)
  - Normalizes/fixes links so they all resolve in the web app (the `coach/` compendium has inconsistent link formats; the script fixes them)
  - Is **repeatable**: re-running it does not duplicate, does not lose customizations not in `coach/`, and reflects any changes to `coach/` since the last run
  - Is idempotent for the unchanged subset (so the second run with no `coach/` changes is a no-op)
- **Security**: `OPENROUTER_API_KEY` is server-only. No secrets bundled into client JS. System prompts are server-only.
- **No client account system in v1.** Conversations are browser-scoped. If two browsers visit the app, they're separate users.
- **First-visit welcome flow** explaining: (a) what the project is, (b) how it was built, (c) privacy posture (local-only chat storage; LLM provider as only network egress), (d) explicit unaffiliation with Joe Hudson / Art of Accomplishment, (e) brief mention that anonymous feedback may be requested later (no commitment to specific mechanism in v1). Short, dismissible, persists dismissal in localStorage, reachable later via help icon. No dedicated safety step in the welcome — runtime safety handling per [[E-031]] is trusted. See [[E-045]].

### Nice-to-haves

- Per-message cost display
- Export/import of a conversation
- Conversation sharing (URL with conversation state encoded)
- Anonymous feedback mechanism (e.g., 👍/👎 per assistant message + optional one-line comment, no account, no PII). Deferred from v1 per user direction on 2026-05-24; the welcome flow mentions it as a future possibility so the framing is in place for a clean future addition.
- Wiki search (full-text or keyword)
- Mobile-responsive design
- Streaming tool-call rendering (show the tool call appearing in the message in real time)
- Multiple coach configs selectable per conversation (e.g., "try this conversation with the Haiku walker instead")
- User account system (deferred to G-011+)

- 2026-05-24: **R-017 complete: resource-attribution UX pattern decided.** Subtle bottom-of-message strip ("N sources") on each assistant message → click opens a native HTML `<dialog>` modal (65/35 split layout, focused resource in left panel, full resource list in right panel). Modal state is URL-encoded via the `nuqs` library (`?resourceModal=<msgId>&focus=<slug>`), explicitly NOT via Next.js parallel + intercepting routes despite the partial-syntax leftover in E-044. Rationale: the parallel-routes-for-modals pattern causes modal-stacking on in-modal navigation (Next.js issue #71586) and adds a fragile `[...catchAll]/page.tsx` slot maintenance burden. Our UX explicitly requires intra-modal focus changes (clicking a right-panel item, clicking a wikilink in the left panel) which are not route navigations and should not touch the router. nuqs `useQueryState` with `history: 'push'` gives us shareable URLs + browser-back-closes-modal without route files. Per-message resources are read directly from R-016's `data-resources` custom typed part on the assistant `UIMessage` (`message.parts.find(p => p.type === 'data-resources')`) — no separate state store. The planned Zustand `useResourceModalState` collapses to a `useResourceModal()` hook over nuqs + the existing chat messages array. "Expand to full" pushes `/wiki/<slug>?fromChat=<msgId>` and the E-042 wiki page reads `?fromChat=` to render a "Back to chat at message N" link (`router.back()` primary, `router.push('/?scrollToMessage=' + msgId)` fallback). Wikilinks inside the modal must NOT navigate — E-042's wiki renderer needs a new `linkBehavior: 'route' | 'in-modal'` prop (default `'route'`), and E-044 passes `'in-modal'` with an `onLinkClick(slug)` callback wired to `focus(slug)`. Modal shell is the native `<dialog>` + `showModal()` pattern (focus-trap, escape, backdrop, ARIA — all built in); no Radix dependency for v1. Layout: 65/35 split, ~90vw × 85vh, max 1400px wide. E-044 blast radius simplified: drop `app/@modal/(.)/wiki/...`, add `nuqs` dep, single `app/components/resource-modal.tsx` + `app/lib/state/use-resource-modal.ts`. E-042 must add a `linkBehavior` prop and an `onLinkClick` callback to its WikiRenderer + read `?fromChat=` to surface the back-to-chat affordance.
