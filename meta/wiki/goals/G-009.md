# Build the Joe Hudson AI coach

## Outcome

An interactive AI coach that takes a user's message in a multi-turn conversation, draws on the `coach/` compendium produced by G-007, and responds in Joe Hudson's voice using AoA methodology. The coach is runnable from a CLI for development, evaluable end-to-end via the G-008 eval harness, and scores measurably better than the AoA-named naive baseline (E-024) on the G-008 rubrics. Implemented with Bun + TypeScript using the Vercel AI SDK with the Anthropic provider, equipped with whatever tools the experiments determine are needed (compendium navigation, search, file-fetch, etc.).

The output is a coach binary that any of E-023 / E-024 / E-029 can target as a `coach_config`, plus a documented system prompt, a documented retrieval strategy, and an iteration log showing which prompts/retrievals were tried and how they scored.

## Correctness

The coach is built across two coupled work fronts: **system prompt** (the model's stance, methodology priors, anti-pattern warnings) and **retrieval** (how the right `coach/` content surfaces at the right moment). The user has called out system prompt iteration as the dominant work item, with retrieval as the open-ended second front. The eval (G-008) is the feedback loop for both; nothing ships without measurable eval improvement.

### Must-haves

- Implemented in Bun + TypeScript using Vercel AI SDK (`ai` + `@ai-sdk/anthropic`), not raw Anthropic SDK. AI SDK is chosen for tool-use ergonomics and agent-style flows; raw SDK already used by E-023 harness and is fine for that role but not for the coach.
- Runnable from a CLI: `bun run coach/run.ts` for interactive REPL; `bun run coach/run.ts --turn <message>` for one-shot used by the eval harness.
- Coach config exposed as a YAML file at `coach-app/configs/<name>.yaml` consumable by `eval/run.ts` as a `coach_config`, so any coach version can be benchmarked by G-008 without code changes to the eval.
- System prompt is a versioned file (or set of files) at `coach-app/prompts/`, with each iteration tagged (`v1`, `v2`, ...) so prompt regression is detectable.
- Retrieval mechanism is pluggable: the chosen approach (tools, embeddings, hybrid, none) is documented and isolated behind an interface so swapping it does not require rewriting the coach.
- Safety screen integrated: the E-027 safety criteria are enforced live, not just at eval time. The coach refuses to bypass safety even under client pressure (one of the anti-patterns the eval will catch).
- Outscores E-024 (AoA-named baseline) on at least 4 of 6 G-008 rubric dimensions, with no dimension regressing by more than 0.2. The required mean delta is a function of E-024's actual scores: if E-024 mean is <= 1.6 on most dimensions (low floor), the bar is delta >= 0.4; if E-024 mean is between 1.6 and 2.4, the bar is delta >= 0.3; if E-024 mean is >= 2.4 (small headroom), the bar is delta >= 0.2. The bar is set after E-024 produces real numbers.
- Triggers anti-pattern tests at less than 50% the rate of E-024.
- Total development spend stays within $60 budget. Cost tracked per experiment and rolled up on this goal page.
- Iteration log on this page shows every prompt version tried, its retrieval config, its eval score, and the change made for the next iteration. A reader should be able to reconstruct why v7 looks the way it does.

### Nice-to-haves

- Approaches gold-exchange scores (mean >= 2.5 across dimensions on at least one profile that resembles a gold exchange)
- Streaming responses in the CLI (AI SDK supports this natively)
- Per-turn telemetry showing which `coach/` files were retrieved and used
- A simple HTML transcript viewer at `coach-app/viewer/`
- Multi-turn cross-session memory (deferred; out of scope for v1)
