# Maintain a regularly-updated OpenRouter model registry with eval/cost/throughput data

## Outcome

A versioned, periodically-refreshed local registry of LLM models available via OpenRouter, annotated with rough eval scores, costs (input/output/cache-read/cache-write), and observed throughput (tokens per second). The registry is a structured data file (JSON or YAML) plus a script that updates it on demand. Any project in this repo (and any future project) can read the registry to make model-selection decisions grounded in current capability/cost/speed data rather than tribal memory.

The output is a single-command refresh: run the script, get a fresh file. A glance at the file tells you "the cheapest Anthropic Sonnet-tier model right now is X at $Y/$Z per million tokens with ~N tok/s throughput."

## Correctness

### Must-haves

- **Refresh script** at `model-registry/refresh.ts` (or similar) — Bun + TypeScript per the project default. Single command (`bun run refresh`) re-pulls the OpenRouter catalogue and updates the registry.
- **Registry file** at `model-registry/models.json` (or `.yaml`). Versioned in git so historical comparisons work. Each entry has:
  - `id` (OpenRouter's `provider/model` form)
  - `provider` and `family` (e.g., `anthropic` / `claude-4`)
  - `pricing`: `{ input_per_mtok, output_per_mtok, cache_read_per_mtok?, cache_write_per_mtok? }`
  - `context_window` (tokens)
  - `eval_scores`: rough indicators from public benchmarks where available (e.g., LMSYS Arena, livebench, OpenRouter's own routing data). Honesty about which scores are robust vs. impressionistic.
  - `throughput_tok_per_sec`: measured by the script itself via a small standardized prompt
  - `last_updated` (ISO timestamp)
  - `notes`: anything not capturable in fields (deprecation warnings, region-specific availability, etc.)
- **Throughput is measured, not advertised**: the refresh script issues a small standardized prompt against each model (or a configurable subset to control cost) and records tokens/sec from the response. This is what actually matters; vendor-claimed throughput is unreliable.
- **Cost-controlled refresh**: the refresh script accepts a `--measure-throughput=<list>` flag so a full refresh doesn't burn money testing every model. Default: pull catalogue metadata only (free); measure throughput only on the explicitly-requested subset.
- **No secrets in registry**: registry file contains pricing and ID data, no API keys. `OPENROUTER_API_KEY` is read from `.env` at refresh time, never written into the registry.
- **Consumable from any project**: the registry is plain JSON/YAML — any project in this repo (and beyond, via copy or submodule) reads it without needing the refresh script.

### Nice-to-haves

- **Per-model eval mini-suite**: a tiny standardized eval (5-10 prompts) run on a configurable subset of models to give a project-relevant quality signal beyond raw benchmarks
- **Historical comparison**: register changes git-diff cleanly so you can see "Sonnet 4.6 dropped to $2.5/$15 per million tokens last week"
- **Auto-suggestion**: a thin CLI that picks the best-fit model given constraints (`./model-registry/suggest --max-cost-per-mtok=5 --min-tok-per-sec=50 --capability=tool-use`)
- **Cron/GitHub-Action integration**: weekly automated refresh + commit (deferred — interactive refresh is enough for v1)
- **Cache-aware cost projection**: given a prompt template and expected cache-hit rate, compute the realistic per-call cost

## Exit Criteria

- [ ] `model-registry/` directory exists with refresh script and registry file
- [ ] `bun run refresh` from `model-registry/` produces a fresh `models.json` populated with every currently-available OpenRouter model
- [ ] At least the major providers' top-3 models per family have measured throughput (`--measure-throughput` was run for them)
- [ ] Registry file is committed to git; subsequent refreshes produce git-diffable changes
- [ ] One downstream consumer demonstrated: G-010's coach pipeline (E-043) reads the registry instead of hardcoding `anthropic/claude-sonnet-4.6` (or a documented note for why it pins instead)
- [ ] Throughput-measurement cost is documented per full-coverage run
- [ ] README in `model-registry/` documents usage for a future agent/project

## Context

The G-009 and G-010 work has surfaced the practical issue of "which OpenRouter model ID is right, current, and priced reasonably." E-040's playbook fix corrected a model-ID drift (dash form → dot form); E-043 had to manually decide which model to use; future projects will repeat this guessing. A registry external to any single project removes the lookup-and-guess step.

- Tokens per second
