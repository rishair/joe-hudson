# Hetzner personal-PaaS: project.play.ris.hair routing via Cloudflare Tunnel with Cloudflare Access

## Outcome

A lightweight personal-PaaS on the Hetzner box that lets any project running on a local port be exposed at `<project>.play.ris.hair` with Cloudflare Access auth, no public Hetzner ports, and `systemd` keeping processes alive across reboots. Adding a new project takes one command and a tiny YAML file.

Concretely: a CLI tool (`./paas register|up|down|list|logs`) plus one Cloudflare Tunnel, one Cloudflare Access policy, and per-project `systemd` units generated from each project's `.local-dev.yaml`.

The output is a script + small directory of configs that turns "I want this dashboard reachable from my phone" into a 30-second operation.

## Correctness

### Must-haves

- **Subdomain pattern**: each project lives at `<project>.play.ris.hair` (sub-subdomain — each project independent, easy to promote to root subdomain on Cloudflare Pages later).
- **One Cloudflare Tunnel** runs as a `systemd` service on the Hetzner box, exposes local ports to Cloudflare's network. No Hetzner public ports opened for the tunnel.
- **Cloudflare Access** policy applied at the tunnel: only the user's email(s) can reach any `*.play.ris.hair` subdomain. Anyone else gets the CF Access login page and bounces.
- **Per-project `.local-dev.yaml`** (in the project's repo root) declaring at minimum: `subdomain`, `port`, `command` (e.g., `bun run dev` or `bun run start`), `mode` (`dev` or `built`). Optionally: env vars, restart policy.
- **CLI tool** at `~/bin/paas` (or similar) with subcommands:
  - `paas register <path>` — reads the project's `.local-dev.yaml`, generates a `systemd` unit, updates the tunnel config, restarts the tunnel
  - `paas up <project>` / `paas down <project>` — start/stop the systemd unit
  - `paas list` — what's registered, what's running, what's the subdomain
  - `paas logs <project>` — tail `journalctl` for that unit
  - `paas remove <project>` — tear down (systemd unit, tunnel route, etc.)
- **Process supervision via `systemd` user units** (`systemctl --user`) so projects survive reboots and auto-restart on crash. No `tmux`, no `pm2` (one less moving part).
- **Tunnel config is generated, not hand-edited.** The CLI maintains `~/.cloudflared/config.yml` (or equivalent) as a derived artifact of registered projects.
- **Secret handling**: project env vars come from each project's `.env` (gitignored), referenced by the systemd unit's `EnvironmentFile=`. CLI never writes secrets into systemd units directly.
- **The dogfood**: [[G-013]] (status dashboard) is the first user. If G-012 can't host G-013 cleanly, G-012 isn't done.

### Nice-to-haves

- A simple "what's running" status page at `paas.play.ris.hair` (which is the CLI's own dogfood — could be served by the CLI itself as a small HTTP endpoint)
- Per-project resource limits (memory cap via systemd) to prevent one runaway process from killing the box
- Restart-on-config-change (CLI watches `.local-dev.yaml` and re-applies)
- Promotion path: `paas promote <project>` outputs the steps to deploy this project to Cloudflare Pages (the production graduation) — defer to v2

## Exit Criteria

- [ ] `paas` CLI exists at a known path; subcommands (`register`, `up`, `down`, `list`, `logs`, `remove`) all work
- [ ] One project (G-013's status dashboard) is registered and reachable at its `*.play.ris.hair` URL
- [ ] Cloudflare Tunnel is running as a `systemd` service that survives a Hetzner reboot
- [ ] Cloudflare Access blocks unauthenticated requests to `*.play.ris.hair` and lets the user through after Google/GitHub SSO
- [ ] `paas list` shows the project, its port, its subdomain, its current systemd status
- [ ] `journalctl --user -u <unit>` works for project logs (CLI's `logs` subcommand wraps this)

- [ ] Documentation in `paas/README.md` explains the per-project `.local-dev.yaml` format and the "add a new project" workflow

## Context

The user is running multiple personal projects on the Hetzner box (this project's status dashboard, the OpenRouter model registry dashboard, future tools). Each needs to be reachable from any device the user owns, with auth so it's not publicly browsable. Doing this ad-hoc per-project (manually configuring nginx, manually managing TLS, manually starting tmux sessions) doesn't scale.

User-stated framing: **Hetzner = personal/play, Cloudflare = public/production.** The dividing line is "are other people going to use it." G-012 is the Hetzner side of that split. The Cloudflare side is per-project: G-010's [[E-046]] handles the coach's Cloudflare deploy; future public projects will get their own production-deploy experiments.

User design choices recorded in conversation 2026-05-24:
- **Sub-subdomain per project** (`<project>.play.ris.hair`), not path-based — keeps projects independent and lets them graduate to a root subdomain on Cloudflare Pages without code changes.
- **Cloudflare Tunnel**, not Caddy/nginx with exposed ports — no exposed Hetzner ports, automatic TLS, consistent operational surface with the Cloudflare-public side.
- **Cloudflare Access**, not security-by-obscurity — apps don't implement auth; CF Access blocks unauthenticated traffic at the edge.
