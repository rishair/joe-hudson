# Project status dashboard at status.play.ris.hair: live wiki state, recent file changes, checkpoints

## Outcome

A simple browser dashboard at `status.play.ris.hair` (deployed via [[G-012]]'s personal-PaaS) that shows the user the live state of THIS project (`/home/claude/joe-hudson/`). Reading it should answer "what's going on right now without me having to ask Claude" — what wiki items are in flight, what just shipped, what's stuck, what changed in the repo recently and what those changes contain, and the latest checkpoints.

The dashboard reads the filesystem directly. No database. Refresh is on page load (or polling). The first version is intentionally simple — a one-page view that you can glance at from your phone.

The output is a Hono+Bun+JSX server (stack picked by [[R-020]]) running on Hetzner and exposed via G-012.

## Correctness

### Must-haves

- **Pending-on-you callout (top of page)**:
  - Visually-prominent panel listing all `REQ-XXX` items in `meta/wiki/requests/` with `status: pending`
  - Treated as distinct from in-flight wiki items: these are "blocked on YOU", not "blocked on an agent"
  - Shows REQ id, title, what's blocked behind it
  - Empty state is fine ("Nothing waiting on you — agents have the wheel")
- **Live wiki state section**:
  - Active goals with status (one line each — same shape as the PM-style overview in `/wiki checkpoint`)
  - In-flight items (claimed research/experiments) with claim age and the claiming agent
  - Stale-claim warnings (anything past TTL)
- **Recent agent activity section** (the "what is the agent doing right now" surrogate):
  - List of files under `meta/wiki/` modified in the last ~15 minutes, sorted newest first
  - For each: file path, mtime delta, one-line preview of the page title (parsed from `# Heading`)
  - This is the proxy for "what is the main session / wiki-next agents doing right now" — direct main-session inspection is out of scope for v1
- **Recent file changes section**:
  - Last 20 commits with subject + diff stats
  - Click into a commit shows the actual diff (fetched async, doesn't block page render)
  - Untracked / uncommitted changes if any
  - If `status-dashboard/` lives in-repo, its own commits filter out of this view (avoid the recursive-status problem)
- **Checkpoints section**:
  - List of all checkpoint files in `meta/wiki/checkpoints/`, most recent first
  - Click to read the checkpoint body
  - At the top: mtime of the most recent checkpoint AND (if discoverable) next scheduled cron fire — the "is the autonomous loop actually ticking" indicator
- **Backlog section**:
  - The auto-generated `meta/wiki/backlog/index.md` rendered inline
- **Single-page, glance-friendly**: information density over chrome. No nav menu, no settings panel, no theme picker. Just the state.
- **Auto-refresh via fetch-polling**: the page polls a `/api/state` JSON endpoint every ~20-30s and replaces the relevant DOM regions in place. Expanded states (commit diffs, checkpoint bodies) are preserved across polls. A visible "Last refreshed: 12s ago" indicator plus a manual "Refresh now" button. Full meta-refresh is explicitly forbidden because it breaks scroll, focus, and expanded-section state.
- **Deployed via G-012** at `status.play.ris.hair` with Cloudflare Access auth so only the user can reach it.
- **Reads the project filesystem directly**: no API calls to external services for v1. The dashboard's job is to present what's already on disk, not to compute new state. Read wiki frontmatter directly (not via `./wiki.sh` shell-out) to avoid lock contention with the wiki-next agents.
- **Graceful degradation**: if a filesystem read fails (e.g., `backlog/index.md` mid-regeneration) or a git command errors transiently, render a clear "data temporarily unavailable" state for that section. No whitescreens.

### Nice-to-haves

- Per-goal drill-down: click a goal in the overview to see all its research/experiments with status
- Spend tally if extractable from `eval/results/*/cost_log.jsonl` and `eval/results/*/summary.json`
- "What changed in the last hour" view (the must-have recent-activity panel is 15min; expanding to 1h is a small nice-to-have)
- Git history visualization (commit graph)
- Markdown rendering of wiki pages inline (so you can read an experiment writeup without leaving the dashboard)
- Mobile-responsive layout
- Server-Sent Events (SSE) push for "checkpoint just landed" or "REQ just got resolved" — replacing polling with push. Reconsider only if 20-30s polling proves unsatisfying in real use.
