# Project status dashboard at status.play.ris.hair: live wiki state, recent file changes, checkpoints

## Outcome

A simple browser dashboard at `status.play.ris.hair` (deployed via [[G-012]]'s personal-PaaS) that shows the user the live state of THIS project (`/home/claude/joe-hudson/`). Reading it should answer "what's going on right now without me having to ask Claude" — what wiki items are in flight, what just shipped, what's stuck, what changed in the repo recently and what those changes contain, and the latest checkpoints.

The dashboard reads the filesystem directly. No database. Refresh is on page load (or polling). The first version is intentionally simple — a one-page view that you can glance at from your phone.

The output is a Next.js app (or simpler — could be plain HTML + small server) running on Hetzner and exposed via G-012.

## Correctness

### Must-haves

- **Live wiki state section**:
  - Active goals with status (one line each — same shape as the PM-style overview in `/wiki checkpoint`)
  - In-flight items (claimed research/experiments) with claim age and the claiming agent
  - Stale-claim warnings (anything past TTL)
  - Open user requests (REQ-XXX with `status: pending`)
- **Recent file changes section**:
  - Last N commits with subject + diff stats
  - Click into a commit shows the actual diff (or a link to it)
  - Untracked / uncommitted changes if any
- **Checkpoints section**:
  - List of all checkpoint files in `meta/wiki/checkpoints/`, most recent first
  - Click to read the checkpoint body
- **Backlog section**:
  - The auto-generated `meta/wiki/backlog/index.md` rendered inline
- **Single-page, glance-friendly**: information density over chrome. No nav menu, no settings panel, no theme picker. Just the state.
- **Auto-refresh** (or manual refresh button): the data should be at most ~30 seconds stale.
- **Deployed via G-012** at `status.play.ris.hair` with Cloudflare Access auth so only the user can reach it.
- **Reads the project filesystem directly**: no API calls to external services for v1. The dashboard's job is to present what's already on disk, not to compute new state.

### Nice-to-haves

- Per-goal drill-down: click a goal in the overview to see all its research/experiments with status
- Spend tally if extractable from `eval/results/*/cost_log.jsonl` and `eval/results/*/summary.json`
- "What changed in the last hour" view
- Git history visualization (commit graph)
- Markdown rendering of wiki pages inline (so you can read an experiment writeup without leaving the dashboard)
- Mobile-responsive layout
- "What's the agent currently doing" if there's a way to inspect main-session activity (probably not feasible without writing logs the dashboard can read)
