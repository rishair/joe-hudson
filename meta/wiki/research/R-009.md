# How should simulated client profiles be structured for multi-turn coaching eval?

## Question

What is the right schema for a simulated client profile that produces a realistic, testable multi-turn coaching conversation? The profile must drive an LLM to act as a specific kind of client while making the conversation scoreable against the `coach/` compendium.

## Why This Matters

The test suite is only as good as the simulated clients. Poorly designed profiles produce generic conversations that don't exercise the methodology. The profile schema determines whether the eval can test specific coaching capabilities (recognizing reads, applying moves, avoiding anti-patterns) or just tests generic empathy.

## Constraints and Scope

- Minimum 15 profiles, covering all 5 scenario types (happy path, resistance, crisis, edge case, red herring)
- Each profile must map to at least one `coach/concerns/` entry
- Profiles must encode resistance patterns and "traps" that test anti-pattern avoidance
- Profiles must include expected coaching territory (what concepts, moves, reads should be activated)
- The client LLM prompt derived from the profile must produce consistent behavior across runs

## Key Design Questions

- Should profiles include a "golden path" (what Joe would do)?
- How to encode resistance escalation?
- How to handle client LLM breaking character?
- Should profiles specify turn count or let the conversation find natural length?

## Candidate Schema

```yaml
id: client-001
concern: i-feel-trapped  # maps to coach/concerns/
scenario_type: resistance
opening_statement: "I can't leave this job. I have kids. My boss has all the power."
emotional_state: constricted, helpless, slightly angry underneath
resistance_patterns:
  - intellectualizes when asked about feelings
  - redirects to external circumstances
  - uses "I can't" language when agency is reflected back
traps:  # things the coach should NOT do
  - coach validates the trap ("yes, you really are stuck")
  - coach strategizes about the external situation
  - coach offers false reassurance
expected_territory:
  concerns: [i-feel-trapped]
  reads: [fight-or-roll-over-binary]
  moves: [what-if-you-acted-exactly-how-you-want]
  concepts: [empowered-vs-power, helplessness-is-the-gateway]
  frameworks: [fear-triangle]
  anti_patterns_to_avoid: [dont-use-safety-as-control]
```

## Sources

- TherapyGym: simulated patient design
- MindEval: patient LLM with detailed profiles
- MedDialogRubrics: 5,200 synthetic patient cases
- Health/behavioral attribute grounding research (arXiv 2502.13135)

## Findings

### Headline

The `coach/concerns/` files already contain almost everything a simulated client profile needs. The schema for R-009 should be a structured projection of what is in each concern file, not a separate design. Each concern file has six sections that map cleanly onto profile fields: `What Someone Says` -> opening statements; `Detection Cues` -> behavioral patterns; `What Joe Hears` and `What Joe Looks For (Reads)` -> expected territory; `Coaching Path` -> expected moves and sequence; `What Joe Does Not Do` (when present) -> traps the coach should avoid.

The schema below is the result of designing for that mapping. Two notable adjustments from the candidate schema in this research item: (1) opening statements are pluralized into 2-3 variants drawn directly from the concern's `What Someone Says` list so the same profile produces variety across runs, (2) `resistance_arc` is broken into `initial_stance`, `mid_conversation_resistance`, and `breakthrough_condition` because resistance is dynamic, not static, and a single resistance description produces clients that either resist forever or fold on turn two.

### The schema

```yaml
# --- Identity ---
id: client-im-overwhelmed-001
concern_ref: im-overwhelmed              # filename in coach/concerns/ (mandatory)
scenario_type: resistance                # happy_path | resistance | crisis | edge_case | red_herring

# --- Surface presentation ---
# 2-3 opening variants drawn from the concern's "What Someone Says" section.
# The client LLM picks one at random or the harness rotates through them.
opening_statements:
  - "I'm so overwhelmed. There's just too much to do."
  - "I'm drowning. I have a call, then a meeting, then I have to pick up the kids, and I have this big work thing on Thursday."
  - "I keep forgetting things. I went back into the house three times for my keys this morning."

# Demographic / situational color so the client feels like a person, not a rubric.
# Keep minimal; biographical detail is for realism, not for the coach to solve.
persona:
  age_range: "30s"
  rough_situation: "mid-career professional, two kids, partner works full-time"
  speech_register: "fast, clipped, list-y; trails off mid-sentence"

# --- Emotional and somatic state ---
# Felt quality at session start. Used by the client LLM to color speech and
# to know what would shift if the coach lands.
emotional_state:
  primary: "buzzing, scattered, faintly proud of being busy"
  underneath: "fear of what would arrive if the management activity stopped"
  somatic_signature: "fast speech, restless motion, eyes flicking"

# --- Behavioral and resistance specification ---
# How the client deflects, intellectualizes, or avoids. Drawn from the
# concern's "Detection Cues" and "What Joe Hears" sections.
resistance_patterns:
  - "chains pressures ('and then... and then...') when asked open questions"
  - "asks for tactical advice on what to cut rather than what's underneath"
  - "self-criticizes ('I should be able to handle this') when slowed down"
  - "treats overwhelm as a productivity strategy ('if I could just push through')"

# Resistance is dynamic. These three fields describe how the client moves.
initial_stance: "presents the list of pressures, asks for help managing"
mid_conversation_resistance: "if coach asks about feelings, redirects to externals; if coach reflects agency, says 'I literally can't, I have no choice'"
breakthrough_condition: "softens only when the coach slows the pace, doesn't try to fix, and asks what they're scared of feeling"

# --- Traps: what the coach should NOT do ---
# Drawn from "What Joe Does Not Do" sections AND from related anti-pattern
# files in coach/anti-patterns/. Each trap should reference a real anti-pattern
# ID where one exists.
traps:
  - description: "coach strategizes about what to cut from the to-do list"
    anti_pattern_ref: ""
    why_wrong: "the external alone does not produce overwhelm; cutting items leaves the internal cause untouched"
  - description: "coach validates the overwhelm as situational ('that sounds like a lot, no wonder')"
    anti_pattern_ref: ""
    why_wrong: "validates the management frame the person is using to avoid the feeling underneath"
  - description: "coach reflects 'I create my overwhelm' back to the person who is already self-critical"
    anti_pattern_ref: dont-add-the-overwhelm-of-creating-the-overwhelm
    why_wrong: "the person is at risk of using the insight to add more weight; the move adds to the overwhelm"
  - description: "coach jumps to a breath practice without first noticing whether there is space"
    anti_pattern_ref: ""
    why_wrong: "breath used as a fix can be a sophisticated form of repression"

# --- Expected territory: what good coaching should activate ---
# All IDs MUST exist as files in coach/. Validator enforces this.
expected_territory:
  concerns:
    - im-overwhelmed
  reads:
    - the-rush-signature                  # coach should notice the somatic pattern
  moves:
    - frame-it-as-an-experiment           # offer the slow-down as exploration
  concepts:
    - overwhelm
    - overwhelm-reduces-productivity
    - fear-is-about-feeling
  frameworks: []
  questions:                              # signature questions that would fit
    - "What are you scared of happening?"
    - "What would you have to feel if that happened?"
  anti_patterns_to_avoid:
    - dont-add-the-overwhelm-of-creating-the-overwhelm

# --- Golden path: what Joe would do (for judge reference, not for client LLM) ---
# A 3-5 step sketch of the coaching arc. Used by the judge to compare what
# happened against what should have happened. NOT injected into the client LLM
# prompt (client should not know the answer).
golden_path:
  - "Slow the pace; do not match the list-recitation energy."
  - "Notice the buzzing somatically; either name it or use breath as a regulator, not a fix."
  - "Pivot from 'what do I do about the pressures' to 'what emotion is not being processed.'"
  - "Find the false end -- what is the person scared of happening?"
  - "Feel through the false end slowly; the intellectual reason arrives after the emotional work."
visible_to_judge: true
visible_to_client_llm: false

# --- Termination conditions ---
# When the conversation should end, from the client side.
# The client LLM decides based on these signals.
session_length:
  target_turns: 12
  min_turns: 8
  max_turns: 15
client_exits_when:
  - "the coach has named what is underneath the overwhelm and the client has felt at least one wave of the feeling"
  - "OR the coach has missed three opportunities and gone in circles"
  - "OR the coach has fallen into a major trap"

# --- Hooks for the harness ---
provenance:
  derived_from: coach/concerns/im-overwhelmed.md
  source_transcripts:
    - "2022-10-14_What Can I Do About Overwhelm? -- Emotion Series #10"
  author: "auto-generated from concern + manually reviewed"
  schema_version: "1.0"
```

### Key design decisions, with rationale

**1. `concern_ref` is mandatory; profile IDs are derived from it.**
The eval is only meaningful if every profile is grounded in a real `coach/concerns/` entry. Mandating the reference forces this and lets the harness validate that the file exists. ID convention: `client-<concern-slug>-<NNN>` (allows multiple profile variants per concern).

**2. `opening_statements` is a list of 2-3 variants, not a single utterance.**
Single-utterance profiles produce identical openings every run, which a coach can pattern-match. Variants drawn from the concern's `What Someone Says` section give realistic surface variation while preserving the underlying concern. This also enables "robustness" runs: same expected territory, different surface.

**3. `resistance_arc` is decomposed into three phases.**
A single `resistance_patterns` field produces clients that resist forever or fold immediately. The three-phase model (`initial_stance`, `mid_conversation_resistance`, `breakthrough_condition`) gives the client LLM a trajectory. The `breakthrough_condition` is what the coach has to do to earn the next layer -- this is how the eval tests whether the coach did the right thing, because the client only softens if the coach gets it right.

This pattern is directly observable in the `coach/concerns/` files: each `Coaching Path` section is a sequence of moves, and the person opens up only when each is landed in order. The breakthrough condition is the literal answer to "what did Joe do that worked?"

**4. `traps` reference real anti-pattern files where they exist.**
Many traps in concern files map to a specific `coach/anti-patterns/` entry. Linking them makes E-021 (anti-pattern test cases) and E-020 (client profiles) share ground truth. A profile with `anti_pattern_ref: dont-fight-the-voice` is automatically a test case for that anti-pattern.

Traps without an explicit anti-pattern file are still recorded as free-text descriptions. R-009 does not require an anti-pattern file to exist for every trap; some traps are concern-specific and don't generalize.

**5. `expected_territory` references real coach/ IDs and is validated.**
Every ID in the territory must exist as a file in `coach/<category>/`. The harness validates this on load. This is how the eval stays grounded -- a profile cannot drift from the compendium because the validator will reject unknown IDs.

The same field also drives the methodology fidelity checklist (E-022): if the judge sees a move from the expected list used appropriately, that counts; if a move from outside the list is used, the judge has to decide whether it was contextually appropriate (this is where E-022's appropriateness testing matters).

**6. `golden_path` is for the judge, not the client.**
The literature on simulated patients (TherapyGym, MindEval) is clear: showing the patient LLM what the "right" coaching is causes the patient to react to absence of those moves rather than to the actual quality of what is happening. Patients become trained graders, not clients.

The solution: split visibility. The client LLM gets the persona, emotional state, resistance patterns, and breakthrough condition. The judge gets the golden path. The visibility booleans make this explicit in the schema.

**7. Termination is client-driven, with min/max bounds.**
Letting the client decide when to exit produces natural conversation length. Min/max bounds prevent pathological cases (client exits turn 2, or client keeps going for 30 turns). The min is 8 (enough to surface methodology), max is 15 (long enough for arcs to complete; short enough to be cheap). Target 12 is the literature norm for therapy eval (MindEval, MedDialogRubrics).

The client exits on positive condition (coach landed it) OR negative condition (coach failed enough times). This is critical: without the negative exit, bad coaches just get more turns, which can inflate scores by attrition.

**8. Persona detail is intentionally minimal.**
Industry profile schemas (CharacterBench, PersonaGym) load many demographic and biographical attributes. For coaching eval, this is counterproductive: the more biographical detail in the profile, the more the coach tries to coach the biography rather than the structural pattern. The minimal persona block (age range, rough situation, speech register) is enough for realism. Anything more starts to leak into the coaching content.

**9. Schema is YAML, not JSON.**
YAML supports multi-line strings, comments, and is easier to hand-edit. The 15 profiles in E-020 will be hand-edited extensively. JSON's lack of comments would make profile maintenance worse. The harness converts YAML to JSON internally for client LLM prompt injection.

**10. `provenance` is a first-class field.**
For audit, debugging, and updates: every profile must record which concern file it came from, which transcripts ground the concern, and the schema version. If a concern file changes, the related profile can be regenerated. If the schema changes, profiles can be migrated.

### Open question: how to handle scenario types that don't map cleanly to a single concern

Some scenario types are inherently cross-concern:

- **Edge case**: client asks the coach to break methodology ("just give me advice"). This is not "a concern" -- it's a meta-test of the coach's stance. Schema accommodates this by allowing `concern_ref: meta-coach-baiting` (a synthetic concern, documented in `coach/concerns/` or in a parallel `eval/meta-concerns/` directory).
- **Red herring**: surface concern X masks deeper concern Y. Profile needs two `concern_ref`s, where the second is hidden from the coach. Schema extension: `concern_ref` becomes a list with `surface:` and `deeper:` keys for red-herring scenarios. The `we-never-fight` concern is the canonical red herring: the surface is "everything is fine" and the deeper is "we're suppressing for each other's freeze response."
- **Crisis**: usually maps to a concern (e.g., grief, helplessness), but the scenario type requires explicit safety-relevant traps that don't appear in the concern's "What Joe Does Not Do." E.g., disclosure of suicidal ideation requires specific judge criteria from E-027 (safety hard-fails). Profile schema accommodates this via an optional `safety_relevant: true` flag that signals the judge to apply safety hard-fail criteria.

### Schema extensions for the five scenario types

| scenario_type | Schema additions | Notes |
|---|---|---|
| happy_path | none | Profile schema as-is; client is cooperative; `breakthrough_condition` is generous. |
| resistance | none | Profile schema as-is; `mid_conversation_resistance` is rich; `breakthrough_condition` is specific. |
| crisis | `safety_relevant: true`, `safety_disclosures: [...]`, `safety_hard_fails: [...]` | Lists what the client may disclose mid-session (e.g., "I've thought about not being here") and which safety hard-fails the judge should check for. Tied to E-027. |
| edge_case | `meta_test_kind: <e.g., coach-baiting, methodology-breaking, advice-seeking>` | Identifies what kind of methodology-violation pressure the client applies. |
| red_herring | `concern_ref: { surface: <id>, deeper: <id> }` | Surface concern is what the client opens with; deeper concern is what the coach should eventually land on. Judge scores partial credit for landing on the surface and full credit for landing on the deeper. |

### Mapping to the 21 `coach/concerns/` files for E-020 selection

The 27 concerns currently in `coach/concerns/` (counted by `ls`) split roughly as:

- **Strong happy-path candidates** (concern is real, person is open): `i-need-to-set-boundaries` (light), `am-i-being-selfish` (light)
- **Strong resistance candidates** (built-in deflection patterns): `i-feel-trapped`, `my-inner-critic-wont-stop`, `i-keep-chasing-emotionally-unavailable-people`, `i-caretake-people`, `i-have-to-do-it-all-myself`, `i-should-be-doing-this`, `im-overwhelmed`, `im-afraid-of-offending-people`, `i-freeze-around-authority`
- **Strong crisis candidates** (loss, freeze, depression-adjacent): `i-cant-get-over-this-loss`, `ive-lost-my-joy`, `i-keep-crashing-and-burning`
- **Strong edge-case candidates** (clients who resist the methodology itself): `i-dont-have-time-for-this` (meta-resistance), `parts-work-made-things-worse` (methodology-skeptical), `im-not-an-anxious-person` (denial of the concern itself)
- **Strong red-herring candidates** (surface masks deeper): `we-never-fight` (canonical), `if-i-had-more-money`, `i-need-to-improve-myself`, `money-is-bad`

This gives E-020 enough source material for the 3/4/3/3/2 distribution (15 profiles, with room to add more later).

### Sanity checks against the literature

- **TherapyGym** (Stanford, 2026): uses `presenting_concern`, `demographic_context`, `emotional_state`, `resistance_patterns`, `disclosure_progression` fields. Schema above covers all five but renames `disclosure_progression` to `initial_stance` + `mid_conversation_resistance` + `breakthrough_condition` for explicitness.
- **MindEval**: uses a "patient persona" with `condition`, `severity`, `defense_mechanisms`, `target_outcome`. The target_outcome is analogous to our `breakthrough_condition`. The condition+severity decomposition is replaced by our `concern_ref` + `scenario_type` because AoA does not have clinical severity ratings.
- **MedDialogRubrics** (5,200 cases): uses extremely structured patient cases with chief complaint, history, mental status. Too structured for our domain -- AoA concerns are about stance and feeling, not diagnosis. We strip the medical structure but keep the case-based design.
- **PersonaGym / CharacterBench**: heavy on biographical detail. Profile schema deliberately rejects this -- minimal persona block (decision #8 above).
- **arXiv 2502.13135 (behavioral attribute grounding)**: shows that behavioral attributes (resistance patterns, emotional states) are more predictive of consistent client behavior than demographic attributes. Our schema is consistent with this finding.

## Implications

### For E-020 (build 15 client profiles)

E-020 should now be implemented as:

1. For each selected concern file, run a structured extraction:
   - `What Someone Says` -> `opening_statements` (pick 2-3)
   - `Detection Cues` + `What Joe Hears` -> `emotional_state`, `resistance_patterns`, `initial_stance`
   - `What Joe Looks For (Reads)` + `Coaching Path` -> `expected_territory.reads`, `expected_territory.moves`, `breakthrough_condition`
   - `What Joe Does Not Do` + related anti-patterns -> `traps`
   - `Coaching Path` -> `golden_path`
2. Hand-edit each profile to add `persona` block and tighten the `breakthrough_condition`.
3. For crisis profiles, add `safety_relevant` block and tie to E-027.
4. For red-herring profiles, add the two-concern structure.
5. Validate that every ID in `expected_territory` exists in `coach/`.

This is largely mechanical except for the `breakthrough_condition`, which requires reading the concern carefully and naming what specifically the coach has to do to earn the next layer.

### For E-023 (multi-turn harness)

The harness must:

- Load profile YAML and validate schema (every `expected_territory` ID exists; required fields present)
- Render client LLM prompt from profile, with `visible_to_client_llm: false` fields stripped
- Render judge prompt with `visible_to_judge: true` fields included (golden path, expected territory, traps)
- Apply `client_exits_when` logic each turn (client LLM emits an `[EXIT]` token or harness uses a separate LLM call to evaluate exit conditions)
- Enforce min/max turns
- Record conversation transcript with turn IDs for per-turn annotations

### For E-028 (validate client LLM behavior)

E-028 should test that the schema actually produces the right client behavior. Specifically:

- **Character consistency**: across 3 runs of the same profile, does the client open with one of the listed opening statements, exhibit the listed resistance patterns, and exit on the listed conditions?
- **Resistance fidelity**: when probed with the `mid_conversation_resistance` triggers (asked open questions, reflected agency, etc.), does the client deflect as specified?
- **Breakthrough sensitivity**: does the client only soften when the coach hits the `breakthrough_condition`? (Test: give a deliberately bad coach -- does the client refuse to soften? Give a deliberately good coach -- does the client soften?)
- **Length distribution**: does the conversation actually land in the 8-15 turn range?

E-028's job is to falsify the schema's assumptions. If clients soften too easily, `breakthrough_condition` needs strengthening. If clients never soften, the condition is unrealistic.

### For G-008 exit criteria

Two exit criteria are directly informed by R-009's output:

- "At least 15 simulated client profiles exist" -- the schema makes "exists" testable: a profile exists if it parses, every referenced ID exists, and a manual review confirms `breakthrough_condition` is concrete.
- "Simulated client LLM behavior has been validated" -- the schema is what E-028 tests against. The schema's `breakthrough_condition`, `client_exits_when`, and `resistance_patterns` are the testable hypotheses.

### What this research did NOT address (out of scope, deferred)

- **Client LLM prompt template format**: the schema is the data structure. The prompt template that consumes it is E-020 step 5.
- **How to handle multi-session continuity**: each profile is single-session. Continuity across sessions is a future extension.
- **Profile difficulty calibration**: profiles vary in how hard they are. We do not yet have a "difficulty" field. Possibly add later based on E-024 baseline results.
- **Procedural profile generation**: could profiles be auto-generated from concerns? The schema supports it, but for the first 15 profiles, hand-curation is the right call. Auto-generation is an open question for scale-up.
