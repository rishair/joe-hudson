# How should the G-009 retrieval pipeline be adapted from coach-app to the web app?

## Question

G-009 produced a working retrieval pipeline at `coach-app/retrieval/` (graph-walk, guided-walk variants, embeddings). It runs as a server-invocable Bun module. The G-010 web app needs the same retrieval to serve the chat route handler. Three options: (a) reuse `coach-app/` as-is from the Next.js app (cross-package import or shell-out), (b) lift the code into the web app's lib (copy + adapt), (c) factor the retrieval into a shared package both consume. Each has trade-offs around code duplication, drift risk, runtime environment, and dependency management.

Also: the v5b coach (Sonnet walker) is the chosen substrate. The web app needs to invoke it in a way that emits per-turn retrieval telemetry the chat UI can render as the per-message resource attribution. The G-009 telemetry shape (logged to conversation JSON) needs to flow through AI SDK's message-part stream so the client can render it.

## Why This Matters

E-043 (wire v5b coach into route handler) depends on this. The decision shapes the web app's directory structure, deployment story, and how G-007/G-009 fixes propagate to the running app. It also determines how compendium changes (G-007 absorption) reach the running coach.

## Constraints and Scope

In scope (these are MUST-answers — E-043 cannot start without all four resolved):

1. **Where does retrieval-orchestration actually live today?** Per the G-010 audit: `coach-app/run.ts` does NOT invoke retrieval. The retrieval-to-coach stitching (trigger-policy logic, injection-into-user-message placement, history preservation, retry policy) lives in the EVAL harness at `eval/lib/conversation.ts` (search for the retrieval-injection logic around lines 270-352) and `eval/lib/retrieval-adapter.ts` (around lines 122-181). The web-app needs equivalent orchestration but in a Next.js route handler, not an eval harness. Specify the minimal port: which functions to lift, which to rewrite, which to leave behind.

2. **Anthropic prompt-cache markers under OpenRouter.** `coach-app/lib/anthropic-client.ts` uses `providerOptions.anthropic.cacheControl` for prompt caching. These markers will NOT propagate cleanly through OpenRouter's OpenAI-shaped API via `@ai-sdk/openai`. Specify whether: (a) we lose prompt caching entirely for the web app (acceptable cost overhead?), (b) we use a different SDK to preserve caching (e.g., `@openrouter/ai-sdk-provider` if it preserves Anthropic options), or (c) we route the web app direct-to-Anthropic instead of through OpenRouter for the coach path (diverges from playbook; needs explicit justification).

3. **AI SDK message-annotation transport for per-turn retrieval telemetry.** The chat UI needs to render per-message resource attribution (E-044). This data must flow from the route handler back to the client attached to each assistant message and persist into SQLite (E-041). AI SDK has several mechanisms: `experimental_attachments`, `data` parts via `streamData`, custom message annotations, or shimmed-in tool calls. Pick one and document the shape. E-041 and E-044 are blocked on this decision.

4. **Compare the three adaptation strategies** on: duplication, drift risk, deployment (single Next.js artifact), Node-vs-Bun runtime in production, and how each handles items 1-3 above.

In scope (other):
- Confirm whether the web-app needs the `bge-small-en-v1.5` embedding model (it does NOT for v5b graph-walk; v5b uses graph traversal only). If the wiki search nice-to-have ever happens, embeddings come back; not for v1.
- Confirm the smallest possible change to `coach-app/` that makes the retriever consumable from a Next.js route handler

Out of scope:
- Swapping the v5b architecture (decided in G-009)
- Multiple coach configs simultaneously (nice-to-have; v1 is one config)
- Hot-reloading compendium changes without a restart (defer)

## Findings

### Answer 1 — Orchestration port location

The retrieval-to-coach orchestration lives entirely in `eval/lib/conversation.ts:270-368`. It is NOT in `coach-app/run.ts` (the CLI never calls retrieval). The orchestration does five things per coach turn:

1. **Trigger-policy gate** (`eval/lib/conversation.ts:284-294`): is this turn one where retrieval should fire? Uses `coachConfig.trigger_policy` and `coachConfig.retrieval.strategy`. V5b defaults to `every_turn`.
2. **Retrieval call** (`eval/lib/conversation.ts:296-313`): invokes the strategy-specific retriever with `{ profile_id, turn, clientMessage, history }`. Returns `{ injection, telemetry, cost_usd, ms, call_record }`.
3. **Injection placement** (`eval/lib/conversation.ts:344-352`): the retrieval bundle is prepended to the **user message**, not the system message. Format: `${injection}\n\n${cleanClient}`.
4. **History preservation** (`eval/lib/conversation.ts:365-366`): the coach's `messages` array retains the BARE client message after the turn, not the augmented one. This avoids double-paying for stale retrievals across multi-turn.
5. **Graceful degradation on retrieval failure** (`eval/lib/conversation.ts:331-341`): a thrown error inside retrieval is caught, telemetry records the failure, the turn proceeds without injection.

The adapter `eval/lib/retrieval-adapter.ts:122-181` (`buildGuidedWalkRetriever`) is what bridges the orchestration to the retrieval module. It:
- Reads coach-config retrieval params (`walker_model`, `seed_model`, `k_max`, `step_budget`, `max_edges_per_step`, `walker_variant`)
- Slices history to the last 4 turns (`history.slice(-4)`) before passing to the walker
- Dynamically imports `coach-app/retrieval/guided-walk.ts` and calls `retrieveByGuidedWalk(...)`
- Re-shapes the walker's return into the eval's `CoachRetriever` interface (combines seed + walker token totals into a single `call_record`)

**Port specification for E-043:**

- **Lift verbatim from `coach-app/`** (these are pure-compute modules):
  - `coach-app/retrieval/guided-walk.ts` (the v5b walker)
  - `coach-app/retrieval/graph-walk.ts` (the seed-detection + loader the walker depends on)
  - `coach-app/lib/safety.ts` (pre/post safety screens — pure regex/keyword logic, no Anthropic dependency)
  - `coach-app/prompts/v1.md` (the v1 system prompt — actually loaded by config)

- **Rewrite from `eval/`** (orchestration that doesn't fit a Next.js route handler):
  - The `eval/lib/conversation.ts:270-368` orchestration becomes part of the `/api/chat` route handler. It loses: the multi-profile loop, the cost-attribution-by-profile-id, the exit-token detection, the call-log array. It keeps: trigger policy, retrieval call, injection placement, history preservation, graceful degradation.
  - The `eval/lib/retrieval-adapter.ts:122-181` adapter becomes a thin function in `web-app/app/lib/coach/retriever.ts` that calls `retrieveByGuidedWalk` with the v5b config baked in (no need for the multi-strategy switch since the web app only runs v5b in v1).
  - Cost-tracking becomes the process-level circuit breaker spec'd in E-043 method step 5, not a per-profile call log.

- **Leave behind**:
  - `coach-app/run.ts` (CLI is unused by the web app)
  - `coach-app/retrieval/embeddings.ts` and `coach-app/retrieval/build_index.py` (v5b doesn't use embeddings)
  - `coach-app/retrieval/pre_screen.ts` (E-035 variant; v5b uses the default `v5b` walker variant)
  - `coach-app/lib/anthropic-client.ts` (web app calls OpenRouter, not Anthropic direct — see Answer 2)
  - `eval/lib/conversation.ts` profile-loop, exit-token logic, call-log attribution
  - `eval/lib/retrieval-adapter.ts` multi-strategy switch (`graph-walk`, `embeddings`, etc.)

**Path-and-import detail**: `guided-walk.ts` line 63 has `const REPO_ROOT = resolve(import.meta.dir, "..", "..");` and `graph-walk.ts` reads files via `join(REPO_ROOT, "coach/...")`. For the web app, these MUST be re-anchored to `web-app/content/wiki/` (E-039 output) instead of `coach/`. Add a `WIKI_ROOT` parameter or env var override at module load.

### Answer 2 — Prompt cache markers under OpenRouter

**Recommendation: keep the playbook (OpenRouter), use `@openrouter/ai-sdk-provider`, set cache_control via `providerOptions.openrouter.cacheControl` on the system message.**

The `@openrouter/ai-sdk-provider` package (not the `@ai-sdk/openai` package with a custom `baseURL` as the playbook currently illustrates) accepts Anthropic-style `cache_control` markers and forwards them to Anthropic via OpenRouter's API. Supported namespaces (all equivalent):

- `providerOptions.openrouter.cache_control`
- `providerOptions.openrouter.cacheControl`
- `providerOptions.anthropic.cache_control`
- `providerOptions.anthropic.cacheControl`

The provider auto-normalizes a string `system` argument into a content block so the cache_control marker attaches correctly (Issue #389 in the provider repo, fixed). Only the `ephemeral` cache type is supported, which matches `coach-app/lib/anthropic-client.ts`'s current usage.

**Why this path over the alternatives:**

- *Option (a) lose prompt caching*: For v1's ~3.1K-token prompt at ~$0.001 amortized per turn (with cache) vs ~$0.01/turn (without), this is ~10x overhead. Across 100 user conversations of 12 turns each that's $12 vs $1.20. Acceptable for v1 dev, painful at any real volume. NOT recommended.
- *Option (b) use `@openrouter/ai-sdk-provider`*: Preserves caching, single key, single provider switch as the playbook intended. RECOMMENDED.
- *Option (c) direct-to-Anthropic from the web app*: Forces the codebase to manage two LLM clients (Anthropic for the coach turn, OpenRouter or Anthropic for any other model swap). Diverges from the playbook's "one gateway, one key" rationale. Only use if (b) hits an undocumented blocker.

**Required playbook patch** (separate work item — see Implications): `ai-web-app.md` currently shows `createOpenAI({ baseURL: 'https://openrouter.ai/api/v1', apiKey: process.env.OPENROUTER_API_KEY })`. For models that need provider-specific options (like Anthropic prompt caching), replace with `createOpenRouter({ apiKey: process.env.OPENROUTER_API_KEY })` from `@openrouter/ai-sdk-provider`. The model-ID convention (`anthropic/claude-sonnet-4-6`) is the same.

**Caveat — known issues to validate:**

- GitHub issue OpenRouterTeam/ai-sdk-provider#35 reports cache-write tokens are not always surfaced on the usage object as of mid-2026; verify during E-043 that cache_read/cache_write fields populate in `result.providerMetadata.openrouter.usage` (or wherever the provider exposes them). If they don't surface, the cap-tracking math still works because cache_read is just cheap input, but the per-turn cost telemetry will under-report cache savings.
- OpenRouter uses "provider sticky routing" to maximize cache hits. Cache hits are NOT guaranteed; under provider load OpenRouter can fall back to a different upstream and the cache effectively cold-starts. Treat caching as a cost-reduction optimization, not a correctness requirement.

### Answer 3 — Telemetry transport via AI SDK custom data parts

**Recommendation: use AI SDK v5+ custom typed data parts (`createUIMessageStream` + a `data-resources` part) attached to the assistant message. Persist the message via E-041 with the data part inline in the UIMessage.parts array — no schema change needed; AI SDK serializes custom data parts as part of the standard UIMessage type.**

Of the four mechanisms surveyed:

- `experimental_attachments`: client-only, designed for user-uploaded files. Wrong tool.
- `streamData` / `writeMessageAnnotation`: REMOVED in AI SDK v5. Migration guide directs everyone to custom data parts.
- Shimmed-in tool calls: technically works (tool result becomes a message part) but pollutes the assistant message with a fake tool call the user never asked for. Wrong abstraction.
- **Custom typed data parts**: the v5+ canonical answer. Type-safe, persistable, reconcilable via stable IDs, and the part appears in `message.parts` so E-041's full-fidelity persistence handles it transparently.

**Shape for E-043:**

```typescript
// web-app/app/lib/coach/types.ts
import type { UIMessage } from 'ai';

export type ResourceAttribution = {
  resources: Array<{
    slug: string;              // e.g. "concepts/limiting-belief"
    category: string;          // e.g. "concepts"
    title: string;             // display label (file H1 or filename)
    walker_reason: string;     // the per-file rationale from guided-walk telemetry
  }>;
  seeds: string[];             // the 1-3 seed slugs the walker started from
  walker_model: string;        // "claude-sonnet-4-6" for v5b
  total_cost_usd: number;      // per-turn retrieval cost
  step_count: number;          // walker steps taken
  stop_reason: string;         // "bundle_full" | "frontier_empty" | "step_budget_exhausted"
};

export type CoachUIMessage = UIMessage<
  never, // no per-message metadata
  {
    resources: ResourceAttribution;
  }
>;
```

**Server-side write (E-043):**

```typescript
// web-app/app/api/chat/route.ts
import { createUIMessageStream, createUIMessageStreamResponse, streamText } from 'ai';

export async function POST(req: Request) {
  const { messages } = await req.json();
  const stream = createUIMessageStream<CoachUIMessage>({
    execute: async ({ writer }) => {
      const userMsg = messages[messages.length - 1];
      const retrieval = await runRetrieval({ clientMessage: userMsg, history: messages });

      // Attach attribution as a typed data part BEFORE the text stream begins.
      // Use a stable id so the client can later refer to it (and SQLite
      // round-trips it).
      writer.write({
        type: 'data-resources',
        id: `resources-${Date.now()}`,
        data: telemetryToAttribution(retrieval.telemetry),
      });

      const result = streamText({
        model: openrouter('anthropic/claude-sonnet-4-6'),
        system: { /* with cacheControl */ },
        messages: withInjection(messages, retrieval.injection),
      });
      writer.merge(result.toUIMessageStream());
    },
  });
  return createUIMessageStreamResponse({ stream });
}
```

**Client-side read (E-044):**

```typescript
const { messages } = useChat<CoachUIMessage>({ api: '/api/chat' });

// In the message renderer:
const resourcesPart = message.parts.find(p => p.type === 'data-resources');
if (resourcesPart) {
  const count = resourcesPart.data.resources.length;
  // render the subtle indicator + click target for the modal
}
```

**Persistence implications for E-041:** None beyond what E-041 already specifies. The custom data part lives inside `UIMessage.parts`. E-041's tightened success criterion ("byte-equality of UIMessage JSON round-trip") covers it as long as E-041 stores the full `parts` array verbatim. E-041 should add the `CoachUIMessage` type as the message type parameter for its repositories rather than the bare `UIMessage`.

**Order-of-emission note**: writing the `data-resources` part BEFORE `streamText` begins means the client sees the resource list during/before the streaming reply, which lets the indicator appear with the message rather than as a post-hoc tag. Persistence semantics are the same either way.

### Answer 4 — Adaptation strategy comparison

| Strategy | Duplication | Drift risk | Deployment | Runtime parity (Node vs Bun) | Handles A1 port | A2 caching | A3 telemetry |
|---|---|---|---|---|---|---|---|
| (a) Reuse `coach-app/` as-is | None | None | Two services to deploy or one with monorepo build | `coach-app` is Bun-only (`#!/usr/bin/env bun`, `import.meta.dir`). Next.js prod is Node. Mismatch. | Cross-package import needs `transpilePackages: ['coach-app']` + path alias + handling `import.meta.dir` (which behaves differently). Risky. | Same as (b)/(c) — independent of strategy | Same as (b)/(c) — independent of strategy |
| (b) Lift to `web-app/app/lib/coach/` | Two copies in repo | High (bug fixes need to land twice) | Single Next.js artifact | Pure-Node code, no Bun bindings | Clean — adapt path constants to `web-app/content/wiki/` on lift | Same | Same |
| (c) Extract to shared `packages/coach-retrieval/` | None | None | Single artifact (Next.js + workspace package) | Pure-Node code (drop Bun-isms during extraction) | Cleanest from coupling perspective | Same | Same |

**Recommendation: (b) lift to `web-app/app/lib/coach/`.**

Reasoning:

- The "drift risk" of (b) is overstated for the actual scope. `coach-app/` IS still used by `eval/` to invoke retrieval (see `eval/lib/retrieval-adapter.ts:137` `import("../../coach-app/retrieval/guided-walk.ts")`). But G-009 is done; G-010 is the consumer. Once G-010 ships, the eval-side `coach-app` consumption fades into rare re-eval runs, not active development. The window where both copies diverge is small.
- (c) is architecturally clean but doubles the deployment complexity: needs a workspace package, build step, type generation, version syncing. The codebase has no other workspace packages today; this would be a precedent-setting structural change for marginal gain.
- (a) has a Bun-vs-Node runtime mismatch. `coach-app/retrieval/guided-walk.ts` imports use `.ts` extensions (Bun-style) and `import.meta.dir` (Bun-only path API). Making these work under Next.js's Webpack/Turbopack pipeline requires `transpilePackages` AND code changes to handle missing `import.meta.dir`. At which point you're already editing the source — you might as well lift it.
- (b) takes ~30 minutes of file copies + path-constant edits + replacing `.ts` extension imports with extensionless (or `.js`) imports for Node compatibility. The drift cost across the remaining lifetime of G-009 is low because G-009 is feature-frozen.

**The lift checklist** (E-043 method step 2 reference):

1. `cp coach-app/retrieval/guided-walk.ts web-app/app/lib/coach/guided-walk.ts`
2. `cp coach-app/retrieval/graph-walk.ts web-app/app/lib/coach/graph-walk.ts`
3. `cp coach-app/lib/safety.ts web-app/app/lib/coach/safety.ts`
4. Replace `import.meta.dir` with `process.cwd()` or a passed-in root (parameterize on call). The web app should pass `path.join(process.cwd(), 'content/wiki')` as the wiki root.
5. Replace `.ts` extension imports (`from "./graph-walk.ts"`) with no extension (`from "./graph-walk"`) for Next.js/Webpack compatibility.
6. Replace `coach-app/lib/anthropic-client.ts`'s `generateCoachTurn` with a route-handler-local call using `@openrouter/ai-sdk-provider`'s `streamText` (so we get streaming for the chat UI; the CLI used `generateText` because it's one-shot).
7. Set `OPENROUTER_API_KEY` + `WIKI_ROOT` env vars; default `WIKI_ROOT` to `process.cwd() + '/content/wiki'`.

### Other in-scope answers

**Embeddings (`bge-small-en-v1.5`)**: NOT needed for v1. v5b graph-walk uses model-guided traversal over `related:` edges, not vector similarity. The HuggingFace token and embedding sidecar are not part of the web-app dependency surface. If the wiki search nice-to-have is ever built, it can either: (i) lift `coach-app/retrieval/embeddings.ts` + the Python `embed_query.py` sidecar (heavy), or (ii) use a JS-only embedding library (e.g., `@xenova/transformers`) entirely in-process (lighter). Both are out-of-scope for v1.

**Smallest possible change to `coach-app/` to consume from Next.js**: None. The recommendation is the lift (strategy b), not modifying `coach-app/`. If we instead pursued (a), the minimum change to `coach-app/` would be: parameterize `REPO_ROOT` to accept an env-var override, replace `import.meta.dir` with a runtime check, and document the Node-vs-Bun import-extension behavior. None of this is needed under (b).

## Implications

For E-043 (unblocked by this research):

- E-043's Method step 1 ("Read R-016's adaptation finding") is now satisfiable. E-043 can proceed.
- E-043's Method step 2 should be expanded with the lift checklist above (copy 3 source files, parameterize `REPO_ROOT`, replace extension imports, replace `anthropic-client` with route-handler-local `streamText`).
- E-043's Method step 4 (telemetry transport) should bind to the `CoachUIMessage` type and `data-resources` part shape in Answer 3.
- E-043's prompt-cache strategy is now resolved: use `@openrouter/ai-sdk-provider` (not `@ai-sdk/openai`) and set `providerOptions.openrouter.cacheControl: { type: 'ephemeral' }` on the system message.

For E-041 (also indirectly affected):

- E-041 should use `CoachUIMessage` (defined in Answer 3) as the persisted message type, not the bare `UIMessage`. The interfaces in step 3 of E-041 already use `UIMessage` and persist `message.parts` verbatim, so the change is just a type parameter swap; no schema change.

For E-044 (also indirectly affected):

- The resource-attribution UI consumes `message.parts.find(p => p.type === 'data-resources')`. R-017's UX work can proceed against this shape.

For the `ai-web-app` playbook:

- Replace the `createOpenAI({ baseURL: 'https://openrouter.ai/api/v1', ... })` example with `createOpenRouter` from `@openrouter/ai-sdk-provider`. Add a note that this provider preserves Anthropic-specific options (cache_control) where the OpenAI-compatibility shim does not. This is a separate small work item — flagged in G-010's decision log below.

For G-007/G-009 propagation:

- Because of strategy (b), G-007 content changes (new compendium files) flow through the E-039 ingestion (re-mirror `coach/` to `web-app/content/wiki/`), then the lifted `guided-walk.ts` reads from the new path. No change needed to coach-app for the web app to see new content.
- G-009 retrieval bug fixes do NOT auto-propagate. If a bug is found in `guided-walk.ts` post-G-009, it must be fixed in both `coach-app/retrieval/guided-walk.ts` AND `web-app/app/lib/coach/guided-walk.ts`. This is the explicit drift cost of strategy (b). Mitigation: file-level identical-content check in CI or a brief manual diff on any retrieval-touching PR. For now, document the drift risk in E-043's Dead Ends section after the lift.
