# How should the G-009 retrieval pipeline be adapted from coach-app to the web app?

## Question

G-009 produced a working retrieval pipeline at `coach-app/retrieval/` (graph-walk, guided-walk variants, embeddings). It runs as a server-invocable Bun module. The G-010 web app needs the same retrieval to serve the chat route handler. Three options: (a) reuse `coach-app/` as-is from the Next.js app (cross-package import or shell-out), (b) lift the code into the web app's lib (copy + adapt), (c) factor the retrieval into a shared package both consume. Each has trade-offs around code duplication, drift risk, runtime environment, and dependency management.

Also: the v5b coach (Sonnet walker) is the chosen substrate. The web app needs to invoke it in a way that emits per-turn retrieval telemetry the chat UI can render as the per-message resource attribution. The G-009 telemetry shape (logged to conversation JSON) needs to flow through AI SDK's message-part stream so the client can render it.

## Why This Matters

E-043 (wire v5b coach into route handler) depends on this. The decision shapes the web app's directory structure, deployment story, and how G-007/G-009 fixes propagate to the running app. It also determines how compendium changes (G-007 absorption) reach the running coach.

## Constraints and Scope

In scope (these are MUST-answers — E-043 cannot start without all four resolved):

1. **Where does retrieval-orchestration actually live today?** Per the G-010 audit: `coach-app/run.ts` does NOT invoke retrieval. The retrieval-to-coach stitching (trigger-policy logic, injection-into-user-message placement, history preservation, retry policy) lives in the EVAL harness at `eval/lib/conversation.ts` (search for the retrieval-injection logic around lines 270-352) and `eval/lib/retrieval-adapter.ts` (around lines 122-181). The web-app needs equivalent orchestration but in a Next.js route handler, not an eval harness. Specify the minimal port: which functions to lift, which to rewrite, which to leave behind.

2. **Anthropic prompt-cache markers under OpenRouter.** `coach-app/lib/anthropic-client.ts` uses `providerOptions.anthropic.cacheControl` for prompt caching. These markers will NOT propagate cleanly through OpenRouter's OpenAI-shaped API via `@ai-sdk/openai`. Specify whether: (a) we lose prompt caching entirely for the web app (acceptable cost overhead?), (b) we use a different SDK to preserve caching (e.g., `@openrouter/ai-sdk-provider` if it preserves Anthropic options), or (c) we route the web app direct-to-Anthropic instead of through OpenRouter for the coach path (diverges from playbook; needs explicit justification).

3. **AI SDK message-annotation transport for per-turn retrieval telemetry.** The chat UI needs to render per-message resource attribution (E-044). This data must flow from the route handler back to the client attached to each assistant message and persist into SQLite (E-041). AI SDK has several mechanisms: `experimental_attachments`, `data` parts via `streamData`, custom message annotations, or shimmed-in tool calls. Pick one and document the shape. E-041 and E-044 are blocked on this decision.

4. **Compare the three adaptation strategies** on: duplication, drift risk, deployment (single Next.js artifact), Node-vs-Bun runtime in production, and how each handles items 1-3 above.

In scope (other):
- Confirm whether the web-app needs the `bge-small-en-v1.5` embedding model (it does NOT for v5b graph-walk; v5b uses graph traversal only). If the wiki search nice-to-have ever happens, embeddings come back; not for v1.
- Confirm the smallest possible change to `coach-app/` that makes the retriever consumable from a Next.js route handler

Out of scope:
- Swapping the v5b architecture (decided in G-009)
- Multiple coach configs simultaneously (nice-to-have; v1 is one config)
- Hot-reloading compendium changes without a restart (defer)
