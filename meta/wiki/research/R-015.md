# What's the best browser-side SQLite approach for full-fidelity AI SDK message persistence?

## Question

The G-010 chat persists conversations to client-side SQLite (per the user transcript). Browser SQLite is not built-in; multiple options exist: `wa-sqlite` + OPFS (Origin Private File System), `sqlocal` (wrapper around wa-sqlite), `sql.js` (pure WASM, in-memory + IndexedDB persist), `absurd-sql`, and the rapidly-stabilizing native browser SQLite (still experimental in most browsers). Each trades off persistence semantics, write performance, file-size durability, and integration complexity.

For full-fidelity AI SDK `UIMessage` persistence — including tool calls, tool results, structured message parts — we need: reliable writes, persistence across reload/tab-close, ability to store JSON blobs efficiently, and a clean TypeScript API for use inside React components and Zustand stores.

## Why This Matters

E-041 (client-side SQLite persistence) directly depends on this choice. The choice also constrains the repository pattern: an OPFS-backed SQLite is essentially synchronous from the consumer's perspective (worker-mediated), while `sql.js` is purely in-memory with explicit `export()` calls to IndexedDB. The Repository API for chat persistence is shaped by this choice.

## Constraints and Scope

In scope:
- Survey wa-sqlite/sqlocal/sql.js/absurd-sql/native (as of 2026)
- For each: persistence semantics, write perf, max DB size, browser support, TypeScript story, dependency size, worker requirements
- The interaction with Next.js App Router (dynamic import in client component, worker file serving, OPFS access policy)
- Practical recommendation for E-041

Out of scope:
- Cross-device sync (deferred to a future goal)
- Server-side fallback (v1 is local-only)
- Performance benchmarking under realistic load (a follow-up if a candidate is borderline)
