# What's the best browser-side SQLite approach for full-fidelity AI SDK message persistence?

## Question

The G-010 chat persists conversations to client-side SQLite (per the user transcript). Browser SQLite is not built-in; multiple options exist: `wa-sqlite` + OPFS (Origin Private File System), `sqlocal` (wrapper around the official `@sqlite.org/sqlite-wasm`), `sql.js` (pure WASM, in-memory + IndexedDB persist), `absurd-sql`, and the official `@sqlite.org/sqlite-wasm` build directly. Each trades off persistence semantics, write performance, file-size durability, and integration complexity.

For full-fidelity AI SDK `UIMessage` persistence — including tool calls, tool results, structured message parts — we need: reliable writes, persistence across reload/tab-close, ability to store JSON blobs efficiently, and a clean TypeScript API for use inside React components and Zustand stores.

## Why This Matters

E-041 (client-side SQLite persistence) directly depends on this choice. The choice also constrains the Repository pattern: an OPFS-backed SQLite is essentially async from the consumer's perspective (worker-mediated, postMessage round-trip), while `sql.js` is purely in-memory with explicit `export()` calls to IndexedDB. The Repository API for chat persistence is shaped by this choice (the playbook's four-method shape is fine either way — `find/create/update/delete` returning `Promise<T>` — but the implementation must handle worker bootstrapping, COOP/COEP requirements, and SQLITE_BUSY semantics for OPFS-based options).

## Constraints and Scope

In scope:
- Survey wa-sqlite/sqlocal/sql.js/absurd-sql/official `@sqlite.org/sqlite-wasm` (as of May 2026)
- For each: persistence semantics, write perf, max DB size, browser support, TypeScript story, dependency size, worker requirements
- The interaction with Next.js App Router (dynamic import in client component, worker file serving, OPFS access policy, COOP/COEP requirements vs same-origin route handler)
- Practical recommendation for E-041

Out of scope:
- Cross-device sync (deferred to a future goal)
- Server-side fallback (v1 is local-only)
- Performance benchmarking under realistic load (a follow-up if a candidate is borderline)

## Findings

### State of the field, May 2026

The browser-SQLite landscape consolidated significantly in 2025–2026. There are now effectively three viable production paths plus one fallback. `absurd-sql` is no longer recommended (creator-acknowledged dormancy since 2021, no 2026 activity), and native browser SQLite (the long-rumored SQLite-as-a-platform-API) has not shipped in any browser.

Each option below is keyed against G-010's actual needs: **full-fidelity AI SDK `UIMessage` persistence for a single-tab chat app, with browser-scoped local storage, no cross-device sync in v1, no concurrent writer requirement**.

#### Option 1: `sqlocal` (wraps official `@sqlite.org/sqlite-wasm`)

- **What it is**: A TypeScript-first wrapper around `@sqlite.org/sqlite-wasm` (the official build) that hides worker bootstrapping, exposes a tagged-template SQL API, ships first-class Drizzle and Kysely drivers, and provides reactive-query hooks for React/Vue/Angular.
- **Version & activity**: v0.18.0 (April 25, 2026), 730 stars, 35 forks, 283 commits, 100% TypeScript. Actively maintained.
- **Persistence**: OPFS via the official sqlite-wasm worker. Data survives reload, tab-close, and browser restart. Each origin gets a private filesystem; cleared only on storage eviction or explicit user action.
- **Write performance**: OPFS sync access handles deliver ~4× IndexedDB throughput in benchmarks; SQLocal puts queries on a Web Worker so they don't block the main thread. Tagged-template queries auto-parameterize.
- **TypeScript story**: Native TypeScript, full type coverage. Drizzle and Kysely drivers give type-safe schema-bound queries.
- **Worker requirements**: Yes — the OPFS sync access handle API is only available in dedicated workers, and sqlocal handles the bootstrap. Caller sees an async API (`await sql\`SELECT ...\``).
- **COOP/COEP requirement**: Yes — sqlocal documents that the page needs `Cross-Origin-Opener-Policy: same-origin` and `Cross-Origin-Embedder-Policy: require-corp` for the `opfs` VFS path. (It can fall back to `opfs-sahpool` without those headers, at the cost of single-connection-per-origin — see "Note on opfs-sahpool" below.)
- **Bundle**: A few hundred KB for the sqlite WASM (~1 MB before gzip, ~400-500 KB transferred). One-time download, cached aggressively.
- **Next.js integration**: Works in App Router. The chat component is a client component (`'use client'`). sqlocal instantiated lazily inside the component (or a custom hook) — never imported at module top-level by a server component. Worker file is served by Next.js automatically; sqlocal has a Vite plugin for build-time worker bundling, but Next.js's webpack/turbopack handles `new Worker(new URL(...))` patterns natively.
- **Multi-tab story**: With the default `opfs` VFS, multiple tabs can each open their own connection (subject to SQLITE_BUSY handling). For our v1 single-conversation-per-tab use case this is fine; even concurrent edits across two tabs of the same browser to the same conversation are rare and the SDK can degrade gracefully.

#### Option 2: `wa-sqlite` directly

- **What it is**: Roy Hashimoto's pre-official SQLite WASM build with a rich VFS ecosystem (OPFSCoopSyncVFS, OPFSWriteAheadVFS, IDBBatchAtomicVFS). Historical leader; still maintained.
- **Persistence**: OPFS via OPFSCoopSyncVFS (recommended general-purpose) or OPFSWriteAheadVFS (April 2026 addition, supports concurrent reads alongside writes, Chrome 121+ only via `readwrite-unsafe`).
- **Write performance**: Excellent. OPFSCoopSyncVFS is PowerSync's documented production recommendation as of May 2026.
- **TypeScript story**: Types ship; ergonomics are lower-level than sqlocal. You write the worker plumbing, the message protocol, and the type-safe query layer yourself (or pick up a community wrapper).
- **Worker requirements**: Yes. You author the worker.
- **COOP/COEP requirement**: Yes, for OPFS-based VFSes.
- **Maturity**: Battle-tested. PowerSync, Notion, and others run this in production.
- **Tradeoff vs sqlocal**: Lower-level. More code to write. For a chat app where the data model is simple (two tables, JSON blob in one of them), sqlocal's higher-level wrapper removes a meaningful chunk of incidental complexity.

#### Option 3: Official `@sqlite.org/sqlite-wasm` directly

- **What it is**: The SQLite project's own WASM build with a thin ES Module wrapper and TypeScript types. Version 3.53.0-build1 as of April 2026.
- **Persistence**: `opfs` VFS (requires COOP/COEP, true multi-connection) or `opfs-sahpool` (no COOP/COEP needed, but single-connection-per-origin lock).
- **Tradeoff vs sqlocal**: Sqlocal IS this library plus a usable wrapper. Picking the raw build means re-implementing the worker bootstrap, the message protocol, the parameter binding, and any reactive layer. For a small team / one-off app, no reason to pay that tax. Pick this only if you have a strong reason to avoid the wrapper dependency.

#### Option 4: `sql.js` with IndexedDB serialize/deserialize

- **What it is**: Pure WASM SQLite running in memory. Persistence is "serialize the full DB to a `Uint8Array`, save that blob to IndexedDB, reload on next visit."
- **Persistence**: All-or-nothing. You decide when to snapshot. Risk: snapshot during a streaming write is awkward; you typically snapshot after each completed conversation turn or on visibility-change.
- **Write performance**: Writes are in-memory and fast. The persistence cost is the snapshot — for a database that grows to a few MB it's milliseconds; for hundreds of MB it gets expensive and stalls the main thread unless you move sql.js into a worker.
- **TypeScript story**: Community types (`@types/sql.js`). API is procedural.
- **Worker requirements**: Not strictly required. Common for performance.
- **COOP/COEP requirement**: No — does not use OPFS, just IndexedDB.
- **Browser support**: The most broadly compatible option (no OPFS dependency).
- **DB size cap**: Limited by available RAM (the whole DB lives in memory) and by IndexedDB blob size limits (Chrome: ~2 GB per object, Firefox: 1 GB). Practical cap ~100–500 MB before things get painful.
- **When this wins**: When you can't set COOP/COEP headers, when you need to support older browsers, or when the data is small and snapshot-on-turn is acceptable.

#### Option 5: `absurd-sql`

- **Status**: Not recommended. Creator's 2021 maintenance note ("excited but time-constrained") never resulted in renewed activity; no meaningful 2026 commits. Use sqlocal or wa-sqlite + OPFS instead.

### Note on `opfs-sahpool` (no-headers fallback)

`opfs-sahpool` is a VFS for the official build that uses OPFS *without* requiring COOP/COEP, by pre-allocating sync access handles into a pool. The catch: **single connection per origin**. If a second tab opens the same DB, it gets a busy error or has to wait. For a chat app where users typically have one tab open at a time this is acceptable, but it's a real degradation from the default `opfs` VFS.

If COOP/COEP can be set in the Next.js deployment (it can — see "Next.js + COOP/COEP" below), prefer the default `opfs` VFS. Use `opfs-sahpool` only as a fallback path inside the same wrapper.

### Next.js + COOP/COEP interaction

The chat app's network architecture matters here. The client posts to `/api/chat` on the same origin; the route handler is what calls OpenRouter. So **OpenRouter is never a cross-origin fetch from the browser** — COEP `require-corp` does not block it because the client never calls OpenRouter directly. The route handler runs on the Node.js side, where COOP/COEP has no effect.

Setting headers in Next.js App Router is one config block in `next.config.js`:

```javascript
async headers() {
  return [{
    source: '/(.*)',
    headers: [
      { key: 'Cross-Origin-Opener-Policy', value: 'same-origin' },
      { key: 'Cross-Origin-Embedder-Policy', value: 'require-corp' },
    ],
  }];
}
```

The risk: if the app later embeds third-party iframes, images from unrelated origins, or scripts from CDNs that don't send `Cross-Origin-Resource-Policy: cross-origin`, those will break. For G-010's app — pure first-party content, all assets served from the Next.js origin, no third-party iframes — this is not a practical concern. Document it in the welcome flow's privacy section if needed.

### AI SDK `UIMessage` persistence shape

AI SDK 5+'s `UIMessage` type is the canonical persistence unit. Per the Vercel docs:

- `UIMessage` is the **source of truth for application state**: it contains text parts, tool-call parts, tool-result parts, metadata, and (in v5) typed data parts.
- The `useChat` hook's `onFinish` callback hands back all messages already in the correct shape — no conversion needed.
- The `ModelMessage` type is a stripped-down format for sending to the model. It is **lossy**. Persisting `ModelMessage` and rehydrating loses tool-call structure. **Never persist `ModelMessage`.**
- Migration guide (Discussion #7988) and persistence guide (Discussion #4845) both reinforce: store `UIMessage` JSON; rehydrate `UIMessage[]` into `useChat`'s `initialMessages`.

The implication for storage schema: one row per message, with a single JSON column for the full serialized `UIMessage`. Don't try to normalize tool calls into separate tables — the round-trip semantics get fragile, and SQLite handles JSON columns well.

```sql
CREATE TABLE conversations (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

CREATE TABLE messages (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  position INTEGER NOT NULL,   -- ordering within conversation
  role TEXT NOT NULL,           -- 'user' | 'assistant' | 'system'
  payload TEXT NOT NULL,        -- JSON.stringify(UIMessage) — the whole thing
  created_at INTEGER NOT NULL
);
CREATE INDEX idx_messages_conv_pos ON messages(conversation_id, position);
```

Byte-equality round-trip is achievable because `JSON.parse(JSON.stringify(msg))` is the persistence path; nothing in the middle reshapes the object. E-041's revised success criterion (byte-equality of the serialized `UIMessage` JSON before persist and after rehydrate) is achievable with this schema.

### Resource-attribution metadata

R-016 will decide the transport for retrieval telemetry into AI SDK messages. Whichever route is chosen (custom `metadata` field, typed `data-*` part, or message annotation), the metadata lands in the `UIMessage` and is captured by the same `payload` column. No schema change required regardless of R-016's outcome. This is a happy consequence of "persist the whole UIMessage as JSON" over "normalize into typed columns."

## Implications

1. **Recommendation: `sqlocal` (with the official `@sqlite.org/sqlite-wasm` underneath), backed by OPFS, served from a same-origin Next.js client component.**
   - Reasoning: most-actively-maintained, TypeScript-native, hides the worker bootstrap, ships Drizzle/Kysely drivers (we won't need Drizzle for two tables but the option is there), and uses the official SQLite WASM build so it inherits future SQLite improvements automatically. The dependency is small (~1 MB WASM, ~50 KB JS wrapper) and the API surface is small enough that ejecting later (to raw wa-sqlite or raw `@sqlite.org/sqlite-wasm`) is a one-week project, not a one-month one.

2. **Fallback / no-headers path: `sqlocal` configured to use `opfs-sahpool`.** If COOP/COEP turns out to cause unforeseen embedding problems (e.g., late-added third-party widgets), drop to `opfs-sahpool` without changing the Repository code — just a one-line VFS option flip. Trade: single-connection-per-origin, which is fine for a chat where the user mostly has one tab.

3. **Set COOP/COEP headers in `next.config.js` from day one.** Cheap to add, expensive to retrofit later (because you don't notice the missing concurrent-tab support until users complain). Add `Cross-Origin-Opener-Policy: same-origin` and `Cross-Origin-Embedder-Policy: require-corp` globally; nothing in the v1 surface area is cross-origin.

4. **Schema: two tables (`conversations`, `messages`), JSON blob for the message payload.** Do NOT normalize `UIMessage` into typed columns — that path loses round-trip fidelity and gets ugly fast. The JSON column is a deliberate choice that aligns with the playbook's "store the whole UIMessage, never `ModelMessage`" guidance.

5. **Don't use Drizzle for v1.** Two tables and three queries (`find conversations`, `find messages by conversation_id`, `insert message`) is not enough surface area to justify the schema-as-code overhead. Plain tagged-template SQL via sqlocal's `sql\`...\`` is fine. Re-evaluate at G-011 if the schema grows.

6. **Repository pattern shape** (per `coding-architecture` playbook): two repositories, both implementing the four-method shape (`find/create/update/delete` with rich criteria objects). `ConversationRepository` and `MessageRepository` are both backed by the same sqlocal instance, passed in via constructor. The instance is lazily created inside the client-component-tree (never at module top-level) and stored in module scope on the client side for re-use across renders.

7. **Worker bootstrap timing**: sqlocal opens its worker on first query. To avoid a first-message latency spike, kick off a no-op `SELECT 1` query during app mount (inside `useEffect`) so the WASM is loaded and the OPFS handle is open before the user sends their first message.

8. **SQLITE_BUSY handling**: With the `opfs` VFS (concurrent connections allowed), occasional `SQLITE_BUSY` is possible if two tabs write at the same instant. The Repository's `create` should retry up to N times with exponential backoff before surfacing the error. SQLocal handles a lot of this internally; verify the failure path during E-041 testing by opening two tabs and writing simultaneously.

9. **OPFS is per-origin, persistent until user clears site data.** For a deployed app (e.g., on Vercel), conversations survive page closes, restarts, and refreshes — but are lost if the user opens an incognito window (no persistent storage), clears site data, or switches browsers. This matches the goal's "browser-scoped users" decision. Document this in the welcome flow's privacy section per E-045.

10. **For tests/dev**: sqlocal supports an in-memory mode (skip the OPFS path) which makes unit tests fast and deterministic. Use it in vitest/jest test runs; OPFS isn't available in Node.

## Decision Summary

| Criterion | Winner | Why |
| --- | --- | --- |
| Best TypeScript ergonomics | **sqlocal** | Native TS, tagged-template SQL, types throughout |
| Best raw performance | wa-sqlite (OPFSCoopSyncVFS) | Mature, PowerSync-blessed |
| Best maintenance trajectory | sqlocal / official sqlite-wasm | Actively shipping in 2026 |
| Smallest bundle | sql.js | But pays the cost on every snapshot |
| Best persistence semantics | sqlocal (uses official OPFS) | True file-backed SQLite, survives reload reliably |
| Best fit for our two-table chat schema | **sqlocal** | Sweet spot between control and ergonomics |
| Easiest swap-out later | sqlocal | API surface is small; Repository pattern hides the choice |

**Overall pick: `sqlocal`.** Use it in E-041.
