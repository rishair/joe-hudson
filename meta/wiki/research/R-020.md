# Status dashboard stack: Next.js per playbook vs simpler Bun-HTTP for a 4-section read-only view

## Question

Pick the stack for G-013's status dashboard. The dashboard is a four-section, read-only, server-rendered view of `/home/claude/joe-hudson/`: wiki state, recent commits with diffs, checkpoints list, backlog inline. Single user. Polling auto-refresh. Hosted under G-012's PaaS as `status.play.ris.hair`.

Candidates evaluated:

- **Next.js (App Router)** — the [[ai-web-app]] playbook default
- **Bun.serve + plain HTML strings** — the lightest possible thing
- **Hono on Bun with `hono/jsx`** — middle ground; JSX templating, no build step
- **Astro 5 with SSR adapter** — content-site framework with SSR support

The deciding question: does the dashboard need anything Next.js gives but the lighter options don't? And: does anything lighter than Next.js have a hidden tax that erases its simplicity advantage?

## Why This Matters

E-050 (dashboard build) can't start without a stack choice. Picking Next.js by default would copy the web-app pattern from G-010, where it's earning its keep (streaming chat, client-side SQLite, complex client state). The dashboard has none of that. Choosing a heavier stack than the problem warrants creates ongoing tax: longer cold starts, a build step in the deploy loop, more dependency churn, more surface area to debug when something breaks at 11pm. Choosing too light a stack means re-implementing what frameworks give for free (routing, templating, type-safe handlers).

## Constraints and Scope

In scope:
- Match the chosen stack to G-013's actual requirements (read-only, server-rendered, polling refresh, FS reads, shell-out to `git` and `./wiki.sh`)
- Account for G-012's runtime model: `systemd --user` unit, `bun run start` (or equivalent), single port
- Account for existing project conventions: Bun is already the script runtime (coach-app, web-app/scripts both `#!/usr/bin/env bun`)
- Produce a concrete recommendation E-050 can pick up and start

Out of scope:
- Picking a CSS strategy (defer to E-050 — Tailwind, vanilla CSS, or inline `<style>` are all viable for a single-page dashboard)
- Evaluating frameworks not on the candidate list (Express, Fastify, Remix, SvelteKit — all would solve this, none would solve it better than the four evaluated)
- Mobile-responsive design (nice-to-have, not a stack-driver)

## Findings

### Recommendation: Hono on Bun with `hono/jsx`

Build `status-dashboard/` as a single Bun process: Hono router, JSX server-side rendering, `Bun.$` for shelling out to `git` and `./wiki.sh`, `node:fs` for filesystem reads. No bundler, no build step, no client JavaScript framework. The systemd unit runs `bun run server.tsx` directly.

### Why Hono+Bun+JSX wins for this specific problem

**It's the only candidate that gets four desirable properties simultaneously:**

1. **No build step** — `bun run server.tsx` executes the JSX source directly. Edit-and-reload is instant. No `next build`, no Astro `pnpm build`, no Vite. The G-012 deploy loop becomes `git pull && systemctl --user restart paas-status`, full stop.
2. **Type-safe JSX templating** — `c.html(<Page data={state} />)` reads like React but ships zero client JS. The dashboard's four sections are four small components in one file.
3. **Tiny memory footprint** — Hono + Bun + JSX baseline is ~30 MB resident. Next.js dev server idles at 200+ MB; Next.js production server is leaner but still ~80-150 MB. On a personal Hetzner box hosting multiple projects via G-012, RSS pressure adds up.
4. **No client-side JavaScript at all (v1)** — auto-refresh is `<meta http-equiv="refresh" content="30">`. Click-to-expand is `<details>/<summary>`. The dashboard is HTML over the wire, which is exactly what a polling read-only view should be. Adding `useChat`-style interactivity later is a deliberate decision, not a default.

### Why Next.js loses for this specific problem

Next.js earns its weight in the web-app (G-010) because that app has streaming chat (`useChat`), client-side SQLite (`sqlocal`), and complex client state (`zustand`). The status dashboard has none of those. Using Next.js here would mean:

- Shipping ~80-120 KB of React runtime to render four HTML sections that never re-render client-side.
- Running `next build` in the deploy loop, which dominates cold-start time on first run after pull.
- Pulling in Turbopack / OpenNext / `wrangler` tooling that exists to solve problems the dashboard doesn't have.
- Stack overlap with G-010's web-app would mean copy-pasting `package.json` patterns that don't apply (no need for `@ai-sdk/react`, no need for `sqlocal`, no need for `next.config.js`).

The [[ai-web-app]] playbook is explicit on its scope: "Use when the goal involves a web UI that streams or invokes LLM responses, not when the work is a CLI tool, a one-shot eval, or a non-AI web app." The dashboard is a non-AI web app. The playbook's prescription does not bind here.

### Why plain Bun.serve loses

Bun.serve directly works, and would be ~150 lines of code. The hidden tax: every section's HTML is a template string. After three or four edits, you're hand-managing escape sequences and concatenating template strings inside loops. Hono's JSX gives the same runtime footprint with much better author ergonomics. The marginal cost of pulling `hono` (a small dep, no build) is negligible.

### Why Astro loses

Astro is built for content sites with islands of interactivity. It needs a build step (`astro build`) even in SSR mode, which fights G-013's "instant edit-and-reload" property. Its file-based routing is overkill for a single-page dashboard. The 2026 framework comparisons (Cosmic, Cipher) both push Astro toward marketing/docs and Next.js toward dashboards; Hono+JSX sits in a different niche that's better for this specific minimal-server use case than either.

### Bun is presumed installed on the Hetzner box

Confirmed by repo evidence: `coach-app/run.ts` and `web-app/scripts/ingest-coach.ts` both use `#!/usr/bin/env bun`, `package.json` defines `"ingest": "bun scripts/ingest-coach.ts"`. The Hetzner box that runs this project already has Bun. The status dashboard runtime requirement is "Bun present", which the box satisfies.

Note: in the dev shell where this research was performed, Bun is not on PATH (`which bun` returned not-found, but `node` is `v24.16.0`). This is a dev-machine quirk, not a deployment blocker. The Hetzner box where G-012 deploys G-013 has Bun (or will; that's a G-012 prereq either way since `paas` itself will need a runtime for project commands like `bun run dev`).

### Boundary conditions on this recommendation

This pick is correct **for v1 of G-013 as currently scoped**: four read-only sections, server-rendered, polling refresh, single user, no LLM calls, no auth (CF Access handles it). If the dashboard scope expands to include any of the following, revisit the choice:

- **Streaming LLM responses** (e.g., "ask the agent why E-046 is stuck") → swap to Next.js + AI SDK per [[ai-web-app]]
- **Client-side state with optimistic updates** (drag-and-drop reorderable backlog, in-place wiki editing) → Next.js + Zustand is the existing pattern
- **Multi-user with per-user views** → still works on Hono, but CF Access needs per-user JWT extraction and the rendering layer needs user-scoping that adds enough complexity to reconsider
- **Real-time push** (WebSockets / SSE for "checkpoint just landed") → Hono supports both, but polling is intentionally simpler for v1; only revisit if 30-second polling proves unsatisfying in real use

### Concrete shape for E-050

Project layout:
```
status-dashboard/
  server.tsx              # Hono app + Bun.serve, single entrypoint
  lib/
    wiki-state.ts         # parses ./wiki.sh next/stale, returns typed state
    git-recent.ts         # Bun.$ shells to git log + git show
    checkpoints.ts        # readdir meta/wiki/checkpoints/, parse frontmatter
    backlog.ts            # reads meta/wiki/backlog/index.md
  views/
    Page.tsx              # top-level layout JSX component
    WikiState.tsx         # section 1
    RecentCommits.tsx     # section 2
    Checkpoints.tsx       # section 3
    Backlog.tsx           # section 4
  styles.css              # one stylesheet, served via hono/bun serveStatic
  package.json            # deps: hono, gray-matter (for frontmatter); devDeps: @types/bun
  tsconfig.json           # jsx: "react-jsx", jsxImportSource: "hono/jsx"
  .local-dev.yaml         # for G-012 paas register
  README.md
```

`package.json` start script: `"start": "bun run server.tsx"`. No `build` script. No `dev` script (start IS dev — Bun reloads on file edit with `--hot`).

`server.tsx` skeleton:
```typescript
import { Hono } from 'hono'
import { Page } from './views/Page'
import { getWikiState } from './lib/wiki-state'
// ... etc

const app = new Hono()

app.get('/', async (c) => {
  const state = {
    wiki: await getWikiState(),
    commits: await getRecentCommits(20),
    checkpoints: await getCheckpoints(),
    backlog: await getBacklogMarkdown(),
  }
  return c.html(<Page state={state} />)
})

app.get('/api/commit/:hash', async (c) => {
  const hash = c.req.param('hash')
  const diff = await getCommitDiff(hash)
  return c.html(<pre>{diff}</pre>)
})

export default { port: Number(process.env.PORT ?? 4001), fetch: app.fetch }
```

`tsconfig.json` snippet (the JSX configuration is the non-obvious bit):
```json
{
  "compilerOptions": {
    "jsx": "react-jsx",
    "jsxImportSource": "hono/jsx"
  }
}
```

`.local-dev.yaml` for G-012:
```yaml
subdomain: status
port: 4001
command: bun run server.tsx
mode: built
working_dir: /home/claude/joe-hudson/status-dashboard
restart: always
```
