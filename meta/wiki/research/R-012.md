# What retrieval strategies fit a deeply-linked methodology compendium like coach/?

## Question

The `coach/` compendium is ~470 small interlinked markdown files (reads, moves, questions, concepts, distinctions, principles, anti-patterns, frameworks, practices, applications, arcs, patterns, concerns, examples) with rich `related:` cross-references. Generic chunked-RAG is designed for long-document retrieval. This compendium is structured more like a hypertext encyclopedia than a document corpus. What retrieval strategies fit this shape? Which deserve experimental evaluation under G-009's $40 budget?

## Why This Matters

The user explicitly wants empirical comparison of multiple retrieval strategies (E-032 tool-based, E-033 embedding-based, plus E-031 no-retrieval as floor). Without a structured survey of the space, we risk testing two arbitrary approaches and missing the one that fits the shape of the data. This research produces a ranked shortlist of approaches with rationale, so E-032 and E-033 are not arbitrary picks. Also names the specific design choices within each approach (chunking unit, embedding model, top-K, reranker, etc.) so the experiments have unambiguous defaults.

## Constraints and Scope

In scope:
- Surveying retrieval patterns: chunked embedding RAG, document-level embedding, hybrid (BM25 + dense), graph-walk via `related:` links (now first-class as E-036), tool-based navigation (E-032), full-context (load index, model picks what to fetch), summary-embedding, question-aspect embedding, hierarchical retrieval, late chunking
- For each: cost profile, latency profile, fit-to-compendium-shape, evidence from public benchmarks
- The user's "question-aspect embedding" idea (extract "what question does this answer" per doc, embed those, match query against question embeddings) is named as inspiration but not pre-committed; R-012 evaluates whether it deserves an experiment slot
- The graph structure of `related:` links is unique to this compendium and deserves explicit consideration (compendium-as-knowledge-graph rather than corpus)
- Recommend a primary approach for E-033 and a fallback if the primary underperforms
- **Multi-turn retrieval-trigger policy** (per audit Important #8): in a 12-turn conversation with retrieval, do you re-retrieve every turn (pollutes cache, costs more), retrieve only on the first turn (risk going stale), retrieve every N turns, or retrieve only when the client message signals a topic shift? This is a separate axis from the retrieval strategy and has both quality and cost implications. R-012 must document a recommended default and the tradeoffs. E-032/E-033/E-036 each set the trigger policy in their config based on this finding.

Out of scope:
- Building any retrieval pipeline (E-033 does that)
- Choosing the embedding model itself in detail (R-012 names 2-3 candidates; E-033 picks one)
- Reranker selection beyond naming whether one is needed
- Hybrid search infrastructure (only consider if the survey says hybrid wins on this shape)

Resources: the actual `coach/` files (read 10-20 to sample structure and link density), `meta/wiki/_backlinks.json` (which encodes the same kind of link graph and may inform a graph-walk approach), the `/ai_toolbelt` skill for local embedding generation (free), AI experiment playbook for cost-tracking discipline.

Output: this page ships with a "Findings" section that names 2-3 candidate approaches in ranked order, the design choices each entails, the predicted cost-per-eval-run, and which one E-033 should implement. If the survey concludes embeddings are wrong for this shape entirely, the finding is that E-033 should be replaced with a graph-walk experiment; G-009's decision log records the pivot.
