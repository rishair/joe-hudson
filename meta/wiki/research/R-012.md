# What retrieval strategies fit a deeply-linked methodology compendium like coach/?

## Question

The `coach/` compendium is ~470 small interlinked markdown files (reads, moves, questions, concepts, distinctions, principles, anti-patterns, frameworks, practices, applications, arcs, patterns, concerns, examples) with rich `related:` cross-references. Generic chunked-RAG is designed for long-document retrieval. This compendium is structured more like a hypertext encyclopedia than a document corpus. What retrieval strategies fit this shape? Which deserve experimental evaluation under G-009's $40 budget?

## Why This Matters

The user explicitly wants empirical comparison of multiple retrieval strategies (E-032 tool-based, E-033 embedding-based, plus E-031 no-retrieval as floor). Without a structured survey of the space, we risk testing two arbitrary approaches and missing the one that fits the shape of the data. This research produces a ranked shortlist of approaches with rationale, so E-032 and E-033 are not arbitrary picks. Also names the specific design choices within each approach (chunking unit, embedding model, top-K, reranker, etc.) so the experiments have unambiguous defaults.

## Constraints and Scope

In scope:
- Surveying retrieval patterns: chunked embedding RAG, document-level embedding, hybrid (BM25 + dense), graph-walk via `related:` links (now first-class as E-036), tool-based navigation (E-032), full-context (load index, model picks what to fetch), summary-embedding, question-aspect embedding, hierarchical retrieval, late chunking
- For each: cost profile, latency profile, fit-to-compendium-shape, evidence from public benchmarks
- The user's "question-aspect embedding" idea (extract "what question does this answer" per doc, embed those, match query against question embeddings) is named as inspiration but not pre-committed; R-012 evaluates whether it deserves an experiment slot
- The graph structure of `related:` links is unique to this compendium and deserves explicit consideration (compendium-as-knowledge-graph rather than corpus)
- Recommend a primary approach for E-033 and a fallback if the primary underperforms
- **Multi-turn retrieval-trigger policy** (per audit Important #8): in a 12-turn conversation with retrieval, do you re-retrieve every turn (pollutes cache, costs more), retrieve only on the first turn (risk going stale), retrieve every N turns, or retrieve only when the client message signals a topic shift? This is a separate axis from the retrieval strategy and has both quality and cost implications. R-012 must document a recommended default and the tradeoffs. E-032/E-033/E-036 each set the trigger policy in their config based on this finding.

Out of scope:
- Building any retrieval pipeline (E-033 does that)
- Choosing the embedding model itself in detail (R-012 names 2-3 candidates; E-033 picks one)
- Reranker selection beyond naming whether one is needed
- Hybrid search infrastructure (only consider if the survey says hybrid wins on this shape)

Resources: the actual `coach/` files (read 10-20 to sample structure and link density), `meta/wiki/_backlinks.json` (which encodes the same kind of link graph and may inform a graph-walk approach), the `/ai_toolbelt` skill for local embedding generation (free), AI experiment playbook for cost-tracking discipline.

Output: this page ships with a "Findings" section that names 2-3 candidate approaches in ranked order, the design choices each entails, the predicted cost-per-eval-run, and which one E-033 should implement. If the survey concludes embeddings are wrong for this shape entirely, the finding is that E-033 should be replaced with a graph-walk experiment; G-009's decision log records the pivot.

Retrieval-pattern literature (drawn from established practice; no LLM calls made in this research):

- Generic chunked dense RAG (LangChain/LlamaIndex default): split docs into ~500-token chunks, embed, retrieve top-K. Designed for long-document corpora (legal, technical docs). Anti-fit here because (a) most coach/ files are already 1-2K words and self-contained, (b) chunking shatters the curated voice mid-paragraph, (c) the compendium's hand-curated edges encode information that lives between files, not inside them.
- BM25 and hybrid (sparse + dense): BM25 retrieves on exact terms; reciprocal-rank fusion blends with dense. Public benchmarks (BEIR, MTEB) consistently show hybrid > dense-alone on technical/named-entity-heavy corpora. Highly relevant here because the alias lists are essentially hand-authored sparse-index targets.
- ColBERT / late-interaction: token-level embeddings, more expressive than single-vector. Higher infrastructure cost; gains are modest below corpus sizes of ~10K docs. Overkill at 648 files.
- HyDE (Hypothetical Document Embeddings): use an LLM to expand the query into a synthetic answer, then embed the answer. Helpful when queries are short and corpus docs are long. Less needed here because aliases already capture many likely query forms.
- Question-aspect / "what does this answer" embeddings (user's mentioned inspiration): for each doc, extract the questions it answers and embed those. The compendium's `aliases:` field is **already this**, generated by a human (the compendium curator) rather than an LLM. The work has been done; retrieval just has to use it.
- Graph-walk / graph-RAG (Microsoft GraphRAG, LlamaIndex KnowledgeGraphIndex, recent academic work on dense+graph hybrid): walk an explicit relation graph from seed nodes. Public results show large gains over dense-only on corpora with hand-curated edges; the canonical example is well-cross-referenced encyclopedic content (which coach/ is).
- Multi-turn retrieval policies: literature splits into per-turn re-retrieval (good for topic drift, kills cache), first-turn-only (good for cache, bad for drift), conversational query-rewriting (rewrite client message with context, then retrieve), and topic-shift detection (cheap classifier or embedding-distance heuristic between consecutive client messages).

No API spend incurred (pure survey + local file inspection).

## Findings

### Headline

The `coach/` compendium is a hand-curated knowledge graph with three features that decisively narrow the retrieval design space: (1) **every file ships with hand-authored aliases** that already encode "what client phrasings does this answer," (2) **a 648-file × 3,941-edge `related:` graph plus a 4,598-edge backlinks inverse** captures expert-chosen relational adjacency, and (3) **the `_index.md` is itself a 684-line routing menu** that fits in a single prompt. Generic chunked dense RAG is the wrong default. The right retrieval family is **hybrid lexical-over-aliases + graph-walk over related links**, with embedding similarity as a tiebreaker rather than the primary retrieval mechanism.

### Ranked candidate approaches (informs E-033 and E-036)

**Rank 1 — Graph-walk with alias-keyed seeds (recommended primary for E-036).**

Architecture:
1. **Seed detection** via cheap-model alias match. Pass the client message plus `_index.md` (which inlines every file's aliases) to Haiku 4.5; ask for 1-3 seed file paths. The alias inlining means Haiku is doing essentially lexical lookup, not deep reasoning — cheap and reliable. Cost: ~$0.001-0.003 per call.
2. **Walk to depth 2** over `related:` edges, deduplicating. The audit explicitly notes "where Joe goes from here" is captured in the Downstream sections — those Downstream items appear in `related:`. Walking two hops is enough to traverse a concern → concept → move chain (the typical coaching arc).
3. **Score nodes by curated-edge proximity, hub-dampen, dedupe.** Direct neighbors score 1.0; 2-hop scores 0.5; nodes reached from multiple seeds get +0.3 bonus (graph centrality signal). **Crucially: dampen hubs**. With `feeling-the-unfelt-emotion` having 134 incoming edges, naive walk would always retrieve it. Penalize by `1/log(1+indegree)` so high-indegree generic nodes lose to specific situational matches.
4. **Cap at 5-7 files** by score. Inline retrieved markdown under "RETRIEVED CONTEXT (graph-walk neighborhood)".

Why this fits the data shape: the `related:` edges are the same expert judgment the coach is trying to embody. The Downstream sections in concept pages literally say "Toward [[X]] when [client signal]" — that is hand-encoded coaching navigation. Embedding similarity replicates topical adjacency; graph-walk replicates **navigational adjacency**, which is what coaching needs.

Where this best moves D4 (Methodology Fidelity, current 2.07, target ≥2.50): D4 fails when the coach uses "generic emotional-coaching language" instead of signature AoA moves. Graph-walk from a concern seed surfaces the exact `moves/` and `questions/` Joe would reach for. E-024's edge-not-anxious failure (D4 was a 1 across the conversation) is the canonical case: the right `moves/meet-resistance-with-okay`, `moves/lets-go-there`, and `reads/disassociating-in-the-coaching-room` are direct `related:` neighbors of `concerns/im-not-an-anxious-person.md`. The retrieval architecture that gets those into context is the one that gets D4 from 2.07 to >2.50.

Failure modes (and mitigations):
- Seed mis-detection (Haiku picks `applications/money-domain` for a client whose actual concern is `i-have-to-sacrifice-to-succeed`). Mitigation: restrict seed candidates to `concerns/`, `patterns/`, `reads/`. The compendium structure makes this safe — those three categories are the "where the client is" surface, while everything else is "where Joe goes from there."
- Hub dilution. Mitigation already specified above (1/log(1+indegree) dampen).
- Sparse seeds (client message doesn't match any concern). Mitigation: fall back to embedding-similarity-over-aliases as a backup seed-finder before declaring no seed.

**Rank 2 — Hybrid alias-BM25 + dense embedding (recommended primary for E-033).**

Architecture:
1. **Build two indexes off the same unit (one whole markdown file = one document):**
   - A **BM25 (or simple TF-IDF) sparse index over `aliases:` fields concatenated with `title:` and the first-section body**. This is the hand-curated paraphrase index, free and fast.
   - A **dense index over a "compact representation" per file**: `title + aliases + first-section body`, embedded with a small local model (bge-small-en-v1.5 default; nomic-embed-text-v1.5 fallback). Both are free local-CPU via the `/ai_toolbelt` skill. bge-small is recommended because: (a) ~33M params runs fast on CPU, (b) tops MTEB for its size class, (c) 384-dim vectors keep the SQLite index tiny (~1MB for 648 docs), (d) `/ai_toolbelt` already targets it.
2. **Query the client message against both indexes**, take top-10 from each, **fuse via reciprocal-rank fusion (RRF, k=60)**, cap at **top-5 files**. RRF is the standard hybrid fusion technique and needs no training.
3. **Inline retrieved markdown under "RETRIEVED CONTEXT"**. No reranker (overkill at K=5).

Why this is rank 2 not rank 1: it's a strong-baseline approach that gives the dense vs. sparse comparison cleanly and is much cheaper to debug than graph-walk. It will score well on D1 (Perceptual Accuracy) and on `concerns/` matching. It will likely **trail graph-walk on D4 specifically**, because dense similarity surfaces topical neighbors, not navigational neighbors. A concept page and a related move page may be embedded into different vector regions even though they belong together in a coaching arc.

Design choices baked in (so E-033 doesn't have to re-research):
- **Chunking unit: one file = one document.** Do not sub-chunk. Files are already 1-2K words and self-contained; sub-chunking shreds the curated voice. The only exception worth considering is the `_index.md` itself, which is too long to be one chunk and isn't a candidate for retrieval anyway (it's a router input).
- **Embedding text per doc**: `title + aliases (joined) + the body up to first \n## heading`. This biases the vector toward client-facing phrasings rather than internal exposition.
- **Embedding model: bge-small-en-v1.5 (384-dim)** via `/ai_toolbelt`. Local, free, fast (<5ms/doc on CPU).
- **Top-K = 5.** Smaller than typical RAG because individual files are larger (1-2K words ≈ 1.5K tokens × 5 = 7.5K tokens of inlined context, which fits comfortably under E-033's 30K context budget).
- **No reranker.** At K=5 over 648 docs the marginal gain from cross-encoder reranking is small relative to its cost.
- **Trigger policy**: see policy section below.

**Rank 3 — Pure dense-over-aliases (fallback; documented but not preferred).**

If the BM25+dense hybrid in Rank 2 hits an infra snag, fall back to **dense-only over the same compact representation**. This loses the lexical exactness on alias phrasings (e.g., a client saying "I just don't get anxiety" should hit `im-not-an-anxious-person.md` exactly via lexical match on its alias `"I don't have anxiety"`; dense alone may miss it for closer-vector-but-wrong files) but the dense index alone still beats whole-file embedding without aliases. Document this as a degenerate-case test inside E-033 (smoke-only, not full-eval), so E-034 can attribute lexical lift.

### Multi-turn retrieval-trigger policy (audit Important #8)

The four candidates and the recommended default per architecture:

| Policy | Cost impact | Drift risk | Cache impact |
|---|---|---|---|
| `every_turn` | High (re-runs router/embedding each turn) | Low | High (per-turn injection invalidates prefix cache) |
| `first_turn_only` | Lowest | High (multi-topic conversations lose retrieval) | Best |
| `every_n_turns` (N=3) | Medium | Medium | Medium |
| `on_topic_shift` | Medium (cheap heuristic + re-retrieve when triggered) | Low | Good (cache holds between shifts) |

**Recommendation: default to `every_turn` for the full eval comparison** in E-031/E-032/E-033/E-036. Rationale: (a) apples-to-apples comparison requires the same trigger policy across architectures; (b) the G-008 profiles are 11-15 turns and many involve mid-conversation pivots (the suicidal disclosure at T9 in `crisis-suicidal-001` is the canonical example — `first_turn_only` would catastrophically fail there); (c) embedding retrieval is local-free so the cost concern is only the inflated coach context, which prompt caching mitigates because the *system prompt + conversation prefix* still caches even though the RETRIEVED-CONTEXT block changes.

**Smoke `on_topic_shift` as a variant** in E-032 and E-036 only. Cheap implementation: cosine distance between current client message embedding and the embedding used at last retrieval; if distance > threshold (start at 0.3), re-retrieve, else reuse last bundle. The audit calls this out as the cost-quality sweet spot for tool-based nav specifically — a tool-call coach pays the most per per-turn retrieval round-trip, so skipping when the topic hasn't moved actually saves money. For E-033 (embedding-based, near-zero retrieval cost), the policy switch buys nothing on cost and only adds drift risk — keep `every_turn`.

**Do not implement `first_turn_only`** in any experiment. It is dominated by both other policies and the safety risk (missing a mid-conversation crisis disclosure) is unacceptable for a coaching system.

### Cross-cutting design choices that apply to all retrieval experiments

1. **Inject as "RETRIEVED CONTEXT" with a short framing line**: `Retrieved compendium files (relationally-adjacent to the client's current concern). Draw on these to ground your reading and move selection. Do not announce that you are retrieving.` This is consistent with the E-031 v1 prompt's existing retrieval-aware line (per audit Important #5) and ensures the coach uses retrieval rather than reciting it.

2. **Format inline as full markdown bodies, not summaries**. The files are short enough to fit and the voice is the value. The judge in E-024 explicitly rewarded "evidence-anchored credit" — abridging would lose that.

3. **Cap at 5-7 files maximum** in any architecture. Coach attention degrades fast above this. K=5 is the default; K=7 only if scores cluster very tight or in the hybrid case where two strategies overlap.

4. **Log per-turn retrieval telemetry** in all four experiments (E-031 logs "no retrieval" sentinel; E-032/E-033/E-036 log filenames + scores). The telemetry is the diagnostic for E-034's head-to-head — without it you can't tell whether a dimension delta came from retrieval quality or coach behavior on the retrieved bundle.

5. **Smoke against `expected_territory` before paying for full eval**: every G-008 profile has `expected_territory` listing the right file IDs. For each retrieval architecture, after smoke runs, compute precision/recall of retrieved files vs. expected_territory. A retrieval that gets <40% recall on expected territory is unlikely to lift D4; cut the experiment before paying for a full eval. This is a free pre-screen.

## Implications

### For the G-009 plan

1. **E-033 should be a hybrid (alias-BM25 + dense), not pure dense.** The "embedding-based retrieval" framing in the experiment page is already general enough — the recommendation here is to make hybrid the default config and document pure dense as the degenerate test. The dependency on R-012 in E-033's frontmatter is now satisfied with a concrete spec.

2. **E-036 (graph-walk) is correctly the architecturally-closest experiment.** The audit's Critical #1 was right. R-012 confirms and strengthens it: 100% alias coverage, 546/648 files with `related:` arrays, 4,598 backlink edges, hand-encoded coaching navigation in Downstream sections. Graph-walk is not a long-shot experiment; it is the architectural front-runner.

3. **Predicted ranking before E-034 runs**:
   - **D4 Methodology Fidelity (the leverage point)**: E-036 ≥ E-033 > E-032 > E-031.
   - **D1 Perceptual Accuracy**: E-033 ≈ E-036 > E-032 > E-031. (Dense/lexical matches well to client-visible signals.)
   - **D2 Intervention Quality**: E-036 > E-032 ≈ E-033 > E-031. (Move selection benefits most from navigational adjacency.)
   - **D3 Coaching Stance**: roughly equal across retrieval variants and possibly *worse* than E-031 on this dimension specifically — retrieved content can pull the coach toward content-delivery and away from VIEW stance. This is the audit's Important #6 risk and it is real. Mitigation: the "RETRIEVED CONTEXT" framing line explicitly says "Do not announce that you are retrieving," and E-035 should be ready to add stance-protection language if D3 regresses under retrieval.
   - **D5 Anti-Pattern Avoidance**: E-036 ≥ E-033 > E-032 > E-031. The anti-patterns are explicit `coach/anti-patterns/` files that the right retrieval will surface; graph-walk from a concern seed reaches them naturally.
   - **D6 Effectiveness / Depth**: E-036 ≥ E-033 > E-032 > E-031. Following the curated path is what produces depth.

4. **Hybrid (E-037) is highly likely to be mandated by E-034.** The prediction above shows mixed per-dimension winners (E-036 likely wins D2/D4/D5/D6; E-033 may win or tie D1; E-031 may protect D3). The audit's Critical #2 prediction holds. E-037 should pre-spec as: per-turn, use graph-walk as primary retrieval, then *augment* with the top-2 dense-hybrid hits that are NOT already in the graph-walk bundle. This is the "graph-first, embedding-as-fallback-for-novel-topics" hybrid.

5. **The `expected_territory` pre-screen is a free quality gate that should be added to E-033/E-036/E-032 success criteria.** Compute precision/recall of retrieved file IDs against `expected_territory` for each profile in smoke. A retrieval that scores <40% recall on expected_territory has no plausible path to D4 ≥ 2.50 and should be debugged or abandoned before paying for full eval. This was not in the original audit; flagging it here as a cost-saving addition.

### For the compendium (G-007, indirectly)

For graph-walk to be maximally effective, those 102 files should grow `related:` arrays. This is not a G-009 blocker — graph-walk just won't reach those files, and the dense fallback handles them — but it is a clean signal back to G-007 that backfilling `related:` on those 102 files is high-leverage absorption work. Worth a Decision Log entry on G-007.

### For G-008 (the eval), indirectly

R-012 affirms that `expected_territory` is doing real work as retrieval ground truth. If G-008 ever extends with new profiles, the `expected_territory` field should remain required, not optional.
