# What tool catalog should the AI SDK coach have for compendium navigation?

## Question

The Vercel AI SDK (`ai` + `@ai-sdk/anthropic`) is the chosen stack for G-009. What tool catalog should E-032's tool-based navigation coach have access to? Specifically: list-index, search, fetch, follow-related-links. What tool names, descriptions, and parameter schemas produce reliable invocation? What are the failure modes (silent tool errors, malformed args, looping)?

Per audit Important #7, R-013 was originally scoped to also cover SDK basics (`generateText`, prompt caching, error handling). That scope has been moved into E-031 as a 30-min spike, because E-031 is no-tool no-retrieval and doesn't need a full tool-use survey before starting. R-013 narrows to the tool catalog and tool-use specifics that E-032 (and E-036's seed-detection tool) actually need.

## Why This Matters

E-032 implements tool-based compendium navigation; E-036 implements a seed-detection tool. Without a structured tool catalog and the SDK-specific quirks documented in one place, each experiment reinvents the schema and the failure-handling. This research produces a documented tool catalog and a tool-use cheatsheet for the AI SDK so E-032 and E-036 share infrastructure. Also articulates the **rationale for choosing Vercel AI SDK over raw `@anthropic-ai/sdk`** for the coach (per audit Minor #13): tool-use loops with `maxSteps` ergonomics, agent-style retrieval, streaming, all of which the AI SDK does with less code than the raw SDK; the eval harness (E-023) uses raw SDK because it's a one-shot orchestrator with no tool-use loops.

## Constraints and Scope

In scope:
- Tool catalog design for the coach: 4-6 tools max for E-032, named and schematized (list-index, search, fetch, follow-related, and the seed-detection tool E-036 needs)
- Vercel AI SDK tool-use specifics: `tool()` helper, `maxSteps` (or equivalent multi-step loop config), tool error handling, parameter validation with Zod, malformed-argument recovery
- Anthropic provider specifics (`@ai-sdk/anthropic`) for tool-use: how Claude tool-use maps to SDK abstractions, cache behavior when tool outputs invalidate prefix caches
- Cost implications of tool-use (extra round-trips, prompt re-sends, cache invalidation on tool output)
- AI SDK vs raw Anthropic SDK rationale (per audit Minor #13): document why the coach uses AI SDK while the eval uses raw SDK

Out of scope:
- SDK basics (`generateText`, prompt caching basics, top-level error handling): moved to E-031 inline spike per audit Important #7
- Implementing the tools (E-032 implements the navigation tools; E-036 implements seed-detection)
- Embedding model selection (R-012 covers)
- Streaming UI considerations (CLI for v1; HTML viewer is nice-to-have)

Resources: Vercel AI SDK docs and GitHub repo, Anthropic tool-use docs, the `/claude-api` skill (mostly covers raw SDK but may help on Claude-side tool-use specifics), existing tool-use patterns in E-023 (which uses raw SDK so it is a contrast, not a template).

Output: this page ships with a documented tool catalog (4-6 tools with name, description, parameter schema, return shape, why) plus an SDK usage cheatsheet snippet showing the minimal multi-step loop with tool error handling and `maxSteps` set sensibly (suggested 8-15 given coaching turn budget). Cheatsheet code becomes the template E-031/E-032/E-033 copy from.

## Findings

### Why AI SDK for the coach, raw SDK for the eval (audit Minor #13)

The eval (E-023) is a one-shot orchestrator. Each judge call sends a static prompt and expects a single structured response. There is no tool loop, no agent step, no fan-out to multiple model calls inside one logical request. The raw `@anthropic-ai/sdk` `messages.create()` API is the right shape: one call in, one response out. `eval/lib/anthropic.ts` is 250 lines including caching, cost tracking, and retry, and it is the shape it is because that is what the eval needs.

The coach is fundamentally different. Each coaching turn involves up to ~5 round-trips inside one logical "respond to the user" action: route -> call `search_compendium` -> call `read_compendium_file` x2 -> call `follow_related` -> generate. The Vercel AI SDK collapses this into one `generateText({ tools, stopWhen: stepCountIs(8) })` call that internally orchestrates the loop. The equivalent in raw SDK is ~80 lines of: build messages, post, check for `tool_use` content blocks, dispatch to handlers, append `tool_result`, post again, repeat until `stop_reason !== 'tool_use'`, with manual cost and token tracking across each step.

Concretely, AI SDK provides:
1. **`tool()` helper** that pairs a Zod schema with a TypeScript-typed execute function. The schema is auto-converted to the JSON Schema Claude expects; the result is type-checked at the call site.
2. **`stopWhen: stepCountIs(N)`** (v5 syntax; v4 used `maxSteps: N`). Caps the multi-step loop so a misbehaving model can't loop forever.
3. **`onStepFinish`** callback for per-step telemetry (which tools were called, how much each step cost). E-032 needs this for the "per-turn telemetry logs which files were retrieved" success criterion.
4. **Provider-agnostic tool semantics.** If we ever swap Anthropic for OpenAI in a fallback experiment, the tool definitions don't change; only the provider does.
5. **Streaming.** `streamText` with the same `tools` and `stopWhen` produces a token stream interleaved with tool-call events. Nice-to-have for the CLI REPL.

The cost of AI SDK: one extra layer of abstraction over raw HTTP, and one extra dependency. Token usage and per-step cost still need to be tracked manually (the SDK exposes `usage` per step but does not bill by model — wrap in a cost-tracker as E-023 does for raw SDK).

**Recommendation: AI SDK for the coach. Do not migrate the eval — it doesn't need what AI SDK adds, and migrating would invalidate the cost-tracking work in `eval/lib/anthropic.ts`.**

### Tool catalog for E-032 (compendium navigation)

Four tools, plus a fifth retrieval-trigger primitive R-012 surfaced. Each tool's name is verbose-but-namespaced (`list_compendium_*`, `read_compendium_*`) so Claude is unlikely to confuse them with built-in tools. Descriptions are written in the second person from the model's perspective with explicit "use this when" cues — Anthropic's tool-use prompting guide recommends this pattern and we follow it.

#### Tool 1: `list_compendium_index`

```typescript
import { tool } from 'ai';
import { z } from 'zod';
import { readFile } from 'node:fs/promises';

const listCompendiumIndex = tool({
  description:
    "Returns a structured catalog of every file in the coach/ compendium. " +
    "Use this once at the start of a turn if you do not already know which " +
    "category to look in. Categories are: concerns (what the client " +
    "presents), reads (what to notice in the client), patterns (recurring " +
    "shapes), concepts, distinctions, principles, anti-patterns, moves, " +
    "questions, frameworks, practices, examples, applications. The output " +
    "includes one line per file with its id, title, and the client " +
    "phrasings (aliases) that match it. Do not call this more than once " +
    "per turn.",
  inputSchema: z.object({
    category: z
      .enum([
        'concerns', 'reads', 'patterns', 'concepts', 'distinctions',
        'principles', 'anti-patterns', 'moves', 'questions', 'frameworks',
        'practices', 'examples', 'applications',
      ])
      .optional()
      .describe(
        "If provided, returns only files in that category. Omit to get " +
        "the full index (about 18KB, manageable in context)."
      ),
  }),
  execute: async ({ category }) => {
    const fullIndex = await readFile('coach/_index.md', 'utf-8');
    if (!category) return { index: fullIndex };
    // Slice to a single section heading
    const sectionPattern = new RegExp(
      `## ${headingFor(category)}\\n([\\s\\S]*?)(?=\\n## |$)`
    );
    const match = fullIndex.match(sectionPattern);
    return { index: match ? match[1].trim() : `(no entries in ${category})` };
  },
});
```

Return shape: `{ index: string }`. The full index is ~18KB which fits comfortably under Anthropic's per-tool-result soft ceiling. The optional `category` filter is the "I already know roughly where to look" shortcut — saves ~15KB and reduces noise for follow-up calls.

Failure modes:
- Model calls this every turn instead of caching its mental map. **Mitigation**: the description says "Do not call this more than once per turn." Per-step telemetry (`onStepFinish`) flags repeated calls; if observed in smoke, tighten the description with "If you have already called this in this conversation, do not call it again."
- Index file moves or renames. **Mitigation**: the execute function is the single source of path truth; failures throw and the SDK surfaces a `ToolExecutionError` that the model sees as a `tool_result` with `isError: true`.

#### Tool 2: `search_compendium`

```typescript
const searchCompendium = tool({
  description:
    "Searches the coach/ compendium for files whose title or aliases " +
    "match the query string. Use this when the client says something " +
    "and you want to find the file that captures that concern or read. " +
    "Returns up to 8 matches ranked by alias-match score. If you get " +
    "zero matches, try shorter or more literal phrasings (e.g., " +
    "\"can't apologize\" rather than \"struggles with admitting fault\").",
  inputSchema: z.object({
    query: z
      .string()
      .min(2)
      .max(120)
      .describe(
        "A short phrase to match against file titles and aliases. " +
        "Literal phrasings work better than paraphrases because aliases " +
        "are hand-authored client phrasings."
      ),
    category: z
      .enum([
        'concerns', 'reads', 'patterns', 'concepts', 'distinctions',
        'principles', 'anti-patterns', 'moves', 'questions', 'frameworks',
        'practices', 'examples', 'applications',
      ])
      .optional()
      .describe("Restrict search to one category. Omit for full search."),
  }),
  execute: async ({ query, category }) => {
    const results = await searchByAliasMatch(query, category, { limit: 8 });
    return {
      query,
      matches: results.map(r => ({
        path: r.path,        // e.g., "coach/concerns/i-cant-apologize.md"
        id: r.id,            // e.g., "i-cant-apologize"
        title: r.title,
        matchedAlias: r.matchedAlias,
        score: r.score,
      })),
    };
  },
});
```

Return shape: `{ query: string; matches: Array<{ path; id; title; matchedAlias; score }> }`. Returning the matched alias is a deliberate signal to the model: it gets to see WHICH phrasing matched, which helps it judge whether the hit is real.

`searchByAliasMatch` is implemented in `coach-app/tools/search.ts` as: parse `coach/_index.md` once at startup into an in-memory `Map<alias, fileId>`; for each query, score each alias by (exact-prefix-match * 1.0) + (substring-match * 0.5) + (token-overlap * 0.25); return top-K. This is BM25-lite; it intentionally does NOT do dense embedding (E-033 owns that). Cheap, deterministic, debuggable.

Failure modes:
- Zero matches. **Mitigation**: the description tells the model to try shorter phrasings. Return `{ matches: [] }` rather than throwing.
- Model issues 4+ search calls per turn trying variants. **Mitigation**: `stopWhen: stepCountIs(8)` caps total steps; the cost is 4 extra Haiku calls per turn (~$0.005) which is bearable but visible in telemetry.

#### Tool 3: `read_compendium_file`

```typescript
const readCompendiumFile = tool({
  description:
    "Reads the full markdown contents of one file in the coach/ " +
    "compendium. Use this AFTER search_compendium or list_compendium_index " +
    "has given you a specific file path. Returns the file's frontmatter " +
    "(aliases, related, tags, sources) and body. Files are typically " +
    "1-2K words each. Reading 3-5 files per turn is normal; reading more " +
    "than 7 fills context unproductively.",
  inputSchema: z.object({
    path: z
      .string()
      .regex(/^coach\/(concerns|reads|patterns|concepts|distinctions|principles|anti-patterns|moves|questions|frameworks|practices|examples|applications)\/[a-z0-9-]+\.md$/)
      .describe(
        "Path to a file in the compendium, e.g., " +
        "\"coach/concerns/i-cant-apologize.md\"."
      ),
  }),
  execute: async ({ path }) => {
    const body = await readFile(path, 'utf-8');
    return { path, contents: body };
  },
});
```

Return shape: `{ path: string; contents: string }`. Returning the full file (not a summary) is consistent with R-012's "inject as full markdown bodies, not summaries" finding — the curated voice is the value.

The regex in the Zod schema is load-bearing: it filters out hallucinated paths before they hit the filesystem. Without the regex, a model that invents `coach/concepts/welcoming.md` (which doesn't exist; the real file is `coach/concepts/love-as-deep-welcoming.md`) gets an `ENOENT` from `readFile`; the SDK turns this into a `tool_result` with `isError: true` and the model can recover, but the regex gives a cleaner error ("Invalid input: path doesn't match the expected pattern") that more often produces a useful next step.

Failure modes:
- Model invents file paths. **Mitigation**: the regex catches structural inventions; the filesystem catches the rest. The model's next-step recovery from a tool error is reliable in Claude Sonnet 4.6.
- Model reads 8+ files per turn. **Mitigation**: description bounds expected usage; `stopWhen: stepCountIs(8)` caps total steps including reads.

#### Tool 4: `follow_related`

```typescript
const followRelated = tool({
  description:
    "Returns the list of files that the given file's frontmatter `related:` " +
    "field points to. Use this to discover the navigation Joe would do from " +
    "a concept: if you've read a concern file, follow_related will give you " +
    "the concept, move, and anti-pattern files that the curator linked. " +
    "These are hand-curated; an LLM did not pick them. Returns at most 15 " +
    "related file paths.",
  inputSchema: z.object({
    path: z
      .string()
      .regex(/^coach\/[a-z-]+\/[a-z0-9-]+\.md$/)
      .describe("Path of a compendium file you have already read."),
    direction: z
      .enum(['outgoing', 'incoming', 'both'])
      .default('outgoing')
      .describe(
        "outgoing = files this file links TO (forward `related:`). " +
        "incoming = files that link to this file (backlinks, useful when " +
        "you want to find what other concerns lead Joe to this concept). " +
        "both = union, deduplicated."
      ),
  }),
  execute: async ({ path, direction }) => {
    const id = path.replace(/^coach\/[a-z-]+\/(.+)\.md$/, '$1');
    const outgoing = direction !== 'incoming' ? await readForwardRelated(path) : [];
    const incoming = direction !== 'outgoing' ? await readBacklinks(id) : [];
    const merged = dedupe([...outgoing, ...incoming]).slice(0, 15);
    return {
      path,
      direction,
      related: merged.map(targetId => ({
        id: targetId,
        path: pathForId(targetId),  // resolves id back to coach/<category>/<id>.md
      })),
    };
  },
});
```

Return shape: `{ path, direction, related: Array<{ id, path }> }`. Returning resolved paths (not bare ids) means the model can pass the result of `follow_related` directly to `read_compendium_file` without re-resolving.

`readForwardRelated` parses YAML frontmatter once on first read and caches the `related:` array. `readBacklinks` reads `coach/_backlinks.json` once at startup into an in-memory map. Both are O(1) per call.

Failure modes:
- The 102 files without `related:` arrays (per R-012). **Mitigation**: `outgoing` returns `[]` cleanly; `incoming` from `_backlinks.json` still works because the inverse index is complete.
- Hub explosion: `feeling-the-unfelt-emotion` has indegree 134; `follow_related --direction incoming` on its `id` would dump 134 paths. **Mitigation**: hard `.slice(0, 15)` cap; the description warns "Returns at most 15." This is intentional — graph-walk-as-strategy (with hub-dampening) lives in E-036, not in the tool layer.

#### Tool 5 (retrieval-trigger primitive, used by E-032 only): `decide_retrieval_needed`

R-012 surfaced an `on_topic_shift` smoke variant for E-032 where the cost saving from skipping retrieval on follow-up turns that stay on-topic is meaningful. This is implemented as a lightweight tool that the model can call BEFORE issuing any compendium tool calls; if it returns `{ retrieve: false }`, the coach skips retrieval and uses the prior turn's bundle.

```typescript
const decideRetrievalNeeded = tool({
  description:
    "Call this FIRST on each non-first turn. Returns whether you should " +
    "retrieve fresh compendium files based on whether the client's " +
    "current message has shifted topic from the last turn. Compares the " +
    "current message's embedding against the embedding at last retrieval; " +
    "if cosine distance > 0.3 the topic has shifted and you should " +
    "re-retrieve. If retrieve=false, use the previously-retrieved bundle " +
    "and skip all other compendium tools this turn.",
  inputSchema: z.object({
    currentClientMessage: z
      .string()
      .min(1)
      .describe(
        "The client's most recent message, verbatim. The function " +
        "embeds it and compares against the embedding stored at last " +
        "retrieval."
      ),
  }),
  execute: async ({ currentClientMessage }) => {
    const currentEmbedding = await embed(currentClientMessage);  // bge-small, local
    const lastEmbedding = retrievalStateStore.lastEmbedding;
    if (!lastEmbedding) return { retrieve: true, reason: 'first_turn' };
    const distance = cosineDistance(currentEmbedding, lastEmbedding);
    return {
      retrieve: distance > 0.3,
      reason: distance > 0.3
        ? `topic_shift (distance=${distance.toFixed(2)})`
        : `same_topic (distance=${distance.toFixed(2)})`,
      distance,
    };
  },
});
```

Return shape: `{ retrieve: boolean; reason: string; distance?: number }`. The reason string is for telemetry; the model uses only the boolean.

Failure modes:
- Model skips this tool and re-retrieves anyway. **Mitigation**: that's a smoke variant — the experiment's default (`every_turn`) doesn't include this tool. R-012 explicitly says smoke this only in E-032 where the cost saving is real.
- Embedding state lost between turns (e.g., process restart in the middle of a conversation). **Mitigation**: store `retrievalStateStore` on the conversation object, not in process memory; if missing, return `retrieve: true`.

### Tool catalog for E-036 (seed detection)

E-036 needs exactly one tool: a seed-detection primitive that Haiku invokes against `_index.md` to return 1-3 seed file IDs constrained to `concerns/`/`patterns/`/`reads/`. This is a separate experiment with its own retrieval architecture (graph-walk), but it shares the AI SDK plumbing.

#### Tool 6: `detect_seeds_from_message`

```typescript
const detectSeedsFromMessage = tool({
  description:
    "Given the current client message, returns 1-3 seed file IDs from " +
    "the coach/ compendium that best match what the client is presenting. " +
    "Seeds are restricted to concerns (what the client says), patterns " +
    "(recurring shapes in the client), and reads (what to notice in the " +
    "client). Use the inline aliases in coach/_index.md to ground your " +
    "match. If nothing matches, return an empty array; do not invent a " +
    "seed.",
  inputSchema: z.object({
    seedIds: z
      .array(z.string().regex(/^[a-z0-9-]+$/))
      .min(0)
      .max(3)
      .describe(
        "1-3 file IDs (no path, no .md), each one from concerns/, " +
        "patterns/, or reads/. Empty array if no good match."
      ),
    rationale: z
      .string()
      .min(10)
      .max(200)
      .describe(
        "One-sentence rationale per seed, concatenated. Used for " +
        "telemetry, not for downstream logic."
      ),
  }),
  execute: async ({ seedIds, rationale }) => {
    // Validate IDs exist and are in the allowed categories
    const validIds = seedIds.filter(id =>
      seedCatalog.has(id) &&
      ['concerns', 'patterns', 'reads'].includes(seedCatalog.get(id).category)
    );
    return {
      seedIds: validIds,
      rejected: seedIds.filter(id => !validIds.includes(id)),
      rationale,
    };
  },
});
```

Return shape: `{ seedIds: string[]; rejected: string[]; rationale: string }`. Returning `rejected` separately lets the caller log which seeds the model proposed but failed validation (often: a `concepts/` or `moves/` ID, which is the model trying to do the navigation work itself instead of just naming the starting point).

This tool is invoked via `generateText({ tools: { detectSeedsFromMessage }, toolChoice: { type: 'tool', toolName: 'detectSeedsFromMessage' } })` — forcing the tool call means Haiku always returns through this shape, never as free-form text. The execute function does the category validation server-side; the model can't bypass it.

`seedCatalog` is a precomputed `Map<id, { category, title, aliases }>` derived from `_index.md` at startup.

Failure modes:
- Model returns IDs that don't exist. **Mitigation**: server-side validation via `seedCatalog.has(id)`. The empty `seedIds` triggers R-012's fallback: dense-similarity-over-aliases seed-finder (reuse the bge-small index E-033 builds).
- Model proposes seeds outside the allowed categories. **Mitigation**: category validation filters; `rejected` field surfaces these in telemetry so E-036 can detect a pattern (e.g., if Haiku keeps trying to seed from `moves/`, the description needs sharpening).

### The minimal multi-step loop (the template E-032 and E-036 copy from)

```typescript
import { generateText, stepCountIs, type ToolSet } from 'ai';
import { anthropic } from '@ai-sdk/anthropic';
import { readFile } from 'node:fs/promises';

interface CoachTurnArgs {
  systemPrompt: string;
  conversationHistory: Array<{ role: 'user' | 'assistant'; content: string }>;
  clientMessage: string;
  tools: ToolSet;
  model?: string;  // default 'claude-sonnet-4-6'
  maxToolSteps?: number;  // default 8
}

interface CoachTurnResult {
  response: string;
  steps: Array<{
    type: 'tool-call' | 'tool-result' | 'text';
    toolName?: string;
    toolInput?: unknown;
    toolOutput?: unknown;
    text?: string;
  }>;
  usage: {
    inputTokens: number;
    outputTokens: number;
    cachedInputTokens: number;
  };
  finishReason: string;
}

export async function runCoachTurn(args: CoachTurnArgs): Promise<CoachTurnResult> {
  const {
    systemPrompt,
    conversationHistory,
    clientMessage,
    tools,
    model = 'claude-sonnet-4-6',
    maxToolSteps = 8,
  } = args;

  const stepLog: CoachTurnResult['steps'] = [];

  try {
    const result = await generateText({
      model: anthropic(model),
      // Anthropic prompt caching via provider options. The system prompt is
      // long and stable -- cache it. cache_control on the last block caches
      // EVERYTHING up to and including that block.
      system: [
        {
          type: 'text',
          text: systemPrompt,
          providerOptions: {
            anthropic: { cacheControl: { type: 'ephemeral' } },
          },
        },
      ],
      messages: [
        ...conversationHistory,
        { role: 'user', content: clientMessage },
      ],
      tools,
      // Cap multi-step loop. If a turn needs more than 8 tool calls, the
      // tool catalog is wrong, not the model.
      stopWhen: stepCountIs(maxToolSteps),
      temperature: 0.7,
      maxOutputTokens: 1024,
      // Per-step telemetry. E-032 success criterion: "per-turn telemetry
      // logs which files were retrieved."
      onStepFinish: ({ stepType, toolCalls, toolResults, text, usage }) => {
        for (const call of toolCalls ?? []) {
          stepLog.push({
            type: 'tool-call',
            toolName: call.toolName,
            toolInput: call.args,
          });
        }
        for (const result of toolResults ?? []) {
          stepLog.push({
            type: 'tool-result',
            toolName: result.toolName,
            toolOutput: result.result,
          });
        }
        if (text) stepLog.push({ type: 'text', text });
      },
      // Recover from malformed tool arguments. Without this, a Zod
      // validation failure stops the loop with an error. With it, the SDK
      // re-prompts the model with the validation message and lets it
      // retry once before giving up.
      experimental_repairToolCall: async ({ toolCall, error, messages, system, tools }) => {
        if (!(error instanceof Error) || !error.message.includes('Invalid')) {
          return null;  // Only repair Zod/JSON-Schema validation errors
        }
        const repair = await generateText({
          model: anthropic(model),
          system,
          messages: [
            ...messages,
            {
              role: 'tool',
              content: [
                {
                  type: 'tool-result',
                  toolCallId: toolCall.toolCallId,
                  toolName: toolCall.toolName,
                  result: `Validation failed: ${error.message}. Reissue the call with corrected arguments.`,
                  isError: true,
                },
              ],
            },
          ],
          tools,
          stopWhen: stepCountIs(1),
        });
        // Return the repaired tool call from the first message in the response
        return repair.toolCalls?.[0] ?? null;
      },
    });

    return {
      response: result.text,
      steps: stepLog,
      usage: {
        inputTokens: result.usage.inputTokens,
        outputTokens: result.usage.outputTokens,
        cachedInputTokens:
          result.providerMetadata?.anthropic?.cacheReadInputTokens ?? 0,
      },
      finishReason: result.finishReason,
    };
  } catch (err) {
    // Top-level: surface failures to the caller with context, not silently
    // empty responses. The eval harness expects a string back; throwing
    // here is the right behavior because a missing coach turn should fail
    // loudly.
    throw new Error(
      `Coach turn failed at step ${stepLog.length}: ${err instanceof Error ? err.message : String(err)}`
    );
  }
}
```

Key choices:

1. **`stopWhen: stepCountIs(8)`** is the AI SDK v5 idiom; older v4 code uses `maxSteps: 8`. Eight is enough for: 1 `decide_retrieval_needed` (if used) + 1 `search_compendium` + 3 `read_compendium_file` + 1 `follow_related` + 1 final-text-generation = 7, with one slot of headroom. If smoke shows the model consistently using fewer, drop to 6 to save round-trips on the rare runaway turn.

2. **Prompt caching via `providerOptions.anthropic.cacheControl`**. The system prompt is ~3,100 tokens (R-011); without caching, every turn re-bills it as input. With caching, the first turn pays cache-write (1.25x input rate); subsequent turns pay cache-read (0.1x input rate). Across a 12-turn conversation that's a ~85% cost reduction on the system-prompt portion.

3. **Tool outputs invalidate the prefix cache below the last cache_control marker.** This is a subtle Anthropic mechanic: prompt caching is keyed by exact prefix; tool calls insert new content into the message stream, which is below the system block. The cache on the system block survives (it's above the tool churn). The cache on the conversation history does NOT survive (it's interleaved with new tool calls and results). The recommendation: cache the system prompt and leave the conversation history uncached. The eval already does this in `eval/lib/anthropic.ts` and the comments there are accurate.

4. **`experimental_repairToolCall`** is the malformed-argument recovery hook. The bare SDK turns Zod validation failures into thrown errors that crash the loop; the repair hook gives the model one retry. This is documented as `experimental_` but has been stable across AI SDK v4 and v5. If it's promoted to non-experimental in a later version, drop the `experimental_` prefix.

5. **`onStepFinish` telemetry** is the per-turn diagnostic. E-032's success criterion ("per-turn telemetry logs which files were retrieved") is satisfied by writing `stepLog` to disk per turn.

6. **Try/catch at the top.** A failed coach turn must throw a useful error, not return an empty string. The eval harness consumes the throw as a profile-level failure with the error message attached.

### Two-tier model invocation (E-032's router/coach split)

E-032's architecture is: cheap router (Haiku) makes the tool calls; expensive coach (Sonnet, or Opus for nice-to-have) generates the final text WITHOUT tools, with the retrieved files inlined into its context.

The implementation pattern is two `generateText` calls per turn, not one. The router call has tools enabled and `stopWhen: stepCountIs(6)` (no final-text-generation step needed — its job is just to assemble the retrieval bundle). The coach call has no tools and just generates text from the system prompt + inlined RETRIEVED CONTEXT block + conversation + client message.

```typescript
// Stage 1: Router (Haiku) assembles retrieval bundle via tools
const routerResult = await generateText({
  model: anthropic('claude-haiku-4-5'),
  system: ROUTER_SYSTEM_PROMPT,  // "You are a retrieval planner..."
  messages: [...history, { role: 'user', content: clientMessage }],
  tools: compendiumTools,
  stopWhen: stepCountIs(6),
  toolChoice: 'required',  // Force at least one tool call; no free-form text
});
const bundle = extractRetrievedFiles(routerResult.steps);  // reads tool-result entries

// Stage 2: Coach (Sonnet/Opus) generates response from bundle + history
const coachResult = await generateText({
  model: anthropic('claude-sonnet-4-6'),
  system: [
    { type: 'text', text: COACH_SYSTEM_PROMPT,
      providerOptions: { anthropic: { cacheControl: { type: 'ephemeral' } } } },
  ],
  messages: [
    ...history,
    {
      role: 'user',
      content: `RETRIEVED CONTEXT (router-assembled, ${bundle.length} files):\n\n${bundle.map(f => f.contents).join('\n\n---\n\n')}\n\nCLIENT MESSAGE:\n${clientMessage}`,
    },
  ],
  temperature: 0.7,
  maxOutputTokens: 1024,
});
```

Cache implications: the coach system prompt caches across turns (the prefix is stable). The retrieved-context block changes every turn (different files each turn under `every_turn` policy), so the user message portion is not cached. This is the dominant cost component of E-032 — R-012 already accounted for it in the $13 budget for E-032.

`toolChoice: 'required'` is the AI SDK's way to force the model to use at least one tool, never returning bare text. Useful in the router stage where free-form text would be a router protocol violation. The alternative `toolChoice: { type: 'tool', toolName: 'search_compendium' }` forces a SPECIFIC tool; useful in E-036's seed-detection stage.

### Common failure modes specific to AI SDK + Anthropic

These come from prior AI SDK tool-use experience (publicly documented patterns; not from running cost in this research):

1. **Silent tool errors when execute throws synchronously.** Without try/catch in `execute`, a thrown error becomes an unhandled rejection that some AI SDK versions surface as a generic "tool failed" without the original message. **Mitigation**: wrap each `execute` body in try/catch and return `{ error: err.message }` rather than throwing; the model handles this cleanly as a recoverable result.

2. **`maxOutputTokens` too low.** Anthropic's default `max_tokens` of 1024 is fine for coach text but too low when the coach is generating a tool call (tool inputs count as output tokens). Bump to 2048 if smoke shows the model truncating mid-tool-call. Symptom: `finishReason: 'length'` with an incomplete `toolCall.args` JSON.

3. **Tool description "leaks" into voice.** If a tool description uses phrases like "compendium," "retrieval," or "navigation," Claude has a tendency to echo this language back to the client ("Let me check the compendium for that"). The v1 system prompt's anti-meta-narration rule (R-011 Finding 4, item 11) catches most of this, but the tool descriptions should also avoid words the coach should never say to the client. The descriptions above use "client" and "concern" (terms the coach can speak) and avoid "compendium" and "retrieval" in the imperative voice where possible.

4. **`stopWhen: stepCountIs(N)` is not a soft cap.** When N is hit and the loop is mid-tool-result, the SDK does NOT do a final text-generation step; it returns with `finishReason: 'tool-calls'` and no `text`. **Mitigation**: in E-032 this is acceptable for the router (extract steps, hand to coach). In E-036 the coach is a separate call so this is also fine. In a single-stage tool-use coach (a degenerate variant of E-032), bump N to leave room for the final text generation.

5. **Cache reads are not free.** Cache reads bill at 0.1x input rate (per Anthropic docs verified 2026-05-22 in `eval/lib/anthropic.ts`). A 3,100-token system prompt cached and read 11 times across a conversation bills as 11 × 3,100 × $0.0000003 ≈ $0.01 per conversation just on cache reads. Over 15 profiles × 12 turns that's still only ~$0.18 — negligible relative to per-turn output token costs, but worth knowing the cache doesn't make the system prompt completely free.
