# What system prompt structure makes an AoA coach behave like Joe?

## Question

What is the right structure, content, and length for an AoA coach system prompt? Specifically: which `coach/` principles, distinctions, and anti-pattern warnings belong in the prompt itself versus being retrieved on demand? What stance instructions reliably produce the "welcome rather than fix" orientation? How long can the prompt be before it stops paying off?

## Why This Matters

The user named system prompt iteration as the dominant lever in G-009. Without a structured answer to "what goes in the prompt", every iteration is exploratory. This research produces a draft v1 prompt plus a structured rationale for every block in it, so subsequent prompt iterations (E-035) are diffs against an articulated baseline rather than wholesale rewrites. The draft is what E-031 ships.

## Constraints and Scope

In scope:
- Reading `coach/principles/`, `coach/frameworks/`, `coach/distinctions/`, `coach/anti-patterns/`, and the 7 gold exchanges at `eval/gold-exchanges/` to extract what the prompt must encode
- Surveying public AoA-style system prompt approaches and prior coaching-AI system prompts (Woebot, Ginger.io, etc.) for structural patterns
- Token budget analysis: what is the cost-per-turn of various prompt lengths under prompt caching?
- Structural decisions: stance section, methodology guardrails, anti-pattern warnings, response shape, refusal conditions, safety screen language
- **Retrieval-awareness line** (per audit Important #5): include one sentence at the end of the stance section that primes the model to use retrieved context if present and not to announce retrieval. This makes v1 reusable across no-retrieval (E-031) and retrieval-equipped (E-032/E-033/E-036) architectures without architecture-tuning iterations later. Suggested wording: "When relevant `coach/` content is retrieved into your context, draw on it to ground your reading and your move selection. Do not announce that you are retrieving."

Out of scope:
- Implementing the coach (that is E-031)
- Retrieval decisions (R-012 covers that)
- Tool-use specifics (R-013 covers that)
- The actual content of every iteration (E-035 iterates against this baseline)

Resources: `coach/` compendium, `eval/gold-exchanges/`, `eval/checklists/methodology.yaml` (E-022's distillation of which moves matter), `eval/anti-pattern-tests/` (E-021's identification of AI-prone failures the prompt should pre-empt).

Output: a draft `coach-app/prompts/v1.md` file plus a rationale document on this page explaining each block. The R-011 page does not need to ship the prompt itself if the rationale is clear; E-031 produces the file.
