# Cloudflare Tunnel + Access setup specifics for a multi-subdomain personal-PaaS

## Question

What's the canonical setup pattern for Cloudflare Tunnel + Cloudflare Access (as of May 2026) for a Hetzner box hosting multiple Next.js / Bun apps each at their own `<project>.play.ris.hair` sub-subdomain? Specifically:

- How is `cloudflared` configured to route N subdomains to N local ports? (One config file, one tunnel, multi-ingress.)
- How is Cloudflare Access scoped to `*.play.ris.hair` so the policy applies to every new project automatically (no per-subdomain CF dashboard click)?
- Is there a `cloudflared`-side wildcard route, or does each subdomain need its own ingress entry?
- What's the right SSO option for an "only me" Access policy — Google, GitHub, One-time PIN, all of them?
- Tunnel running as a `systemd` system service (not user) — the standard pattern, but the per-project apps will run under `systemd --user`. Coordination concerns? Specifically the **boot ordering and lingering** question: how do user units survive a reboot when the user isn't logged in, and what happens between `cloudflared` coming up and the project's user unit being ready?
- Are there gotchas with WebSockets (Next.js HMR in dev mode), streaming responses (chat SSE), or large request bodies?
- DNS: does `ris.hair` need to be on Cloudflare nameservers, or is a CNAME from another DNS provider sufficient?
- **Does cloudflared support SIGHUP-based ingress reload, or does config change require full `systemctl restart cloudflared`?** Restart drops in-flight connections — this matters for `paas register` behavior.
- **Alternatives survey**: Tailscale Funnel/Serve, Caddy with Cloudflare DNS-01, Pomerium, wireguard+Caddy — should `Cloudflare Tunnel + Access` actually win for this goal, or did the conversation foreclose a better option?
- **User-side dashboard prerequisites**: what's the minimum set of dashboard actions the user must do before any agent can run `cloudflared tunnel create`? (Needed to file the right REQ.)

## Findings

### Alternatives surveyed: why Cloudflare Tunnel still wins (with caveats)

The audit (rec #3) correctly flagged that R-019 inherited "Cloudflare Tunnel + Access" without surveying the field. Comparison below; chosen primitive last.

| Option | Multi-subdomain at `<project>.play.ris.hair` | Public access | "Only me" auth | Open ports on Hetzner | Recurring cost | Verdict for G-012 |
|---|---|---|---|---|---|---|
| **Cloudflare Tunnel + Access** | Yes (one ingress per project, all on one tunnel) | Yes | Yes (CF Access; Google/OTP) | None (tunnel is outbound) | $0 (free Universal SSL) OR $10/mo (ACM) — see SSL section | **Chosen.** |
| **Tailscale Funnel** | No — Funnel only serves `<machine>.<tailnet>.ts.net`. No custom domain support. Open issues #11563 and #16389 request it; not shipped as of 2026. Only ports 443/8443/10000. | Yes | Yes (just-my-tailnet) | None | $0 | **Disqualified.** Cannot satisfy the `<project>.play.ris.hair` URL pattern. |
| **Tailscale Serve** (private-tailnet only, no public Funnel) | One MagicDNS name per device only — multi-service needs subpaths (`vps.tailnet.ts.net/coach`), NOT subdomains. `tsnet` can give per-service tailnet IPs but with its own bespoke setup. | No (private tailnet only) | Yes (tailnet membership) | None | $0 | **Disqualified.** Loses the "phone-to-Hetzner-from-anywhere-with-real-domain" affordance the goal wants. Also no subdomain story. |
| **Caddy + Cloudflare DNS-01** | Yes (Caddy reverse-proxies `<project>.play.ris.hair` → `localhost:port` per Caddyfile rule). Auto-issues wildcard cert via DNS-01. | Yes | No, by itself. Would need additional layer (Caddy plugin verifying CF Access JWT, or Authelia/Pomerium in front). | **Yes — port 443 must be open on Hetzner**. Violates G-012's "no Hetzner public ports" must-have. | $0 (with `xcaddy build` for the CF DNS plugin) | **Disqualified by the no-public-ports must-have.** |
| **Pomerium / Authelia in front of nginx** | Yes (proxy-level subdomain routing) | Yes | Yes (IDP-backed) | Yes (public 443) | $0 | **Disqualified by no-public-ports** AND adds a major auth-system surface for a personal box. |
| **wireguard + Caddy** | Yes (Caddy on the box) | No (requires WG peer) | Yes (WG keys = auth) | One UDP port for WG | $0 | Reasonable for strictly-personal use, but loses public addressability — the user wants `coach.play.ris.hair` reachable from any device including untrusted phones/browsers. |

**Reflection on the user's "Hetzner=personal, Cloudflare=public" framing**: the audit raised the philosophical concern that putting personal-play traffic through Cloudflare violates the separation. Counterargument: the framing is about **deployment target** (where the app lives + who's expected to use it), not about which CDN paths the traffic crosses. The Cloudflare Tunnel + Access path keeps the app on Hetzner (right side of the line) but uses Cloudflare's edge for *transport and auth*. The G-010 web-app side (which is the "public" side) deploys the app code *to* Cloudflare Workers (a fundamentally different boundary). So the framing is preserved as "deployment target," not as "traffic untouched by Cloudflare." The user-stated design choices recorded in the G-012 context (Cloudflare Tunnel chosen explicitly) align with this interpretation.

**Verdict**: Cloudflare Tunnel + Access remains the right primitive because it's the only option that satisfies the full conjunction of {custom-domain subdomain pattern, no public ports on Hetzner, IDP-backed auth, single-config-file model, zero or low recurring cost}. Tailscale's products would be cleaner if either the URL pattern requirement or the public-reachability requirement were relaxed; neither can be relaxed for the goal as written.

### Two ways to manage a tunnel: local vs remote (2026 default)

cloudflared supports two configuration models. **Locally-managed**: a `~/.cloudflared/config.yml` on the host declares the tunnel UUID, credentials file, and ingress rules. **Remotely-managed** (the 2026 default per Cloudflare): tunnel config lives in the dashboard or behind the API; the host's `cloudflared` only needs a connector token. Wrangler's tunnel subcommands (released 2026-03-19) only create remotely-managed tunnels — `npx wrangler tunnel create <name>` returns a UUID and the message "configure its ingress rules in the Cloudflare dashboard."

**Implication for G-012's CLI**: the original E-048 design assumes locally-managed config (the `paas` CLI rewrites a `config.yml` and reloads cloudflared on each register). That's still valid and is the lower-friction path because the CLI owns the source of truth on disk. Choosing remote-managed would mean the CLI hits `PUT /accounts/{ACCT}/cfd_tunnel/{TUNNEL_ID}/configurations` for each register/remove — fine, but introduces an API token dependency and online-only operations.

Recommendation: **locally-managed for v1**. The `paas` design is "edit a YAML, generate a systemd unit, reload" — fitting a remote API call into that adds complexity for no immediate gain. Revisit if/when running >1 cloudflared host or if zero-downtime ingress changes become a hard requirement.

### cloudflared ingress configuration

The canonical `config.yml` shape for a multi-subdomain locally-managed tunnel:

```yaml
tunnel: <tunnel-UUID>
credentials-file: /etc/cloudflared/<tunnel-UUID>.json

ingress:
  - hostname: coach.play.ris.hair
    service: http://localhost:3001
  - hostname: status.play.ris.hair
    service: http://localhost:4001
  - hostname: hello.play.ris.hair
    service: http://localhost:8080
  - service: http_status:404   # MANDATORY catch-all
```

Rules:

- **Catch-all is required**. Without a final `- service: http_status:404`, cloudflared refuses to start.
- **Top-down match** on hostname + optional path regex. First match wins.
- **Wildcards in `hostname`** are supported but only as a leading label: `*.play.ris.hair` matches `coach.play.ris.hair`. `test.*.example.com` is NOT supported (no mid-hostname wildcards).
- **`originRequest`** can be set globally or per-ingress: documented fields include `connectTimeout`, `disableChunkedEncoding`, `noTLSVerify`, `tcpKeepAlive`, `http2Origin`. Run `cloudflared tunnel help` for the full set.

### Wildcard ingress vs per-project ingress

A single `- hostname: *.play.ris.hair` line cannot route to multiple local ports (you can only have one `service:` per match). So the **CLI must add one ingress block per project** — wildcard ingress isn't useful here.

Wildcards ARE useful for the DNS record and the Access policy (one-time setup, applies to every new project).

### DNS prerequisites

- `ris.hair` zone must be **on Cloudflare nameservers** (full DNS delegation). CNAME-flattening from another DNS provider is NOT sufficient for Tunnel-routed hostnames; the route command (`cloudflared tunnel route dns <tunnel> <hostname>`) creates a proxied CNAME in the CF zone.
- For per-subdomain routing, run `cloudflared tunnel route dns <tunnel> <subdomain>.play.ris.hair` once per new project, OR create a one-time wildcard CNAME `*.play.ris.hair → <tunnel-UUID>.cfargotunnel.com` (proxied, orange-cloud). The wildcard CNAME is preferred for `paas` — the CLI doesn't need to touch DNS per registration.
- The `*.play.ris.hair` wildcard CNAME does NOT cover the apex `play.ris.hair` itself.

### SSL certificate gotcha for two-level wildcards

Cloudflare's free Universal SSL covers `ris.hair` and `*.ris.hair` (one-level wildcard) but does NOT cover `*.play.ris.hair` (two-level wildcard).

Without an additional cert, `coach.play.ris.hair` will produce browser SSL errors because the edge cert presented for `play.ris.hair` doesn't include the sub-subdomain SANs.

**Three resolution paths**:

1. **Advanced Certificate Manager** — $10/month per zone, supports multi-level wildcards. Covers `*.play.ris.hair` as a SAN. Up to 100 edge certs per zone, up to 50 hostnames each. Canonical fix.
2. **Flatten the subdomain scheme** — use `coach-play.ris.hair` instead of `coach.play.ris.hair`. Free Universal SSL covers `*.ris.hair` (one level), so flat-but-namespaced names work. Loses the visual `<project>.play.ris.hair` clarity but costs nothing.
3. **Per-subdomain edge certs** — issue individual certs for each `<project>.play.ris.hair` via Cloudflare's regular cert manager. Technically possible without ACM but requires manual provisioning per hostname; automation cost outweighs the $10/mo.

**Recommendation**: surface this as a decision in G-012. Either file a REQ for ACM ($10/mo recurring billing) or shift the scheme to flat-namespaced subdomains.

### Cloudflare Access setup for `*.play.ris.hair`

**Self-hosted application** is the right Access type for HTTP apps behind Tunnel.

Dashboard path: Zero Trust > Access controls > Applications > Add an application > Self-hosted.

Configuration:

- **Application Domain**: `*.play.ris.hair` (wildcard at the subdomain position, supported since the May 2024 "Wildcard and multi-domain support in Cloudflare Access" release). One Access app covers every `<project>.play.ris.hair` automatically.
- **Identity providers**: enable Google (no Workspace account required — any Google account) AND One-time PIN. OTP fallback covers cases where a browser isn't signed into Google.
- **Policy**: single Include rule, `Emails | <user-email>`. This restricts every covered subdomain to that single email.
- **Precedence**: if a future need arises to scope a specific subdomain differently (e.g., `public.play.ris.hair` with no auth), create a more-specific Access app for `public.play.ris.hair` — most-specific hostname definition wins.
- **Wildcard nesting**: per the May 2024 blog, wildcards can appear "anywhere in the subdomain or path." The single-level interpretation (covers `coach.play.ris.hair` but NOT `a.b.play.ris.hair`) is consistent with Cloudflare's other wildcard products (DNS, SSL), though the Access docs don't pin this down explicitly. Acceptable for `paas` — projects use single-label names like `coach.play.ris.hair`, never three-level depth.
- **JWT cookies** are issued and validated across all hostnames in a multi-hostname Access app — one Google login covers all subdomains for the session duration.

### Cloudflared as a systemd service

The canonical install: `sudo cloudflared service install` (after `cloudflared` itself is installed via apt). This:

- Creates `/etc/systemd/system/cloudflared.service`
- Copies `~/.cloudflared/config.yml` and `<tunnel-UUID>.json` to `/etc/cloudflared/` (the service runs as root by default and reads from `/etc/cloudflared/config.yml`)
- Enables and starts the unit (`systemctl enable --now cloudflared`)

**Config file location for the service**: `/etc/cloudflared/config.yml` — NOT `~/.cloudflared/config.yml`. The home-dir location is for interactive `cloudflared tunnel run` only. This is the most common gotcha — surfaces as "I edited the YAML and nothing changed."

**Implication for `paas`**: the CLI must edit `/etc/cloudflared/config.yml` (requires sudo) and run `systemctl restart cloudflared` after each change. Options:

- Run `paas` itself with sudo for register/remove (simple, ugly).
- Configure a sudoers rule allowing the user to `systemctl restart cloudflared` and edit `/etc/cloudflared/config.yml` without password (cleaner).
- **Symlink workaround** (recommended for v1): install cloudflared with its config dir as a symlink — `sudo ln -s /home/<user>/paas/cloudflared /etc/cloudflared`. The CLI then writes to `/home/<user>/paas/cloudflared/config.yml` as the user and only needs sudo for `systemctl restart cloudflared`. One-time sudo setup.

**Logs**: `journalctl -u cloudflared -f` (system service, not `--user`).

**The systemd user-unit pivot**: cloudflared as a system service (correct, single tunnel for the box) and per-project apps under `systemctl --user` is fine — they don't interact directly. cloudflared connects to `localhost:<port>` regardless of which user owns the listening process. The only coordination point is the port number, which `paas` manages.

### Boot survival: `loginctl enable-linger` is mandatory (added per audit rec #1)

By default, the systemd user instance is started **after a user's first login** and killed after their last session closes. This means a `systemctl --user enable paas-<sub>.service` is **not enough** to survive a Hetzner reboot — without a logged-in user, the per-project units never start, while the system `cloudflared` service comes up regardless and forwards traffic to `localhost:<port>` where nothing is listening. Result: every reboot leaves `<project>.play.ris.hair` returning 502 until someone SSHes in.

**Fix**: a one-time `sudo loginctl enable-linger <user>` makes the user's systemd instance start at boot and stay running. Verify via `loginctl show-user <user> | grep Linger=yes`.

This must be done **once** as part of E-047 (or a thin E-047a). The check should also live in `paas`: at first run, `paas` verifies `loginctl show-user $USER | grep -q Linger=yes` and either auto-applies the fix (with sudo prompt) or refuses to register projects with a clear message.

**Boot ordering between system `cloudflared` and per-user `paas-*` services**:

- `cloudflared.service` starts in the `multi-user.target` system scope.
- After linger is enabled, `user@<uid>.service` starts in `multi-user.target` (or earlier).
- The per-user `systemctl --user` units start at the user-scope's `default.target`, which begins after `user@<uid>.service` is up.
- Order of operations on reboot: kernel → systemd-system → cloudflared starts opening tunnel ~5s in → user manager starts ~10s in → user services start ~12s in.
- Race window: cloudflared may receive traffic for `<project>.play.ris.hair` during the 5-10s before the user service is listening on its port. Result: brief 502s.

**Mitigations (in order of complexity)**:

1. **`Restart=always` with short `RestartSec=2s`** on every generated paas user unit. Even if the project crashes or hasn't started yet, systemd retries quickly. The user-facing window of 502s shrinks to a few seconds at boot — acceptable for personal use.
2. **`After=user-runtime-dir@%U.service` and explicit `Wants=` on a network-online target** in the unit template, plus a small startup delay if the project is slow to bind its port.
3. **`After=cloudflared.service` cross-scope** — NOT possible directly because the units are in different scopes. Would require a polkit rule or a small system-scope helper service.

**Recommendation for `paas`**: ship option 1 by default (`Restart=always`, `RestartSec=2s`) and document the brief-startup-window caveat in the README. Don't add complexity for option 2/3 until a user actually hits the problem.

### SIGHUP reload behavior (added per audit recs #2 and #7)

After research into cloudflared issues #301, #512, and #1171 (all open as of May 2026) and the Cloudflare docs, the state of play for **locally-managed tunnels**:

- **`cloudflared` does NOT support SIGHUP-based ingress reload.** The feature request has been open since 2019 (#301) and 2021 (#512); both remain open. Sending `SIGHUP` to cloudflared is currently a no-op for ingress changes.
- **`cloudflared tunnel ingress validate <config>`** exists and is the right way to validate a new config before applying it. Returns non-zero on parse errors or invalid ingress.
- **The canonical reload primitive is `systemctl restart cloudflared`.** A restart re-establishes the four connections to Cloudflare's edge. Reconnect time is typically 2-5 seconds in good network conditions, occasionally 10-30 seconds.
- **In-flight connections are dropped on restart**: long-lived HTTP requests (WebSocket, SSE, in-progress AI SDK streaming responses) get cut. TCP and UDP flows are dropped. Short requests in flight may still complete if they're under cloudflared's drain timeout.
- **The hot-reload-via-replicas pattern** (documented for managed tunnels) requires running multiple cloudflared instances against the same tunnel; the orchestration cost (two services to manage, ensure one is always up) is too high for a single-host PaaS.

**E-048's "sub-second downtime" claim is inaccurate.** Correct estimate: **2-30 seconds of tunnel unavailability per `paas register` invocation**, during which any in-flight long-lived connection to ANY project on this tunnel is dropped.

**Implications for `paas` design**:

1. **`paas register` MUST be atomic via `flock`** (a single global lock on `~/paas/cloudflared/.lock`) so two concurrent invocations serialize.
2. **The reload pattern** should be: write new `config.yml.new` → `cloudflared tunnel ingress validate` → `mv` into place atomically → `sudo systemctl restart cloudflared` → poll `systemctl is-active cloudflared` until active or timeout (30s).
3. **Document the tradeoff in `paas/README.md`**: every register/remove causes a brief tunnel reconnect that drops in-flight long-lived connections. Suggest batching: register multiple new projects in sequence inside one `flock` if possible, or schedule registrations during off-hours.
4. **`paas register --dry-run`** (mentioned in audit rec #16) should be the default for first-run iteration so the user sees what changes before pulling the trigger.
5. **Future**: migrate to remotely-managed tunnel for hot-reload — the API path is documented in this research's "Two ways to manage a tunnel" section. Defer until pain materializes.

### User-side dashboard prerequisites checklist (added per audit rec #6)

The following actions can only be performed by the user (no API token grants them):

1. **Verify `ris.hair` is on Cloudflare nameservers.** Dashboard: DNS section shows zone status "Active." If not, the user must change nameservers at the domain registrar. This is a one-time, irreversible-feeling step.
2. **(Decision) Enable Advanced Certificate Manager** on the `ris.hair` zone for `*.play.ris.hair` SSL coverage. $10/mo recurring. Required if `<project>.play.ris.hair` (two-level wildcard) is kept. Skip if subdomain scheme is flattened to `<project>-play.ris.hair`.
3. **Enable Zero Trust** (Cloudflare Access requires this — free for up to 50 users). Dashboard: Zero Trust > Settings (one-time activation; no credit card needed for the free tier).
4. **Set up identity providers** in Zero Trust > Settings > Authentication > Login methods:
   - Add **Google** as a login method (any Google account, no Workspace) — guided wizard, no API setup required.
   - Add **One-time PIN** as a fallback (no setup beyond toggling on).
5. **Create the Cloudflare Tunnel** via dashboard OR mint an API token with `Cloudflare Tunnel:Edit` + `DNS:Edit` scopes so an agent can do it via `cloudflared tunnel create`. The token-based path requires:
   - Dashboard: Profile > API Tokens > Create Token > Custom token.
   - Scope: Account > Cloudflare Tunnel > Edit. Zone > DNS > Edit (for the `ris.hair` zone). Account > Access: Apps and Policies > Edit (if creating the Access app via API too).
   - Save token securely — shown only once.
6. **Create the Access application** for `*.play.ris.hair` (either via dashboard per the recipe in the "Cloudflare Access setup" section above, OR via API call to `POST /accounts/{ACCT}/access/apps` with the token from #5).

### Config reload without restart — the SIGHUP question

There is **no SIGHUP-based config reload** for cloudflared as of May 2026 (open feature requests since 2019 — issues #301 and #512). Editing `config.yml` requires `systemctl restart cloudflared` to take effect.

Restarting drops all in-flight long-lived connections (WebSockets, SSE streams, in-progress AI SDK responses). For `paas`, restart-on-register is acceptable for a personal box — a sub-second cut, and registrations are rare. If zero-downtime ingress changes become a hard requirement later, switch to **remotely-managed tunnels with replicas**: dashboard/API config changes propagate to all running replicas without restart. The 2026-03-20 multi-replica streaming update makes this operationally easier to monitor.

### Streaming and WebSockets gotchas

- **WebSockets work out of the box** through cloudflared tunnels (enabled by default since 2022). Next.js dev-mode HMR works fine. Set the HMR WebSocket URL to use WSS over port 443 if behind the tunnel.
- **SSE-over-GET bug** (cloudflared issue #1449, reported April 2025, open as of May 2026): events are buffered in the tunnel and flushed only when the server closes the connection. Documented for Quick Tunnels; not confirmed for named tunnels but author speculates same root cause. **Impact on the coach app and other AI SDK apps**: the Vercel AI SDK's default `useChat` hook sends a **POST** to `/api/chat` and streams the response body. POST streaming works correctly through cloudflared per the same issue. AI SDK chat apps are unaffected. EventSource-based clients (which only support GET) would be affected; switch to POST + fetch streaming or skip the tunnel for those endpoints.
- **HMR caveat**: when accessing the dev server through `coach.play.ris.hair`, the browser will try to open the HMR WebSocket at `wss://coach.play.ris.hair/_next/webpack-hmr`. This requires either (a) Next.js dev server configured with the public URL for HMR, or (b) developing locally on `localhost:3001` and skipping the tunnel for dev iteration. Document (b) as the default in the `paas` README — devs run `bun run dev` locally for iteration, and `paas register` is for "built mode" deployments.
- **Large request bodies**: Cloudflare's edge has a 100 MB request body limit on Free/Pro tiers. Not relevant for chat apps but worth noting for future projects.

### For E-048 (paas CLI design adjustments — apply when E-048 is claimed)

The audit raised these specific items; this research confirms or refines each:

1. **Config file location** (audit-implicit): `/etc/cloudflared/config.yml` via the symlink workaround. E-048's current method says `~/.cloudflared/config.yml` — update.
2. **Atomicity** (audit rec #2): write `config.yml.new` → `cloudflared tunnel ingress validate` → `mv` into place → trigger reload. Wrap all of `paas register`/`remove` in `flock` on `/home/<user>/paas/cloudflared/.lock`.
3. **Reload primitive** (audit rec #7): `sudo systemctl restart cloudflared` (NOT SIGHUP — confirmed no-op as of May 2026). Honest "2-30s tunnel reconnect" downtime, NOT "sub-second" as currently claimed. Document in the CLI's `--help` output and the README.
4. **No per-project DNS step**: with the one-time wildcard CNAME, `paas register` only updates ingress + reloads. No DNS API call needed.
5. **Boot story** (audit rec #1): generated systemd user units include `Restart=always` and `RestartSec=2s`. The unit template should also include `After=user-runtime-dir@%U.service` for cleanliness. `paas` itself should verify linger is enabled at first run and offer to enable it.
6. **Sudoers requirement**: document the one-time sudoers entry; `paas` first-run can offer to set it up or print the exact line to add.
7. **Generated config header guardrail** (audit rec #9): every `config.yml` rewrite includes a `# Generated by paas. Do not hand-edit. Run \`paas <subcommand>\` to modify.` header. `paas` checks for this header before overwriting; if absent, back up the existing file with a timestamp before overwriting (preserves user edits).
8. **Atomicity success criterion**: add to E-048 a falsifiable test — run two `paas register` concurrently and verify both projects end up registered cleanly (no orphan systemd units, no missing ingress entries, no corrupted YAML).

### For the schema (audit rec #5)

E-048's `.local-dev.yaml` schema needs four refinements when E-048 is claimed:

1. **Port conflict detection**: at `paas register` time, read all `paas-*.service` units, check `Environment=PORT=`, AND run `ss -tlnp | awk '{print $5}'` for non-paas processes. Error on collision with a clear message naming both projects.
2. **`env_file` path resolution**: document as **relative to `working_dir`** if relative; absolute paths accepted as-is.
3. **`paas restart <name>` subcommand**: add to the subcommand list. Equivalent to `paas down && paas up` but in one operation. Needed so `.env` changes propagate (systemd doesn't auto-reload `EnvironmentFile=` on file change).
4. **Secrets ownership doc**: one-line explainer in the README: secrets stay in `.env` (gitignored, user-owned, mode 0600); the systemd unit's `EnvironmentFile=` reads them at unit start; `paas list` and `journalctl` outputs are safe to share because the unit file itself never embeds secret values.
