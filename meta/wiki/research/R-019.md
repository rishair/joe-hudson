# Cloudflare Tunnel + Access setup specifics for a multi-subdomain personal-PaaS

## Question

What's the canonical setup pattern for Cloudflare Tunnel + Cloudflare Access (as of May 2026) for a Hetzner box hosting multiple Next.js / Bun apps each at their own `<project>.play.ris.hair` sub-subdomain? Specifically:

- How is `cloudflared` configured to route N subdomains to N local ports? (One config file, one tunnel, multi-ingress.)
- How is Cloudflare Access scoped to `*.play.ris.hair` so the policy applies to every new project automatically (no per-subdomain CF dashboard click)?
- Is there a `cloudflared`-side wildcard route, or does each subdomain need its own ingress entry?
- What's the right SSO option for an "only me" Access policy — Google, GitHub, One-time PIN, all of them?
- Tunnel running as a `systemd` system service (not user) — the standard pattern, but the per-project apps will run under `systemd --user`. Coordination concerns?
- Are there gotchas with WebSockets (Next.js HMR in dev mode), streaming responses (chat SSE), or large request bodies?
- DNS: does `ris.hair` need to be on Cloudflare nameservers, or is a CNAME from another DNS provider sufficient? (Tunnel-routed hostnames require CF DNS for the apex or the relevant zone.)

## Why This Matters

E-047 (Cloudflare Tunnel + Access setup) can't start without this. The "spec it once, get it right" research saves the experiment from doing trial-and-error against the Cloudflare dashboard.

## Constraints and Scope

In scope:
- Survey Cloudflare's current Tunnel + Access docs (May 2026 — they update frequently)
- Map answers to all the questions above
- Produce a concrete `~/.cloudflared/config.yml` template for E-048's CLI to generate
- Concrete Access policy JSON / dashboard-setting recipe
- DNS-side prerequisites checklist (what needs to be true before E-047 starts)

Out of scope:
- Comparing Tunnel against alternatives (already decided: Tunnel)
- Auth providers beyond what's needed for "only the user" access
- Building the CLI (E-048's job)
