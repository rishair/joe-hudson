# What's the canonical pattern for resource-attribution UX in chat (subtle in-flow + modal deep-dive)?

## Question

The user transcript specifies a specific UX for per-message resource attribution: subtle indicator inline with the message (does not break flow), click → modal with split layout (left = wiki entry for focused resource, right = list of all resources used in that message), clicking another right-panel resource switches the left panel, "back to chat" closes the modal, "expand" promotes the wiki entry to a full page, internal wikilinks within the modal navigate the left panel. This is a specific interaction model that needs concrete UI patterns and component choices.

What are existing well-designed implementations of this pattern (chat with citations + drill-down)? Examples worth studying: Perplexity's citation chips, Phind's source panel, Claude.ai's artifacts panel, ChatGPT's reference display. What URL routing strategy supports both modal-in-context AND "expand to full page" without losing conversation state?

## Why This Matters

E-044 (build the resource modal) depends on this. The wrong abstraction here makes "expand to full" awkward or loses chat context. The right abstraction makes the modal a thin overlay on routable URLs.

## Constraints and Scope

In scope:
- Survey 3-5 well-designed chat-with-citations interfaces; identify common patterns
- URL routing options: modal-as-route (Next.js parallel routes / intercepting routes), modal-as-state, modal-as-route-with-state-fallback
- State management for "current resource focus" + "all resources in this message" + "back-to-chat target message"
- Wikilink handling: in-modal navigation should update the focused-resource state without closing the modal
- "Expand" → routes to the full wiki page while preserving "back to this chat message" affordance

Out of scope:
- Visual design beyond layout/structure (Tailwind defaults; visual polish is a later concern)
- Animations and transitions (later)
- Mobile layout (nice-to-have; v1 is desktop-first)

## Findings

### 1. The pattern the user spec'd is a hybrid that does not match any single existing implementation cleanly

The closest analogues split into two camps:

**Camp A — inline numbered chips next to claims (Perplexity, Open-WebUI deep-link proposal):** Citation density is the point — each claim gets its own chip; the chip is hover-rich; click opens the source in-place or in a side panel. This works for retrieval where the model genuinely cites specific spans. It does NOT fit our compendium-walker case, where retrieval returns 3–8 wiki entries that informed an entire reply, not chunk-by-claim.

**Camp B — dedicated right-side panel or drawer (ChatGPT Deep Research, Perplexity Sources tab, AnythingLLM proposal):** A single panel that holds the current message's sources, persistently visible or toggleable. Some implementations switch the panel contents as you scroll/select different assistant messages. The drawer is part of the main layout, not a modal.

**The user's spec is closer to Camp B but routed through a modal rather than a persistent drawer.** The reasoning is sensible: a persistent drawer eats 30–40% of conversation width permanently for a feature most users will not need most of the time. The modal-on-demand keeps chat at full width by default but escalates to a focused split view on click.

The Claude.ai artifacts panel is a different beast — it shows a single live artifact, not a list of references — but it validates the split-pane mental model: the chat stays semantically anchored on the left edge of the user's attention while structured content occupies the bulk of the viewport on the right.

**ShapeOfAI's "panel layout" pattern** (right-side, scannable metadata, used by Bing) is the nearest documented match, but it's typically a persistent panel rather than a click-to-open modal. The user's design is best described as **"on-demand panel-layout, rendered modally."**

### 2. Recommended routing strategy: nuqs URL-state, NOT Next.js parallel/intercepting routes

Initial intuition (and the unresolved partial path in E-044, `app/@modal/(...)/page.tsx`) was that Next.js's parallel + intercepting routes give us "modal as URL" for free. After investigating, this is the **wrong abstraction for this specific UX** because of three concrete blockers.

**Blocker 2a — Modal-to-modal stacking bug.** The Next.js parallel+intercepting modal pattern only handles the *first* navigation from an underlying page into the modal slot. If a user opens the modal (e.g. via `/wiki/concepts/limiting-belief`) and then clicks a wikilink to another wiki page (e.g. `/wiki/concepts/love-as-deep-welcoming`) intending to stay in the modal with the left panel updated, the soft-navigation either (a) leaves both modal contents stacked in the DOM (issue #71586) or (b) closes the modal and routes to the full page. Workarounds exist (consolidate slots, conditional rendering via `usePathname`, `<Link replace>`) but they're hacks against the routing system, not idiomatic usage.

Our UX explicitly requires in-modal navigation: clicking a right-panel resource switches the left panel; clicking a wikilink in the left panel switches the left panel. Both are intra-modal state changes that shouldn't touch routes at all.

**Blocker 2b — The catch-all slot is fragile.** Per the official Next.js docs, closing a modal via `<Link>` (rather than `router.back()`) requires a `[...catchAll]/page.tsx` inside the slot that returns `null` — otherwise the modal stays mounted because the slot doesn't match a real route. This works but means every new top-level route in the app must be considered against the modal slot's match set. For an app with one main `/` route and one `/wiki` route, it's tractable. For an app that later grows (settings, conversations sidebar, etc.) it becomes a footgun.

**Blocker 2c — "Expand to full" needs a route push, but the chat must be preserved.** The expand affordance routes the user to `/wiki/<slug>` as the primary page (not a modal). When they hit "back to chat at message N," we need to restore conversation scroll position. With parallel+intercepting routes, the chat is the underlying `children` slot and is preserved across the modal opening — but once we route the user *away* to a non-intercepted full-page `/wiki/<slug>`, chat state has to be re-hydrated by `useChat` on return, and scroll position is lost without manual restoration. The parallel-routes pattern gives us nothing extra here.

**Recommendation: use `nuqs` for URL-encoded modal state, plus a regular `/wiki/[...slug]` route for the full-page case.**

Concretely:
- The modal is a regular React component conditionally rendered when URL search params include `?resourceModal=<messageId>&focus=<slug>` (or omits them entirely).
- The chat page (`app/page.tsx`) renders both the conversation AND the modal component, gated on the URL state. No `@modal` slot, no intercepting routes.
- Clicking a resource indicator → `setQueryState({ resourceModal: msgId, focus: slug })`. URL updates, modal opens, chat stays mounted underneath.
- Clicking another right-panel resource → `setQueryState({ focus: newSlug })` (keeps `resourceModal` the same). URL updates, left panel re-renders, modal stays open, no route navigation.
- Clicking a wikilink inside the modal → same thing: `setQueryState({ focus: newSlug })`. Wikilinks inside the modal are intercepted by the modal's own click handler, NOT rendered as `<Link>`s.
- "Back to chat" → `setQueryState({ resourceModal: null, focus: null })`. Modal unmounts. Chat is untouched (was always mounted underneath).
- "Expand" → `router.push('/wiki/' + focusedSlug)` AND `setQueryState({ resourceModal: null, focus: null })`. The wiki page is the primary page from this URL. We add `?fromChat=<msgId>` so the wiki page can show a "back to chat" affordance that does `router.back()` or `router.push('/?scrollToMessage=' + msgId)`.

This design has four properties the parallel-routes pattern can't easily match:
1. Modal contents change without route nav — no stacking, no soft-navigation surprises.
2. URL is shareable in every state: opening message X's modal focused on slug Y, then sending that URL, reproduces the exact view.
3. "Expand" is a clean route push, not a slot-shuffle. The "back to chat" path is one extra search param the wiki page reads.
4. Adding new routes later does not affect the modal — nuqs is layout-agnostic.

The cost: we lose the small ergonomic win of the browser back button closing the modal via `router.back()` in the nextgram pattern. nuqs gives us the same thing via the `history: "push"` option on `useQueryState`, which writes each modal-state change to the history stack. Back button → previous URL state → previous modal state (closed). Equivalent UX, simpler implementation.

### 3. The nextgram reference implementation, and what we'd keep from it

For completeness, the official Next.js modal-as-route pattern (per `/tmp/nextgram-probe`):

```
app/
  layout.tsx                          # accepts {children, modal} props, renders both
  default.tsx                         # returns null (children fallback)
  page.tsx                            # feed page
  photos/[id]/page.tsx                # full photo page (direct URL or hard nav)
  @modal/
    default.tsx                       # returns null (modal-slot fallback)
    (.)photos/[id]/
      page.tsx                        # server component returning <Modal>{...}</Modal>
      modal.tsx                       # 'use client' component: <dialog> + showModal() + router.back()
```

We could lift the `<Modal>` shell pattern (a `'use client'` component wrapping a portaled `<dialog>` with `showModal()`, escape-key dismissal, and a close button) regardless of whether we use parallel routes or not. The `<dialog>` element gives us focus-trap, escape handling, backdrop, and correct ARIA for free in modern browsers; no Radix dependency needed if we don't want one. (Radix Dialog is a reasonable upgrade if we later want richer animation hooks or more cross-browser polish.)

**Recommendation: take the `<Modal>` shell idea from nextgram, drop the parallel-routes scaffolding, use nuqs for the URL state.**

### 4. State management: minimum viable shape

The modal needs three pieces of state per opening:
- `messageId` — which assistant message's resources are being viewed (for "back to chat at message N" scroll restoration on close)
- `focusedSlug` — which resource is rendered in the left panel
- `resources[]` — the full list of resources for that message (right panel)

`messageId` and `focusedSlug` are URL state (nuqs). `resources[]` is derived from the message itself — it's already in the AI SDK `UIMessage.parts` array for that message as a `data-resources` part (per R-016's transport decision; see §6 below). So the resource list is looked up from the chat's `messages` array given `messageId`, not stored separately.

The Zustand layer therefore holds essentially nothing: the modal is a controlled component reading nuqs URL state and the chat's `messages` array. A `useResourceModal()` hook can wrap the nuqs reads + the message lookup:

```typescript
// app/lib/state/use-resource-modal.ts
'use client';
import { useQueryState } from 'nuqs';
import { useChatStore } from '@/app/lib/state/chat-store'; // exposes messages

export function useResourceModal() {
  const [messageId, setMessageId] = useQueryState('resourceModal', { history: 'push' });
  const [focusedSlug, setFocusedSlug] = useQueryState('focus', { history: 'push' });
  const messages = useChatStore((s) => s.messages);
  const message = messages.find((m) => m.id === messageId);
  const resourcesPart = message?.parts.find((p) => p.type === 'data-resources');
  const resources = resourcesPart?.data?.resources ?? [];

  return {
    isOpen: !!messageId,
    messageId,
    focusedSlug,
    resources,
    open: (msgId: string, initialSlug: string) => {
      setMessageId(msgId);
      setFocusedSlug(initialSlug);
    },
    focus: (slug: string) => setFocusedSlug(slug),
    close: () => {
      setMessageId(null);
      setFocusedSlug(null);
    },
  };
}
```

This collapses E-044's planned `useResourceModalState` Zustand store into a thin hook. No new state primitive is introduced; nuqs holds URL state, the chat store already holds the message list, and the hook joins them.

(If E-041's chat persistence ends up exposing messages via a Repository call rather than a Zustand store directly, swap `useChatStore` for the appropriate hook — the shape is the same.)

### 5. Wikilink-in-modal: intercept clicks, don't use `<Link>`

The wiki renderer from E-042 emits wikilinks as Next.js `<Link href="/wiki/...">` components — the right behavior on the wiki view. Inside the modal, those links must NOT navigate; they must update `focusedSlug`.

Two clean implementations:

**(a) Renderer accepts a `linkBehavior` prop** — `'route'` (default, for E-042 wiki view) or `'in-modal'` (for E-044 left panel). The wikilink component checks the prop and renders either a `<Link>` or a `<button>` that calls `focus(slug)`.

**(b) Modal-level click delegation** — wrap the rendered wiki entry in a container that captures clicks: if the click target is an `<a href="/wiki/...">`, `preventDefault()` and call `focus(slug)` extracted from the href.

(a) is more explicit, (b) is more separation-friendly. I recommend (a) — the link behavior is part of the renderer's API and shouldn't be a hidden capture. E-042 should expose the prop; E-044 passes `linkBehavior="in-modal"`.

External links (anything not starting with `/wiki/`) always behave normally — open in new tab, etc.

### 6. Per-message resource transport: use AI SDK v5 custom typed data parts (`data-resources`)

**Note: This question is owned by R-016 and was resolved there on 2026-05-24, before R-017 completed.** R-017 originally drafted a `messageMetadata`-based answer, but R-016's deeper investigation found `messageMetadata` is the wrong tool when the payload is structured content (versus per-message-level annotations like latency or token counts). The chosen mechanism is AI SDK v5+'s custom typed data parts via `createUIMessageStream`.

Server (from R-016's Findings §3):

```typescript
// app/api/chat/route.ts
export type CoachUIMessage = UIMessage<never, { resources: ResourceAttribution }>;

const stream = createUIMessageStream<CoachUIMessage>({
  execute: async ({ writer }) => {
    const resources = await runRetrieval(/* ... */);
    writer.write({
      type: 'data-resources',
      data: { resources },   // [{slug, title, category, snippet}, ...]
    });
    const result = streamText({ /* ... */ });
    writer.merge(result.toUIMessageStream());
  },
});
return createUIMessageStreamResponse({ stream });
```

Client:

```typescript
const { messages } = useChat<CoachUIMessage>({ api: '/api/chat' });
// For an assistant message:
const resourcesPart = message.parts.find(p => p.type === 'data-resources');
const resources = resourcesPart?.data?.resources ?? [];
```

This shape is what E-044 consumes: `message.parts.find(p => p.type === 'data-resources')` returns the array used to populate the right panel of the modal, and `focusedSlug` indexes into it (or any other resource the user navigates to).

**Update to the §4 hook signature:** the `useResourceModal()` hook reads the resources via the parts array, not via `message.metadata.resources`:

```typescript
const message = messages.find((m) => m.id === messageId);
const resourcesPart = message?.parts.find((p) => p.type === 'data-resources');
const resources = resourcesPart?.data?.resources ?? [];
```

The rest of the hook (URL state via nuqs, focus state, open/close/focus actions) is unchanged from §4.

**Persistence implication for E-041:** `message.parts` is a top-level field of `UIMessage`. The byte-equality round-trip criterion E-041 already mandates handles `data-resources` parts automatically — no schema change, no special persistence logic. This matches R-016's finding §3.

**Order of emission:** R-016 specified writing `data-resources` *before* `streamText` begins. This means the resource list is available on the client during streaming, so the bottom-of-message strip in §8 can render the indicator as the message streams in rather than only after `finish`. UX win: no flash of un-indicator'd message.

### 7. "Expand to full" — concrete mechanics

Trigger: a header button in the modal's left panel.

Action:
```typescript
function expand() {
  router.push(`/wiki/${focusedSlug}?fromChat=${messageId}`);
  close();  // clear modal URL state
}
```

The destination `/wiki/[...slug]/page.tsx` (built by E-042) reads `?fromChat=` from search params. If present, it renders a small "Back to chat at message N" link in the top bar that either:
- Calls `router.back()` (if the user got here via expand — back goes to chat with the modal still in the URL? No, we cleared it. Back goes to chat without the modal.), OR
- Calls `router.push('/?scrollToMessage=' + msgId)` for an explicit return path.

I recommend `router.back()` as the primary action with the push as fallback (when `history.length` is 1, e.g., the user opened the wiki URL directly). The chat page on mount checks `?scrollToMessage=` and scrolls the conversation pane to that message id, then clears the param (`replace`).

### 8. Visual layout: 60/40 split with right-panel scannability

The user spec says split layout, left = focused wiki entry, right = resource list. Concrete sizing recommendations:

- **Desktop (≥1024px):** left 65% / right 35%. The wiki entry is the dominant visual element; the right panel is a vertical list of resource cards, each ~80–120px tall (title, category badge, 1–2 line snippet, subtle indicator if it's the currently focused resource).
- **Modal sizing:** ~90vw / ~85vh, max 1400px wide. Padding around the dialog so it visibly overlays the conversation.
- **Currently focused resource** in the right panel is highlighted (left border + background tint). Hover state on non-focused cards (background tint only). Click → switches focus.
- **Right panel header:** "Sources used (N)" with the count. No filter UI in v1; lists are small (3–8 items per E-038's retrieval).

For the subtle in-flow indicator on the message itself, the ShapeOfAI "nested layout" pattern guides us: small bottom-of-message strip showing the count and a chevron icon (e.g., `5 sources >`). Click → opens modal. Hover state on the strip; not hover-rich (no preview popovers in v1 — those are nice-to-haves that the modal already does better). Visual treatment is intentionally restrained — gray text, small font, single line — to keep the strip from interrupting conversational flow.

### 9. Accessibility checklist (carries into E-044's success criteria)

- Native `<dialog>` via `showModal()` OR Radix Dialog — either gives focus-trap, escape, ARIA. Going with native `<dialog>` for v1 (zero deps, matches nextgram).
- Tab navigates within the modal only; Shift+Tab reverses; Escape closes (calls `close()` from §4).
- Focus returns to the resource indicator that opened the modal on close.
- The modal has `aria-labelledby` pointing to the focused resource's title (announces what's being viewed when the modal opens).
- Right-panel list items are `<button>`s with `aria-current="true"` on the focused one.
- Wikilink buttons in the left panel have visible focus rings.
- Screen-reader announcement: when the focused resource changes (right-panel click or wikilink), an `aria-live="polite"` region announces "Now viewing: <title>".

## Implications

**For E-044 (build the modal):**
1. Drop the parallel/intercepting route scaffolding. Use a regular React component conditionally rendered from URL state. Update the blast radius: no `app/@modal/(.)wiki/...` files; instead `app/components/resource-modal.tsx` rendered from `app/page.tsx` (or wherever the chat lives).
2. Adopt nuqs as a new dependency. Set `history: "push"` for both `resourceModal` and `focus` query state hooks so the browser back button closes the modal naturally.
3. Use the native `<dialog>` + `showModal()` pattern lifted from nextgram for the modal shell (focus-trap, escape, backdrop for free). No Radix dependency needed for v1.
4. Collapse the planned Zustand store into the `useResourceModal()` hook described in §4. No new state primitive.
5. Layout spec: 65/35 split, ~90vw × 85vh, focused-resource highlighting in the right panel, header showing count.
6. Inline indicator: bottom-of-message strip, "N sources" + chevron icon, subtle gray small-font single line.
7. Wikilink behavior: pass `linkBehavior="in-modal"` to the E-042 renderer; renderer must support this prop.
8. Expand: `router.push('/wiki/<slug>?fromChat=<msgId>')` + close modal. E-042 wiki page reads `?fromChat=` and renders a "Back to chat at message N" link.

**For E-042 (wiki view):** Add `linkBehavior?: 'route' | 'in-modal'` prop to the wiki renderer (default `'route'`). When `'in-modal'`, render wikilinks as buttons that call an `onLinkClick(slug)` callback rather than `<Link>` components. E-044 supplies the callback.

**For E-041 (persistence):** `message.parts` is a top-level UIMessage field; the byte-equality round-trip criterion already covers `data-resources` parts persistence — no additional work. Use R-016's `CoachUIMessage` type as the `UIMessage<>` parameter when wiring `useChat`.

**For R-016 (the parallel research item):** R-016 completed first and resolved the transport question with `data-resources` custom typed parts via `createUIMessageStream`. R-017 originally proposed `messageMetadata` but defers to R-016's deeper investigation. §6 of this document has been rewritten to reflect R-016's decision. No further coordination needed.

**For E-043 (coach wire-in):** Already covered by R-016. The right-panel resource cards in E-044 consume the exact shape R-016's `ResourceAttribution` defines.

**For G-010's exit criteria:** The "Each AI message displays a subtle resource attribution; clicking it opens the split modal" criterion is fully scoped by this research. No change needed.
