# Scaffold Next.js + AI SDK + OpenRouter minimal chat

## Hypothesis

A minimal Next.js (App Router) chat surface using Vercel AI SDK + OpenRouter can be stood up in a self-contained `web-app/` directory and demonstrate end-to-end streaming chat with a generic system prompt (no retrieval, no compendium tools, no persistence yet). This validates the stack works as the playbook specifies and provides the scaffolding all subsequent experiments build on.

## Method

1. Read [[ai-web-app]] playbook. Follow stack defaults.
2. Create `web-app/`:
   - `package.json` with Next.js, React, TypeScript, `ai`, `@ai-sdk/react`, `@openrouter/ai-sdk-provider`, Zustand
   - `tsconfig.json` with `strict: true`
   - `next.config.js` minimal + COOP/COEP headers from day one (R-015 prereq for E-041)
   - `app/layout.tsx`, `app/page.tsx` (client component using `useChat`)
   - `app/api/chat/route.ts` (server route handler with `streamText` + OpenRouter)
   - `app/lib/openrouter.ts` (configured OpenRouter client per R-016 — uses `@openrouter/ai-sdk-provider`, NOT `@ai-sdk/openai` with custom baseURL, so E-043 can use cache_control without swapping providers)
   - `app/lib/system-prompt.ts` (placeholder system prompt: "You are a helpful coach. Help the user with whatever they bring." — v5b prompt slots in via E-043)
3. Wire `useChat` → POST `/api/chat` with `messages` payload
4. Run `bun run dev`, open browser, send a test message, verify streaming response renders
5. Test multi-turn: 3-5 exchanges, verify conversation context is preserved across turns
6. Verify no secrets leak: scan the rendered HTML and every client JS chunk for `OPENROUTER_API_KEY` and the system prompt string; both must be absent

## Success Criteria

- `bun run dev` starts the Next.js dev server with no errors
- Chat UI renders, accepts input, streams response
- Multi-turn works (model has context of prior messages)
- `OPENROUTER_API_KEY` is not present in any client-served file or network response
- System prompt text is not present in any client-served file or network response
- `useChat` correctly serializes a full message array on each turn
- The chat works on whichever OpenRouter model the playbook recommends (`anthropic/claude-sonnet-4.6` default; note the dot, not dash, in OpenRouter's ID convention)

## Blast Radius

Creates `web-app/` directory tree (new, no conflict). Reads `.env` for `OPENROUTER_API_KEY` (mirrored to `web-app/.env.local`, which is gitignored). Does not modify `coach/`, `coach-app/`, `eval/`, or any prior work.

## Result

Scaffold succeeded end-to-end. The Next.js dev server starts in <500ms, the page renders, and a POST to `/api/chat` returns a streamed `text/event-stream` response from `anthropic/claude-sonnet-4.6` via OpenRouter.

## Analysis

Three deviations from the original method writeup, all upgrades:

1. **Used `@openrouter/ai-sdk-provider` from the start, not `@ai-sdk/openai`** with a custom baseURL. R-016 already decided this is the right gateway for the project; using it at E-040 means E-043 doesn't have to swap providers. The playbook (`ai-web-app.md`) still recommends the OpenAI compat path; that text is the source of the "ai-web-app.md should swap createOpenAI for createOpenRouter" patch R-016 flagged as a separate follow-up. Logging that as a follow-up item rather than addressing it inline (the playbook is shared with future projects; the swap warrants its own assessment).

2. **Set COOP/COEP headers in `next.config.js` from day one.** R-015 lists this as a hard requirement for `sqlocal` + OPFS in E-041, and there's no cost to having them at E-040 stage (no embedded third-party widgets). Defending E-041's environment up front is cheaper than back-fixing once a regression lands.

3. **Used `bun` for install + dev/build commands** even though `bun` was not pre-installed in this environment. Installed at `~/.bun/bin/bun` (1.3.14). The lockfile is `bun.lock`. Next.js 16's Turbopack runs cleanly under Bun. No issues observed.

The model ID `anthropic/claude-sonnet-4.6` (dot) is the OpenRouter convention; the playbook's example used `claude-sonnet-4-6` (dash) which would 404. Followed up by checking the OpenRouter model catalogue directly. Worth a note in the playbook.

AI SDK v6 made `convertToModelMessages` async (it must fetch remote file/image attachments). Typecheck caught the missing `await` immediately — without it, the route handler would have type-erred and the chat would not stream. Cheap mistake, but the kind of thing that the strict-mode typecheck pays for.

The page UI is deliberately barebones (~140 lines including inline styles). E-041 will replace the in-memory state with `useChat`'s `initialMessages` hydrated from sqlocal, E-044 will add the resource attribution strip, E-045 will add the welcome flow. Not styling beyond a clean baseline because the visual design is downstream of those experiments.

**Scope discipline:** No persistence layer, no repository setup, no resource attribution, no welcome modal, no multi-conversation, no abort/regenerate UI. All deferred to their respective experiments. The audit's minor #13 ("E-040 should not set up Repository conventions until E-041 needs them") is honored.
