# Build web-app coach-version selector UI and wire selected profile through chat route; deploy

## Hypothesis

With the profile registry from [[E-053]] in place, a small client control plus a request-body field is enough to let the user choose which coach variant answers each conversation. Switching the selector will produce observably different coach behavior on the same input. Deploying it gives the user a single app to vibe-check all four variants back to back.

## Method

1. **Read [[E-053]]'s output** (`web-app/app/lib/coach/profiles.ts`) to learn the profile ids and shape. This experiment depends on E-053 being complete.

2. **Wire the route handler.** In `web-app/app/api/chat/route.ts`, read a `coachProfile` (or `variant`) field from the request body. Look it up in the profile registry; fall back to `v5b` if missing/unknown. Replace the hardcoded `SYSTEM_PROMPT` (line ~183/331) and `DEFAULT_COACH_MODEL` (line ~180/328) and the hardcoded `walker_model: 'claude-sonnet-4-6'` (line ~259) with the selected profile's fields. Keep the cache_control / prompt-caching behavior. Keep cost-cap and safety wiring unchanged.

3. **Client control.** In `web-app/app/page.tsx` (and wherever the AI SDK `useChat`/transport request body is assembled), add a selector (dropdown or segmented control) listing the four profiles by `label`, defaulting to `v5b`. Pass the selected id in the chat request body so the route handler receives it. The selection applies to the current conversation; switching mid-conversation is allowed (the next turn uses the new profile).

4. **Persistence tagging (nice-to-have, do if cheap).** Record the active profile id on each assistant message in the client SQLite layer, so a saved conversation shows which variant produced each reply. If the schema change is non-trivial, note it and defer to a follow-up rather than risk the persistence layer.

5. **Local verification.** Run the dev server, switch the selector across all four variants, send the same opening message to each, confirm the replies differ in stance (baseline asks a bare question; reflective/grounded/joe-voiced share substance). Confirm no console errors, streaming still works, the attribution modal still works.

6. **Deploy.** Follow the [[cloudflare-deploy]] playbook / [[E-046]] path: `bunx wrangler deploy` (or the established build+deploy command). Confirm the live URL serves the selector. Record the deployed URL and bundle size in the result.

## Success Criteria

- Selecting a variant in the deployed UI routes the chat request to that profile's system prompt + params server-side.
- All four variants are reachable from the selector and produce stance-distinct replies on the same input.
- Streaming, the attribution modal, persistence, and cost-cap all still work (no regression to E-043/E-044/E-052 behavior).
- The change is live at the Cloudflare URL.
- A [[Q-002]] QA pass confirms the above end to end.

## Analysis

The wiring was the pure-plumbing job E-053 set up: three constants → one `resolveCoachProfile(body.coachProfile)` lookup, threaded into both the happy-path stream and the safety-regen helper. Holding retrieval at `V5B_CONFIG` for all four meant `runRetrieval` needed no change at all — the system prompt is the only thing that varies, which is exactly the G-014 one-variable discipline.

**Stance contrast is input-dependent.** On a thin one-line opener ("I don't know how to let people in") the four replies were close — the higher-authority variants didn't have enough material to reflect a read or land a frame, so they fell back toward curious questions. On the substantive concern they separated cleanly along the predicted monotonic spectrum (questions-only → +read → +frame → +challenge). This is a useful finding for the REQ-001 session-2 vibe-check and for Q-002: judge stance distinctness on a concern with enough texture to reflect, not a bare opener. Recorded as a note on Q-002's test plan.

**Persistence-as-data-part was the right call.** It avoided touching the shipped E-041 persistence layer entirely (the blast-radius warning), gave reload-survival for free via the existing byte-equality round-trip, and required no migration. The `VariantTag` type is kept loose (`id: string`) so `types.ts` doesn't have to import the server-only registry.

**Selector state scoping.** State lives in `HydratedChat`, which React-remounts via `key={activeId}` on conversation switch, so each conversation starts at the baseline. Switching mid-conversation is allowed and takes effect next turn (read from a ref at send time). This matches the method's "selection applies to the current conversation; switching mid-conversation is allowed."

## Dead Ends

- **CI deploy via the GitHub Actions workflow failed twice; deployed manually instead.** The push-to-main pipeline (`.github/workflows/deploy.yml`) failed on run #1 (Typecheck) and run #2 (wrangler-action). Two distinct causes:
  1. **Typecheck (run #1).** `tsconfig.json` `include: ["**/*.ts"]` swept `scripts/*.ts` (Bun build/ingest scripts using `import.meta.dir`) into `tsc --noEmit`, but `bun-types`/`@types/bun` isn't installed, so `tsc` errored `TS2339: Property 'dir' does not exist on type 'ImportMeta'`. This was a **latent pre-existing bug** — the workflow had never actually run before (E-046 deferred GitHub Actions to v2 and deployed from the developer machine; my push was run #1 of this workflow ever). Masked locally by the `incremental` tsbuildinfo cache. Reproduced via a fresh clone + frozen install + ingest + typecheck. Fixed by excluding `scripts/` from the app typecheck (those files are run/typechecked by Bun against its own types; the app typecheck should gate the Next.js app, not the build scripts).
  2. **wrangler-action (run #2).** The GitHub repo secrets `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID` were never provisioned — E-046 explicitly deferred CI deploys to v2 and runs live deploys from the developer machine via `bunx wrangler deploy` (token in the gitignored `/home/claude/joe-hudson/.env`). So the wrangler-action had no credentials. This is **not** a code problem and not in scope to fix here (it needs the user to set GitHub repo secrets — a v2 follow-up E-046 already documented). I deployed via the established `bun run deploy:cf` local path with the `.env` token, which succeeded.
  - Diagnosis without `gh` auth: used the public GitHub Actions REST API (repo is public) to read run status + per-step conclusions; logs need auth (403) so I reproduced the typecheck failure locally instead.
- **Considered: tagging the variant via a new SQLite column.** Rejected — the blast-radius note flagged the shipped E-041 persistence layer as risky to touch, and a data part gives the same reload-survival with no migration. Documented in `types.ts`.
