# Build web-app coach-version selector UI and wire selected profile through chat route; deploy

## Hypothesis

With the profile registry from [[E-053]] in place, a small client control plus a request-body field is enough to let the user choose which coach variant answers each conversation. Switching the selector will produce observably different coach behavior on the same input. Deploying it gives the user a single app to vibe-check all four variants back to back.

## Method

1. **Read [[E-053]]'s output** (`web-app/app/lib/coach/profiles.ts`) to learn the profile ids and shape. This experiment depends on E-053 being complete.

2. **Wire the route handler.** In `web-app/app/api/chat/route.ts`, read a `coachProfile` (or `variant`) field from the request body. Look it up in the profile registry; fall back to `v5b` if missing/unknown. Replace the hardcoded `SYSTEM_PROMPT` (line ~183/331) and `DEFAULT_COACH_MODEL` (line ~180/328) and the hardcoded `walker_model: 'claude-sonnet-4-6'` (line ~259) with the selected profile's fields. Keep the cache_control / prompt-caching behavior. Keep cost-cap and safety wiring unchanged.

3. **Client control.** In `web-app/app/page.tsx` (and wherever the AI SDK `useChat`/transport request body is assembled), add a selector (dropdown or segmented control) listing the four profiles by `label`, defaulting to `v5b`. Pass the selected id in the chat request body so the route handler receives it. The selection applies to the current conversation; switching mid-conversation is allowed (the next turn uses the new profile).

4. **Persistence tagging (nice-to-have, do if cheap).** Record the active profile id on each assistant message in the client SQLite layer, so a saved conversation shows which variant produced each reply. If the schema change is non-trivial, note it and defer to a follow-up rather than risk the persistence layer.

5. **Local verification.** Run the dev server, switch the selector across all four variants, send the same opening message to each, confirm the replies differ in stance (baseline asks a bare question; reflective/grounded/joe-voiced share substance). Confirm no console errors, streaming still works, the attribution modal still works.

6. **Deploy.** Follow the [[cloudflare-deploy]] playbook / [[E-046]] path: `bunx wrangler deploy` (or the established build+deploy command). Confirm the live URL serves the selector. Record the deployed URL and bundle size in the result.

## Success Criteria

- Selecting a variant in the deployed UI routes the chat request to that profile's system prompt + params server-side.
- All four variants are reachable from the selector and produce stance-distinct replies on the same input.
- Streaming, the attribution modal, persistence, and cost-cap all still work (no regression to E-043/E-044/E-052 behavior).
- The change is live at the Cloudflare URL.
- A [[Q-002]] QA pass confirms the above end to end.

## Blast Radius

Modifies `web-app/app/api/chat/route.ts` (coach setup), `web-app/app/page.tsx` (selector UI + request body), and possibly the client SQLite schema/persistence module (only for the nice-to-have message tagging — guard this carefully; the persistence layer is shipped and working from E-041). Redeploys the live Cloudflare Worker, replacing the current deployment. Coordinate timing: this is the experiment that changes deployed behavior, so it should land after [[E-053]] and after the [[Q-001]] QA of the current deploy is captured (a redeploy invalidates an in-flight Q-001 run). Does not touch `coach-app/`, the eval harness, or the wiki beyond this page, Q-002, and the G-014 decision log.
