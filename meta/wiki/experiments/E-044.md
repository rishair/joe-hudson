# Build resource-attribution modal split layout

## Hypothesis

Implementing the resource-attribution UX per R-017's chosen pattern produces the interaction the user specified: subtle in-flow indicator on each assistant message, click → split modal (left = wiki entry, right = resource list), inter-resource navigation in the modal, back-to-chat, expand-to-full-page. R-017 explicitly resolved this with: native `<dialog>` modal (NOT Next.js parallel/intercepting routes), `nuqs` URL-state, and resources read directly from R-016's `data-resources` part on the assistant message. The E-042 wiki renderer powers the left-panel via a new `linkBehavior="in-modal"` prop (E-042 adds the prop); the `data-resources` part on each `UIMessage.parts` array populates the right-panel.

## Method

1. **Add dependency:** `bun add nuqs` and wire `<NuqsAdapter>` at the root layout (see nuqs docs — for App Router this goes in `app/layout.tsx`).
2. **Per-message indicator** (`app/components/message.tsx` or wherever assistant messages render):
   - For each assistant message, find `message.parts.find(p => p.type === 'data-resources')` (typed via R-016's `CoachUIMessage`).
   - If present and resources non-empty, render a subtle bottom-of-message strip: small gray text "N sources" with a chevron icon. Click → calls `useResourceModal().open(message.id, resources[0].slug)`.
   - If no resources part (e.g., a pre-retrieval intro message), render nothing — do NOT render an empty strip.
3. **State hook** at `app/lib/state/use-resource-modal.ts`:
   ```typescript
   'use client';
   import { useQueryState } from 'nuqs';
   import { useChat } from '@ai-sdk/react';  // or read from wherever messages live
   import type { CoachUIMessage } from '@/app/lib/coach/types';

   export function useResourceModal() {
     const [messageId, setMessageId] = useQueryState('resourceModal', { history: 'push' });
     const [focusedSlug, setFocusedSlug] = useQueryState('focus', { history: 'push' });
     // Read messages from the chat layer (concrete source TBD per E-041's hook surface)
     const messages = useChatMessages() as CoachUIMessage[];
     const message = messages.find((m) => m.id === messageId);
     const resourcesPart = message?.parts.find((p) => p.type === 'data-resources');
     const resources = resourcesPart?.data?.resources ?? [];

     return {
       isOpen: !!messageId,
       messageId, focusedSlug, resources,
       open: (msgId: string, slug: string) => { setMessageId(msgId); setFocusedSlug(slug); },
       focus: (slug: string) => setFocusedSlug(slug),
       close: () => { setMessageId(null); setFocusedSlug(null); },
     };
   }
   ```
4. **Modal component** at `app/components/resource-modal.tsx`:
   - `'use client'`. Reads `useResourceModal()`. Returns `null` when not open.
   - Native `<dialog ref>` with `useEffect(() => { ref.current?.showModal(); }, [])`. Renders via `createPortal` into a `#modal-root` div (added once to root layout).
   - Layout: 65% left (focused wiki entry), 35% right (resource list).
   - Left panel: renders the focused-slug page via E-042's `<WikiRenderer slug={focusedSlug} linkBehavior="in-modal" onLinkClick={(slug) => focus(slug)} />`. Header shows page title + "Expand" button + close button (X).
   - Right panel: list of resource cards (title, category badge, 1–2 line snippet) — `<button>` per item, `aria-current={slug === focusedSlug}`, click → `focus(slug)`. Focused card has a left border + tinted background.
   - `aria-labelledby` on `<dialog>` points to the left-panel title.
   - `aria-live="polite"` region: "Now viewing: <title>" updates on focus changes.
   - Modal sizing: ~90vw × 85vh, max-width 1400px.
5. **Expand action** (left-panel header "Expand" button):
   ```typescript
   function expand() {
     const slug = focusedSlug;
     const msgId = messageId;
     close();  // clears nuqs state first
     router.push(`/wiki/${slug}?fromChat=${msgId}`);
   }
   ```
6. **Render the modal** at the root of the chat surface (e.g., `app/page.tsx`):
   ```tsx
   export default function Page() {
     return (
       <>
         <ChatSurface />
         <ResourceModal />  {/* renders null when no URL state */}
       </>
     );
   }
   ```
   Add `<div id="modal-root" />` to `app/layout.tsx` for the portal target.
7. **Test plan** (vibe + automated):
   - Open conversation with retrieval. Confirm bottom strip shows "N sources" on each assistant message after the retrieval phase.
   - Click strip → modal opens with first resource focused, URL updates to `?resourceModal=<id>&focus=<slug>`.
   - Click a right-panel resource → left panel changes, URL `focus` updates, modal stays open, no flash.
   - Click a wikilink inside the left panel (an internal `[[...]]` link from the focused entry) → left panel changes to the linked entry, modal stays open.
   - Click Expand → URL becomes `/wiki/<slug>?fromChat=<msgId>`, full-page wiki view renders, "Back to chat at message N" link appears.
   - Click Back-to-chat → returns to chat, scroll position restored at message N (chat surface reads `?scrollToMessage=` on mount).
   - Browser back from open modal → modal closes (nuqs `history: 'push'` recorded each state change).
   - Escape key in modal → close.
   - Tab navigation stays inside modal (focus-trap via native `<dialog>`).
   - Share URL with `?resourceModal=...&focus=...` to another tab/browser → modal opens to the correct state on load.

## Success Criteria

- Resource strip renders subtly per assistant message that has a `data-resources` part; passes a "does not disrupt flow" eyeball test.
- Modal opens on click with the correct focused resource and the full resource list visible in the right panel.
- Switching between right-panel resources updates the left panel without route change or visible flash; URL `?focus=<slug>` updates.
- Wikilinks inside the modal's left panel navigate the left panel only (right panel and modal state unchanged).
- "Expand" pushes to `/wiki/<slug>?fromChat=<msgId>` and that view renders a working "Back to chat at message N" link.
- Closing the modal (close button, escape, browser back, or backdrop click) returns to the conversation with scroll position at the originating message.
- Modal is keyboard-accessible: Escape closes, Tab/Shift+Tab cycle within the dialog, focus returns to the resource strip that opened the modal.
- Direct URL load (`/?resourceModal=<msgId>&focus=<slug>`) opens the modal to the correct state on first paint.

## Analysis

### What landed cleanly

- Native `<dialog>` + `createPortal` pattern from R-017 §3 worked as advertised. Focus trap, escape-to-close, ARIA modal semantics, inert background — all free from the browser primitive. The only addition needed for the listed accessibility checklist was the `aria-live="polite"` region for focus-change announcement and explicit focus restoration on close (the dialog doesn't do that itself).
- nuqs URL state with `history: 'push'` is the right primitive. Each focus change writes a new URL state, so browser back closes the modal (or steps back through focus changes). Sharing the URL `?resourceModal=msgX&focus=slug` reproduces the modal view on load with no extra work — the hook's read side is pure.
- Resources read off `message.parts.find(p => p.type === 'data-resources').data` worked first try because E-043 already emits the part in the exact shape `extractResources` expects, and E-041's byte-equality round-trip persists it. The contract chain (R-016 → E-043 → E-041 → E-044) held.
- The Expand → wiki page → Back-to-chat round-trip reused the existing `?fromChat=<msgId>` mechanism E-042 built. No changes needed to BackToChat or ChatScrollRestorer.
- WikiRenderer's `linkBehavior="in-modal"` prop (E-042) just worked. In-modal wikilink clicks call `focus(slug)`, which writes nuqs state, which re-runs the fetch effect on the modal, which loads the new page. The user never leaves the modal.

### What I had to adjust from the spec

- **The `useResourceModal` hook's `messages` source**: the spec showed a `useChatStore`/`useChatMessages` hook. The actual chat architecture (E-041) doesn't expose messages through a separate hook — `useChat()` is the source of truth and lives in `HydratedChat`. I pass the messages array as a hook argument instead of trying to read it from a global, keeping the modal a pure function of (URL state, messages). The cost is the modal must be rendered inside the same component that calls `useChat`. The benefit is no shared mutable state and no race between useChat updates and modal reads.
- **Page-body fetching**: R-017 §4 assumed the modal could render a page directly from a wiki source. In practice the wiki content is filesystem-backed (`FsWikiRepository` is `'server-only'`), so I added the `/api/wiki/page` endpoint as a thin client→server bridge. The endpoint returns the same wikilink-rewritten body the wiki route renders, so the in-modal experience matches the full-page experience pixel-for-pixel.
- **Resource card snippet**: R-017 §8 mentions "1–2 line snippet" on each card. I used `walker_reason` from the attribution payload (the per-file rationale the walker emitted) rather than a body excerpt — the walker reason is more compact and directly answers "why is this resource here?" which is the right question for a list-of-sources view. Body snippets would have required either a snippet field in `ResourceAttribution` (would touch E-043) or fetching every resource's body on modal open (latency).
- **Backdrop dismissal**: native `<dialog>` doesn't dismiss on backdrop click by default; the dialog element is clipped to its content box but clicks outside the content land on the dialog element itself (since the backdrop is `::backdrop`, which doesn't receive clicks). I detect `event.target === dialogRef.current` and call `close()`. The inner layout div stops propagation so clicks on content don't bubble up and dismiss.
- **Type narrowing on data parts**: TypeScript can't narrow `UIMessage["parts"][number]` to the `data-<name>` shape because the discriminator union doesn't include the `data` field on text/reasoning/tool parts. Cast through `unknown` with runtime check on `part.data` presence is the safest workaround until AI SDK types catch up.

### Choices vs spec

- **Inline styles, not Tailwind** — consistent with E-040/E-041/E-042/E-045. Adding Tailwind for one component would inflate the dependency surface; a future visual-polish work item can sweep all of `web-app/app/components` at once.
- **Cache page bodies per modal mount** — `pageCache.current = new Map()` cleared on `messageId` change. Avoids re-fetching pages the user already viewed in the same modal session, while not bloating memory across conversation turns. Trade-off: navigating between two conversations and back hits the API again. Acceptable for v1; the API is fast (filesystem read + JSON parse, no network).
- **`aria-labelledby` on dialog points to a stable id** (`resource-modal-title`) on the focused page's h2. When the focused page changes, the h2 text changes and the SR re-reads the label. Combined with the `aria-live="polite"` region's "Now viewing: <title>" announcement, the focus change is double-announced — slightly verbose but unambiguous, which is the right trade for a modal that switches content under the same dialog frame.
- **Strip icon and chevron** — `◇` for the indicator and `›` for the chevron. Both Unicode glyphs so no SVG dependency; both small/gray to stay subtle per R-017 §8.

### Not done (intentional)

- **No browser-driven test**: there's no headless Chrome / Playwright in the repo. The verify script + SSR HTML inspection + end-to-end chat smoke + manual code reading cover the load-bearing pieces. A future work item can layer in browser tests if the cost of writing them is judged worthwhile against vibe-checks.
- **No focus-trap unit test for Tab/Shift+Tab cycling**: native `<dialog>` provides this. Verifying the browser implementation is out of scope.
- **No animation/transition on modal open/close**: spec defers visual polish.
- **No mobile-responsive grid**: R-017 §8 calls out desktop-first; the 65/35 grid is fixed. A future media-query swap to a stacked layout under 768px is straightforward.
- **No per-resource snippet of the actual body** (only walker_reason). See the spec-adjustment note above.

## Dead Ends

None notable. The build went linearly: install nuqs → add adapter + portal target → write hook → write strip → write modal → write API route → wire into page → verify. The only stumble was the TypeScript narrowing on the data-resources part, resolved with a one-line `as unknown as` cast and a runtime guard.
