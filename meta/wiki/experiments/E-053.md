# Define server-side coach-profile abstraction and author authority-spectrum prompt variants

## Hypothesis

A text coach earns trust by sharing substance (reflecting reads, naming what it notices, offering frames as statements), not only by asking curious questions. There is a calibration band between "too passive" (the current v5b prompt) and "overdoing it" (advice/interpretation the methodology forbids). Authoring four prompts along an authority spectrum, all on the same retrieval substrate, will let a vibe-check locate that band. This experiment produces the prompts and the server abstraction that makes them selectable; [[E-054]] builds the UI; the user's vibe-check ([[REQ-001]] session 2) picks the winner.

## Method

1. **Read the current baseline.** `web-app/app/lib/system-prompt.ts` (the deployed `SYSTEM_PROMPT`, which is the v5b/v1 prompt — pure curious-questioning stance) and `coach-app/prompts/v1.md` (its source). Understand its structure: SAFETY FIRST, the "Don't" list, signature questions, retrieval-aware closing.

2. **Define a coach-profile abstraction** server-side. Create `web-app/app/lib/coach/profiles.ts` (or similar) exporting a typed registry: `{ id, label, blurb, systemPrompt, walkerModel, coachModel, retrievalParams }`. All four profiles share v5b retrieval (walker `claude-sonnet-4-6`, same K_max / step_budget / every_turn as the deployed default) so the **system prompt is the only variable**. The route handler ([[E-054]]) will look up the profile by id from the request and use its `systemPrompt` + params instead of the hardcoded `SYSTEM_PROMPT` / `DEFAULT_COACH_MODEL`.

3. **Author four prompt files** under `web-app/app/lib/coach/prompts/` (or `web-app/prompts/`), one per stance. Each is the v5b prompt as the base, with a stance layer added/modified. Keep SAFETY FIRST and the anti-pattern guardrails intact in all four — the authority dial must not loosen the safety floor.

   - **`v5b` (baseline / control)** — unchanged deployed prompt. Pure curious questioning. The thing the vibe-check found hollow; kept as the comparison anchor.
   - **`reflective` (mild)** — coach names a read/reflection back as a statement before (or instead of) asking. Lowest risk. Example, on "the glimmer pops up but I feel forgotten": *"Something in you catches the glimmer and covers it just as fast — like letting it land would cost something. What's it like to hear that?"*
   - **`grounded` (medium)** — coach offers a relevant frame or distinction from the compendium as a statement, takes a gentle stance. The "earn trust by sharing substance" sweet spot. Example: *"There's a guard that won't let the wish reach the part of you it's meant for. That guard showed up for a good reason once. I'm curious what it's protecting."*
   - **`joe-voiced` (high)** — coach is direct, makes observations, gently challenges, welcomes resistance out loud, uses Joe's signature moves. Closest to in-person Joe; highest risk of tipping into interpretation/advice. Example: *"I don't think you don't know how to let people in — I think a part of you is working hard to stop it today, on your birthday of all days, which tells me it matters. Can we get curious about that part instead of trying to fix you?"*

4. **Ground the stance language in the compendium.** Pull stance/voice cues from [[E-026]] (gold-standard exchanges) and `coach/` reads/moves so the variants reflect how Joe actually shares substance, not a generic "be more assertive" instruction. The distinction to encode: Joe offers reads and frames the client can reject; he does not diagnose or prescribe. Each prompt should make that line explicit ("offer it as something to try on, not a verdict").

5. **Smoke-check locally** that each profile loads and produces a reply (one turn each via the local dev server or the coach-app harness if a profile bridge is easy). Do NOT run the full eval (per the goal decision: vibe-check first, eval gates only winner-promotion).

## Success Criteria

- `web-app/app/lib/coach/profiles.ts` exports a typed registry of exactly four profiles (`v5b`, `reflective`, `grounded`, `joe-voiced`) all sharing v5b retrieval params.
- Four prompt files exist, each a complete system prompt with SAFETY FIRST and anti-pattern guardrails intact, differing only in stance layer.
- The three new prompts demonstrably instruct sharing-substance behavior distinct from baseline (a reviewer reading them can predict the example phrasings above).
- The abstraction is shaped so [[E-054]] can select a profile by id without further refactor.
- No change to the eval harness, the seed-detection module, the retrieval algorithm, or `coach-app/` (the web app is the surface; coach-app stays the reproducible reference).

## Result

Landed the coach-profile data layer and the four authority-spectrum prompts. No route, eval, or `coach-app/` changes (E-054 owns the wiring).

**Files created:**

| File | Purpose |
|---|---|
| `web-app/app/lib/coach/prompts/base.ts` | `BASE_PROMPT` — the v5b body, factored out (byte-identical to deployed `SYSTEM_PROMPT`, verified) |
| `web-app/app/lib/coach/prompts/reflective.ts` | `REFLECTIVE_PROMPT` = base + mild stance layer |
| `web-app/app/lib/coach/prompts/grounded.ts` | `GROUNDED_PROMPT` = base + medium stance layer |
| `web-app/app/lib/coach/prompts/joe-voiced.ts` | `JOE_VOICED_PROMPT` = base + high stance layer |
| `web-app/app/lib/coach/profiles.ts` | `COACH_PROFILES` typed registry of exactly 4 profiles + `resolveCoachProfile()` / `isCoachProfileId()` / `DEFAULT_COACH_PROFILE_ID` / `COACH_PROFILE_ORDER` |

**Registry shape** (`CoachProfile`): `{ id, label, blurb, systemPrompt, coachModel, retrievalParams }`. The four ids are the typed union `'v5b' | 'reflective' | 'grounded' | 'joe-voiced'`.

**Key design decisions:**

1. **`v5b` imports `SYSTEM_PROMPT` directly** from `app/lib/system-prompt.ts` rather than re-authoring it, so the baseline/control option is byte-identical to the currently-deployed default — zero drift risk. `system-prompt.ts` was left untouched (blast radius).
2. **Shared base, not 4 copies.** The three new variants compose `BASE_PROMPT + STANCE_LAYER`. This makes the "SAFETY FIRST + anti-pattern guardrails intact in all four" requirement true *by construction* — a variant cannot drop or weaken a safety clause because it never owns its own copy of the safety section. A reviewer diffs only the short stance layer, not 130 identical lines.
3. **Retrieval held constant.** All four profiles' `retrievalParams` reference the SAME `V5B_CONFIG` object (imported from `retriever.ts`), so the system prompt is provably the only variable. The field is per-profile (not global) so a later experiment can vary retrieval per variant without reshaping the registry.
4. **Graceful resolution for E-054.** `resolveCoachProfile(id)` never throws: an unknown / undefined / non-string id falls back to `v5b`, so a bad client payload degrades to baseline rather than 500-ing the chat route.

**Stance layers (the only variable), grounded in [[E-026]] gold exchanges:**

- **reflective (mild)** — "lead with a reflection, not only a question." Names a read back as a statement, then optionally one question. Encodes the bright line *"offer the read as something to try on, not a verdict."* Example it predicts: *"Something in you catches the glimmer and covers it just as fast — like letting it land would cost something. What's it like to hear that?"*
- **grounded (medium)** — "put a frame on the table, gently." States a methodology frame (the guard, want-under-the-should, joy-as-matriarch) as substance + gentle read, hand left open. Restates anti-pattern #4 (no fix) and #5 (no lecture) as the line between *offering* a frame and *installing* one. Example: *"There's a guard that won't let the wish reach the part of you it's meant for. That guard showed up for a good reason once. I'm curious what it's protecting."*
- **joe-voiced (high)** — "speak with Joe's directness — and keep the guardrails tight." Direct observations, challenge-as-hypothetical ("what if X were true?"), welcome resistance by name, signature live moves (mirror-the-judgment, "just like right now," paradoxical instruction). Because directness is the move most likely to strain them, this layer restates the no-diagnosis / no-advice / respect-the-no / welcome-emotion-not-action guardrails MORE forcefully than the others. Example: *"I don't think you don't know how to let people in — I think a part of you is working hard to stop it today, on your birthday of all days, which tells me it matters. Can we get curious about that part instead of trying to fix you?"*

## Analysis

The data layer is shaped so [[E-054]] is a pure wiring job: replace `route.ts`'s three hardcoded constants (`SYSTEM_PROMPT`, `DEFAULT_COACH_MODEL`, the `V5B_CONFIG` baked into `runRetrieval`) with a `resolveCoachProfile(requestProfileId)` lookup and pass `.systemPrompt` / `.coachModel` / `.retrievalParams` through. The `generateCoachAnswer` helper and the happy-path `streamText` both currently close over `SYSTEM_PROMPT` / `DEFAULT_COACH_MODEL`; E-054 will thread a `profile` argument to both. `runRetrieval` reads `V5B_CONFIG` internally, so to honor per-profile `retrievalParams` E-054 will either pass params into `runRetrieval` or (simplest for now, since all four share v5b) leave `runRetrieval` as-is and document that the per-profile retrieval field is reserved. Either is a small change with no reshape of this registry.

The authority spectrum is deliberately monotonic in *how much declarative substance the coach volunteers*: v5b (none — questions only) → reflective (names a read) → grounded (names a read + offers a frame) → joe-voiced (names a read + offers a frame + challenges + welcomes resistance aloud). Each step strictly adds a sharing-move while the safety floor and the no-diagnosis/no-advice line are constant. That monotonicity is what makes the vibe-check ([[REQ-001]] session 2) able to *locate a band*: if reflective already feels like enough authority, grounded/joe-voiced over-shoot; if grounded lands and joe-voiced tips into interpretation, the winner is grounded. The prompts are written so a reader can predict where each will sit.

One judgment call worth flagging for the vibe-check: the example phrasings in the three variant prompts are intentionally written around the SAME concern the REQ-001 session 1 used ("the glimmer / can't let it land / birthday"), so when Rishi re-runs that concern across the selector the contrast between variants is maximally legible against the session-1 baseline he already has felt-memory of.

## Dead Ends

- **Considered: re-authoring the v5b prompt inline in `profiles.ts` for the baseline.** Rejected — it would create a second copy of the deployed prompt that could drift from `system-prompt.ts`. Importing `SYSTEM_PROMPT` directly guarantees the control is the real deployed thing.
- **Considered: four fully-independent prompt files (the method's literal "each is the v5b prompt as the base, with a stance layer added").** Rejected the copy-paste reading in favor of `base + layer` composition. Same observable result (each export IS a complete system prompt), but the safety floor cannot drift between copies and the reviewer-legibility is far better. Documented the rationale in `base.ts`. This is a faithful implementation of the method's intent ("differing only in stance layer"), not a deviation from success criteria.
- **Considered: running the full eval here.** Explicitly out of scope per the goal decision (vibe-check first; eval gates only winner-promotion). Not run.
