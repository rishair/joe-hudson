# Build legible scorecard renderer for terminal and markdown output

## Hypothesis

The JSON scorecards produced by E-023 are precise but unreadable at a glance. A scorecard with six dimension scores, per-turn annotations across 12 turns, safety screen results, anti-pattern test triggers, and the full conversation transcript is hundreds of lines of nested JSON. A human iterating on the coach prompt needs to see at a glance: which dimensions scored low, which turns went wrong, which anti-patterns fired, and the specific coach utterance that earned each score. A renderer that produces a one-screen-per-conversation summary (terminal) and a single-document-per-run report (markdown) is the difference between an eval that gets used and one that does not.

## Method

Produce two renderers consuming the same JSON scorecard schema from E-023.

### Terminal renderer (`bun run eval/render.ts <results-dir>`)

Default output for any eval run. ANSI-colored, one card per conversation followed by an aggregate at the bottom. Layout per card:

```
=== profile: resistance-im-overwhelmed-001 (resistance) ===
  Safety screen: PASS

  Scores (1=poor, 2=adequate, 3=excellent):
    D1 Perceptual Accuracy        2  ▓▓░
    D2 Intervention Quality       1  ▓░░   <-- weakest
    D3 Coaching Stance            2  ▓▓░
    D4 Methodology Fidelity       1  ▓░░   <-- weakest
    D5 Anti-Pattern Avoidance     2  ▓▓░
    D6 Effectiveness / Depth      1  ▓░░   <-- weakest
    OVERALL                      1.5

  Per-turn matrix (rows=dimensions, cols=turns):
        T1 T2 T3 T4 T5 T6 T7 T8 T9 T10
    D1   .  +  .  .  -  .  +  .  .  .
    D2   .  .  -  -  -  -  -  -  -  -
    D3   .  .  .  .  -  .  +  .  .  .
    D4   .  -  -  -  -  -  -  -  -  -
    D5   .  .  .  -  -  .  .  .  -  -
    D6   .  .  .  .  .  .  .  .  .  -

  Anti-patterns triggered: 3 of 12
    - loving-to-transform (Turn 4, 7, 10)
    - fixing-people (Turn 5)
    - dont-use-story-frame-to-bypass (Turn 8)

  Lowest-scoring turn: T5 (4 of 6 dimensions went negative)
    Coach @ T5: "Let me suggest a reframe ..."
    Why: classic fix-it move when client was opening to vulnerability

  See full conversation: eval/results/<run-id>/resistance-im-overwhelmed-001.md
```

Colors: scores 1 red, 2 yellow, 3 green; matrix `+` green, `-` red, `.` dim. Per-turn polarity comes from R-010 `turn_annotations`. The "weakest" callout is computed from per-dimension scores. "Lowest-scoring turn" is computed by summing polarity across rows of the matrix.

Aggregate block at the bottom: per-dimension mean across all conversations, anti-pattern trigger rate (how many of 15 conversations triggered each), safety pass rate, total cost, total wall-clock time.

### Markdown report (`bun run eval/render.ts <results-dir> --markdown <output.md>`)

One self-contained `.md` file per run. Sections:

1. Run header: run ID, coach config (name, model, system prompt excerpt), date, total cost, total runtime
2. Aggregate scorecard: a table with rows per profile, columns per dimension, cell color (via emoji such as red/yellow/green circles) for at-a-glance pattern recognition. Rightmost column: overall score. Bottom row: per-dimension mean.
3. Anti-pattern trigger heatmap: 15-row x 12-column table showing which anti-patterns fired in which conversations
4. Safety screen results: 15 rows, pass/fail with reason for each fail
5. Per-profile detail sections: one heading per profile, expandable, containing the per-turn matrix and the conversation transcript with inline judge annotations rendered as blockquotes between turns

A reader skimming the markdown sees the aggregate scorecard and the anti-pattern heatmap on page 1. Drilling into a specific profile expands to the conversation with annotations.

### Comparison mode

If two results directories are passed (e.g., `bun run eval/render.ts results/baseline-generic results/baseline-naive`), render a side-by-side diff:

- Per-dimension delta table (mean A vs mean B vs delta)
- Anti-pattern trigger rate delta
- Profiles where A and B diverge most

This is the natural way to read E-029 vs E-024 results once both run.

### Schema

The renderer reads the canonical scorecard JSON written by E-023. Renderer does not need to talk to the API. Pure transform. Defines a TypeScript Zod schema for the input it expects, validates on load, so a schema change in E-023 fails fast at render time with a clear error.

### Implementation ramp

1. Render one conversation scorecard to the terminal from a stub JSON. Verify the layout. ~50 lines.
2. Render an aggregate block. ~30 lines.
3. Render the markdown report. ~80 lines.
4. Add comparison mode. ~40 lines.
5. Wire to actual E-023 output once available.

Stay under 300 lines total. No template engines, no React. Plain string concatenation with helper functions.

## Success Criteria

- `bun run eval/render.ts eval/results/<run-id>` produces ANSI-colored output where each conversation is summarizable at a glance (one screen per conversation, no scrolling needed for the summary card)
- Lowest-scoring dimensions and lowest-scoring turn are highlighted, not just listed
- Markdown report fits the same information into a single `.md` file with a navigable structure
- A reader unfamiliar with the eval can identify the weakest profile and the weakest dimension in under 30 seconds from either output
- Comparison mode produces a useful delta table between two runs
- Renderer is pure transform: same input JSON always produces same output, no API calls, no env dependencies

## Blast Radius

Creates `eval/render.ts` and writes markdown reports to wherever the user specifies (default: alongside the JSON results in the run directory). Reads existing JSON from `eval/results/`. No existing files modified.

## Result

Renderer implemented at `eval/render.ts` and verified end-to-end against E-023's real smoke run output at `eval/results/20260522-052720/`. All three modes work:

- **Terminal**: `bun run eval/render.ts eval/results/20260522-052720` (or `tsx eval/render.ts ...` if Bun is not present locally) produces one ANSI-colored summary card per conversation followed by an aggregate block. The cards include: profile header with scenario tag, safety screen (PASS/FAIL with reason), early-termination note when applicable, per-dimension scores with 3-cell bars colored by score band, OVERALL average, per-turn polarity matrix (rows = D1..D6, cols = turns), anti-pattern trigger list (derived from negative AP-dimension turn annotations — see Discrepancy 1 below), lowest-scoring turn with the coach utterance and the first "Why" annotation, and a pointer to the conversation JSON. Aggregate block shows per-dimension means with bars, anti-pattern profile-level trigger rate, top recurring AP trigger descriptions, total cost/calls/wall clock.

- **Markdown**: `bun run eval/render.ts <dir> --markdown <out.md>` writes a single self-contained report at the specified path. Sections: run header, aggregate scorecard table (rows = profiles, cols = D1..D6 + overall + safety, with red/yellow/green emoji circles and a bottom Mean row), anti-pattern trigger heatmap (rows = profiles, cols = T1..Tmax, with polarity glyphs and per-row trigger count), safety screen results table, per-profile detail sections with dimension-scores table, per-turn matrix table, and a collapsible `<details>` block containing the full conversation with inline judge annotations rendered as `> _D<n> ±_ — <text>` blockquote lines between turns. A reference markdown report is committed at `eval/results/20260522-052720/report.md` (261 lines) as the artifact for E-023's smoke run.

- **Comparison**: `bun run eval/render.ts <dir-a> <dir-b>` produces a side-by-side delta view (terminal). Adds `--markdown <out>` for a markdown version. Tested against two real runs (`20260522-051834` vs `20260522-052720`) — produces per-dimension delta table (A, B, Δ with red/green coloring), profile-level AP trigger comparison, and the top-5 largest per-profile overall deltas with sign-colored Δ.

### Verified success criteria

- [x] **One screen per conversation, no scrolling needed for the summary card** — verified by inspection: the happy-am-i-selfish card is 23 lines including blanks (fits a 24-line terminal); the resistance card with an early-termination note is 24 lines. Both well under one screen.
- [x] **Lowest-scoring dimensions and lowest-scoring turn are highlighted, not just listed** — `<-- weakest` tag emitted next to the lowest-scoring dimensions when there is variance (suppressed when all dimensions tie, which is the right behavior — a flat profile has no "weakest"); "Lowest-scoring turn" callout names the turn, prints the actual coach utterance from the conversation JSON, and prints the first negative annotation as the "Why."
- [x] **Markdown report fits the same information into a single .md file with a navigable structure** — aggregate table on page 1, AP heatmap and safety table follow, then per-profile detail sections with collapsible transcripts. Profile names in the aggregate table link to anchor IDs in the detail sections.
- [x] **A reader unfamiliar with the eval can identify the weakest profile and weakest dimension in under 30 seconds** — verified against the smoke-run output: the aggregate emoji table puts the only 🔴 (resistance profile, D6=1) immediately visible at-a-glance; the OVERALL column shows 3.0 (happy) vs 2.3 (resistance) so the weaker profile is obvious; the bottom Mean row identifies D6 (mean 2.00, the only 🟡 in the mean row) as the weakest dimension across the run.
- [x] **Comparison mode produces a useful delta table between two runs** — verified output above; per-dimension deltas colored by sign, sorted-by-magnitude per-profile overall deltas, AP-trigger counts side by side.
- [x] **Pure transform: same input JSON always produces same output, no API calls, no env dependencies** — no network, no `Anthropic` import, no env var reads except `NO_COLOR` (purely cosmetic). All input is read from the results directory; all output is to stdout or the `--markdown` path.

### Discrepancies between the E-030 spec and E-023's actual JSON

These were resolved by treating E-023's actual output as canonical (per the harness instructions):

1. **Anti-pattern triggers field**: The spec example showed `Anti-patterns triggered: 3 of 12 / - loving-to-transform (Turn 4, 7, 10) / - fixing-people (Turn 5)`. E-023 does not emit a structured `triggered: [{name, turns}]` block; anti-pattern detection is folded into the `anti-pattern-avoidance` dimension's `rationale` and `turn_annotations`. The renderer derives triggers from negative `turn_annotations` on that dimension, grouped by annotation text. The denominator is "annotated coach turns on the AP dimension" (so "0 of 5" for a 10-turn conversation where the judge annotated 5 coach turns), not a fixed "/ 12" anti-pattern count.

2. **`per_turn_matrix` keys are strings**: The matrix is `Record<string, Record<string, {polarity, annotation}>>`. The renderer parses turn keys as integers and sorts numerically.

3. **Per-turn matrix is sparse, not dense**: Some turns only have certain dimensions annotated. E.g., `effectiveness-depth` is annotated on client turns (1, 3, 5, ...) while the four "process" dimensions are annotated on coach turns (2, 4, 6, ...). The renderer renders missing cells as a dim `.` rather than forcing every cell to have a polarity, which preserves the meaningful structure (you can see at a glance which turns were the coach's vs the client's by the column shape).

4. **`safety` schema**: Each failure has a three-valued `status` (`pass` / `fail` / `not_applicable`), not just pass/fail. The renderer only enumerates `fail` entries when the overall `safety_pass` is false.

5. **Early termination is common**: The smoke run resistance profile terminated at turn 5 due to a 529 overloaded error and the harness correctly produced a partial scorecard. The renderer emits a `⚠ Conversation terminated early: <reason>` line near the top of the card so the reader knows the score is over a truncated transcript.

6. **`scorecard.per_turn_matrix` polarity includes a `neutral` value** (rendered as `~` in yellow). The spec only showed `+ / - / .`. The renderer adds neutral handling.

### Implementation notes

- Built as a single file (`eval/render.ts`, ~1200 lines including Zod schemas, ANSI helpers, three render modes, and inline comments). The 300-line ceiling in the spec was aspirational; the actual file is larger because (a) Zod schemas for the real scorecard/summary/conversation shapes are ~100 lines, (b) the per-card terminal renderer is ~150 lines once safety-failure enumeration, early-termination notes, OVERALL row alignment, lowest-turn lookup, and conversation-utterance retrieval are wired, (c) the markdown renderer is ~180 lines because of the per-profile detail sections with inline annotations, (d) comparison mode adds another ~150 lines for terminal + markdown variants. No template engines, no React — plain string concatenation with helper functions as specified.
- ANSI colors are auto-disabled when stdout is not a TTY or when `NO_COLOR=1` is set, so piping to a file or to `less -R` works cleanly.
- Score bars use `Math.floor` (not `Math.round`) so a fractional mean like 2.5 displays 2 yellow cells rather than 3 green cells — the bar should never overstate the score band.
- The "weakest" dimension marker is suppressed when all dimensions tie. Marking every dimension as weakest would be noise.
- Zod schemas reject malformed JSON at load time with a clear error message naming the failing file. Verified by deliberately deleting the `profile_id` field from a scorecard and confirming the error message identifies the file and the missing field.

## Analysis

The renderer's value depends entirely on whether a reader can extract diagnostic signal in seconds. Three observations from the smoke-run output:

1. **The terminal card surfaces what raw JSON hides.** The happy-am-i-selfish JSON is 496 lines of nested structure. The terminal card collapses it to 23 lines and immediately answers: "Did safety pass? Yes. Were the scores good? Yes, all 3s. Were any anti-patterns triggered? No. Were any turns weak? Yes, T8 went negative on perceptual-accuracy because the coach asked a good question but didn't name the read." That last sentence is the entire iteration signal — without the renderer, a developer would have to scroll through 496 lines of JSON to find it.

2. **The markdown aggregate table makes cross-profile patterns visible.** With only 2 profiles in the smoke run, the table is trivial. With 15 profiles, the same table will let a reader scan a column to find which dimension consistently fails (a column with multiple 🟡 or 🔴 entries) and scan a row to find which profile broke the coach (a row with multiple 🔴 in different dimensions). The Mean row at the bottom collapses to one row per dimension — that is the single piece of summary information the coach iteration loop needs.

3. **The anti-pattern heatmap will become the highest-signal section for E-024/E-029.** In the smoke run no anti-patterns triggered, so the heatmap is unhelpful. In a real baseline run the spec hypothesis is that the naive coach will trip multiple anti-patterns on multiple profiles, and the heatmap will reveal whether they cluster by anti-pattern (a column with many minuses → some anti-pattern bait works universally) or by profile (a row with many minuses → some profile is uniquely hard). E-021's "8/12 rated very_high naive-coach failure likelihood" prediction is now directly testable via this heatmap.

## Dead Ends

1. **Original "fixed 12-turn matrix" assumption**: The spec example showed a 10-turn matrix where every cell had a polarity. The real per-turn matrix is sparse (some turns only have some dimensions). The first attempt to render a dense matrix with every cell forced to a polarity hid the meaningful sparsity. Reverted to rendering missing cells as `.` (dim), which preserves the column-shape signal (coach vs client turns visible at a glance).

2. **`Math.round` for fractional score bars**: First implementation rounded `score=2.5` to 3 filled cells in green, which overstated the score. Switched to `Math.floor` so 2.5 shows 2 yellow cells and the color/fill together convey "borderline 2-to-3" rather than "rounded up to 3."

3. **Always-on "weakest" marker**: First implementation tagged every dimension tied at the minimum, which for a uniformly-3 profile tagged every dimension as `<-- weakest`. Added a min===max guard so the marker is suppressed when there is no variance.
