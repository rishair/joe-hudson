# Progressive streaming with retrieval progress events

## Hypothesis

Wrap the entire `/api/chat` POST handler in `createUIMessageStream` so the SSE response opens BEFORE the slow retrieval pipeline runs, and emit `data-progress` UI message parts at each stage (seed-detection â†’ walker steps â†’ coach compose). The user sees incremental feedback in the UI within <1s of pressing Send instead of staring at silence for 10-25s while retrieval completes.

## Method

1. Add `ProgressEvent` type to `app/lib/coach/types.ts` with stages: `analyzing` / `retrieving` / `walking` / `composing` and add it as a second data-part key on `CoachUIMessage`.
2. Plumb an optional `onProgress?: (event: ProgressEvent) => void` callback through `runRetrieval` (`app/lib/coach/retriever.ts`) â†’ `retrieveByGuidedWalk` (`app/lib/coach/guided-walk.ts`). The callback fires:
   - Once at start (`analyzing`, "Reading what you wrote")
   - Once after seed-detection (`retrieving`, "Found N starting points")
   - Per walker iteration (`walking`, step N of M)
3. Rewrite the route handler: move `createUIMessageStream` to wrap the ENTIRE handler body (not just the post-retrieval section). Pass `emit` as the `onProgress` callback. Emit a final `composing` event right before the coach `streamText` merge.
4. The safety-regen path runs INSIDE the same stream now (no separate `safetyRegenerationFlow` Response builder). It emits its own `composing` events: "Considering safety" / "Regenerating with stronger safety reminder" / "Serving safety fallback".
5. Client (`app/page.tsx`): extract the latest `data-progress` part on each assistant message and render it as a small italic gray inline status line. Hide it once any `text` part has non-empty content.

## Success Criteria

- Local: send a chat message; first `data-progress` event arrives in <1s; per-walker-step events tick through over the next 30-40s; resources + composing events fire just before text; UI shows the progress text replacing as stages change.
- Build: `bun run build:cf` clean with 0 secret leaks.
- Deploy: `bunx wrangler deploy` succeeds; `curl -N -X POST` against the deployed `/api/chat` shows progress events arriving incrementally, NOT all-at-once at the end (cloudflared SSE-buffering caveat).
- No regressions: existing `data-resources` part still fires, resource modal still works, safety regen path still produces compliant text, cost-cap still records spend.

## Analysis

The fix was structural: `createUIMessageStream`'s `execute` callback gets a `writer` that emits chunks AS the callback runs. By moving the stream-creation BEFORE the retrieval call, the SSE response opens immediately and each `writer.write({type:'data-progress',...})` flushes to the wire instantly. The retrieval pipeline's existing 10-25s wall-clock is unchanged; the UX win is purely about giving the user visible feedback during that window.

Three small implementation choices worth recording:

1. **Progress events have unique ids per emission** (`progress-<ts>-<seq>`). The client-side helper just picks the LATEST progress part on each render, so it doesn't matter if older events are kept or filtered â€” the rendered state is always "the most recent stage". Persisting the events to SQLite alongside the message is harmless (~20 small JSON entries per assistant message at most); we could filter them out at persistence time if storage proves an issue, but the byte-equality round-trip explicitly preserves them as part of `UIMessage.parts` per the E-041 contract.
2. **Safety regen runs inline inside the same stream now**, not as a separate `safetyRegenerationFlow` that builds its own Response. This means the safety path also gets the analyzing/retrieving/walking progress events FOR FREE â€” before this change, the safety branch was just as silent as the happy path because both branches finished retrieval first. The inlined helper `runSafetyRegenInline` writes the `start`/`start-step`/`text-start`/`text-delta`/`text-end`/`finish-step`/`finish` chunks directly to the writer, which is what `streamText`'s `toUIMessageStream()` does for the happy path.
3. **Client hides progress once any text part has content**. This is a one-line condition: `latestProgress && !hasAnyText(m)`. The progress line disappears as soon as the first text delta arrives, transitioning visually from "Composing response" to the actual response text. No animation; the CSS `transition: opacity 0.2s` is there for future polish if we want a fade.

`UIMessageStreamWriter<T>` is exported from `ai` v6 (verified in `node_modules/ai/dist/index.d.ts`). The progress emitter helper closes over the writer and a sequence counter â€” emits as `data-progress` parts that the typed `CoachUIMessage` recognizes by virtue of the `progress` key in its data-parts map.

## Dead Ends

None. The whole change was straightforward once the AI SDK shape was clear. The only minor friction was matching the existing safety-regen Response shape inside a `writer` (which expects the raw chunks like `start` / `start-step` / `text-start` / `text-delta` / `text-end` / `finish-step` / `finish`); the original `safetyRegenerationFlow` already had this same pattern with the `as never` casts to dodge AI SDK v6's stricter chunk type discrimination, so I lifted that verbatim.
