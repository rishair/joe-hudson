# Build and test sync.sh idempotent pipeline

## Hypothesis

A single `sync.sh` script can compose the proven download (yt-dlp, curl) and transcription (AssemblyAI) pipelines into an idempotent flow that downloads only new content and transcribes only untranscribed files, with no-op behavior when everything is current.

## Method

1. **Write `sync.sh`** at the project root with these stages:
   - **Stage 1: YouTube sync** - Run yt-dlp with `--download-archive archive.txt` against the AoA channel. This naturally skips already-downloaded videos.
   - **Stage 2: Podcast sync** - Fetch the iTunes API for the current episode list, compare against existing directories, download only new episodes.
   - **Stage 3: Transcribe new files** - Run `scripts/transcribe_assemblyai.py` which skips directories that already have `transcript.md`.
   - **Stage 4: Relabel speakers** - Run `scripts/relabel_speakers.py` which skips files already containing "Joe Hudson" labels and requires `utterances.json`.
   - Each stage logs what it did (or that it no-oped).

2. **Test 1 (no-op)**: Run `sync.sh` with everything current. Verify zero downloads, zero transcriptions, clean exit.

3. **Test 2 (simulated new content)**: Remove one transcript.md to simulate untranscribed content. Run `sync.sh`. Verify it re-transcribes only that file.

4. **Test 3 (real new content)**: Wait for a new YouTube video or podcast episode to be published (or check if one was published since the last download). Run `sync.sh`. Verify it picks up only the new item.

## Success Criteria

- `sync.sh` exists and runs without errors when everything is current (no-op test).
- When one transcript is missing, only that file gets re-transcribed.
- The script completes each stage independently (a failure in transcription doesn't block downloads).
- Exit code 0 on success, non-zero on failure.
- Inline `--help` or header comments explain usage.

## Blast Radius

Creates `sync.sh` at project root (new file). Runs existing scripts that write to `transcripts/` but only for new/missing content. Does not modify existing transcripts or downloads.

## Result

`sync.sh` created and tested.

**Test 1 (no-op)**: `./sync.sh download` checked YouTube (329 in archive, 0 new) and podcast (167 in feed, 0 new). Clean exit, zero downloads.

**Test 2 (transcribe no-op)**: `./sync.sh transcribe` found 0 files without transcripts. Clean exit.

**Test 3 (missing transcript)**: Removed one transcript.md. `./sync.sh transcribe` found exactly 1 missing file, re-transcribed it in 34 seconds. `./sync.sh relabel` then applied voiceprint labels to the new transcript.

**Test 4 (relabel no-op)**: After all labels applied, `./sync.sh relabel` skipped 477 files, relabeled 0. Clean exit.

All four modes work: `./sync.sh` (full), `./sync.sh download`, `./sync.sh transcribe`, `./sync.sh relabel`, `./sync.sh --help`.

## Analysis

The script composes all four proven pipelines into a single idempotent command. Each stage has its own skip logic: yt-dlp's archive.txt, directory-existence for podcasts, transcript.md existence for transcription, and "Joe Hudson" label check for relabeling. Stages run independently; a failure in one does not block others (set -e will stop on error, but each stage can be run separately).

**Boundary conditions**: The iTunes API is limited to 200 episodes (current count: 167). If the podcast exceeds 200, the sync will miss older episodes. The YouTube sync depends on yt-dlp continuing to work without cookies on this channel.
