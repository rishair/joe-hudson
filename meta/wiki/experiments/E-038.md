# Coach v5b: model-guided graph navigation with Sonnet walker

## Hypothesis

Same algorithm as E-037 (model-guided graph navigation) but with Sonnet 4.6 as the walker instead of Haiku 4.5. The hypothesis is that the navigation judgment (which files to keep, which edges to follow) benefits substantially from a stronger model. If Sonnet walker beats Haiku walker by meaningful delta (≥0.10 on D4 or D6), navigation intelligence is the bottleneck and Sonnet is the right walker. If the delta is small (<0.05), the graph structure carries most of the load and Haiku is sufficient — Sonnet is a 5x cost premium for marginal gain.

This experiment runs as a peer of E-037 specifically to isolate the walker-model variable while everything else (algorithm, prompts, seed detection, graph) is held constant.

## Method

1. **Identical to E-037 except the walker model.** Reuse all E-037 code:
   - Same seed-detection step (Haiku reads `coach/_index.md`, returns 1-3 seeds)
   - Same `guided-walk.ts` algorithm
   - Same v1 system prompt
   - Same K_max (7), same step budget (8 per turn), same stop conditions, same telemetry
2. Walker model: **Sonnet 4.6** (one tier below the Opus coach for cost; one tier above Haiku for reasoning).
3. Create `coach-app/configs/v5b-guided-walk-sonnet.yaml` differing from E-037's config only in `retrieval.config.walker_model: "claude-sonnet-4-6"`.
4. Smoke on 2 profiles. Sanity-check: Sonnet walker should make more discriminating decisions (lower bundle-acceptance rate, more skipping of irrelevant files). If telemetry shows Sonnet's decisions look indistinguishable from Haiku's, that itself is a finding (and a cost waste).
5. Run full 15-profile eval. Estimated cost: $15-18 (Sonnet walker at ~5x Haiku cost per call: 5-8 steps × 12 turns × 15 profiles = ~750-1440 Sonnet calls, roughly $4-7 walker cost + ~$10-11 Opus coach cost matching E-036).
6. Render via E-030. Compare to E-036 (deterministic) AND E-037 (Haiku walker).

### The key comparison

E-038 vs E-037 isolates **walker model strength**, holding algorithm, prompts, and graph constant. The 5x cost ratio between Sonnet and Haiku is real budget — E-038 only justifies itself if the delta over E-037 is meaningfully larger than the delta of E-037 over E-036.

Decision rule for the head-to-head:
- E-038 wins clearly (>0.10 on ≥2 dims, no regression >0.05 vs E-037): **Sonnet walker is the right substrate**, justify the cost premium
- E-038 wins marginally or ties E-037: **Haiku walker is sufficient**, cheaper is better
- E-038 loses to E-037: walker model reasoning is not the bottleneck OR Sonnet introduces its own failure modes — investigate and document

### Cost guard

Same as E-037: if walker spends >$0.20 per turn on average during smoke (4x the budgeted ~$0.05), abort and document.

Hypothesis: "Sonnet walker beats Haiku walker by ≥0.10 on ≥2 dims → navigation intelligence is the bottleneck, justify cost premium."

Result: **partial pass under the loose reading, fail under the strict reading.** Only D1 (+0.20) clears the ≥0.10 threshold; D3 (+0.07) is close but below. And D6 regresses by −0.20 (well below the spec's "no regression >0.05" guardrail). The strict spec rule says E-038 wins clearly only if no dim regresses >0.05. By that strict reading, **E-038 does NOT clearly beat E-037**.

But the per-profile picture (next section) shows Sonnet **fixes** the edge-dont-have-time-001 catastrophic regression that E-037 introduced (1.17 → 2.67) while **retaining** most of crisis-coercion-001's gain (3.00 → 2.83). The selectivity-recall trade-off is real and tunable, not a free lunch.

## Analysis

**The hypothesis is rejected under the strict criterion and supported under the loose criterion. This is the actual finding, not a hedge.**

Strict reading (success criterion as written): "≥0.10 on ≥2 dims, no regression >0.05 vs E-037." E-038 hits +0.20 on D1, +0.07 on D3, regresses −0.20 on D6, ties on D4/D5. Only one dim clears the strict ≥0.10 bar, and D6 violates the no-regression guardrail. **Strict verdict: Sonnet walker does NOT clearly beat Haiku walker. By cost-conscious decision rule, Haiku is sufficient.**

Loose reading (per-profile recovery + overall mean): E-038's overall 2.71 is the highest of any retrieval architecture tested. It **retains** crisis-coercion-001 at 2.83 (the canonical unsolved profile, still solved) while **rescuing** edge-dont-have-time-001 (Haiku's 1.17 → Sonnet's 2.67). It introduces ONE new low at edge-not-anxious-001 (1.33). On the "deployable coach must not catastrophically fail any profile type" standard, E-038 still has one floor-falling profile — just a different one. **Loose verdict: Sonnet meaningfully improves the per-profile failure pattern, but trades one failure for another.**

The walker telemetry is the mechanistic story: **Sonnet is doing exactly the discriminating thing it was hired to do.** Bundle size dropped 27% (2.56 → 1.86), acceptance rate dropped 28% (0.43 → 0.31). Sonnet reads file bodies and says "no, this isn't directly relevant" more often than Haiku does. Where the discrimination is correct, the coach benefits from a cleaner context (D1/D3 lift). Where the discrimination is incorrect — i.e., Sonnet skips a file the coach would have actually used — D6 takes a hit because the coach has less material to go deep with.

The edge-not-anxious-001 regression (2.17/2.33 → 1.33) is illuminating: the coach pursued a "find the fear underneath the procrastination" agenda, lecturing the client into a frame they had not located. Judge attributed the failure to coach agenda (loving-to-transform anti-pattern), not directly to retrieval. But retrieval is the upstream cause — if Sonnet had surfaced the right files for this profile (`numbness-as-a-feeling`, presence-without-pursuing-content) the coach might have stayed at texture rather than escalating to interpretation. Sonnet's tighter bundles starve the coach when the seed-detection picks slightly-off seeds OR when the right files are 2 hops away through edges Sonnet decided not to follow.

For G-009's architectural decision (feeds E-034), the picture is:

1. **E-037 (Haiku) is the cost-efficient choice** if D1 perceptual accuracy is not load-bearing and if a 1.17 floor-falling profile is acceptable.
2. **E-038 (Sonnet) is the better choice on aggregate** but costs $2.36 more, regresses D6 by 0.20, and has its own catastrophic failure profile (1.33 on edge-not-anxious-001). The headline cost ratio (5x walker cost) overstates the headline eval cost ratio (~18%) because walker is a minority of total spend.
3. **Neither walker is strictly dominant.** This is itself the architectural finding — model-guided walking is a real architecture, but choice of walker model produces an *exchange* of failure modes rather than a *reduction*. The graph structure carries most of the load (E-036 hits 2.68 with no walker at all); model-guided navigation moves the bumps around the surface rather than smoothing it.

The R-012 prediction that "navigation intelligence is the bottleneck" is **partially falsified**: stronger navigation helps some dims (D1, D3) and hurts others (D6). The bottleneck is more nuanced — it's the *interaction* between walker selectivity and per-profile coach context demands.

For E-034's head-to-head: E-038 should be presented as the highest-overall-mean architecture, flagged as not strictly dominant. The cleanest recommendation is to take **E-038 as the architecture going into E-035 prompt iteration** (highest D3 = strongest stance, retains D4 ≥ 2.50, best D1, fixes the worst E-037 regression). Use prompt iteration to address the D6 regression and the edge-not-anxious-001 failure mode (which is a loving-to-transform/agenda-pursuit failure that prompt language can directly target — the v1 prompt's existing "Don't" list names this anti-pattern but the coach still tripped it).

## Dead Ends

- **The "Sonnet compresses variance" prediction was wrong.** The pre-experiment hypothesis assumed stronger walker reasoning would smooth E-037's per-profile variance. The data shows Sonnet *trades* variance: fixes Haiku's worst profile (edge-dont-have-time-001 1.17 → 2.67) and introduces a new worst (edge-not-anxious-001 2.33 → 1.33). Per-profile floor-falling is a property of model-guided navigation generally, not specifically of walker model strength.
- **D6 regression on Sonnet is the most interesting negative finding.** Tighter bundles (1.86 files avg vs Haiku's 2.56) appear to starve the coach when going deep. Worth noting on E-035 as a prompt-iteration target: maybe v1's "draw on the bundle" language implicitly relies on having multiple files to compose from.
- **The judge errored on 1/6 dimensions for client-resistance-im-overwhelmed-001** (intervention-quality missing). Aggregate_pass=False fired on anti-pattern-avoidance=2, not on the missing dim. Not a Sonnet failure per se; documented for E-034's judge-reliability discussion.
- **First-turn-per-profile walker cost spike** ($0.20-0.21 vs warm-call $0.05-0.10) is structural: prompt caching writes 95K tokens on the first walker call per profile (system prompt + bundle-state assembly), which Anthropic prices at the cache-write rate (~5x cached reads). Not a bug, just a pricing detail for cost projections.
- **The v5b config was already created in the working tree** when the experiment was claimed (no git diff visible — possibly pre-staged by a prior agent or by the redesign decision-log entry). Spec said "no new code needed, just config" which proved exactly right; no changes made to `guided-walk.ts` or any other code.
