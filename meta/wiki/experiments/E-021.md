# Build anti-pattern test cases

## Hypothesis

Each `coach/anti-patterns/` entry describes a specific failure mode that an AI coach is particularly likely to fall into (e.g., loving to transform, VIEW as technique, vulnerability as control). Designing client scenarios that specifically trigger these failure modes will test whether the coach avoids them. These tests are the highest-signal component of the eval because anti-patterns are exactly where naive AI coaching fails.

## Method

1. Read all 31 `coach/anti-patterns/` files
2. For each of the top 10-12 most critical anti-patterns (prioritize those an LLM is most likely to exhibit), design a test case at `eval/anti-pattern-tests/<anti-pattern-id>.yaml` containing:
   - anti_pattern_id: maps to coach/anti-patterns/ filename
   - trigger_scenario: a client statement or behavior specifically designed to bait the coach into this anti-pattern
   - what_failure_looks_like: concrete behavioral description of the coach falling into it
   - what_success_looks_like: what the coach does instead
   - detection_signals: specific phrases, patterns, or structural features the judge should look for
3. Priority anti-patterns for AI coaches (most likely to fail):
   - loving-to-transform (AI loves to "help" people change)
   - view-as-technique (AI does communication techniques without state)
   - dont-use-story-frame-to-bypass (AI might reframe too quickly)
   - dont-fight-the-voice (AI might argue with the inner critic)
   - vulnerability-as-control (AI might perform vulnerability)
   - dont-evangelize-the-work (AI might preach the methodology)
   - impartial-apathy (AI might be flat rather than warmly impartial)
   - dont-half-ass-the-experiment (AI might suggest weak experiments)
   - dont-use-safety-as-control (AI might over-use safety framing)
   - limiting-belief-meta-trap (AI might label things as limiting beliefs prematurely)

## Success Criteria

- 10+ anti-pattern test YAML files exist at `eval/anti-pattern-tests/`
- Each test is specific enough that a judge can determine pass/fail
- Manual inspection confirms: a naive AI coach WOULD fail at least 7/10 of these tests
- Each test references the source `coach/anti-patterns/` file

Each YAML conforms to a consistent schema with 14 top-level fields: `test_id`, `anti_pattern_id`, `anti_pattern_file`, `title`, `version`, `why_ai_prone`, `trigger_scenario`, `what_failure_looks_like`, `what_success_looks_like`, `detection_signals`, `scoring`, `judge_instructions`, `output_fields`, `references`, `plausibility`. Validated by `yaml.safe_load` and per-field presence check (12/12 pass).

The schema is deliberately richer than the experiment spec required. Beyond the five required structural fields (`anti_pattern_id`, `trigger_scenario`, `what_failure_looks_like`, `what_success_looks_like`, `detection_signals`), every file also includes:
- `why_ai_prone`: a short paragraph naming the specific LLM training pressure that produces this failure mode. Lifted from the rubric's `priority_anti_patterns.why_ai_prone` annotations where applicable, expanded where the rubric was brief. This makes each test self-contained for judge prompting.
- `trigger_scenario.bait_turns`: sequenced client moves at specific turn ranges (e.g., turns 3-5, 7-9) that escalate the bait. This converts the test from a static scenario into a multi-turn pressure trajectory. The harness or client LLM can use these directly.
- `what_failure_looks_like.concrete_tells` and `.example_failure_utterances`: detection-grade specificity. Each "tell" is a phrase pattern or structural feature the judge can verify. Example utterances give the judge concrete failures to pattern-match.
- `what_success_looks_like.concrete_tells`, `.example_success_utterances`, `.positive_avoidance_marker`: parallel positive structure. The positive_avoidance_marker names the strongest single signal that the coach has actively avoided the anti-pattern (the un-trap), which is what scoring uses to discriminate score-3 from score-2.
- `detection_signals` decomposed into `failure_triggered_if_any_of`, `success_demonstrated_if_any_of`, and `edge_cases`: gives the judge clear binary rules plus boundary clarifications. Edge cases prevent the test from over-firing or under-firing on adjacent behaviors that should fire other tests.
- `scoring.score_1/score_2/score_3.when`: explicit thresholds in terms of the detection signal counts and bait-turn outcomes. The judge can compute the score deterministically from the signal evidence.
- `judge_instructions`: a self-contained instruction paragraph for the Pass-2 D5 judge call. Appended to the D5 rubric's `judge_guidance` when running this specific test.
- `output_fields`: the per-test record the harness records. Common fields (`test_id`, `score`, `failure_signals_fired`, `success_signals_fired`, `turn_citations`, `rationale`) appear everywhere; test-specific fields appear where they discriminate (e.g., `arc_bending` for view-as-technique, `experiment_specificity` for dont-half-ass-the-experiment, `safety_language_count` for dont-use-safety-as-control).
- `plausibility.naive_coach_failure_likelihood` and `.rationale`: explicit hypothesis about how often a naive AI coach will fail each test. 8/12 tests are rated `very_high` (~85-95% naive failure), 4/12 are `high` (~70-85% naive failure). The aggregate is the plausibility-check for the experiment as a whole.

Bait-turn design across the 12 files is consistent: each test has 2-3 client moves that escalate the bait in known turn windows. Common pattern: turn 3-5 mild bait (client offers natural opportunity), turn 5-7 stronger bait (client explicitly invites the failure mode), turn 8-10 hardest bait (client makes an explicit request or surfaces meta-observation about the coach). The hardest bait is the diagnostic moment â€” a naive coach will take it; a coach in stance will decline. The detection signals reference these turn ranges explicitly so the judge can localize evidence.

Both can fire on the same conversation from different lenses (anti-pattern lens at D5 vs. safety screen at Pass-1). The seam was already documented in `eval/rubrics/anti-pattern-avoidance.yaml#seam_with_safety`; the dont-use-safety-as-control test references this and instructs the judge that this is acceptable double-coverage. Similarly, `vulnerability-as-control.yaml` references its pairing with `dont-use-safety-as-control` per the coach/ files' related links.

Coverage analysis of co-occurring anti-patterns: `limiting-belief-meta-trap.yaml` notes co-occurrence with `dont-evangelize-the-work` (when coach uses too much AoA category language) and `fixing-people` (when coach treats the freezing as a problem to solve). `dont-evangelize-the-work.yaml` notes the parent relationship to `fixing-people`. `wonder-as-strategy.yaml` notes the parent relationship to `view-as-technique`. Edge cases throughout explicitly route ambiguous behavior to whichever test should fire, preventing both over-firing (every test marks every bad behavior) and under-firing (no test catches a behavior because each thinks another would).

## Analysis

The richer-than-required schema pays off in three places downstream:

**(1) Judge prompt construction (R-010 Pass-2 D5 call).** The harness can pull `judge_instructions` + `detection_signals.failure_triggered_if_any_of` + `scoring.*.when` directly into the D5 prompt without further synthesis. The few-shot slots in `eval/rubrics/anti-pattern-avoidance.yaml` get populated by referencing the `example_failure_utterances` and `example_success_utterances` from these files. Without this structure, the judge prompt would have to invent the failure forms at scoring time.

**(2) Client profile generation (E-020).** The `trigger_scenario` blocks are themselves partial profile specs. The `client_opening` and `bait_turns` can be lifted into anti-pattern-focused profiles in E-020, especially for the 3 edge-case slot in the 3/4/3/3/2 distribution. E-020 can either build these as additional profiles or use them as overlays on top of resistance profiles.

**(3) Calibration (E-025).** Each test's `plausibility` block is a hypothesis to validate. E-025 calibration on a naive coach should produce failure rates close to the predicted likelihoods. Wide divergence (predicted very_high, observed pass) means either the test isn't biting or the naive coach is unexpectedly strong; either way, the calibration generates useful debugging signal.

Three risks identified that E-025 should address:

**Risk A: Over-firing on adjacent behaviors.** Several tests cover behaviors that adjacent tests might also flag. The edge_cases blocks attempt to route, but in practice the judge may fire all candidates. E-025 should check whether each test fires only on its intended failure mode by running a conversation crafted to trip exactly one anti-pattern and verifying only that test reports score 1.

**Risk B: Bait-turn alignment with E-020 profiles.** The bait turns are specified at turn ranges (e.g., "turn 3-5"). When merged with E-020 profile-based conversation generation, the actual turns where the bait fires may shift if the client LLM diverges from the script. The harness needs to either (a) inject the bait moves at the specified turns regardless of conversation flow, or (b) let the client LLM produce equivalent baits naturally and accept turn drift. The current schema accommodates both but does not commit. E-025 should determine the right answer.

**Risk C: Judge sensitivity to language vs. structure.** Several detection signals are language-pattern-based ("Coach uses 'I love' / 'beautiful' density increase across turns"). A coach who has internalized the anti-pattern but uses different surface language might escape detection. The signals should be tested for both false positives (coach uses the words in a good way and still gets flagged) and false negatives (coach falls into the pattern without the language tells). E-025 calibration should specifically include a "stylistic outlier" case for at least one test to probe this.

## Dead Ends

The first draft attempted to keep references minimal (3-4 per file). After verifying that the rubric (E-019) wanted to populate few-shot slots by pointing at these files, and that R-010's judge prompt structure expects per-rubric reference grounding, I expanded references to 5-9 per file with explicit parent/related/co-occurrence relationships. The verbosity is a feature: the references encode the navigation graph the judge can use to ground scoring decisions.

I considered using a single shared `bait_pattern_template` with named variants per test but rejected this â€” the bait scenarios are sufficiently different that DRY-ing them produces less readable files. The slight repetition across files is worth the per-test self-containment, which lets each file be loaded independently into a judge prompt.

I considered cutting `plausibility` out as overkill but kept it because (a) it's the success-criterion check for the experiment itself ("a naive AI coach WOULD fail at least 7/10"), and (b) it gives E-025 calibration explicit per-test predictions to falsify.
