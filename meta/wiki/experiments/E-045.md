# Build first-visit welcome flow explaining project, build, privacy, unaffiliation

## Hypothesis

A short first-visit modal (or step sequence) covering project intro, how it was built, privacy posture, and unaffiliation gives new users enough context to use the coach intentionally — without overwhelming them or hiding the chat behind reading. The flow is dismissible, persists dismissal in localStorage, and is reachable later via a help menu.

## Method

1. Build `web-app/app/components/welcome.tsx`:
   - Trigger: shows on mount if `localStorage.welcomeAcknowledged !== 'true'`
   - Reachable later via a help icon (top-right) that re-opens it
   - Layout: single modal with 4 short sections (or a 4-step flow — pick whichever feels less bureaucratic for a "short and sweet" intent; recommend single modal with scrollable sections)
2. Section content (final copy is editable; this is the structural spec):

   **§1 — What this is**
   > An AI coach inspired by Joe Hudson's Art of Accomplishment methodology. Bring something you're stuck on, something you're avoiding, or something you can't quite name. The coach asks rather than advises.
   >
   > *This project is independent. It is not affiliated with, endorsed by, or operated by Joe Hudson or Art of Accomplishment. It draws on publicly available content (podcasts, videos, talks).*

   **§2 — How it was built**
   > Joe's publicly available conversations were transcribed and structured into a compendium of his reads, moves, questions, frameworks, and anti-patterns. A retrieval system surfaces the right pieces of that compendium per turn; an LLM generates the coach response grounded in what's surfaced. You can browse the compendium directly via the Wiki tab.

   **§3 — Privacy**
   > Your conversations are stored **only in your browser** (local SQLite via OPFS). They never reach our servers. Your messages do travel to OpenRouter and onward to an LLM provider for the coach's response — that is the only network egress. Closing your browser keeps your conversations safe locally; clearing site data deletes them.
   >
   > We may add an optional anonymous feedback mechanism later (e.g., 👍/👎 on a message). Nothing is sent without your explicit click.

   **§4 — Get started**
   > One sentence to start: *"Something I've been avoiding deciding."* / *"A pattern I keep falling into."* / *"I'm stuck on something but I can't name what."* Or anything else on your mind.
   >
   > [I got it →]  (button dismisses the modal and persists the flag)

3. Track a single localStorage key: `welcomeAcknowledged: 'true'`. No other state. No analytics in v1.
4. Help icon (top-right of the app shell) opens the welcome again. Same content, but the button reads [Close] instead of [I got it].
5. Style: Tailwind defaults consistent with the rest of the app. Modal width ~640px, readable line length.
6. Manual test: load app in incognito → welcome shows → dismiss → reload → welcome does not show → click help icon → welcome shows → close → reload → welcome does not show.

## Success Criteria

- Welcome modal renders on first visit only (controlled by `localStorage.welcomeAcknowledged`)
- All 4 sections present with the spec'd content (final copy can be tuned)
- Dismissal persists across reloads
- Help icon re-opens the modal
- Modal is keyboard-accessible (escape closes, button is focused on open)
- No analytics or remote calls fire on welcome open/dismiss
- The unaffiliation language is present and prominent in §1
- The privacy language explicitly names: (a) local SQLite storage, (b) LLM provider as the only network egress

**Design choices vs the spec:**

- **Used vanilla CSS / inline styles, not Tailwind.** The spec mentioned "Tailwind defaults consistent with the rest of the app" but E-040 actually ships plain `globals.css` + inline styles — no Tailwind installed. Stayed consistent with the existing scaffold rather than introducing a Tailwind dependency that none of E-041, E-042, or E-043 currently need. If a later experiment installs Tailwind, the welcome's inline styles are trivial to migrate.
- **Used the native `<dialog>` element with `showModal()` / `close()`** rather than a custom div+overlay. This aligns with R-017's choice for E-044's resource modal (focus trap, escape, backdrop click, ARIA modal role all free) and means the codebase has one modal pattern, not two. E-045 ships the pattern first.
- **HelpIcon dispatches a `window.dispatchEvent(new Event('open-welcome'))`** rather than sharing state via Zustand or a context. Welcome subscribes via `window.addEventListener`. This keeps both components as siblings under `layout.tsx` with no shared parent or store needed. If multiple consumers of the help-open signal appear later, lifting to Zustand is a one-screen refactor.
- **Initial render returns `<></>` until the localStorage read completes** (state starts as `null`, the `useEffect` flips it to a definite boolean). Without this gate, returning users would briefly see the welcome modal flash open on hydration before the localStorage read demoted it. The cost is one render of empty fragment for new users; trivial.
- **Button label switches between "I got it →" (first-visit) and "Close" (help re-open).** The first-visit phrasing is the spec's wording; the post-acknowledgement phrasing avoids the "I got it" claim once they already have. Determined by `hasAcknowledged` state, set on first dismissal.

**localStorage state machine (verified by code inspection — no browser available to drive interactively):**

```
INITIAL              hasAcknowledged=null, isOpen=false → render <></>
                     useEffect reads localStorage
NEW USER             hasAcknowledged=false, isOpen=true → dialog opens, focus on button
  click "I got it →" hasAcknowledged=true, localStorage set, isOpen=false → dialog closes
  reload             hasAcknowledged=true (from storage), isOpen=false → dialog hidden, button rendered
  click ? icon       'open-welcome' event → isOpen=true → dialog opens with "Close" label
  press escape       dialog 'close' event → isOpen=false → dialog hidden
RETURNING USER       hasAcknowledged=true on mount, isOpen=false → dialog hidden, button rendered
```

The state machine handles three reentry paths uniformly: explicit dismiss-click, escape key (via dialog `close` event), and backdrop click (treated the same as escape). Each fires `onClose` which both syncs React state and writes the localStorage flag.

**What was not done (intentional):**

- No automated browser test. Without a Chrome/jsdom dep already in the repo, adding one for this single component would balloon scope. The state-machine logic is small, the verification via SSR HTML inspection + production-chunk grep + regression test on `/api/chat` covers the load-bearing checks. The first user to actually open the app in a browser is the live test. If interactive flakiness shows up later, install vitest+jsdom then.
- No copy editing pass on the welcome text. Spec's literal copy used verbatim (with appropriate HTML entity escaping for apostrophes and en-dashes). User can iterate copy in a follow-up edit if desired; the structure is correct.
- No analytics opt-in surface. The spec explicitly defers anonymous feedback mechanism. §3 mentions it as a future possibility — that is the entire "future framing" commitment per the goal's decision log.

**Architecture observations:**

The native `<dialog>` + window-event channel is a noticeably lighter pattern than what most React tutorials reach for (typically: portals, focus-trap library, custom escape-handler, click-outside handler). Roughly 200 lines across two components plus a 4-line CSS rule, vs. typical 400+ lines + a focus-trap dep. R-017 anticipated this for E-044 in a more complex form; E-045 validates the pattern works at a smaller scale first.

The HelpIcon is intentionally a fixed-position button rather than part of a future top-bar layout. If E-042 (wiki view) introduces a header bar with navigation, the help icon would migrate into that header. For now it sits in the top-right corner of any page and survives layout changes downstream.
