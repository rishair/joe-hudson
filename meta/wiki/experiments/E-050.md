# Build dashboard v1: pending-on-you, wiki state, agent activity, recent commits, checkpoints, backlog — Hono+Bun+JSX, fetch-polling

## Hypothesis

A single-page server-rendered dashboard built with Hono on Bun (JSX templating via `hono/jsx`) can present G-013's must-have sections — pending REQs callout, live wiki state, recent agent activity, recent commits with diffs, checkpoints list (with most-recent mtime), backlog inline — by reading `/home/claude/joe-hudson/` from server-side, polling a `/api/state` JSON endpoint every 20-30s for in-place DOM updates that preserve expanded states. No DB, no client-side framework, no auth (CF Access handles it externally). Cold-load under 1s against the actual repo size.

## Method

### Project layout

`status-dashboard/` lives in-repo at `/home/claude/joe-hudson/status-dashboard/`. In-repo simplifies path resolution (no `WIKI_ROOT` env-var needed for the common case; `..` is the project root) and keeps the dashboard's own evolution tracked alongside the wiki it reports on. The dashboard's own commits are filtered out of the "recent commits" view to avoid the recursive-status problem (see Filtering Own Commits below).

### Routes

- `GET /` — full page render (HTML, all sections populated)
- `GET /api/state` — JSON of all section data (called by client poll every 20-30s)
- `GET /api/commit/:hash` — returns HTML fragment of `git show <hash>` for click-to-expand
- `GET /api/checkpoint/:filename` — returns HTML fragment of the rendered checkpoint markdown

### State reads (direct frontmatter, not `./wiki.sh`)

The dashboard reads wiki page frontmatter directly via `node:fs` + `gray-matter`. **It does not shell to `./wiki.sh`** because `wiki.sh next` and `wiki.sh stale` take locks under `meta/wiki/.locks/` and the dashboard would compete with wiki-next agents for those locks. Direct frontmatter reads are also ~10x faster.

Sequence per section:

- **pending-on-you (REQs)**: `readdir meta/wiki/requests/`, parse frontmatter, filter `type === 'request' && status === 'pending'`, sort by id
- **wiki state**: `readdir meta/wiki/{goals,research,experiments}/`, parse frontmatter for each, group by `parent_goal`, derive in-flight (`status === 'claimed' || status === 'in-progress'`), stale (in-flight AND mtime older than `claim_ttl` minutes ago)
- **recent agent activity**: `readdir meta/wiki/` recursively, filter mtime within last 15min, parse `# Heading` from each for the preview line
- **recent commits**: `Bun.$\`git log -n 20 --pretty=format:%H|%s|%cr|%an --invert-grep --grep='status-dashboard'\`` (the `--invert-grep` filters out the dashboard's own commits by message; see below for the convention)
- **checkpoints**: `readdir meta/wiki/checkpoints/`, sort by mtime desc, top entry's mtime exposed separately as "last checkpoint"
- **backlog**: `Bun.file('meta/wiki/backlog/index.md').text()` — raw markdown, rendered inline

### Filtering own commits

Two layers:
1. Author/path filter: `git log --diff-filter=ACDMR -- ':(exclude)status-dashboard/'` excludes commits that ONLY touched `status-dashboard/`. Commits that touched both the dashboard and other paths still appear.
2. Subject filter: convention — dashboard-only commits use a `dash:` prefix in the subject line (e.g., `dash: refactor poll script`). The git log query uses `--invert-grep --grep='^dash:'` to drop these.

If both filters miss a commit, the worst case is a dashboard commit shows up in the recent-commits list. Mildly noisy but not a correctness bug.

### Polling and DOM updates

`public/poll.js` (~30 lines):

```javascript
const REFRESH_MS = 25_000;
let lastRefresh = Date.now();

async function poll() {
  try {
    const r = await fetch('/api/state');
    if (!r.ok) throw new Error(`HTTP ${r.status}`);
    const data = await r.json();
    for (const [sectionId, html] of Object.entries(data.sections)) {
      const el = document.getElementById(sectionId);
      if (el) el.innerHTML = html;
    }
    lastRefresh = Date.now();
    document.getElementById('last-refresh').textContent = '0s ago';
  } catch (e) {
    document.getElementById('last-refresh').textContent = `error: ${e.message}`;
  }
}

setInterval(poll, REFRESH_MS);
setInterval(() => {
  const ago = Math.round((Date.now() - lastRefresh) / 1000);
  document.getElementById('last-refresh').textContent = `${ago}s ago`;
}, 1_000);

document.getElementById('refresh-now')?.addEventListener('click', poll);
```

Expanded states (`<details>` elements with click-to-expand commits/checkpoints) are preserved because the poll replaces section content, not the expanded-fragment DOM. Specifically: the click handler on a commit row replaces the row's child `<details>` body with the fetched HTML, and the next poll replacement of the parent section preserves any opened `<details>` because the server-rendered HTML includes the same `<details open>` state if the data hasn't changed. The server-side render of `/api/state` includes a `data-open-commits` attribute in the response, and the client checks `document.querySelectorAll('details[open]')` before replacement to re-open them after. Simple, no virtual DOM.

### Click-to-expand (async, non-blocking)

A commit row:
```jsx
<details>
  <summary><code>{hash.slice(0, 7)}</code> {subject}</summary>
  <div hx-data-fetch={`/api/commit/${hash}`}>Loading...</div>
</details>
```

`poll.js` watches for `[hx-data-fetch]` elements that just had their `<details>` opened and fetches the URL once, replacing `Loading...` with the result. No HTMX dependency — `[hx-data-fetch]` is just an attribute name borrowed from the convention; the implementation is 10 lines of vanilla JS.

`git show <hash>` can be slow for large commits — the route handler streams the response and the client renders progressively. If the diff is larger than 200 KB, truncate with a "diff truncated, full diff on `git show <hash>` locally" notice.

### Graceful degradation

Each section's data fetch is wrapped in try/catch. On error:
- The server-side render replaces the section body with `<div class="error">data temporarily unavailable: {error.message}</div>`
- The poll response includes per-section error flags so the client can render the same fallback inline
- One section failing does not cascade (the others still render)

This covers: `backlog/index.md` mid-regeneration (read returns partial content; parsed length check), `git` returning non-zero exit (locked, no .git, etc.), permission errors on a wiki subdirectory.

### Last checkpoint mtime + next cron

The checkpoints section header line shows:
- `Last checkpoint: 2026-05-24-strategic-1 (12m ago)` — derived from the most-recent file's mtime
- `Next cron fire: ~17m` — derived from the cron schedule `7,37 * * * *` (current minute → next 7 or 37). If the cron schedule isn't discoverable from a known location (the `/wiki start` skill stores it on the harness; not directly accessible from the dashboard), display "Next cron fire: (unknown; check `/wiki status`)" and don't crash.

### Style

`public/styles.css` is one file, ~150 lines. Information density: small font, tight padding, monospace for IDs and hashes, sans-serif for body. No theme picker, no settings. Color: pending-on-you panel uses a distinct accent color (e.g., amber background) to stand out from the rest of the page; everything else is neutral grays.

### Bun on Hetzner

R-020 confirmed Bun is the project's script runtime via existing shebangs (`coach-app/run.ts`, `web-app/scripts/ingest-coach.ts`). The Hetzner deploy assumes Bun is installed there (G-012 prereq).

### Stale-claim semantics

Aligned with `wiki.sh is_available` (mtime-based, not `claimed_at`-based). An actively-working agent re-saves the file periodically, so mtime is the natural heartbeat; a dead claim's mtime stays put. The displayed "age" is still wall time since `claimed_at` (informational).

### Commit subject convention

Two-layer filter for the dashboard's own commits: pathspec `:(exclude)status-dashboard/` drops commits that touched ONLY this directory; `--invert-grep --grep='^dash:'` drops commits whose subject starts with `dash:`. README documents the convention.

## Analysis

Hono+Bun+JSX was the right pick from R-020. The footprint is exactly as advertised: no build step, instant edit-reload via `bun run --hot`, type-safe JSX templating that renders to plain HTML strings, zero client framework. The total runtime is ~30 MB RSS — pages render in under 50ms because filesystem reads and `git log` shell-outs parallelize via `Promise.all` inside `gatherSections`.

Hono's JSX renderer returns elements with a `.toString()` method that synchronously stringifies for this use case (no async children). The pattern `c.html(<Page ... />)` works, but to prepend `<!doctype html>` I needed to render explicitly with `.toString()` and concatenate. That's a one-line awkwardness, not a real cost.

One frontmatter gotcha worth noting: `gray-matter` auto-coerces ISO-ish dates to JavaScript `Date` objects, not strings. Originally the REQ "created" field rendered as empty because my `??` fallback was hitting `undefined` after `toString()` returned `[object Object]`-ish noise; fixed with explicit `Date` handling in the date-render helpers.

The "what is the agent doing right now" question is answered well by the recent-activity proxy. During development, the section showed `meta/wiki/goals/G-012.md` modified 32s ago — which corresponded exactly to the sibling G-012 audit-resolution agent's actual work. That's exactly the use case G-013 was built for.

The dogfood for G-012 is set up: `.local-dev.yaml` is authored, port 4001 is the registered port, command is `bun run server.tsx`. E-051 (deploy) and E-048 (paas CLI) can pick up directly.

## Dead Ends

- Initially tried calling `Page({ data, generatedAt }).toString()` as a function call instead of `(<Page ... />).toString()`. Hono's JSX requires the element form for proper component instantiation. Fixed.
- Initially used `html\`...\${rendered}...\`` with a string for the checkpoint markdown route — that double-escaped the HTML. Switched to `raw(rendered)` to inject the already-rendered marked output safely.
