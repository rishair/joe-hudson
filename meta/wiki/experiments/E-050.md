# Build dashboard v1: 4 sections (wiki state, recent commits, checkpoints, backlog), single page, server-side filesystem reads

## Hypothesis

A single-page dashboard following R-020's stack choice can present G-013's 4 must-have sections — live wiki state, recent file changes with diffs, checkpoints list, backlog inline — by reading `/home/claude/joe-hudson/` from server-side. No DB, no client-side state library, no auth (CF Access handles it externally). Cold-load in under 1s, refresh-friendly, glance-readable.

## Method

1. Read R-020's stack pick. Use the recommended stack.
2. Project layout (assuming Bun-HTTP if R-020 picked simpler; adjust for Next.js):
   ```
   status-dashboard/
     bin/server.ts         # Bun.serve entrypoint, single port (defaulted by .local-dev.yaml)
     lib/wiki-state.ts     # Reads claims, in-flight, stale, requests; returns structured JSON
     lib/git-recent.ts     # Shells to git log + git diff for recent commits
     lib/checkpoints.ts    # Lists meta/wiki/checkpoints/*.md, parses frontmatter, returns sorted list
     lib/backlog.ts        # Reads meta/wiki/backlog/index.md (already auto-generated)
     templates/index.html  # Or .tsx component if Next.js
     .local-dev.yaml       # For G-012 paas to pick up
     README.md
     package.json
   ```
3. Each section reads the project filesystem directly. Path is configurable via `WIKI_ROOT` env var (defaults to `/home/claude/joe-hudson`).
4. Wiki-state section reuses the logic from `./wiki.sh next` and `./wiki.sh stale` — call the script via subprocess, parse output. Bonus: a shared TypeScript type for the parsed state so future tools can consume it.
5. Recent-commits section: `git log -n 20 --pretty=format:'%H|%s|%cr|%an'` for the list; click on a commit POSTs to `/api/commit/:hash` returning `git show <hash> --stat`. (If Bun-HTTP, /api routes are just URL prefixes in the server's request handler.)
6. Checkpoints: list `meta/wiki/checkpoints/*.md`, extract `kind:` from frontmatter (operational vs strategic), render most-recent-first with click-to-expand showing the rendered markdown.
7. Backlog: render `meta/wiki/backlog/index.md` inline (it's already a curated view).
8. Auto-refresh: simple page-level `<meta http-equiv="refresh" content="30">` for v1. Polling beats SSE for this read-only use case.
9. Style: one CSS file (or Tailwind if Next.js). Information density over chrome. No nav.
10. Local test: run `bun run start` from the project dir; visit `localhost:<port>`; verify all 4 sections render with real data.
11. The G-013 nice-to-haves (drill-down, spend tally, etc.) are explicitly NOT in this experiment — they're follow-ups once v1 is live and the user has actually used it for a day.

## Success Criteria

- `bun run start` launches the server on a configurable port
- Visiting the URL shows 4 sections with real data from `/home/claude/joe-hudson/`
- Wiki-state section accurately reflects current `./wiki.sh next` / `./wiki.sh stale` output
- Recent-commits section shows the last 20 commits; click expands to the diff
- Checkpoints section lists checkpoints, most-recent-first; click expands to markdown
- Backlog section shows the auto-generated backlog index
- Page loads in under 1 second
- `.local-dev.yaml` is authored and ready for G-012's `paas register`
- README documents how to run it locally and what env vars it expects

## Blast Radius

- Adds `/home/claude/joe-hudson/status-dashboard/` project directory
- Reads `/home/claude/joe-hudson/` filesystem (read-only)
- Shells to `git` and `./wiki.sh` (read-only operations)
- Does NOT modify any wiki content, repo state, or other projects
