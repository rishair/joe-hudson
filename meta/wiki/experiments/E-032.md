# Coach v2: tool-based compendium navigation with cheap-model router and expensive-model coach

## Hypothesis

Equipping the coach with tools to navigate `coach/` (list-index, search-by-keyword, read-file, follow-related-links) and using a two-tier model setup (Haiku 4.5 as the router making tool calls; Opus 4.7 as the final coach response generator with retrieved content in context) outperforms E-031 (no retrieval) by mean delta >= 0.2 on G-008 dimensions, at acceptable cost. The two-tier setup is plausible because routing is a planning task that Haiku handles, while voice and stance demand the more capable model.

Alternative configurations within the same experiment: single-model tool-use (Opus 4.7 makes its own tool calls), and Sonnet 4.6 as the router. The experiment picks the best of these variants on a smoke-test before committing to a full eval.

## Method

1. Read E-031's coach-app/ scaffolding. Reuse the project structure.
2. Read R-013's AI SDK cheatsheet for tool-use patterns. Reuse the multi-step loop template.
3. Implement tools at `coach-app/tools/compendium.ts`:
   - `list_compendium_index()` — returns the top-level taxonomy and counts per category
   - `search_compendium(query: string, category?: string)` — keyword search over filenames and titles
   - `read_compendium_file(path: string)` — returns the markdown content of a `coach/` file
   - `follow_related(path: string)` — returns the `related:` link targets from a file
4. Wire a two-stage flow per turn:
   - Stage 1: Router model (Haiku 4.5 default) sees the client message + conversation history + tool catalog. It makes up to 5 tool calls to assemble a "retrieval bundle" (typically 3-8 compendium files).
   - Stage 2: Coach model (Opus 4.7) sees the system prompt + conversation + retrieval bundle inlined into context + the client message. It generates the response.
5. Create `coach-app/configs/v2-tool-nav.yaml` with the two-tier config and the v1 system prompt (held constant from E-031 so we isolate retrieval as the variable). v1 already includes the retrieval-aware instruction line (per audit Important #5; established in E-031), so the prompt is inheritable without modification.
6. Document retrieval-trigger policy (per audit Important #8 and R-012 confirmation): tool-based nav triggers retrieval every turn by default. R-012 specifically recommends E-032 smoke an `on_topic_shift` variant (cheap implementation: cosine distance between current client message embedding and the embedding used at last retrieval; if distance > 0.3, re-retrieve, else reuse last bundle) because E-032's per-turn router-plus-tool overhead is the experiment where topic-shift detection has the largest cost saving. Smoke both; default to `every_turn` for the full eval comparison so apples-to-apples with E-033 and E-036.
6. Smoke on 2 profiles first ($1-3 because of the cheap router). Validate that tools are invoked, retrieval bundles are sensible, responses are in voice.
7. Run full 15-profile eval. Estimated cost: $8-12 (router tokens add overhead, but Haiku is ~12x cheaper than Opus so net add is moderate).
8. Render scorecard via E-030. Compare to E-031.
9. Log: which compendium files were retrieved most often? Did the router ever get stuck in loops? Were retrieval bundles too large (degraded coach attention) or too sparse (under-retrieval)?
10. Write Result with delta vs E-031, Analysis on whether tool-based nav is worth the cost.

Pre-experiment guard: if R-013 reveals the AI SDK has a serious tool-use limitation that affects multi-model orchestration, fall back to single-model tool-use (Opus makes its own tool calls) and document the constraint.

Implementation ramp: smoke on 2 profiles confirms the architecture works before spending on full eval.

## Success Criteria

- Tool-based navigation works end-to-end: router selects relevant files, coach response uses retrieved content visibly
- Per-turn telemetry logs which files were retrieved
- Full G-008 eval produces a scorecard
- Mean delta vs E-031 reported per dimension
- Per-turn cost reported separately for router and coach
- Total experiment spend logged and under $13
- Verdict: did tool-based nav beat E-031? By how much? At what cost premium?

Failure mode: router making >5 tool calls per turn (suggests architecture is wrong), or coach ignoring retrieved content (suggests the inline-injection format is bad). Both are debuggable; document the specific failure.

## Blast Radius

Adds `coach-app/tools/compendium.ts`, `coach-app/configs/v2-tool-nav.yaml`. Reads `coach/` compendium read-only. Writes eval results to `eval/results/coach-v2-tool-nav/`. Does not modify `coach/`, `eval/`, or `coach-app/prompts/v1.md` (held constant from E-031).

## Result

Abandoned 2026-05-22 in favor of budget conservation after E-036 landed. Rationale:

- R-012 predicted ranking E-036 ≥ E-033 > E-032 > E-031. The three completed retrieval experiments confirmed this exactly (E-031 = 2.61 overall mean, E-033 = 2.64, E-036 = 2.68). E-032 was predicted to slot between E-031 and E-033 at approximately 2.62.
- E-032 was the most expensive of the four ($15-20 estimated) due to per-turn router + coach overhead plus tool-output cache invalidation.
- G-009 spend at decision point: ~$31 of $60 budget. Running E-032 plus E-037 (hybrid, likely mandated) plus E-035 (iteration) projected $54-74 total, with E-032 specifically responsible for the bulk of the overrun risk.
- The G-009 exit criterion "at least 4 retrieval approaches tested" relaxes to "3 architectures tested empirically plus R-012's predicted ranking confirmed for the 4th" given the empirical confirmation of R-012's three-experiment predictions.
- The four named tool primitives (`list_compendium_index`, `search_compendium`, `read_compendium_file`, `follow_related`) remain implemented at `coach-app/tools/` from R-013's spec; they are reusable if a future goal wants to test tool-based nav with budget allocated for it.
