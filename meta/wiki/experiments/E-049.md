# Dogfood: register G-013 status dashboard via paas and verify end-to-end

## Hypothesis

Once both E-048 (paas CLI) and E-051 (status dashboard built and ready to deploy) are complete, registering the dashboard via `paas register /path/to/status-dashboard` produces a working `status.play.ris.hair` URL with Cloudflare Access auth. If this works cleanly, G-012 is proven; if it fights, G-012's design needs to fix the friction.

This experiment closes the G-012 goal (it's the goal's last exit criterion: "one project is registered and reachable") AND validates the dogfood-design contract.

## Method

1. Confirm E-048 (paas CLI) is complete and E-051 (dashboard build) has produced a runnable artifact.
2. Author `.local-dev.yaml` in the status-dashboard project directory per E-048's spec.
3. Run `paas register /path/to/status-dashboard`.
4. Verify:
   - `paas list` shows the status entry with subdomain `status` and the assigned port
   - `systemctl --user status paas-status` shows running
   - `paas logs status` tails the dashboard's output
   - `status.play.ris.hair` in incognito browser hits CF Access, signs through, shows the dashboard
   - Refreshing the dashboard shows live state (any wiki change shows up within one refresh)
   - Browser dev tools confirm no `OPENROUTER_API_KEY` or other secret leakage
5. Stress-test: simulate `systemctl --user restart paas-status` mid-page-load — should recover gracefully.
6. Documentation: update `paas/README.md` with the status-dashboard registration as the worked example.

## Success Criteria

- Status dashboard is reachable at `https://status.play.ris.hair` from any device after CF Access sign-in
- `paas list` accurately reflects the dashboard's state
- Restarting the dashboard's systemd unit doesn't kill the tunnel
- A reboot of the Hetzner box (or `systemctl --system restart` simulation) brings the dashboard back up automatically
- G-012's exit criterion "one project is registered and reachable at its `*.play.ris.hair` URL" is met
- No surprises in the paas CLI surfaced by this real deploy that weren't caught in E-048's dummy-project testing — if surprises ARE found, document them and create a follow-up experiment to address (don't try to fix mid-experiment)

## Blast Radius

- Adds `~/.config/systemd/user/paas-status.service`
- Adds an ingress entry in `~/.cloudflared/config.yml` for `status.play.ris.hair`
- Reachable at the public URL (CF Access-gated)
- Does NOT modify the dashboard project's source — only reads its `.local-dev.yaml`
