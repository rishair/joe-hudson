# Set up Cloudflare Tunnel + Access on Hetzner with a hello-world subdomain

## Hypothesis

Following R-019's recipe, a Cloudflare Tunnel can be installed on the Hetzner box as a `systemd` system service, routed to a single hello-world local port, and protected by Cloudflare Access. End state: hitting `hello.play.ris.hair` in an incognito browser shows the CF Access SSO page; signing in with the user's Google account passes through to a "hello world" page served from a local port. A simulated reboot confirms the tunnel comes back AND a `systemctl --user`-managed test service is reachable through it within 30 seconds.

This establishes the infrastructure foundation before building the CLI in E-048.

## Method

### Pre-flight

1. Verify [[REQ-004]] is resolved: `ris.hair` on Cloudflare DNS, SSL coverage path chosen, Zero Trust enabled, IDPs set up, API token (or dashboard-only path) committed to.
2. Read [[R-019]] in full — the "Cloudflared as a systemd service", "Cloudflare Access setup for `*.play.ris.hair`", "Boot survival" and "SIGHUP reload behavior" sections are load-bearing.

### Linger enablement (audit rec #1 — must be done BEFORE any project units)

3. Run `sudo loginctl enable-linger $USER`.
4. Verify: `loginctl show-user $USER | grep Linger=yes` returns a match. If not, fix before proceeding.

### Sudoers entry for the CLI's restart capability

5. Add the line `<user> ALL=(ALL) NOPASSWD: /bin/systemctl restart cloudflared` to a new file `/etc/sudoers.d/paas`. Validate with `sudo visudo -cf /etc/sudoers.d/paas` (should report "parsed OK").

### Cloudflared install + tunnel creation

6. Install `cloudflared` on the Hetzner box: `curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64.deb -o cf.deb && sudo dpkg -i cf.deb`.
7. Authenticate: `cloudflared tunnel login` (opens browser to authorize against `ris.hair` zone).
8. Create the tunnel: `cloudflared tunnel create paas`. Note the UUID printed.
9. Create the wildcard DNS route ONCE (depends on the SSL decision in REQ-004):
   - If SSL path (A) — ACM enabled: `cloudflared tunnel route dns paas '*.play.ris.hair'`
   - If SSL path (B) — flattened scheme: `cloudflared tunnel route dns paas '*-play.ris.hair'`

### Config + service setup

10. Set up the symlink layout:
    - `mkdir -p /home/<user>/paas/cloudflared`
    - `mv ~/.cloudflared/* /home/<user>/paas/cloudflared/`
    - `sudo ln -s /home/<user>/paas/cloudflared /etc/cloudflared`
11. Write the initial `/home/<user>/paas/cloudflared/config.yml`:
    ```yaml
    # Generated by paas (initial hello-world setup; will be managed by `paas register` going forward).
    # DO NOT HAND-EDIT after E-048 ships — run `paas <subcommand>` to modify.
    tunnel: <TUNNEL_UUID>
    credentials-file: /etc/cloudflared/<TUNNEL_UUID>.json
    ingress:
      - hostname: hello.play.ris.hair   # or hello-play.ris.hair under SSL path (B)
        service: http://localhost:8080
      - service: http_status:404
    ```
12. Validate: `cloudflared tunnel ingress validate --config /etc/cloudflared/config.yml`. Should report "OK."
13. Install as system service: `sudo cloudflared service install`. This creates `/etc/systemd/system/cloudflared.service` and starts it.
14. Verify: `sudo systemctl status cloudflared` shows active+running. Tunnel connections to Cloudflare's edge are listed in the journal: `journalctl -u cloudflared --since "1 minute ago"`.

### Hello-world origin

15. Write a 5-line Bun HTTP server in `/home/<user>/paas/hello/`:
    ```typescript
    // server.ts
    Bun.serve({ port: 8080, fetch: () => new Response('hello from play.ris.hair') });
    ```
16. Create a `systemctl --user` test unit to verify the user-scope boot-survival story:
    - `~/.config/systemd/user/paas-hello.service` with `[Service] ExecStart=/usr/bin/bun /home/<user>/paas/hello/server.ts`, `Restart=always`, `RestartSec=2s`.
    - `systemctl --user daemon-reload && systemctl --user enable --now paas-hello.service`.
    - Verify locally: `curl http://localhost:8080` returns "hello from play.ris.hair".

### Cloudflare Access setup

17. Follow the dashboard recipe in R-019's "Concrete Access policy recipe":
    - Login methods: Google + One-time PIN
    - Self-hosted app: name "play.ris.hair PaaS", domain `*.play.ris.hair` (or `*-play.ris.hair` if path B), 24h session
    - Policy: Allow / Emails / `<user-email>`

### Verification

18. From an incognito browser, hit `https://hello.play.ris.hair` — should display the Cloudflare Access login page.
19. Sign in with Google as the policy-included email — should land on "hello from play.ris.hair".
20. From a different incognito browser, sign in with a different Google email — should be denied at the Access layer.
21. On the Hetzner box: `sudo netstat -tlnp | grep -v '127.0.0.1\|::1\|sshd\|cloudflared'` should be empty (no new public-bound ports added for the tunnel; only outbound connections from cloudflared).

### Reboot survival (added per audit rec #1)

22. Snapshot the system state: capture `systemctl is-active cloudflared`, `systemctl --user is-active paas-hello.service`, the output of `curl https://hello.play.ris.hair` after CF Access auth.
23. `sudo systemctl reboot` (or, if a real reboot is not desired, simulate the effect: `sudo systemctl restart cloudflared user@$(id -u)`).
24. Wait 30 seconds.
25. Verify: `systemctl is-active cloudflared` returns "active". `systemctl --user is-active paas-hello.service` returns "active" (this is the linger payoff). `curl https://hello.play.ris.hair` after CF Access auth returns the same hello message.
26. If any step in #25 fails, the boot story is broken — investigate per R-019's "Boot survival" section before declaring the experiment succeeded.

## Success Criteria

- `systemctl status cloudflared` shows active+running, survives `systemctl restart cloudflared` and a `systemctl reboot`
- `hello.play.ris.hair` (or the path-B equivalent) resolves via Cloudflare and serves a CF Access page to unauthenticated requests
- User can sign in via Google SSO and reach the hello-world page
- Anyone else hitting the URL stays at the Access page (verified by signing in with a different account — should be denied)
- `netstat -tlnp` on the Hetzner box shows no new public-bound ports added for the tunnel (only outbound `cloudflared` connections)
- `/etc/cloudflared/config.yml` (or its symlinked location) is in place with the "Generated by paas" header
- `loginctl show-user <user>` reports `Linger=yes`
- The sudoers entry `/etc/sudoers.d/paas` parses cleanly with `visudo -cf`
- **Reboot test passes**: after `systemctl reboot` (or its simulation), within 30 seconds the tunnel is back AND the `systemctl --user`-managed `paas-hello` service is reachable through it. This validates the boot-ordering story.

## Blast Radius

- Installs `cloudflared` system-wide on Hetzner (small dep, well-trusted)
- Adds a `systemd` system service: `cloudflared.service`
- Adds Cloudflare-side resources: a Tunnel, a wildcard DNS record for `*.play.ris.hair` (proxied), an Access application + policy
- Adds `/home/<user>/paas/cloudflared/` (symlinked from `/etc/cloudflared/`) with `config.yml` and the credentials JSON
- Adds `/etc/sudoers.d/paas` granting passwordless `systemctl restart cloudflared` to the user
- Enables linger for the user (affects every `systemctl --user` unit henceforth — desired side effect)
- Adds `~/.config/systemd/user/paas-hello.service` (test unit; may be removed by the cleanup step in [[E-051]] — the merged successor of E-049)
- Adds `/home/<user>/paas/hello/server.ts`
- Does NOT modify any existing project, port, or DNS record outside `play.ris.hair`
