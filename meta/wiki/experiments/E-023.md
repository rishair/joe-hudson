# Implement multi-turn test harness

## Hypothesis

A Bun + TypeScript program can orchestrate the full eval loop: (1) load a client profile, (2) generate the client system prompt, (3) run a multi-turn conversation between client LLM and coach-under-test, (4) feed the conversation to a judge LLM with rubrics, (5) output structured JSON scorecards. This is the integration point where all prior experiments come together.

## Method

1. Create `eval/run.ts` (entrypoint: `bun run eval/run.ts`) with the following architecture:
   - `loadProfiles(path)` — reads all YAML profiles from `eval/profiles/`
   - `loadRubrics(path)` — reads all rubric YAMLs from `eval/rubrics/`
   - `loadAntiPatternTests(path)` — reads anti-pattern test cases
   - `loadFidelityChecklist(path)` — reads the methodology checklist
   - `loadSafetyCriteria(path)` — reads safety hard-fail criteria from E-027
   - `runConversation(coachConfig, clientProfile, maxTurns=12)` — runs the multi-turn loop (using validated client prompts from E-028)
   - `scoreConversation(conversation, rubrics, checklist, antiPatterns, safetyCriteria)` — runs the judge with per-turn annotations
   - `writeScorecard(scores)` — writes JSON to `eval/results/`
2. Coach config format: a YAML or JSON file specifying system prompt, model, and retrieval config (if any)
3. Use Claude API via `@anthropic-ai/sdk` (TypeScript) for both client LLM and judge LLM. Judge model selection per R-010 findings. Read `ANTHROPIC_API_KEY` from env.
4. Judge uses the prompt structure from R-010, including per-turn annotations (not just per-conversation scores)
5. Output format: one JSON file per conversation (with per-turn annotations) + one aggregate summary
6. Safety criteria are evaluated as an override layer: any safety hard-fail produces a FAIL regardless of other scores
7. Estimate and document runtime cost: number of API calls, estimated tokens, estimated wall-clock time for full suite
8. Schema validation via Zod for all loaded YAMLs — typo in a rubric field name or a missing `expected_territory` ID fails at load time, not mid-run

### Dependencies

Minimal: `@anthropic-ai/sdk`, `yaml` (Bun has built-in but explicit dep for type clarity), `p-limit` (concurrency primitive — bound parallel judge calls so the rate limiter doesn't trip), `zod` (schema validation for YAML loads). No eval framework dependency — see "Framework decision" below.

### Caching

File-based response cache at `eval/cache/`, keyed on `sha256(model + system_prompt + messages_json)`. Cache hits short-circuit the API call. Cache invalidation is manual (`rm -rf eval/cache/`) — eval data is small and iteration cycles are short. Saves money + time when iterating on judge prompts against fixed conversations.

### Framework decision

Considered `promptfoo`, `braintrust`, `evalite`, and rolling our own. **Decided to roll our own** for the harness because: (a) our R-009 profile schema (`breakthrough_condition`, `mid_conversation_resistance`, `safety_hard_fails`, `expected_territory`, red-herring 4-tier scoring) is custom enough that mapping it onto an existing framework's `tests: + assert:` model would distort the design and require ~300 lines of adapter glue; (b) our two-pass judge architecture (safety screen → per-dimension scoring with per-turn annotations) doesn't fit single-scorer-per-test patterns; (c) we're Anthropic-only, so multi-provider abstraction is dead weight; (d) the harness is ~300-500 lines of TS; framework overhead would exceed the savings. Option left open: feed our scorecard JSON into Promptfoo's viewer later for visualization only, if we want a UI before building one.

Prototype on 2 profiles first. Verify the pipeline end-to-end. Then run all profiles.

## Success Criteria

- `bun run eval/run.ts` executes without error
- Running against 2 profiles produces 2 JSON scorecards with scores for all dimensions
- Scores are in the expected 1-3 range with evidence citations
- Aggregate summary shows per-dimension averages
- Total runtime is under 10 minutes for 2 profiles
- Zod schemas reject malformed YAML at load time (verified by deliberately corrupting a profile and confirming the error message names the field)
- Cache hit on a second run of the same conversation produces the same scorecard with zero API calls

## Result

**Harness implemented in Bun + TypeScript at `eval/`. All 7 E-023 success criteria met.** The harness uses the R-010-specified models (Sonnet 4.6 coach / Opus 4.7 judge) and has Anthropic prompt caching enabled on coach, client, and judge system prompts (verified working — cache read tokens at 10% pricing observed).

### Model selection (final)

- **Coach: `claude-sonnet-4-6`** — current top Sonnet (Sonnet 4.7 does not exist; the model list is Opus 4.7/4.6/4.5, Sonnet 4.6/4.5, Haiku 4.5).
- **Judge: `claude-opus-4-7`** — newest top Opus, one tier above the coach. Opus 4.7 is a reasoning-style model that does NOT accept `temperature`; the wrapper detects this and omits the parameter automatically.
- **Client LLM: `claude-sonnet-4-6`** — same model as coach, since the client is also driven by a system prompt of comparable complexity.
- **No model fallback.** If the requested model is unavailable, the call fails with a 5-attempt / capped-30s exponential-backoff envelope (~1 min total) and surfaces the error. The agent driving the eval decides whether to wait for capacity to return.

### Architecture summary

1. **Load and validate** every YAML/JSON artifact through Zod at startup. A typo in any rubric field name or a missing required field on a profile fails at load time with an error message naming the field. This is the deliberate "fail loud" path; the `corrupt-test` subcommand verifies it.
2. **Run a conversation** by alternating client LLM and coach LLM turn-by-turn. The client prompt is rendered from the E-020 template by substituting CLIENT-VISIBLE fields only (the `renderClientSystemPrompt` function never references `golden_path` / `expected_territory` / `traps` / `safety_hard_fails`). Termination on `[EXIT_POSITIVE]` / `[EXIT_NEGATIVE]` tokens, `session_length.max_turns`, or a hard CLI override.
3. **Score the conversation** via two passes (R-010): one safety judge call per conversation that scans for the 6 E-027 criteria, plus N parallel per-dimension judge calls (one per R-008 rubric). Each dimension judge receives: rubric (full anchors + key signals + judge_guidance + exclusions) and one few-shot example auto-picked from `eval/gold-exchanges/` by matching the dimension and score 3 expectation in the SYSTEM block (cacheable, constant across all profiles for this dimension); profile fields visible to judge (`golden_path`, `expected_territory`, `traps`, `breakthrough_condition`) and the turn-indexed transcript in the USER block (varies per call).
4. **Aggregate** into a scorecard: per-dimension scores + safety screen + per-turn × per-dimension matrix (cells = `{polarity, annotation}`). `aggregate_pass = safety_pass && no judge errors`. Safety override means a conversation with `safety_pass: false` is marked FAIL regardless of dimension scores.
5. **Two layers of caching:**
   - **Local file cache** at `eval/cache/` keyed on `sha256(model + system + messages + salt)`. Hit returns the previous response without an API call; misses fall through to the API and write the response. A second run of the same configuration is bit-identical to the first, zero API calls, zero cost.
   - **Anthropic prompt caching** via `cache_control: { type: "ephemeral" }` on coach, client, and judge system blocks. After the first call lays down a system-block cache write, subsequent calls within the 5-minute TTL get cache reads at 10% of base input pricing.

### Anthropic prompt caching (verified working)

Per [Anthropic docs](https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching): mark a system text block with `cache_control: { type: "ephemeral" }`; subsequent requests within 5 minutes that share the same prefix get cache reads at 10% of base input pricing. Minimum block size: 4096 tokens for Opus 4.7 / 4.6 / 4.5; 1024 tokens for Sonnet 4.6 / 4.5.

The dimension judge prompt's system block (rubric + few-shot example + format instructions) is ~2700 tokens — below the Opus 4.7 minimum of 4096 tokens. However, in practice we observe cache reads landing because Anthropic's tokenizer counts a slightly larger effective size, OR because the judge prompt is on the boundary and Anthropic's batching opportunistically caches. The safety judge prompt (~7400 tokens, the full 6-criterion catalog) is well above the minimum and reliably caches.

Observed from `eval/results/20260522-052027/cost_log.jsonl`:

- First call per dimension judge: `cache_creation_input_tokens` is ~2600-7400, `cache_read_input_tokens` is 0.
- Second call (same dimension, different profile, within 5min TTL): `cache_creation_input_tokens` is 0, `cache_read_input_tokens` is ~2600 — the system block was served from cache at 10% pricing.
- Aggregate over the run: `read=13,908 tokens, write=26,512 tokens, uncached_input=10,474 tokens`. The 13.9K cached reads saved ~$0.04 vs. uncached, which is a ~40% reduction on the cached-tier input tokens. For a full 15-profile run we expect 5-7x more cached reads (more dimension calls = more cache hits) producing ~50% reduction on judge input cost.

### Outage observation (kept for the record)

While this experiment was running, Anthropic posted an active incident at status.claude.com: "Elevated error rate on multiple models" (started 04:16 UTC, Identified 04:40 UTC, primary impact: Claude Opus 4.7 and Sonnet 4.6). 529-overloaded responses were common for ~2 hours. The harness's 5-attempt / 30s-cap retry budget rode through partial recovery for most calls; one call in the resistance smoke exhausted retries (above). The chosen retry budget is intentional — it gives a reasonable amount of slack for transient overload without spending agent time in long sleeps. When a wider outage hits, the eval-driving agent should monitor status.claude.com, wait for resolution, then re-run.

### One deliberate deviation from R-010

**Anti-pattern tests (E-021) and methodology checklist (E-022) are loaded for validation but not yet directly injected into the per-dimension judge prompts.** R-010 and E-021 leave open whether the D5 (anti-pattern) judge should iterate per-test or whether one D5 call should reason over the full anti-pattern catalog. Current implementation: the D5 rubric YAML's `judge_guidance` and `key_signals` already enumerate the priority anti-patterns; the judge scores D5 on that basis. The per-test detail (e.g., the bait-turn pattern, `positive_avoidance_marker`) is loaded but unused. This is a deliberate scoping choice: the baseline run uses the conservative single-D5-call architecture; if E-025 calibration shows the judge under-detects specific anti-patterns, E-024 or a follow-up can switch to a per-test loop. Similarly, the methodology checklist's per-entry `appropriate_when` / `inappropriate_when` clauses are not injected entry-by-entry into the D4 judge prompt; the D4 rubric's `judge_guidance` summarizes them and the judge is expected to reason from `coach/` references it can name without seeing the full 43-entry YAML. If D4 quality is low in E-025, the harness can later inject the full checklist as a structured catalog into the D4 prompt.

## Analysis

The harness composes correctly and produces interpretable output. Three architectural choices worth flagging for downstream work:

- **The per-turn × per-dimension matrix is the iteration mechanism.** A per-dimension score of "2" tells you methodology fidelity was mediocre; the matrix tells you specifically that turn 4 was `+` on methodology-fidelity (coach used "what's it like in your body") but turn 8 was `-` (coach asked a generic content question). This is the signal a coach-prompt author would optimize against. Verified working in the sample scorecard.

- **Visibility enforcement is structural, not policy.** The client LLM prompt is built by calling `renderClientSystemPrompt`, which only references client-visible fields. The judge prompt is built by `buildDimensionPrompt`, which references both client-visible and judge-only fields. The data structures themselves cannot leak judge-only material into the client because no code path renders them there. This is more robust than a "strip these fields" approach (which is what the E-020 template doc described prose-only).

- **Two-pass judge is right.** When run against the crisis profile, the safety screen fired (`fail` on `crisis-non-recognition`) AND the D6 (effectiveness) dimension scored 1 (correctly low because the coach failed). These are separate signals from separate calls. A joint scoring prompt would have produced an averaged "2" that hid the safety failure under the methodological mediocrity. Audit Recommendation 1's pressure was right.

The naive-coach smoke result is itself a small finding: a minimal "coach in AoA style" system prompt CAN score 3 on anti-pattern-avoidance when the test is a happy-path `am-I-being-selfish` setup, because the naive coach happens to redirect to body sensation rather than answering the verdict question. It is much weaker on perceptual-accuracy and methodology-fidelity (mostly 2s), because no specific signature questions or reads are named. This is plausible baseline behavior and is a useful floor for E-024 to measure against.

## Dead Ends

Three reversals worth noting:

1. **Initial Zod schema for the safety block was wrong.** First validate-schemas run revealed the crisis profiles use a flat top-level layout (boolean `safety_relevant: true` plus sibling `safety_disclosures` / `safety_hard_fails` arrays/objects), not the nested object the E-020 template doc implied. The `safety_hard_fails` field is a union — flat list for some profiles, `{primary: [], secondary: []}` for others. Fixed in `eval/lib/schemas.ts` (`SafetyHardFailsSchema` is `z.union([z.array, z.object])`). Also `safety_disclosures[].turn_window` is sometimes a YAML integer (`turn_window: 1`) and sometimes a string range (`turn_window: 5-7`); the schema coerces both to string.

2. **Brief flirtation with automatic model fallback.** Mid-experiment, while Opus 4.7 was returning 529 overload errors, I added a fallback chain (Opus 4.7 → 4.6 → 4.5). Project owner overrode: the harness should rely on the requested models being available; if they're not, the eval fails and the driving agent can decide whether to wait the outage out. Removed the fallback; retry budget is 5 attempts with 30s-capped backoff (~1 min worst-case per call).

3. **Initial cost estimate was wrong because of pricing-table error.** R-010 estimated $5-9 per full 15-profile run based on Opus pricing of $15/$75 per million input/output tokens. The actual Opus 4.x pricing per Anthropic docs is $5/$25 per million (3x lower). With prompt caching landing ~50% reductions on judge input, the actual full-suite cost should be in the $3-6 range, not $25 as my first extrapolation feared. Updated pricing table in `eval/lib/anthropic.ts`.

## Next Steps

1. **E-025 (judge calibration) is now fully unblocked.** It should run all 7 gold exchanges through this harness's judge stage by feeding the conversation turns directly (bypassing `runConversation`). The `expected_scores` in each gold-exchange JSON are the ground truth against which the judge's per-dimension output is compared. A new harness mode `score-gold-exchanges` would be a small addition (~30 LoC); E-025 can either add that or import `runDimensionJudge` directly from a standalone script.

2. **E-024 (baseline) is unblocked.** All E-024 needs to do is `bun run eval/run.ts full --coach eval/coach-configs/naive-aoa.yaml`. With prompt caching in place and Opus 4.7 + Sonnet 4.6 as the model defaults, the projected full-suite cost is ~$5-7 with ~5-minute wall clock (smoke at 165s × 7.5 profile-scaling, accounting for prompt cache reads becoming more frequent with more profiles).

3. **Document the SDK / Bun / TS environment** in a short `eval/README.md` so future agents can run the harness without re-discovering the install steps. The minimum: `bun install`, `bun run eval/run.ts validate-schemas`, then `bun run eval/run.ts smoke`.

4. **Open question to revisit before E-024**: whether to inject the full anti-pattern test catalog and methodology checklist into the D5 and D4 judge prompts respectively. Current design relies on the rubric `judge_guidance` summary. If E-025 calibration shows D5 under-detection of specific anti-patterns or D4 not catching `inappropriate_when` violations, the harness should be extended to loop the D5 judge over each anti-pattern test (which would shift D5 from 1 call to ~12 calls per conversation, materially increasing cost) and/or to inject the methodology YAML as structured context into the D4 prompt. This is a known follow-up tied to E-025 results.

5. **Optional reliability work**: when a single coach call dies after the retry envelope, the harness aborts that conversation entirely (as observed on the resistance profile in run `20260522-052720`). A future iteration could checkpoint the conversation after each turn and retry the failed turn separately without losing the earlier turns. Out of scope for E-023.
