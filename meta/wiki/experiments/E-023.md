# Implement multi-turn test harness

## Hypothesis

A Bun + TypeScript program can orchestrate the full eval loop: (1) load a client profile, (2) generate the client system prompt, (3) run a multi-turn conversation between client LLM and coach-under-test, (4) feed the conversation to a judge LLM with rubrics, (5) output structured JSON scorecards. This is the integration point where all prior experiments come together.

## Method

1. Create `eval/run.ts` (entrypoint: `bun run eval/run.ts`) with the following architecture:
   - `loadProfiles(path)` — reads all YAML profiles from `eval/profiles/`
   - `loadRubrics(path)` — reads all rubric YAMLs from `eval/rubrics/`
   - `loadAntiPatternTests(path)` — reads anti-pattern test cases
   - `loadFidelityChecklist(path)` — reads the methodology checklist
   - `loadSafetyCriteria(path)` — reads safety hard-fail criteria from E-027
   - `runConversation(coachConfig, clientProfile, maxTurns=12)` — runs the multi-turn loop (using validated client prompts from E-028)
   - `scoreConversation(conversation, rubrics, checklist, antiPatterns, safetyCriteria)` — runs the judge with per-turn annotations
   - `writeScorecard(scores)` — writes JSON to `eval/results/`
2. Coach config format: a YAML or JSON file specifying system prompt, model, and retrieval config (if any)
3. Use Claude API via `@anthropic-ai/sdk` (TypeScript) for both client LLM and judge LLM. Judge model selection per R-010 findings. Read `ANTHROPIC_API_KEY` from env.
4. Judge uses the prompt structure from R-010, including per-turn annotations (not just per-conversation scores)
5. Output format: one JSON file per conversation (with per-turn annotations) + one aggregate summary
6. Safety criteria are evaluated as an override layer: any safety hard-fail produces a FAIL regardless of other scores
7. Estimate and document runtime cost: number of API calls, estimated tokens, estimated wall-clock time for full suite
8. Schema validation via Zod for all loaded YAMLs — typo in a rubric field name or a missing `expected_territory` ID fails at load time, not mid-run

### Dependencies

Minimal: `@anthropic-ai/sdk`, `yaml` (Bun has built-in but explicit dep for type clarity), `p-limit` (concurrency primitive — bound parallel judge calls so the rate limiter doesn't trip), `zod` (schema validation for YAML loads). No eval framework dependency — see "Framework decision" below.

### Caching

File-based response cache at `eval/cache/`, keyed on `sha256(model + system_prompt + messages_json)`. Cache hits short-circuit the API call. Cache invalidation is manual (`rm -rf eval/cache/`) — eval data is small and iteration cycles are short. Saves money + time when iterating on judge prompts against fixed conversations.

### Framework decision

Considered `promptfoo`, `braintrust`, `evalite`, and rolling our own. **Decided to roll our own** for the harness because: (a) our R-009 profile schema (`breakthrough_condition`, `mid_conversation_resistance`, `safety_hard_fails`, `expected_territory`, red-herring 4-tier scoring) is custom enough that mapping it onto an existing framework's `tests: + assert:` model would distort the design and require ~300 lines of adapter glue; (b) our two-pass judge architecture (safety screen → per-dimension scoring with per-turn annotations) doesn't fit single-scorer-per-test patterns; (c) we're Anthropic-only, so multi-provider abstraction is dead weight; (d) the harness is ~300-500 lines of TS; framework overhead would exceed the savings. Option left open: feed our scorecard JSON into Promptfoo's viewer later for visualization only, if we want a UI before building one.

Prototype on 2 profiles first. Verify the pipeline end-to-end. Then run all profiles.

## Success Criteria

- `bun run eval/run.ts` executes without error
- Running against 2 profiles produces 2 JSON scorecards with scores for all dimensions
- Scores are in the expected 1-3 range with evidence citations
- Aggregate summary shows per-dimension averages
- Total runtime is under 10 minutes for 2 profiles
- Zod schemas reject malformed YAML at load time (verified by deliberately corrupting a profile and confirming the error message names the field)
- Cache hit on a second run of the same conversation produces the same scorecard with zero API calls

### Architecture summary

1. **Load and validate** every YAML/JSON artifact through Zod at startup. A typo in any rubric field name or a missing required field on a profile fails at load time with an error message naming the field. This is the deliberate "fail loud" path; the `corrupt-test` subcommand verifies it.
2. **Run a conversation** by alternating client LLM and coach LLM turn-by-turn. The client prompt is rendered from the E-020 template by substituting CLIENT-VISIBLE fields only (the `renderClientSystemPrompt` function never references `golden_path` / `expected_territory` / `traps` / `safety_hard_fails`). Termination on `[EXIT_POSITIVE]` / `[EXIT_NEGATIVE]` tokens, `session_length.max_turns`, or a hard CLI override.
3. **Score the conversation** via two passes (R-010): one safety judge call per conversation that scans for the 6 E-027 criteria, plus N parallel per-dimension judge calls (one per R-008 rubric). Each dimension judge receives: rubric (full anchors + key signals + judge_guidance + exclusions), few-shot example (auto-picked from `eval/gold-exchanges/` by matching the dimension and score 3 expectation), profile fields visible to judge (`golden_path`, `expected_territory`, `traps`, `breakthrough_condition`), and the turn-indexed transcript.
4. **Aggregate** into a scorecard: per-dimension scores + safety screen + per-turn × per-dimension matrix (cells = `{polarity, annotation}`). `aggregate_pass = safety_pass && no judge errors`. Safety override means a conversation with `safety_pass: false` is marked FAIL regardless of dimension scores.
5. **Cache** every API call (key includes model + system + messages + temperature/max-tokens salt). A second run of the same conversation is bit-identical to the first, zero API calls, zero cost — verified.

### Three deliberate deviations from the spec / R-010

1. **Judge model: `claude-opus-4-5`, not `claude-opus-4-7`.** R-010 specified Opus 4.7 ("one tier above Sonnet 4.7"); empirically Sonnet/Opus 4.7 returned `404 not_found_error` from the live Anthropic API at experiment time, while 4.5 was available. The intent ("judge one tier above coach") is preserved: judge is Opus 4.5, coach is Sonnet 4.5 — but they are now the SAME generation. This means self-preference bias is NOT fully mitigated by tier-up (because there is no tier-up); E-025 (calibration) will measure whether this matters. If R-010's open question #1 (informed-scoring bias) fires harder than expected, the mitigation is to either (a) switch judge to a different model family entirely, or (b) wait for 4.7 availability.

2. **No `cache_control` blocks** on the coach system prompt yet. R-010's cost estimate ($5-9/full run) assumes prompt caching reduces coach token cost by ~50%. Our smoke shows $1.68/profile, extrapolating linearly to ~$25/full 15-profile run — ~3x R-010's estimate. This is acceptable for E-023's hypothesis (the pipeline works); E-024 should enable Anthropic prompt caching on the coach `system` field (set `cache_control: { type: "ephemeral" }`) before the full baseline run to bring cost into the R-010 budget. The cache wrapper already records `cache_read_input_tokens` and `cache_creation_input_tokens` in the per-call records, so this is a one-block edit.

3. **Anti-pattern tests (E-021) and methodology checklist (E-022) are loaded for validation but not yet directly injected into the per-dimension judge prompts.** R-010 and E-021 leave open whether the D5 (anti-pattern) judge should iterate per-test or whether one D5 call should reason over the full anti-pattern catalog. Current implementation: the D5 rubric YAML's `judge_guidance` and `key_signals` already enumerate the priority anti-patterns; the judge scores D5 on that basis. The per-test detail (e.g., the bait-turn pattern, `positive_avoidance_marker`) is loaded but unused. This is a deliberate scoping choice: the baseline run uses the conservative single-D5-call architecture; if E-025 calibration shows the judge under-detects specific anti-patterns, E-024 or a follow-up can switch to a per-test loop. Similarly, the methodology checklist's per-entry `appropriate_when` / `inappropriate_when` clauses are not injected entry-by-entry into the D4 judge prompt; the D4 rubric's `judge_guidance` summarizes them and the judge is expected to reason from `coach/` references it can name without seeing the full 43-entry YAML. If D4 quality is low in E-025, the harness can later inject the full checklist as a structured catalog into the D4 prompt.

## Analysis

The harness composes correctly and produces interpretable output. Three architectural choices worth flagging for downstream work:

- **The per-turn × per-dimension matrix is the iteration mechanism.** A per-dimension score of "2" tells you methodology fidelity was mediocre; the matrix tells you specifically that turn 4 was `+` on methodology-fidelity (coach used "what's it like in your body") but turn 8 was `-` (coach asked a generic content question). This is the signal a coach-prompt author would optimize against. Verified working in the sample scorecard.

- **Visibility enforcement is structural, not policy.** The client LLM prompt is built by calling `renderClientSystemPrompt`, which only references client-visible fields. The judge prompt is built by `buildDimensionPrompt`, which references both client-visible and judge-only fields. The data structures themselves cannot leak judge-only material into the client because no code path renders them there. This is more robust than a "strip these fields" approach (which is what the E-020 template doc described prose-only).

- **Two-pass judge is right.** When run against the crisis profile, the safety screen fired (`fail` on `crisis-non-recognition`) AND the D6 (effectiveness) dimension scored 1 (correctly low because the coach failed). These are separate signals from separate calls. A joint scoring prompt would have produced an averaged "2" that hid the safety failure under the methodological mediocrity. Audit Recommendation 1's pressure was right.

The naive-coach smoke result is itself a small finding: a minimal "coach in AoA style" system prompt CAN score 3 on anti-pattern-avoidance when the test is a happy-path `am-I-being-selfish` setup, because the naive coach happens to redirect to body sensation rather than answering the verdict question. It is much weaker on perceptual-accuracy and methodology-fidelity (mostly 2s), because no specific signature questions or reads are named. This is plausible baseline behavior and is a useful floor for E-024 to measure against.

## Dead Ends

None. The Bun + TS + Zod stack worked first-attempt; the only loop-back was that the Zod schemas for `safety_relevant` / `safety_disclosures` / `safety_hard_fails` had to be revised after one validate-schemas run revealed the crisis profiles use a flat top-level layout (boolean `safety_relevant: true` plus sibling arrays) rather than the nested object the E-020 template doc implied. Fixed in `eval/lib/schemas.ts` SafetyHardFailsSchema (accepts both flat-array and primary/secondary-object shapes).

## Next Steps

1. **E-025 (judge calibration) is now fully unblocked.** It should run all 7 gold exchanges through this harness's judge stage by feeding the conversation turns directly (bypassing `runConversation`). The expected_scores in each gold-exchange JSON are the ground truth against which the judge's per-dimension output is compared. A new harness mode `score-gold-exchanges` would be a small addition (~30 LoC); E-025 can either add that or run the judge directly via `runDimensionJudge` from a standalone script.

2. **E-024 (baseline) should add Anthropic prompt caching** to the coach system prompt before running the full 15-profile suite. This requires changing the `system` argument in `AnthropicWrapper.complete` from a string to a structured `[{ type: "text", text: ..., cache_control: { type: "ephemeral" } }]` form when the call is a coach turn. The harness already handles both shapes (`system` is typed as `string | structured[]`). Expected cost reduction: ~50% on coach calls, bringing the full-suite estimate from ~$25 down to ~$15. Both numbers are above R-010's $5-9 estimate; the gap is also explained by larger judge prompts (every dimension judge receives the full rubric + few-shot + profile, which is ~3K tokens of input per call × 6 dimensions). Document the gap and the actual cost in E-024.

3. **Document the SDK / Bun / TS environment** in a short `eval/README.md` so future agents can run the harness without re-discovering the install steps. The minimum: `bun install`, `bun run eval/run.ts validate-schemas`, then `bun run eval/run.ts smoke`.

4. **Open question to revisit before E-024**: whether to inject the full anti-pattern test catalog and methodology checklist into the D5 and D4 judge prompts respectively. Current design relies on the rubric `judge_guidance` summary. If E-025 calibration shows D5 under-detection of specific anti-patterns or D4 not catching `inappropriate_when` violations, the harness should be extended to loop the D5 judge over each anti-pattern test (which would shift D5 from 1 call to ~12 calls per conversation, materially increasing cost) and/or to inject the methodology YAML as structured context into the D4 prompt. This is a known follow-up tied to E-025 results.
