# Coach v5: hybrid retrieval combining best-per-dimension strategies from E-034

## Hypothesis

G-008's six dimensions are orthogonal by design. The empirical outcome from E-034 is most likely a per-dimension split: e.g., tool-based nav wins D1 Perceptual Accuracy because the router actively fetches `reads/` files; embedding-based wins D4 Methodology Fidelity because semantic search surfaces signature questions; graph-walk wins D3 Coaching Stance because the relational structure encodes Joe's "go-to-next" intuitions. A hybrid architecture that routes retrieval based on the dominant client signal (or runs multiple retrievals in parallel and merges results) outperforms any single architecture by mean delta >= 0.15 on at least 4 dimensions.

This is the audit-flagged contingency made first-class. Pre-spec'd, budgeted at ~$8 in the goal plan, so E-034 has a real second path to choose from beyond E-035 (single-architecture iteration).

## Method

E-037 is only run if E-034's analysis mandates a hybrid (no clear single-architecture winner; per-dimension splits). If E-034 picks a clear winner, E-037 is skipped and E-035 runs directly on that winner. The G-009 decision log records which path was taken and why.

If executed:

1. Read E-034's per-dimension winner table and per-profile variance table. These define which architectures handle which dimensions best.
2. Pick a hybrid design from one of these patterns (informed by E-034 evidence):
   - **Per-dimension routing**: a cheap classifier inspects the client message and routes to the best architecture for the predicted dominant dimension (e.g., emotional content → no-retrieval to protect stance; methodology question → embedding; pattern-recognition → graph-walk)
   - **Tiered (cheap-then-strong)**: graph-walk for initial neighborhood, embedding for refinement, tool-based for deep-dives only when graph+embedding both signal low confidence
   - **Parallel-merge**: run 2-3 retrievals in parallel, merge results with deduplication, pass merged bundle to coach. Most expensive, most informative as a baseline for whether hybrid even helps
3. Implement chosen hybrid at `coach-app/retrieval/hybrid.ts` reusing the components from E-032/E-033/E-036.
4. Create `coach-app/configs/v5-hybrid.yaml` with the v1 system prompt (plus retrieval-aware language) and the hybrid retriever.
5. Smoke on 2 profiles ($1-2). Validate: hybrid doesn't double-cost relative to single retrieval; coach handles merged context.
6. Run full 15-profile eval. Estimated cost: $8 (slightly higher than single-retrieval E-033 if parallel-merge; comparable if tiered).
7. Render scorecard via E-030. Compare to the best single-architecture from E-034 across all 6 dimensions.
8. Write Result with delta vs best-single per dimension, Analysis on whether hybrid is worth the complexity for the score improvement.

Implementation ramp: smoke first, validate that the merged retrieval bundle isn't blown out by token count, then full run.

## Success Criteria

- Hybrid retriever runs end-to-end
- Full G-008 eval produces a scorecard
- Mean delta vs best-single architecture from E-034 reported per dimension
- Total experiment spend logged and under $9
- Verdict: is hybrid worth the implementation complexity? Does it move the bar that single-architecture iteration in E-035 could not?

Failure mode: hybrid scores lower than any single architecture (the merged context is noisier than focused retrieval). If this happens, document and conclude that single-architecture iteration (E-035) is the right path; hybrid is a dead end at this scope.

## Blast Radius

Adds `coach-app/retrieval/hybrid.ts`, `coach-app/configs/v5-hybrid.yaml`. Reuses retrieval components from E-032/E-033/E-036 (read-only consumption of their modules). Writes eval results to `eval/results/coach-v5-hybrid/`. Does not modify `coach/`, `eval/`, or the v1 system prompt.
