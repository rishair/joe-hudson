# Coach v5a: model-guided graph navigation with Haiku walker

## Hypothesis

E-036's deterministic graph walk (fixed depth 2, hub-dampened scoring) leaves untapped intelligence: the related edges between coach/ files are not all equally relevant in every situation. A model-guided walk that reads each node's content, decides whether to keep it in the retrieval bundle, and chooses which related edges to follow next — produces more situationally-relevant bundles than the fixed algorithm. Using Haiku as the walker tests whether basic content reasoning over the graph is enough to beat E-036's deterministic baseline, at acceptable cost.

The hypothesis is intentionally split across E-037 (Haiku walker) and E-038 (Sonnet walker) so the delta isolates whether navigation intelligence is the bottleneck, or whether the graph structure itself carries the load and a thin walker is sufficient. Both run the same algorithm with the same prompt; only the walker model differs.

## Method

1. Read E-036's scaffolding at `coach-app/retrieval/graph-walk.ts` — reuse the seed-detection step exactly as-is (Haiku reads `coach/_index.md` aliases inline, returns 1-3 seeds constrained to `concerns/`/`patterns/`/`reads/`).
2. Build the new walker at `coach-app/retrieval/guided-walk.ts`:
   - Input: list of seed file paths
   - State: `bundle: string[]` (chosen file paths), `visited: Set<string>`, `frontier: string[]` (paths to consider next)
   - Per step: read the next frontier path's content, then make TWO decisions via a single LLM call (the walker model):
     - Decision A: should this file be added to the bundle? (yes/no with one-sentence reason)
     - Decision B: of this file's `related:` edges (after filtering already-visited and already-in-bundle), which should be added to the frontier? (subset selection with one-sentence reason)
   - Stop conditions: bundle reaches K_max (default 7), frontier empty, OR step count reaches budget cap (default 8 steps per turn)
   - Return: bundle (full markdown bodies) + telemetry (decision rationales, files skipped, edges not followed)
3. Walker model: **Haiku 4.5** for this experiment. The decision schema is constrained via Zod / structured output so the walker's responses are parseable.
4. Inline retrieved markdown into Opus coach's context under "RETRIEVED CONTEXT (model-guided graph navigation)".
5. Create `coach-app/configs/v5a-guided-walk-haiku.yaml` with the v1 system prompt (held constant) and the Haiku walker wired in.
6. Smoke on 2 profiles. Validate: walker decisions are sensible (read the rationales), bundle size ≤ 7, no infinite loops, total per-turn cost matches projection.
7. Run full 15-profile eval. Estimated cost: $10-12 (Haiku walker at 5-8 steps/turn × 12 turns × 15 profiles = ~750-1440 Haiku calls + same coach cost as E-036).
8. Render via E-030. Compare to E-036 (deterministic graph-walk) and E-038 (Sonnet walker).

### Telemetry

The walker's per-step decision rationales are first-class data. Log every step's `(file_read, kept_in_bundle?, edges_considered, edges_followed, reasoning)` to a JSONL trace so E-034's head-to-head can inspect whether Haiku is making sensible navigation choices or just collecting everything indiscriminately (the failure mode where the walker degenerates to a more expensive E-036).

### Cost guard

If walker spends >$0.05 per turn on average during smoke (5x the budgeted ~$0.01), abort the experiment and document the cost surprise. The architectural test only matters if the cost stays reasonable.

## Success Criteria

- Walker runs end-to-end: seeds detected, walk traversed, bundle assembled, coach response generated
- Per-step telemetry logged
- Full G-008 eval produces a scorecard
- Mean delta vs E-036 reported per dimension (this is the architectural test: does model-guided beat deterministic?)
- expected_territory recall reported (R-012 quality gate)
- **D4 ≥ 2.50 maintained** (must not regress vs E-036 baseline of 2.53)
- Total experiment spend logged and under $13
- Verdict: did Haiku walker beat E-036? On which dimensions? At what cost premium?

Failure mode: walker keeps every file it reads (degenerates to "read 7 files, return all 7", losing the curation benefit). Document with telemetry counts: if `kept_in_bundle == read_count` for >60% of steps, walker is not curating.

## Blast Radius

## Analysis

The model-guided walk validates the core hypothesis from R-012's design space: LLM-driven graph navigation can solve cases the deterministic walk cannot. Crisis-coercion-001 going from 1.83 → 3.00 is the architectural-win signal. The walker presumably reads the somatic-shutdown content at the relevant node and decides to follow edges toward `dont-confuse-embracing-fear-with-inviting-danger` / threshold-calibration rather than the generic "feel the unfelt emotion" path that the deterministic walk's hub-dampened scoring favored.

However, the **per-profile variance pattern** is the key new finding. E-036 had relatively flat per-profile scores (most 2.33-3.00, only crisis-coercion at 1.83). E-037 has the same upper distribution but a NEW low at 1.17 (edge-dont-have-time-001). The trade is: walker excels when it picks the right path, fails harder when it doesn't. This is a different risk profile than the deterministic walk, which had medium-quality retrievals everywhere — predictable mediocrity vs unpredictable highs-and-lows.

For G-009's exit criteria (mean across profiles), this matters less than for production use. A coach that catastrophically fails 1 of 15 client types is not deployable even if the mean is high. E-038 (Sonnet walker) is the test of whether stronger walker reasoning compresses this variance.

The cost ($13.26) is 1.20x E-036's $11.09 — about what was projected. Haiku walker is not free but tractable.
