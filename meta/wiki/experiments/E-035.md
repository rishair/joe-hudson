# Iterate system prompt v4-v7 on winning retrieval architecture until eval bar met

## Hypothesis

Once E-034 picks a winning retrieval architecture, the bottleneck moves back to the system prompt. Iterating the system prompt over 3-4 versions (v4 through v7) on the same retrieval setup, using G-008 eval scores as the feedback loop, brings the coach within the G-009 exit-criteria bar (mean delta >= 0.3 over E-024 on >= 4 dimensions, anti-pattern triggers < 50% of E-024). The user's framing is that prompt is the "big big one"; this experiment is where that bet is cashed.

Each iteration is targeted by the worst-scoring dimension in the previous run. If v4 scores low on D3 Coaching Stance, v5 strengthens the stance-section of the prompt and re-runs. The smoke-test 2-profile mode is the iteration loop (~$1-2 per iteration); the full 15-profile eval is the verification at the end of each version that the smoke result generalizes.

## Method

1. Read E-034's decision. The winning architecture (single or hybrid via E-037) is the fixed substrate. The variable is the system prompt.
2. Pull the current best prompt (v1 from E-031, or whichever the winning config used).
3. For each iteration v4..v6 (cap at 3 iterations, tightened per audit Critical #3):
   - Read the prior version's scorecard. Identify the worst-scoring dimension (or the dimension closest to the exit-criteria bar).
   - Diagnose: read the per-turn matrix and the lowest-scoring turn's coach utterance. What about the prompt allowed that failure?
   - **Two-diagnoses rule (per audit Important #6)**: document TWO candidate diagnoses, not one. Pick the highest-likelihood one to try. Record the alternative. If two iterations in a row fail to move the target dimension, switch to the alternative diagnosis on the next iteration. This prevents confirmation-bias loops where the same diagnosis is doubled-down despite evidence.
   - Make a targeted edit to the prompt addressing the chosen diagnosis. Document edit + hypothesis + alternative diagnosis on this page.
   - Smoke-eval on 2 profiles ($1-2): does the targeted dimension improve? Did anything else regress?
   - Variance re-run: same 2 smoke profiles, second pass (~$1-2). Confirm the delta isn't noise (per audit Minor #14).
   - Full eval ONLY if smoke delta on the targeted dimension is >0.3 (tightened per audit Critical #3). Otherwise revert and try the alternative diagnosis.
4. Stop early when exit criteria are met: mean delta >= 0.3 over E-024 on >= 4 dimensions, anti-pattern triggers < 50% of E-024.
5. Final iteration is named `coach-app/prompts/v-final.md` and configured at `coach-app/configs/v-final.yaml`.
6. Write Result with: every version tried, the diagnosis that motivated each edit, the smoke-eval delta, the full-eval delta, whether exit criteria were met.

Budget guard: cap total spend at $15. If after $15 the bar is not met, document the gap, mark the experiment as inconclusive, and write a Next Steps section with specific further-iteration ideas for a future round. Do NOT silently burn the entire G-009 contingency on this experiment.

Implementation ramp: every iteration is smoke-first. No full eval without smoke evidence of movement.

## Success Criteria

- At least 3 prompt iterations (v4, v5, v6) are tried with documented diagnoses AND alternative diagnoses (per audit Important #6)
- Final coach config at `coach-app/configs/v-final.yaml` exists
- Final eval scorecard meets G-009 exit-criteria bar (conditional delta over E-024 per Must-haves, on >= 4 dimensions; anti-pattern triggers < 50% of E-024) OR a clear documentation of the gap and what was tried is recorded if the bar is not met within budget
- Iteration log on G-009's goal page is updated with every version, score, edit, AND alternative diagnosis
- Total experiment spend logged and under $11

Failure mode: prompt edits oscillate (v5 fixes a dimension but breaks another, v6 fixes that and breaks the first). Mitigation: keep a small running test suite of "must-pass" snippets the prompt must handle correctly, derived from anti-pattern tests, that smoke-eval checks before claiming an iteration succeeded.

## Blast Radius

Adds versioned files at `coach-app/prompts/v4.md`, `v5.md`, `v6.md`, `v7.md`, `v-final.md`. Adds config files at `coach-app/configs/v4-*.yaml`, etc. Writes eval results to `eval/results/coach-v{4,5,6,7,final}/`. Does not modify the chosen retrieval architecture, the eval harness, the rubrics, or the profiles. Updates G-009 goal page decision log and iteration log.
