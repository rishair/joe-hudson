# Build wiki view: routing, rendering, internal wikilinks

## Hypothesis

A Next.js route `/wiki/[...slug]` can render any file from `web-app/content/wiki/` (produced by E-039) as a styled markdown page. Internal `[[wikilinks]]` resolve to the correct routes within the wiki. The wiki view is navigable on its own (separate from chat) and is the surface the chat's resource modal expands into.

## Method

1. Confirm E-039 has produced `web-app/content/wiki/` with normalized links
2. Define interfaces per the [[coding-architecture]] playbook's four-method shape — read-only repo, so only `find`:
   - `WikiPage { slug, title, frontmatter, body, related: string[], category }`
   - `WikiRepository`:
     - `find(criteria?: { slug?: string; category?: string; query?: string; limit?: number }): Promise<WikiPage[]>`
   - Callers needing a singleton (e.g., "get page by slug") do `(await wikiRepo.find({ slug }))[0] ?? null`
   - Callers needing the index do `wikiRepo.find()` (or `find({ limit: ... })` if pagination matters later)
3. Implement `FsWikiRepository` (server-only) that reads from `web-app/content/wiki/`. Parses markdown + frontmatter. Resolves wikilinks at render time (not at ingest — the slug→file mapping is dynamic and a missing file should render as a broken-link span, not error).
4. Add `web-app/app/wiki/[[...slug]]/page.tsx` (catch-all route):
   - If slug is empty → wiki index page (categories + counts)
   - If slug resolves to a file → render the page via `<WikiRenderer>` (see step 5)
   - If slug doesn't resolve → 404 with a "did you mean X" suggestion if there's a close match
   - Read `?fromChat=<msgId>` from `searchParams`. If present, render a small "Back to chat at message N" link at the top of the page (per R-017 §7). On click: `router.back()` primary, falls back to `router.push('/?scrollToMessage=' + msgId)` if `history.length <= 1`. The chat page reads `?scrollToMessage=` on mount, scrolls the conversation pane to that message id, and `replace`s the URL to clear the param.
5. **`<WikiRenderer>` component** at `web-app/app/components/wiki-renderer.tsx`:
   - Props:
     - `page: WikiPage` (or `slug: string` + the renderer does the lookup)
     - `linkBehavior?: 'route' | 'in-modal'` (default `'route'`)
     - `onLinkClick?: (slug: string) => void` (required when `linkBehavior === 'in-modal'`)
   - Markdown renderer: `react-markdown` + `remark-gfm`, with a custom `[[wikilink]]` pre-processor (converts `[[slug]]` and `[[slug|display]]` per R-018 to standard markdown links).
   - When `linkBehavior === 'route'`: internal `/wiki/...` links render as Next.js `<Link>` components.
   - When `linkBehavior === 'in-modal'`: internal `/wiki/...` links render as `<button>` elements; `onClick` calls `onLinkClick(slug)` where `slug` is extracted from the href. The visual styling is identical (looks like a link, behaves like a link, but does not navigate).
   - External links (anything not `/wiki/...`) always render as standard `<a target="_blank" rel="noopener">` regardless of `linkBehavior`.
   - Broken internal links (slug not in the index) render as a visibly-broken span with a `title` tooltip explaining the missing target, NOT as a clickable link in either mode.
6. Style with Tailwind (already in the playbook stack).
7. Test: load 5 random wiki pages in route mode, click through wikilinks, verify navigation works and the URL updates correctly. Render the same renderer with `linkBehavior="in-modal"` and a stub `onLinkClick`, verify clicks call the callback instead of navigating. Verify a `[[broken-slug]]` renders visibly broken, not clickable, not crashing.

## Success Criteria

- `/wiki` shows the category index.
- `/wiki/concepts/limiting-belief` (or any actual slug) renders the entry.
- Internal `[[wikilinks]]` are clickable and navigate correctly in default (`linkBehavior="route"`) mode.
- In `linkBehavior="in-modal"` mode, internal wikilinks invoke `onLinkClick(slug)` instead of routing — verified by E-044 integration test.
- Broken wikilinks render as visibly broken (not crashing) in both modes.
- Navigation is browser-native (back button works, URLs are shareable).
- `?fromChat=<msgId>` on a wiki page renders a "Back to chat at message N" link; clicking it returns to the chat with conversation scroll restored to message N.
- Page renders in under 500ms (file read + markdown parse).

## Blast Radius

Adds `web-app/app/wiki/[[...slug]]/page.tsx`, `web-app/app/components/wiki-renderer.tsx`, `web-app/app/lib/wiki/*` (repository + types). Reads from `web-app/content/wiki/` (read-only). No conflict with the chat surface.

Adds a small handler to the chat page (`app/page.tsx` or equivalent) for `?scrollToMessage=<msgId>` URL param: on mount, scroll to message, then `router.replace('/')` to clean the URL. This is shared with E-044's expand-then-back-to-chat path.
