# Build wiki view: routing, rendering, internal wikilinks

## Hypothesis

A Next.js route `/wiki/[...slug]` can render any file from `web-app/content/wiki/` (produced by E-039) as a styled markdown page. Internal `[[wikilinks]]` resolve to the correct routes within the wiki. The wiki view is navigable on its own (separate from chat) and is the surface the chat's resource modal expands into.

## Method

1. Confirm E-039 has produced `web-app/content/wiki/` with normalized links
2. Define interfaces per the [[coding-architecture]] playbook's four-method shape — read-only repo, so only `find`:
   - `WikiPage { slug, title, frontmatter, body, related: string[], category }`
   - `WikiRepository`:
     - `find(criteria?: { slug?: string; category?: string; query?: string; limit?: number }): Promise<WikiPage[]>`
   - Callers needing a singleton (e.g., "get page by slug") do `(await wikiRepo.find({ slug }))[0] ?? null`
   - Callers needing the index do `wikiRepo.find()` (or `find({ limit: ... })` if pagination matters later)
3. Implement `FsWikiRepository` (server-only) that reads from `web-app/content/wiki/`. Parses markdown + frontmatter. Resolves wikilinks at render time (not at ingest — the slug→file mapping is dynamic and a missing file should render as a broken-link span, not error).
4. Add `web-app/app/wiki/[[...slug]]/page.tsx` (catch-all route):
   - If slug is empty → wiki index page (categories + counts)
   - If slug resolves to a file → render the page (title, body with wikilinks converted to Next.js `<Link>` components)
   - If slug doesn't resolve → 404 with a "did you mean X" suggestion if there's a close match
5. Markdown renderer choice: `react-markdown` + `remark-gfm`, with a custom `[[wikilink]]` plugin (or pre-processor that converts to standard markdown links)
6. Style with Tailwind (already in the playbook stack)
7. Test: load 5 random wiki pages, click through wikilinks, verify navigation works and the URL updates correctly

## Success Criteria

- `/wiki` shows the category index
- `/wiki/concepts/limiting-belief` (or any actual slug) renders the entry
- Internal `[[wikilinks]]` are clickable and navigate correctly
- Broken wikilinks render as visibly broken (not crashing)
- Navigation is browser-native (back button works, URLs are shareable)
- Page renders in under 500ms (file read + markdown parse)

## Blast Radius

Adds `web-app/app/wiki/[[...slug]]/page.tsx`, `web-app/app/lib/wiki/*` (repository + types). Reads from `web-app/content/wiki/` (read-only). No conflict with the chat surface.
