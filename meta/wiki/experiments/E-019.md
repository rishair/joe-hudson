# Define scoring dimensions and rubrics

## Hypothesis

The `coach/` compendium can be mapped into 5-7 scoring dimensions with concrete behavioral anchors that an LLM judge can reliably apply to coaching conversations. A 1-3 scale with clear examples at each level will produce more consistent scores than vague descriptions.

R-008 has now resolved this to **exactly six** quality dimensions (plus a separate safety pass per R-010 Pass 1). The six dimensions, their definitions, behavioral anchors at 1/2/3, key signals, and reference files are all specified in R-008. This experiment is the YAML-isation step.

## Method

1. ~~Complete R-008 research (define the dimensions)~~ — R-008 complete; six dimensions defined.
2. For each of the six dimensions, write a rubric file at `eval/rubrics/<dimension-id>.yaml`:
   - `eval/rubrics/perceptual-accuracy.yaml` (D1)
   - `eval/rubrics/intervention-quality.yaml` (D2)
   - `eval/rubrics/coaching-stance.yaml` (D3)
   - `eval/rubrics/methodology-fidelity.yaml` (D4)
   - `eval/rubrics/anti-pattern-avoidance.yaml` (D5)
   - `eval/rubrics/effectiveness-depth.yaml` (D6)
3. Each YAML file follows the R-010-specified schema:
   ```yaml
   dimension_id: <id>
   name: <human-readable name>
   definition: <1-2 sentence definition from R-008>
   scale:
     1: { label: poor, anchor: "<R-008 anchor 1 text>" }
     2: { label: adequate, anchor: "<R-008 anchor 2 text>" }
     3: { label: excellent, anchor: "<R-008 anchor 3 text>" }
   key_signals: [<bulleted list from R-008>]
   reference_files: [<list of coach/ paths from R-008>]
   few_shot_examples:
     - score: 3
       excerpt: "<from E-026 real-Joe gold>"
       rationale: "<why this is a 3>"
     - score: 1
       excerpt: "<from gold-poor.json>"
       rationale: "<why this is a 1>"
   independence_notes: "<from R-008 — what distinguishes this dimension from adjacent dimensions>"
   ```
4. Start with two rubrics (coaching-stance and methodology-fidelity, the two most-tested-against pair given the view-as-technique failure mode) to verify the format works for judge injection, then write the rest.
5. Validate each rubric against R-008's reference materials: all `reference_files` paths exist in `coach/`.

## Analysis

The YAML-isation step from R-008's specification was largely mechanical, but several design refinements emerged that strengthen the rubrics beyond what R-008 specified:

1. **The `exclusions` field is new and important.** R-008 specifies independence between dimensions but does not have a dedicated field naming what each dimension does NOT score. Adding `exclusions` to each rubric makes the boundary explicit and gives the judge prompt a direct instruction list. This addresses the central R-010 concern about score compression and the audit's concern about dimension independence: the judge calling a single dimension sees both "what to score" (anchors, signals) AND "what to ignore" (exclusions). The exclusions for coaching-stance specifically include "Do not score on whether AoA moves or signature questions were used. That is Methodology Fidelity (D4)." — this is the exact instruction needed to detect view-as-technique.

2. **The `judge_guidance` field exceeds R-008's spec.** R-008 says "judge guidance" should be present but R-010's RCAF prompt template is the operational backing. I rendered the judge_guidance in each YAML as a self-contained instruction paragraph that the harness can either inline into the RCAF prompt or use as a reference. This makes each rubric file independently usable — a judge can be tested against a single dimension in isolation, which is what E-025 will do.

3. **The "use 2 only when neither 1 nor 3 fits" instruction appears in every rubric's judge_guidance.** R-010 specified this anti-central-tendency directive should be in the prompt template. By repeating it in each rubric, the directive survives even if the harness loads only the YAML (e.g., for ad-hoc dimension-only evaluation). This is defense in depth against the most-documented LLM-as-judge bias.

4. **The `priority_anti_patterns` block in D5 is operationally critical.** R-008 names the top-10 anti-patterns in prose; rendering them as a structured block (id, file, why_ai_prone) makes them machine-readable for the judge prompt. The `why_ai_prone` annotation per anti-pattern is new content that didn't exist in either R-008 or in the coach/anti-patterns/ files themselves — it explains to the judge why this anti-pattern is specifically what an AI is likely to do. This shifts the judge from "scan for failures" to "actively look for these specific AI tells."

5. **The `seam_with_safety` block in D5 captures R-008's "the seam between D5 and E-027 needs design" implication.** D5's `dont-use-safety-as-control` and Pass-1's `immersion-breaking-disclaimers` are documented as deliberate double-coverage. The judge prompts for D5 and Pass-1 will both fire on a single coach action (unnecessary safety disclaimer) and both rationales will reference the other. This makes the rationale traceable across passes and accepts the overlap explicitly rather than fighting it.

6. **The `risks_and_mitigations` block in D6 is unique to that dimension.** D6 (Effectiveness) is the most subjective dimension and the most susceptible to judge bias. R-008 names the risks; rendering them in the YAML makes them part of the rubric the judge sees. The judge is therefore not just scoring D6, they are scoring D6 with explicit knowledge of the bias they are most likely to commit. Whether this metacognitive priming reduces bias is a hypothesis E-025 will test.

7. **Few-shot placeholders use descriptive language rather than empty strings.** Each `excerpt` placeholder names what kind of content will fill it (e.g., "real-Joe exchange where Joe holds VIEW state, names own state, slows pace") and each `rationale` explains why that content would score at the target level. This means the rubric is already usable for hand-running an evaluation — the judge has enough scaffolding to score even before the final excerpts arrive. When E-026 and E-025 produce the actual excerpts, they slot into the placeholder structures without re-architecting the rubric.

8. **The forward references (E-021, E-022) are deliberately invalid file paths.** R-008's spec calls for D4 to reference `eval/checklists/methodology.yaml` and D5 to reference `eval/anti-pattern-tests/`. Neither exists yet. I included them as-spec; the validation script flagged them as missing (as expected). When E-021 and E-022 complete, the references will resolve. This is the intended dependency pattern.

### What this validates about R-008 and R-010

The YAML-isation forced explicit decisions on points R-008 and R-010 left implicit:
- **Per-dimension exclusions are a needed field.** R-008 has "independence notes" but those describe theoretical independence. The exclusions field operationalizes it.
- **The judge_guidance per dimension is sufficiently rich to be self-contained.** Each rubric carries the instruction for how the judge should score it; the harness can load these directly into the RCAF context without re-deriving from R-010.
- **The "do not use 2 as hedge" instruction needs to be repeated per dimension.** A single instance in the top-level prompt template, as R-010 specified, is fragile — the per-dimension reinforcement is safer.

### What this surfaces about E-022 (fidelity checklist)

The methodology-fidelity rubric (D4) references `eval/checklists/methodology.yaml` as operational backing. R-008's "Implications for E-022" section noted that E-022's reads section should be consumed primarily by D1 (Perceptual Accuracy), not D4. The D1 rubric I wrote references the same checklist via `coach/reads/` files directly, not via `eval/checklists/reads.yaml`. When E-022 runs, it should produce two files (methodology.yaml and reads.yaml) or a single file with sections — either is fine, but the D1 rubric will need a one-line update once E-022's file structure is final. I note this as a minor follow-up rather than a blocker.

### What this surfaces about E-025 (calibration)

The five open questions R-008 deferred to E-025 are now testable against concrete YAML rubrics. Specifically:
- "Are the six dimensions actually independent in practice?" — E-025 runs each gold conversation through all six rubrics and checks whether scores vary independently.
- "Is Effectiveness biased by client profile difficulty?" — E-025 includes easy + hard profiles and checks correlation.
- "Does Stance discriminate well from Methodology Fidelity?" — the "perfect fidelity, wrong stance" gold conversation should produce D3=1, D4=3, D5=1.
- "Is Anti-Pattern Avoidance redundant with the safety pass?" — the seam_with_safety block in D5 documents the deliberate overlap; E-025 verifies that signals fire consistently from both lenses.
- "Does Perceptual Accuracy reward correct-but-unspoken reads?" — the perceptual-accuracy rubric explicitly accepts this limitation ("score on visible reads only"); E-025 should confirm this does not produce systematic underscoring of competent quiet coaches.

## Dead Ends

None. The YAML schema R-010 specified was sufficient as-is; the additions (`exclusions`, `judge_guidance`, `priority_anti_patterns`, `seam_with_safety`, `risks_and_mitigations`) are extensions, not corrections.

One design choice considered and rejected mid-write: collapsing `key_signals` and `judge_guidance` into a single field. Rejected because `key_signals` is a scan list (the judge reads it as "look for these specific things") while `judge_guidance` is a scoring algorithm (the judge reads it as "here is how to map what you found to a score"). Conflating them produces both a worse scan list and a worse algorithm.

Also considered: producing a single combined rubrics.yaml with all six dimensions as keys. Rejected because R-010's harness invokes one judge call per dimension and benefits from being able to load a single file per call (smaller prompt, easier to cache the prompt for repeat scoring). Six separate files match the call structure.
