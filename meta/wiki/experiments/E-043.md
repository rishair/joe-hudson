# Wire v5b coach (graph-walk + Sonnet walker) into chat route handler

## Hypothesis

The v5b coach pipeline (Sonnet walker over hand-curated graph-walk retrieval, plus the v1 system prompt, plus runtime safety screen) can be reproduced in a Next.js `/api/chat` route handler. The chat route reads from the ingested `web-app/content/wiki/` (E-039's output), not from `coach/` directly. Per-turn retrieval telemetry flows back to the client via the AI SDK transport R-016 specifies, so the client can render the resource attribution UX (consumed by E-044).

**This is bigger than a port.** Per the G-010 audit: the retrieval-to-coach orchestration (trigger-policy, injection placement, history preservation, retry) lives in the EVAL harness at `eval/lib/conversation.ts` and `eval/lib/retrieval-adapter.ts`, NOT in `coach-app/run.ts`. We are rebuilding the orchestration layer in a different runtime (Next.js route handler) against a different SDK (Vercel AI SDK with OpenRouter, not raw Anthropic SDK). Coach-app/retrieval/* (the walker + graph algorithms) is the only directly reusable piece; everything around it must be reconstructed.

## Method

1. R-016 is complete. The four MUST-answers are resolved: (a) adaptation = lift to `web-app/app/lib/coach/`; (b) prompt-cache markers = `@openrouter/ai-sdk-provider` with `providerOptions.openrouter.cacheControl: { type: 'ephemeral' }`; (c) telemetry transport = AI SDK v5+ custom typed `data-resources` part via `createUIMessageStream`; (d) orchestration port = lift three files + rewrite the eval `conversation.ts:270-368` glue inline in the route handler. See R-016 Findings for the full lift checklist and code shapes.
2. Port the v5b pipeline (per R-016 Answer 4 lift checklist):
   - `cp coach-app/retrieval/guided-walk.ts web-app/app/lib/coach/guided-walk.ts`
   - `cp coach-app/retrieval/graph-walk.ts web-app/app/lib/coach/graph-walk.ts`
   - `cp coach-app/lib/safety.ts web-app/app/lib/coach/safety.ts`
   - Replace `import.meta.dir` (Bun-only) with `process.cwd()` + a passed-in or env-var-driven `WIKI_ROOT` parameter; default `WIKI_ROOT = process.cwd() + '/content/wiki'`
   - Replace `.ts` extension imports (`from "./graph-walk.ts"`) with extensionless imports for Next.js/Webpack
   - Coach invocation: use `createOpenRouter({ apiKey: process.env.OPENROUTER_API_KEY })` from `@openrouter/ai-sdk-provider` (NOT `createOpenAI` with custom baseURL — the OpenAI-compat shim does NOT preserve Anthropic provider options like `cacheControl`). Model id `anthropic/claude-sonnet-4-6`. Use `streamText` (not `generateText`) for streaming to the chat UI.
   - Runtime safety screen runs in the route handler before/after `streamText`, per `coach-app/lib/safety.ts`'s pre/post pattern (lifted, no Anthropic dependency to adapt).
   - **Orchestration layer** (new code, NOT a port — the eval harness orchestration doesn't fit a Next.js route handler): trigger policy (v5b defaults to `every_turn`), retrieval call with the last 4 history turns, retrieval-bundle injection into the user message (format: `${injection}\n\n${userText}`), history preservation across multi-turn (the persisted message array keeps the BARE user message — the augmented form is constructed per-turn so we don't double-pay for stale retrievals), graceful degradation on retrieval failure (catch, log telemetry with `error` field, proceed without injection), retry on transient OpenRouter 429/5xx (mirror the 5x exponential backoff from `coach-app/lib/anthropic-client.ts:198-211`).
3. Replace the placeholder system prompt in `app/lib/system-prompt.ts` with the v1 coach prompt from `coach-app/prompts/v1.md`. Pass it as `system: { content: PROMPT, providerOptions: { openrouter: { cacheControl: { type: 'ephemeral' } } } }` to `streamText` so the prompt is cached (~10x cost savings on the ~3.1K-token prompt).
4. Wire per-turn retrieval telemetry per R-016 Answer 3 (AI SDK v5+ custom typed data parts). Define `CoachUIMessage = UIMessage<never, { resources: ResourceAttribution }>` in `web-app/app/lib/coach/types.ts`. Inside `createUIMessageStream`, write `{ type: 'data-resources', id: 'resources-<turn>', data: telemetryToAttribution(walkerTelemetry) }` BEFORE invoking `streamText`. The `data` shape is `{ resources: [{ slug, category, title, walker_reason }], seeds, walker_model, total_cost_usd, step_count, stop_reason }`. The data part lives inside `message.parts`; E-041's byte-equality round-trip handles persistence with no schema change.
5. **Cost cap** (concrete circuit-breaker spec since per-IP is unworkable without auth):
   - Track per-process cumulative spend in memory, reset on process restart
   - When cumulative spend exceeds a configurable budget (default: `OPENROUTER_DAILY_CAP=5` USD), the route handler returns HTTP 503 with a clear error and stops invoking the model
   - Log the running total per request so the cap is observable
   - This protects against runaway loops; auth-gated per-user limits are a v2 concern
6. **Eval-vs-web-app behavioral verification** (NEW success criterion — closes the gap the audit flagged):
   - After E-043 ships, re-run the G-008 eval against the web-app's route handler as the coach target (rather than `coach-app/run.ts`)
   - Verify the scorecard is within ±0.1 mean per dimension of E-038's 2.71 baseline
   - This is a one-shot verification; if it diverges meaningfully, the port introduced a regression that needs diagnosis
7. Test: 10-turn conversation that exercises retrieval; verify each assistant message carries the per-turn resource list; verify the eval-vs-web-app behavioral check passes

## Success Criteria

- `/api/chat` handler streams a coherent v5b-coach response to a multi-turn conversation
- Each assistant message includes per-turn retrieval metadata as a structured annotation
- Conversation persists end-to-end through E-041's SQLite layer
- Runtime safety screen fires correctly on the 3 adversarial prompts from E-031's safety-pressure-test
- Cost cap prevents runaway: simulated burst that exceeds `OPENROUTER_DAILY_CAP` hits the 503 gracefully
- No `OPENROUTER_API_KEY` or system prompt visible in any client payload
- Eval-vs-web-app verification (method step 6): re-running G-008 eval against the route handler produces mean per-dim scores within ±0.1 of E-038's 2.71 baseline; if NOT, the port has regressed and the experiment is failed pending diagnosis

## Blast Radius

Modifies `web-app/app/api/chat/route.ts`. Adds `web-app/app/lib/coach/*` (adapted retrieval, walker, safety per R-016). Reads `web-app/content/wiki/` (read-only). May add a `package.json` dep on a rate-limiter library.

## Result

**Succeeded (2026-05-24).** Full v5b coach pipeline running in the Next.js `/api/chat` route handler against the ingested wiki. Verified end-to-end with a 10-turn conversation, a safety-trigger test, a cost-cap test, and a no-leakage scan.

### What landed

`web-app/app/lib/coach/` — new namespace, ~1500 LOC total:

- `graph-walk.ts` — lifted from `coach-app/retrieval/graph-walk.ts`; `WIKI_ROOT` parameterized via env var (defaults `<cwd>/content/wiki`); `import.meta.dir` removed; `.ts` extension imports stripped; catalog paths now category-relative (no `coach/` prefix); **added `buildSeedIndex()`** to slim the seed-detection index — see "Surprise" below.
- `guided-walk.ts` — lifted from `coach-app/retrieval/guided-walk.ts`; same path/import adaptations; walker step decisions, frontier management, telemetry shape preserved byte-for-byte.
- `safety.ts` — lifted verbatim from `coach-app/lib/safety.ts` (pure regex/keyword logic with no path or Anthropic deps).
- `retriever.ts` — thin replacement for `eval/lib/retrieval-adapter.ts`'s `buildGuidedWalkRetriever`; bakes in V5B_CONFIG (sonnet walker, haiku seed, k_max=7, step_budget=8, max_edges_per_step=4, variant=v5b); converts walker telemetry into the `ResourceAttribution` shape.
- `cost-cap.ts` — process-level cumulative-spend circuit breaker; `OPENROUTER_DAILY_CAP` env var (default $5); 503 on exceed; in-memory only (reset on process restart).
- `types.ts` — `ResourceAttribution` and `CoachUIMessage = UIMessage<never, { resources: ResourceAttribution }>` per R-016 Answer 3.

`web-app/app/lib/system-prompt.ts` — placeholder replaced with v1 prompt (lifted byte-identical from `coach-app/prompts/v1.md`, verified by length+phrase-count comparison).

`web-app/app/api/chat/route.ts` — full rewrite of the orchestration layer:
- Cost-cap pre-check → 503 if exhausted.
- `preResponseScan` on the latest user message.
- Retrieval call with graceful degradation (catch + log + proceed without injection).
- Builds augmented user message (safety reminder + injection + bare user text) for the model only; persisted history stays bare on the client (matches `eval/lib/conversation.ts:344-352`).
- HAPPY PATH (no safety trigger): `createUIMessageStream` writes `data-resources` part FIRST, then `streamText` merge for the streamed coach reply; `providerOptions.openrouter.cacheControl: { type: 'ephemeral' }` on the system message; `onFinish` records coach-turn cost on the cap.
- SAFETY PATH (any trigger fires): non-streaming generate → `postResponseCheck` → regenerate once with stronger reminder if failed → fall back to `SAFETY_FALLBACK_TEMPLATE` if regenerated draft also fails; final answer delivered via synthesized text-delta stream so the client UI is path-uniform.

`web-app/package.json` — added `@ai-sdk/anthropic@^3` (the walker + seed-detector go direct-to-Anthropic for prompt caching; coach turn stays on OpenRouter per R-016).

`web-app/.env.example` — documented `ANTHROPIC_API_KEY`, `OPENROUTER_DAILY_CAP`, and `WIKI_ROOT`.

### Surprise (worth promoting to the goal Decision Log)

The seed-detection index `_index.md` has grown to **1.61 MB / ~400K tokens** since G-007's continuous absorption (the original coach-app fit comfortably in 684 lines). Passing the full index to Haiku via `system` failed with `prompt is too long: 200866 tokens > 200000 maximum` — Anthropic's hard context cap.

Mitigation in `graph-walk.ts`: `buildSeedIndex()` produces a slimmed copy containing ONLY the three seed-allowed categories (concerns/, patterns/, reads/ — 126+117+121 = 364 entries vs the full 2,376) with each entry's "(also: ...)" alias list capped at the first 8 aliases (`MAX_ALIASES_PER_ENTRY_FOR_SEED`). The seed-detector never selects from categories outside those three anyway, so the trimming is lossless for the decision it actually makes; the walker still reaches all 2,376 files via `related:` traversal from the seeds.

This means **the eval-side coach-app would now also fail on the bare index** if re-run today. The slimming logic is a candidate for back-port — but G-009 is feature-frozen, so the back-port lives or dies with G-007's eventual re-eval, not with E-043.

## Analysis

The port itself was straightforward — the three coach-app modules were structured cleanly enough that R-016's "lift" strategy held. The 30-minute estimate in R-016 was accurate for the file copies + path-constant edits + extension fixes; the orchestration rewrite (route handler) took longer but landed in the shape R-016 specified.

The single non-obvious thing was the index-size problem. That's not a "port" bug — it's "the corpus this code reads grew 4x between when the eval last ran and now." The slimming is a clean fix because the seed-detector only ever selects from 3 categories anyway; trimming the other 10 categories from its view is correct, not just expedient.

The safety regeneration flow is appropriately heavier than the happy path: it generates non-streaming so we can post-check the full draft before committing it to the client. Latency cost is real (~5-15s) but it fires on <1% of turns and protects against the most consequential failure mode (crisis-non-recognition). The synthesized stream at the end keeps the client UI path-uniform (always reads `data-resources` then text-delta chunks).

The `data-resources` part flowing through `UIMessage.parts` is the elegant piece. E-041's byte-equality persistence handles it for free; E-044's modal reads it directly off the message. No separate metadata channel, no out-of-band store. R-016's "AI SDK v5+ custom typed data parts" recommendation was the right call.

## Dead Ends

None. The slimmed-index detour was a fix, not a dead end (the original lifted code WOULD have worked for the original corpus size; the slim is a corpus-growth adaptation, not a rejection of the lift strategy).
