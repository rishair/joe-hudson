# Build paas CLI tool: register / up / down / list / logs / remove with systemd user units

## Hypothesis

A single-file Bun/TypeScript CLI at `~/bin/paas` (or `/home/claude/joe-hudson/paas/bin/paas`) can manage the personal-PaaS workflow per G-012's spec. Each subcommand wraps a small set of operations (read `.local-dev.yaml`, generate or remove a `systemd --user` unit, update `~/.cloudflared/config.yml`, reload `cloudflared`, query `systemctl --user status`, tail `journalctl --user -u <unit>`).

## Method

1. Pick the project layout. Recommend: `/home/claude/joe-hudson/paas/` with `bin/paas` (entrypoint), `lib/*.ts` (per-subcommand logic), `templates/systemd-unit.template` (Mustache-style), `README.md`, `.local-dev.yaml` (for paas itself? or excluded — paas is the host, not a managed project).
2. Define the `.local-dev.yaml` schema (per the [[coding-architecture]] playbook's four-method spirit — keep it tight):
   ```yaml
   subdomain: status       # → status.play.ris.hair
   port: 4001
   command: bun run start  # or "bun run dev" for hot-reload mode
   mode: built             # or "dev"
   working_dir: /home/claude/joe-hudson/status-dashboard
   env_file: .env          # optional; defaults to .env if present
   restart: always         # systemd Restart= value
   ```
3. Implement subcommands:
   - `paas register <project-dir>` — reads `<project-dir>/.local-dev.yaml`, validates, generates `~/.config/systemd/user/paas-<subdomain>.service`, runs `systemctl --user daemon-reload`, updates `~/.cloudflared/config.yml` to add an ingress for `<subdomain>.play.ris.hair` → `http://localhost:<port>`, runs `systemctl --user restart cloudflared` (or signals reload).
   - `paas up <subdomain>` — `systemctl --user start paas-<subdomain>`
   - `paas down <subdomain>` — `systemctl --user stop paas-<subdomain>`
   - `paas list` — list all `paas-*.service` units with status + URL
   - `paas logs <subdomain>` — `journalctl --user -u paas-<subdomain> -f`
   - `paas remove <subdomain>` — stop, disable, remove the systemd unit, remove the tunnel ingress, reload tunnel
4. Schema validation via Zod (per the existing project conventions).
5. Test ramp:
   - First: register a small dummy project (5-line Bun HTTP server in `/tmp/test-project`) with `.local-dev.yaml` → `test.play.ris.hair`. Verify `paas up test` brings it up, `curl https://test.play.ris.hair` returns the right thing (through CF Access auth), `paas logs test` tails output, `paas down test` stops, `paas remove test` cleans up.
   - Second: a Next.js project to ensure the Next.js dev server works behind the tunnel (HMR / WebSockets).
6. Document in `paas/README.md`: the `.local-dev.yaml` format, the "add a new project" workflow, the troubleshooting steps (tunnel down, systemd unit failed, Cloudflare Access misconfigured).

## Success Criteria

- All 6 subcommands implemented and pass manual testing
- Schema validation rejects malformed `.local-dev.yaml` with a clear error
- A dummy Bun HTTP project goes from "in this dir" → "reachable at <subdomain>.play.ris.hair" in one `paas register` invocation
- The Next.js dev mode test confirms HMR works through the tunnel (no errors in the browser console; live edits propagate)
- `paas list` output is glance-readable
- `paas remove` is fully reversible (re-registering re-creates the same setup)
- Tunnel config edits are atomic (a registered-then-removed project doesn't leave config residue)
- `paas/README.md` covers the happy path and the 3 most likely failure modes

## Blast Radius

- Adds `/home/claude/joe-hudson/paas/` directory with bin + lib + templates
- Adds (or modifies) `~/.config/systemd/user/paas-*.service` files
- Modifies `~/.cloudflared/config.yml`
- Restarts `cloudflared` system service on each registration change (brief downtime — sub-second)
- Does NOT modify the project repos being registered (only reads their `.local-dev.yaml`)
