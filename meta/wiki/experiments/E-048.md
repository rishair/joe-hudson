# Build paas CLI tool: register / up / down / restart / list / logs / remove with systemd user units

## Hypothesis

A single-file Bun/TypeScript CLI at `/home/<user>/joe-hudson/paas/bin/paas` can manage the personal-PaaS workflow per G-012's spec. Each subcommand wraps a small set of operations (read `.local-dev.yaml`, generate or remove a `systemd --user` unit, edit `/etc/cloudflared/config.yml` atomically via tmpfile-validate-rename, signal `sudo systemctl restart cloudflared`, query `systemctl --user status`, tail `journalctl --user -u <unit>`). Concurrent invocations serialize via `flock` and do not corrupt the tunnel config or leave orphan systemd units. The reload behavior is honest about the 2-30s tunnel reconnect cost.

## Method

### Project layout

1. Code lives at `/home/<user>/joe-hudson/paas/`:
   ```
   paas/
     bin/paas                          # entrypoint (Bun shebang)
     lib/
       schema.ts                       # Zod schema for .local-dev.yaml
       register.ts                     # paas register
       up-down-restart.ts              # paas up/down/restart
       list.ts                         # paas list
       logs.ts                         # paas logs
       remove.ts                       # paas remove
       config.ts                       # cloudflared config.yml read/write/lock
       systemd-unit.ts                 # unit-file generation
       sudo.ts                         # sudo systemctl helpers
       preflight.ts                    # linger check, sudoers check, etc.
     templates/
       systemd-unit.template           # Handlebars-style or simple string template
     cloudflared/                      # symlinked from /etc/cloudflared (created in E-047)
       config.yml                      # Generated. DO NOT HAND-EDIT.
       <tunnel-uuid>.json              # credentials from cloudflared tunnel create
       .paas.lock                      # flock target for atomicity
     README.md
   ```
2. `paas` itself is NOT a managed project (it's the host). No `.local-dev.yaml` in `paas/`.

### `.local-dev.yaml` schema (tightened per audit rec #5)

3. Schema (Zod):
   ```typescript
   const Project = z.object({
     subdomain: z.string().regex(/^[a-z][a-z0-9-]{0,29}[a-z0-9]$/)
       .refine(s => !RESERVED.includes(s), { message: `Reserved subdomain` }),
     port: z.union([z.number().int().min(1024).max(65535), z.literal('auto')]),
     command: z.string().min(1),
     mode: z.enum(['dev', 'built']).optional(),  // OPTIONAL; sets NODE_ENV, otherwise redundant with command ‚Äî see audit rec #14
     working_dir: z.string().min(1),              // absolute path; CLI validates existence
     env_file: z.string().optional().default('.env'), // RELATIVE TO working_dir (documented)
     restart: z.enum(['always', 'on-failure', 'no']).default('always'),
     memory_max: z.string().default('512M'),      // promoted from nice-to-have per audit rec #15
   });
   const RESERVED = ['www', 'api', 'admin', 'dashboard', 'paas', 'mail', 'ftp', 'ns'];
   ```
4. Example `.local-dev.yaml` for a project:
   ```yaml
   subdomain: status
   port: 4001
   command: bun run start
   working_dir: /home/claude/joe-hudson/status-dashboard
   env_file: .env
   ```

### Subcommands

5. **`paas register <project-dir> [--dry-run]`** (audit rec #16):
   - Acquire `flock` on `.paas.lock`.
   - Run preflight checks: linger enabled, sudoers entry present, cloudflared running.
   - Read and validate `<project-dir>/.local-dev.yaml`.
   - Check port conflicts:
     - Read all `~/.config/systemd/user/paas-*.service` files; extract their `Environment=PORT=` values.
     - Run `ss -tlnp | awk '{print $4}'` to catch non-paas binders.
     - If `port: auto`, scan `4000-4099` for a free port; pick the lowest.
     - On collision: error with both project names (or the binding process for non-paas).
   - Generate the systemd unit content from the template:
     ```ini
     [Unit]
     Description=paas-managed: {{subdomain}}
     After=user-runtime-dir@%U.service
     
     [Service]
     WorkingDirectory={{working_dir}}
     EnvironmentFile=-{{working_dir}}/{{env_file}}
     Environment=PORT={{port}}
     {{#if mode == "dev"}}Environment=NODE_ENV=development{{else}}Environment=NODE_ENV=production{{/if}}
     ExecStart={{command}}
     Restart={{restart}}
     RestartSec=2s
     MemoryMax={{memory_max}}
     
     [Install]
     WantedBy=default.target
     ```
   - Read the current `cloudflared/config.yml`, parse YAML, add an ingress entry for `{{subdomain}}.play.ris.hair ‚Üí http://localhost:{{port}}`. Re-serialize WITH the generated-by header preserved.
   - Write to `cloudflared/config.yml.new` (under flock).
   - Validate: `cloudflared tunnel ingress validate --config cloudflared/config.yml.new`. Non-zero ‚Üí abort, delete `.new`.
   - **DNS + cert provisioning** (per the per-subdomain explicit-cert decision in [[REQ-004]] step 2):
     - Call Cloudflare API `POST /zones/{{zone_id}}/dns_records` with `{ type: 'CNAME', name: '{{subdomain}}.play', content: '{{tunnel_uuid}}.cfargotunnel.com', proxied: true, ttl: 1 }`. (Equivalent to `cloudflared tunnel route dns {{tunnel}} {{subdomain}}.play.ris.hair`, but explicit so we control retries.)
     - On 409 (record exists with same content) ‚Äî proceed. On 409 with different content ‚Äî abort with a clear error message naming the existing record.
     - Poll `GET /zones/{{zone_id}}/ssl/verification` filtering for the hostname, every 2s, up to 60s, until certificate_status is `active`. Cloudflare typically issues in 10-30s. If still not active at 60s, print a warning and continue ‚Äî the next user-driven HTTPS request will surface the actual state.
     - This step adds 10-30s to first-register per subdomain. Cached after first registration (re-registering same subdomain skips both DNS create and cert poll).
   - **If `--dry-run`**: print the diff (unit file content + config.yml unified diff + DNS record JSON) and exit. Don't apply.
   - Otherwise: `mv cloudflared/config.yml.new cloudflared/config.yml` (atomic on POSIX).
   - Write unit file to `~/.config/systemd/user/paas-{{subdomain}}.service`.
   - `systemctl --user daemon-reload`.
   - `systemctl --user enable --now paas-{{subdomain}}.service`.
   - `sudo systemctl restart cloudflared` (passwordless via the sudoers entry from E-047).
   - Poll `systemctl is-active cloudflared` every second up to 30s; report when active.
   - Release flock.
   - Print: `Registered {{subdomain}} ‚Üí https://{{subdomain}}.play.ris.hair (port {{port}}). DNS+cert: {{cert_seconds}}s. Tunnel reload: {{reload_seconds}}s.`
6. **`paas up <subdomain>`**: `systemctl --user start paas-{{subdomain}}.service`.
7. **`paas down <subdomain>`**: `systemctl --user stop paas-{{subdomain}}.service`.
8. **`paas restart <subdomain>`** (added per audit rec #5 ‚Äî needed for `.env` reloads): `systemctl --user restart paas-{{subdomain}}.service`. Does NOT touch the tunnel.
9. **`paas list`** (audit rec #11 ‚Äî explicit format):
   - 3-column table: `SUBDOMAIN | PORT | STATUS`
   - `STATUS` is one of `running`, `failed`, `stopped`, `not-installed` (color-coded if TTY)
   - `--verbose` adds `URL`, `WORKING_DIR`, last 3 log lines per project
   - Example:
     ```
     SUBDOMAIN  PORT  STATUS
     status     4001  running
     coach      3001  running
     hello      8080  stopped
     ```
10. **`paas logs <subdomain> [-n N] [--no-follow]`** (audit rec #12):
    - Default: last 100 lines + follow (`journalctl --user -u paas-{{subdomain}} -n 100 -f`)
    - `--no-follow`: print and exit
    - `-n N`: change tail length
11. **`paas remove <subdomain>`**:
    - Acquire flock.
    - `systemctl --user disable --now paas-{{subdomain}}.service`.
    - Delete `~/.config/systemd/user/paas-{{subdomain}}.service`.
    - `systemctl --user daemon-reload`.
    - Read `cloudflared/config.yml`, parse, remove the matching ingress entry, re-serialize.
    - Write to `.new`, validate via `cloudflared tunnel ingress validate`, `mv` into place.
    - **DNS cleanup**: `DELETE /zones/{{zone_id}}/dns_records/{{record_id}}` for the matching CNAME. The cert is auto-revoked when the record disappears. Idempotent on 404.
    - `sudo systemctl restart cloudflared`, poll.
    - Release flock.

### Test ramp

12. **Test 1 ‚Äî Bun HTTP dummy**:
    - Create `/tmp/test-project/` with a 5-line Bun HTTP server (`server.ts`), a `.local-dev.yaml` (`subdomain: test, port: 4099, command: bun run server.ts`), no `.env`.
    - `paas register --dry-run /tmp/test-project`. Verify the printed diff matches expectations.
    - `paas register /tmp/test-project`.
    - `curl -L https://test.play.ris.hair` (after CF Access auth via incognito browser) returns the server's response.
    - `paas list` shows the project as `running`.
    - `paas logs test --no-follow -n 5` prints recent log lines.
    - `paas restart test`, verify it comes back.
    - `paas down test`, `paas list` shows `stopped`.
    - `paas up test`, back to `running`.
    - `paas remove test`. Verify systemd unit gone, ingress entry gone, `cloudflared` reload triggered, `https://test.play.ris.hair` returns 404.
13. **Test 2 ‚Äî Next.js dev mode** (audit rec #8 ‚Äî falsifiable criteria):
    - Bring up a Next.js dev project (any minimal scaffold) on a port.
    - `paas register` it.
    - In browser at `https://<sub>.play.ris.hair`, sign through CF Access:
      - Browser console shows `[HMR] connected` (or equivalent), no `WebSocket connection failed`.
      - Modify a `.tsx` file, save; change appears within 5 seconds without hard reload.
    - (Next.js `experimental.allowedDevOrigins: ['<sub>.play.ris.hair']` may be needed ‚Äî document.)
14. **Test 3 ‚Äî Streaming**:
    - Bring up a project with an SSE-via-POST or streamed-fetch endpoint (e.g., the coach app's `/api/chat`).
    - From the CF-Access-authenticated browser, exercise the streaming endpoint.
    - Verify tokens arrive in real-time (not buffered into a single burst).
15. **Test 4 ‚Äî Cross-subdomain Access cookie**:
    - Register two projects (`a` and `b`).
    - Sign through CF Access once on `a.play.ris.hair`.
    - Visit `b.play.ris.hair` ‚Äî should NOT require re-auth (cookie shared across subdomains in the same Access app).
16. **Test 5 ‚Äî Atomicity / concurrent register**:
    - In two terminals: simultaneously run `paas register /tmp/projA` and `paas register /tmp/projB`.
    - Verify both projects end up registered cleanly: both systemd units exist, both ingress entries present, both reachable. No orphans, no corrupted YAML.
17. **Test 6 ‚Äî Crash mid-register**:
    - Modify `register.ts` temporarily to throw after writing `config.yml.new` but before `mv`. Run `paas register`.
    - Verify: `config.yml` (the live file) is unchanged. `config.yml.new` may exist; next run cleans it up.
    - Revert the throw.

### Documentation

18. `paas/README.md` covers:
    - Setup (assumes E-047 ran first; references the linger + sudoers requirements)
    - `.local-dev.yaml` format with all fields and defaults
    - "Add a new project" workflow (the canonical happy path)
    - The 2-30s reload window: every `register`/`remove` causes a brief tunnel reconnect that drops in-flight long-lived connections. Suggest batching multiple registrations during off-hours.
    - The SSE-over-GET caveat: cloudflared buffers GET-based SSE streams (cloudflared issue #1449). Use POST + fetch streaming for real-time. AI SDK chat apps are unaffected (they POST).
    - The HMR caveat: dev mode may need `experimental.allowedDevOrigins`.
    - Secrets ownership: `.env` is gitignored and user-owned (mode 0600); systemd's `EnvironmentFile=` reads at unit start; `paas list` and `journalctl` are safe to share ‚Äî secret values never appear there.
    - Troubleshooting: tunnel down, systemd unit failed, CF Access misconfigured.

## Success Criteria

- All 7 subcommands (register, up, down, restart, list, logs, remove) implemented and pass manual testing per Tests 1-6
- Schema validation rejects malformed `.local-dev.yaml` (bad subdomain regex, reserved name, missing required field, port out of range, env_file pointing outside working_dir) with clear errors
- A dummy Bun HTTP project goes from "in this dir" ‚Üí "reachable at <subdomain>.play.ris.hair" in one `paas register` invocation
- The Next.js dev-mode test (Test 2) confirms HMR works through the tunnel with the falsifiable criteria above
- The streaming test (Test 3) confirms real-time POST streaming
- The Access cookie test (Test 4) confirms cross-subdomain single-sign-on
- The atomicity test (Test 5) confirms concurrent `paas register` invocations serialize cleanly
- The crash-recovery test (Test 6) confirms tunnel config is never left in a half-applied state
- `paas list` output matches the 3-column format
- `paas remove` is fully reversible (re-registering re-creates the same setup)
- The generated `config.yml` has the "Generated by paas. Do not hand-edit." header (audit rec #9)
- `paas` detects existing hand-edits to `config.yml` (missing header) and backs up the file with a timestamp before overwriting
- `paas/README.md` covers the happy path and the 5 most likely failure modes (tunnel down, systemd unit failed, port conflict, env_file missing, CF Access misconfigured)

## Blast Radius

- Adds `/home/<user>/joe-hudson/paas/` directory with `bin/`, `lib/`, `templates/`, `README.md`
- Adds (or modifies) `~/.config/systemd/user/paas-*.service` files
- Modifies `/home/<user>/paas/cloudflared/config.yml` (which is symlinked from `/etc/cloudflared/config.yml`)
- Restarts `cloudflared` system service on each register/remove (**brief downtime of 2-30s, not "sub-second"** ‚Äî corrected per R-019 SIGHUP findings)
- Holds a `flock` on `cloudflared/.paas.lock` during register/remove (concurrent invocations wait)
- Does NOT modify the project repos being registered (only reads their `.local-dev.yaml`)
- Does NOT modify cloudflared credentials or the tunnel itself (only ingress within the tunnel)
