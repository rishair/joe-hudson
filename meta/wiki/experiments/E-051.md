# Deploy dashboard via G-012 PaaS at status.play.ris.hair with Cloudflare Access auth (CLOSES G-012 AND G-013)

## Hypothesis

The dashboard from E-050, plus a small `.local-dev.yaml`, can be deployed via G-012's `paas` CLI to `status.play.ris.hair` and reached from any browser after Cloudflare Access SSO. The whole flow takes one command (`paas register .`) once the project's YAML is in place. If it works cleanly, both G-012 and G-013 are proven; if it fights, the friction surfaces the design gap.

This experiment is the **single closer** for two goal exit criteria — G-013's "deployed via paas" and G-012's "one project registered and reachable" — per the G-012 audit rec #4 reconciliation. [[E-049]] is abandoned in favor of this consolidation.

## Method

### Pre-flight

1. Confirm [[E-050]] (dashboard build) is complete with a working `bun run server.tsx` and the dashboard listens on a local port.
2. Confirm [[E-048]] (paas CLI) is complete and `paas` is on `$PATH`.
3. Confirm [[E-047]] (tunnel + Access setup) is in place: tunnel running, hello-world reachable through CF Access, linger enabled, sudoers entry present.

### Author `.local-dev.yaml`

4. In `/home/claude/joe-hudson/status-dashboard/.local-dev.yaml`:
   ```yaml
   subdomain: status
   port: 4001          # or 'auto' to let paas pick from 4000-4099
   command: bun run server.tsx
   mode: built
   working_dir: /home/claude/joe-hudson/status-dashboard
   env_file: .env
   restart: always
   memory_max: 512M
   ```

### Dry-run

5. `paas register --dry-run /home/claude/joe-hudson/status-dashboard`. Confirm:
   - Generated systemd unit content matches expectations
   - Proposed ingress entry: `status.play.ris.hair → http://localhost:4001`
   - No port collision detected

### Live register

6. `paas register /home/claude/joe-hudson/status-dashboard`. Wait for the "Registered status" confirmation (should include the 2-30s tunnel reconnect window).

### G-013 closure verification

7. `paas list` shows the `status` entry with port 4001 and STATUS `running`.
8. `systemctl --user status paas-status` shows active+running.
9. `paas logs status --no-follow -n 20` shows the dashboard's startup output (port bind, ready message).
10. From incognito browser at `https://status.play.ris.hair` (or `https://status-play.ris.hair` if SSL path B was chosen in REQ-004):
    - CF Access login page appears
    - Sign in with the policy-included Google account
    - Dashboard renders with all 4 must-have sections populated (per G-013's correctness criteria)
11. Refresh the dashboard: any wiki change made via filesystem should appear within one refresh (live state, not stale).
12. Browser dev tools confirm no `OPENROUTER_API_KEY` or other secret leakage in HTML, JS bundles, or network responses.

### G-012 closure verification

13. `paas list` accurately reflects the dashboard's state (the dogfood proves the CLI works end-to-end for a real project, not just a dummy).
14. `paas remove status` followed by `paas register .` re-creates the same setup (idempotency / reversibility).
15. Concurrent test: open a long-running session in the dashboard (mid-page-load), THEN run `paas register /tmp/another-project` in another shell. The dashboard's session gets dropped (expected — tunnel reconnect drops in-flight connections), but recovers on refresh. Document this observation.
16. **Reboot survival** (extends E-047's reboot test to the real project): `sudo systemctl reboot` (or the simulation). Within 30s, both `cloudflared` and `paas-status` are active, and `status.play.ris.hair` is reachable after CF Access auth.

### Real-world test

17. From mobile browser (real-world test): does the dashboard render readably on a small screen? If not, log as a follow-up nice-to-have (mobile-responsive layout) — don't block on it for v1.

### Documentation

18. Update `paas/README.md` with the status-dashboard registration as the canonical worked example (replace any dummy-project example from E-048).
19. Update [[G-012]]'s exit criteria checklist: tick everything that's now true (the goal closes via this experiment).
20. Update [[G-013]]'s exit criteria checklist: tick the "deployed via paas" criterion.
21. **Add an addendum to `meta/wiki/playbooks/cloudflare-deploy.md`** (audit rec #17): a "Personal-play deployment path" section pointing at the `paas register .` workflow as the alternative to the Workers/Pages path. Cross-reference G-012's `paas/README.md` for the full schema.

- No surprises in the paas CLI surfaced by this real deploy that weren't caught in E-048's dummy-project testing — if surprises ARE found, document them and create a follow-up experiment (don't try to fix mid-experiment)
