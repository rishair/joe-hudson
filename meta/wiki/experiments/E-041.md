# Add client-side SQLite for full-fidelity conversation persistence

## Hypothesis

Adding a client-side SQLite repository (using `sqlocal` over OPFS — the [[R-015]] recommendation) to E-040's chat scaffold lets the user have multiple conversations, switch between them, and have all messages persist across page reloads at full AI SDK message fidelity (text, tool calls, tool results, message parts). The Repository pattern from [[coding-architecture]] keeps the SQLite specifics behind an interface so the implementation can swap later (e.g., to raw `wa-sqlite` or `@sqlite.org/sqlite-wasm` directly). The schema stores each `UIMessage` as a single JSON blob, not normalized, to preserve byte-equal round-trip per the audit-tightened success criterion.

## Method

1. Set COOP/COEP headers in `next.config.js` from day one — `Cross-Origin-Opener-Policy: same-origin` + `Cross-Origin-Embedder-Policy: require-corp` globally. This unlocks the default `opfs` VFS (multi-connection) rather than the single-connection-per-origin `opfs-sahpool` fallback. Safe because `/api/chat` is same-origin; OpenRouter calls go server-side and are unaffected.
2. Add `sqlocal` and `@sqlite.org/sqlite-wasm` packages. Sqlocal handles the worker bootstrap; Next.js webpack/turbopack handles the `new Worker(new URL(...))` pattern natively. No extra Vite plugin needed.
3. Define interfaces in `web-app/app/lib/types/` per the [[coding-architecture]] playbook's four-method shape (`find`/`create`/`update`/`delete` with rich param objects, NOT many narrow methods):
   - `Conversation { id, title, createdAt, updatedAt }`
   - `ConversationRepository`:
     - `find(criteria?: { id?: string; orderBy?: 'createdAt' | 'updatedAt'; limit?: number }): Promise<Conversation[]>`
     - `create(input: { title: string }): Promise<Conversation>`
     - `update(criteria: { id: string }, patch: Partial<Conversation>): Promise<void>`
     - `delete(criteria: { id: string }): Promise<void>`
   - `MessageRepository`:
     - `find(criteria: { conversationId: string; limit?: number; offset?: number }): Promise<UIMessage[]>`
     - `create(input: { conversationId: string; message: UIMessage }): Promise<void>`
     - `delete(criteria: { conversationId: string }): Promise<void>`
   - Message type is AI SDK's `UIMessage` (full fidelity, all message parts)
   - Callers needing a singleton (e.g., "get conversation by id") do `(await convRepo.find({ id }))[0] ?? null` per the playbook convention
4. Implement `SqliteConversationRepository` and `SqliteMessageRepository` using sqlocal's tagged-template `sql\`...\`` API. Schema: two tables (`conversations`, `messages`) with a single `payload TEXT` column on `messages` holding `JSON.stringify(UIMessage)`. Index `messages(conversation_id, position)`. Do NOT normalize tool calls into typed columns — that loses round-trip fidelity. Both repositories share one sqlocal instance (lazily created in module scope on the client, NOT instantiated at import time by any server component). Kick off a `SELECT 1` warmup query in a top-level `useEffect` so the WASM is loaded before the user sends their first message. Wrap `create` with up-to-3-retry exponential backoff on `SQLITE_BUSY` (possible if two tabs write at once on the default `opfs` VFS).
5. Add a thin Zustand reactive layer (`useConversationState`) that holds the active-conversation id and the current snapshot of conversations; on Repository writes, update the Zustand snapshot. The Repository is the source of truth; Zustand is only for re-render reactivity (per the revised [[coding-architecture]] playbook)
6. Update the chat UI:
   - Sidebar listing conversations (click to switch)
   - "New conversation" button
   - On mount: hydrate `useChat` `initialMessages` from `messageRepo.find({ conversationId: activeId })`
   - On every assistant message complete: persist via `messageRepo.create({ conversationId: activeId, message })``
   - Per the playbook: do NOT try to sync `useChat` two-way mid-stream; hydrate on mount, persist on completion
7. Test: send a few messages, refresh, verify everything restores including tool calls if any were made (a tool-using path will be wired in E-043, but the persistence layer should handle the message shape today even without tools firing)

## Success Criteria

- DB initializes on first load, persists across reloads
- Three conversations can be created, switched between, and all retain their message histories on reload
- A message with tool calls (manually crafted or from E-043 once wired) round-trips through SQLite with byte-equality of the serialized `UIMessage` JSON before and after (verify by `JSON.stringify(original) === JSON.stringify(roundTripped)` for a representative sample)
- Repositories are behind interfaces; the Zustand reactive layer talks to interfaces, not concrete classes
- Composition pattern is set up even if no wrapper is needed yet (e.g., `LoggingConversationRepository` available as an opt-in)
- Caching wrapper (if used) handles write-invalidation correctly per the playbook's pitfall callout

## Blast Radius

Modifies `web-app/app/`. Adds `web-app/app/lib/{repos,types,state}/` (state/ holds the thin Zustand reactive layer per the [[coding-architecture]] playbook; not a Stores directory). Adds dependency for chosen browser-SQLite package. Stores DB in OPFS or IndexedDB depending on R-015 choice. Does not modify server-side anything except possibly adding a worker file route.

The byte-equality check is the audit-tightened success criterion. The schema stores the whole `UIMessage` as a single JSON column (per R-015) precisely to make this trivial — `JSON.parse(JSON.stringify(msg))` is the entire transform.

The sample[3] case covers R-016's `data-resources` typed data part shape that E-044's resource attribution will read. No schema change needed for E-044 — the layer is already byte-equality preserving for arbitrary data parts.

## Analysis

Three implementation decisions worth noting.

**FK cascade requires `PRAGMA foreign_keys = ON` per connection.** SQLite ships with FK enforcement disabled — a quirk inherited from before its FK implementation existed. The pragma is per-connection (per-tab in the browser, per-test in the verifier) and is the only reason `ON DELETE CASCADE` in the messages table works at all. Without it, deleting a conversation would leave orphan message rows, and the first behavioral test caught exactly that ("10 stragglers"). Added to `getDb()` right after the SELECT 1 warmup so every tab inherits it.

**ConversationRepository and MessageRepository stayed split.** The audit's issue #2(e) suggested collapsing them into a single Repository. Kept them separate because the sidebar reads the conversation list independently of any specific conversation's messages (the sidebar re-renders on conversation rename, but not on message append; conversely, the message panel re-renders on append but not on rename). Combining them would force the find signature to either fetch messages eagerly on every conversation list query (wasteful) or expose a `findMessages` second-method alongside `find` (the narrow-method anti-pattern the playbook calls out). Two repositories sharing one SQLocal instance at the wiring layer captures the cohesion without fattening either interface.

**Message position is assigned at insert time inside a transaction.** Two concurrent calls to `messages.create` for the same conversation would otherwise race on `MAX(position) + 1`. The transaction wrapper (`db.transaction(async tx => ...)`) acquires SQLite's write lock for both the SELECT MAX and the INSERT, so positions are guaranteed unique. The behavioral test asserts this — 10 concurrent `Promise.all` inserts produce positions 0..9 with no duplicates. Without the transaction, two would collide on the same position and the second's UNIQUE-on-position would fail (or, worse, both would succeed and leave the conversation history with two messages at the same position).

The verify-e041 script uses an in-process `SQLiteMemoryDriver` rather than the real OPFS path. That's a necessary substitution because OPFS only exists in a browser and headless Chrome is unavailable in this environment. The code path differences are limited to (a) the worker-bootstrap hop that the browser path takes for query messages, and (b) the OPFS file persistence. Both are sqlocal-internal and exercised the same way by `sql\`...\`` calls; the Repository code, the migration runner, the byte-equality serialization, and the FK cascade are identical in both modes. The R-015 finding that COOP/COEP headers are needed for OPFS is honored by `next.config.js` (set in E-040); a manual smoke in a real browser would only re-verify the worker hop and `localStorage.opfs` persistence, both of which sqlocal itself tests.

The `onFinish` integration with `useChat` persists user + assistant messages together once the stream settles. An alternative — persisting the user message synchronously on `sendMessage` and the assistant message on `onFinish` — was rejected because it gives the user message a timestamp ahead of the assistant message even when both are produced in the same logical turn. Using `onFinish` for both means they share the same `created_at` group, which matches the chat's conceptual model of a turn.

## Dead Ends

None at the implementation level. Two friction points worth noting for future agents:

**SQLocal in Bun without a real browser requires the explicit memory driver.** Bun defines `globalThis.Worker`, so the default SQLocal constructor path tries to spawn a worker module that fetches the OPFS-backed sqlite-wasm bundle, which fails. The fix in `scripts/verify-e041.ts` is to construct a `SQLocalProcessor` with a `SQLiteMemoryDriver` and pass it as the `processor` option. This is documented sqlocal API and not a hack, but it's the kind of thing the next agent writing a test will run into.

**Boot race between `config` and the first query.** The SQLocal constructor fires off the config message but doesn't await driver init. In a real browser this is masked by the postMessage queue separator between client and worker thread; in-process (e.g., a test using the memory driver), both run on the same microtask queue and the first query can race ahead of init. `verify-e041.ts` blocks on `onConnect` to side-step this. Production code in `getDb()` issues a `SELECT 1` warmup query before any user-triggered query — which serializes correctly in the browser worker path but technically still has the race in-process. Not a production concern, but worth a comment if someone writes a Node-side smoke test in the future.

## Next Steps

The persistence layer is feature-complete for E-041's scope. Downstream effects:

- **E-044 (resource attribution modal)** consumes `message.parts.find(p => p.type === 'data-resources')` per R-017. The persistence layer round-trips arbitrary typed data parts byte-exactly, so E-044 needs no schema work — just UI work to find and render the data-resources part.
- **E-043 (v5b coach)** writes the `data-resources` part via `createUIMessageStream` per R-016. The route handler will set the part shape; the persistence layer will store and rehydrate it without modification. E-043 needs only the typed UIMessage parameterization (`UIMessage<never, { resources: ResourceAttribution }>`) which is a type-level concern, not a schema concern.
- **E-042 (wiki view)** is independent of E-041; the wiki view doesn't write to SQLite at all.

The Repository interface is the seam for any future server-side sync (G-011+). A `RemoteConversationRepository` implementing the same interface, wrapped via composition (`SyncedConversationRepository(local, remote)`), would let chat history sync to a server without touching the chat UI or the message renderer.

A small follow-up item to file (not standalone): the [[ai-web-app]] playbook's "Message persistence" section should be updated to cite `sqlocal` + `getDb()` + `applyMigrations` as the canonical pattern, with a one-line warning about `PRAGMA foreign_keys = ON`. Adding via a future playbook touch rather than a separate work item.
