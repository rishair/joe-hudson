# Deploy web-app to Cloudflare with automated GitHub pipeline

## Hypothesis

The G-010 web app (E-040 scaffold + E-041 persistence + E-042 wiki + E-043 coach pipeline + E-044 modal + E-045 welcome) can be deployed to Cloudflare such that pushing to the main branch on GitHub automatically rebuilds and re-deploys. The pattern that emerges becomes a `cloudflare-deploy` playbook for any future project in this repo, so the second consumer (G-011's model registry or any subsequent web app) doesn't repeat the discovery work.

The web app uses Next.js App Router, Bun, and several browser features (OPFS for SQLite per [[E-041]], COOP/COEP headers per the [[ai-web-app]] playbook). Cloudflare's options for Next.js apps as of May 2026 — Cloudflare Pages with the `@opennextjs/cloudflare` adapter, Cloudflare Workers via `wrangler`, or hybrid setups — need to be evaluated against these requirements.

## Method

1. **Evaluate Cloudflare deployment options for Next.js + Bun.** Decide between Pages with `@opennextjs/cloudflare`, Workers, or another path. Document the tradeoffs (cold start, SSR behavior, edge runtime constraints, OPFS / COOP-COEP compatibility, Bun-specific build needs).
2. **Set up a Cloudflare account and project** for this app. Configure DNS for a chosen subdomain (or use the workers.dev / pages.dev default for v1).
3. **Wire `OPENROUTER_API_KEY`** as a Cloudflare environment secret (NEVER in the repo, NEVER in client bundles). Document the path from local `.env` → Cloudflare secret.
4. **Build pipeline**: configure GitHub Actions (or Cloudflare's built-in Pages git integration) to rebuild on push-to-main. Verify the build runs Bun, produces a deployable artifact, and the artifact serves correctly when deployed.
5. **End-to-end verification**:
   - Public URL serves the welcome flow (E-045) on first visit
   - `/api/chat` route handler streams a coach response
   - Wiki view (E-042) renders entries from the ingested compendium
   - SQLite persistence (E-041) survives a reload (OPFS works in the deployed environment)
   - Resource modal (E-044) opens and switches focus correctly
   - No secrets leaked in any client payload
   - Cost cap (E-043's circuit breaker) is observable in Cloudflare logs
6. **Capture the pattern as a playbook**: `meta/wiki/playbooks/cloudflare-deploy.md` documenting: account setup, secret wiring, build config, Next.js adapter choice + caveats, common pitfalls (Bun on Cloudflare CI, OPFS/COOP-COEP, edge-vs-node runtime tradeoffs), and a "how to apply to a new project" recipe.
7. **Cost note**: Cloudflare Pages has a generous free tier; Workers paid plan is $5/mo. Document the running cost.

Implementation ramp: deploy to a staging-equivalent subdomain first; verify; then update DNS to the chosen production subdomain (or leave at the staging URL for v1 if the project is still being shown to a handful of people).

## Success Criteria

- Web app is reachable at a public URL
- All 6 features (welcome, chat, wiki, persistence, modal, cost-cap) work in the deployed environment
- `OPENROUTER_API_KEY` is configured as a Cloudflare secret and not present in any client payload
- A `git push origin main` triggers a build + deploy without manual intervention
- Build log + deploy log are observable from Cloudflare's dashboard
- `meta/wiki/playbooks/cloudflare-deploy.md` exists with a "how to apply to a new project" recipe
- The cost (CF Pages free tier vs Workers paid) is documented in the playbook

## Blast Radius

Adds Cloudflare account-side resources (project, DNS records, secrets, build config) — these live in the Cloudflare dashboard, not in this repo. Adds `.github/workflows/deploy.yml` (or `wrangler.toml` / `cloudflare-pages` config) to the repo. Adds the `cloudflare-deploy` playbook. Does NOT modify the web app's source code beyond what's needed for Cloudflare compatibility (e.g., adapter config).
