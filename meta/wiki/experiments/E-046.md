# Deploy web-app to Cloudflare with automated GitHub pipeline

## Hypothesis

The G-010 web app (E-040 scaffold + E-041 persistence + E-042 wiki + E-043 coach pipeline + E-044 modal + E-045 welcome) can be deployed to Cloudflare such that pushing to the main branch on GitHub automatically rebuilds and re-deploys. The pattern that emerges becomes a `cloudflare-deploy` playbook for any future project in this repo, so the second consumer (G-011's model registry or any subsequent web app) doesn't repeat the discovery work.

The web app uses Next.js App Router, Bun, and several browser features (OPFS for SQLite per [[E-041]], COOP/COEP headers per the [[ai-web-app]] playbook). Cloudflare's options for Next.js apps as of May 2026 — Cloudflare Pages with the `@opennextjs/cloudflare` adapter, Cloudflare Workers via `wrangler`, or hybrid setups — need to be evaluated against these requirements.

## Method

Done in three phases. Phase A (architecture + code prep) and Phase B (build pipeline) are complete; Phase C (the live deploy) is blocked on [[REQ-003]] (Cloudflare account + API token + GitHub repo secrets — credentials only the user can mint).

**Phase A — architecture**

1. Evaluate Cloudflare deployment options. **Decision**: `@opennextjs/cloudflare` (1.19.11) → Cloudflare Workers (NOT Pages — `next-on-pages` is deprecated as of 2026 per Cloudflare's own framework guide). Node.js runtime (not Edge). `compatibility_date: 2025-09-01` to get `nodejs_compat` + `process.env` auto-population from wrangler vars + Workers VFS for `node:fs` against `/bundle`.
2. Audit web-app for Cloudflare-runtime compatibility. **Surprise**: the 38 MiB / 2,379-file `content/wiki/` directory cannot live inside the Worker bundle (3 MiB compressed cap on Free tier, 10 MiB on Paid; the tar.gz of content alone is 10.01 MiB — right on the line). Solution: serve content as **static assets via the ASSETS binding**, fetched at request time via `env.ASSETS.fetch('https://assets.local/content/wiki/<path>')`.
3. Refactor wiki/coach reads to be runtime-agnostic. Created `app/lib/runtime/wiki-asset-reader.ts` that detects Cloudflare (via `process.env.NEXT_RUNTIME === 'cloudflare'`) and uses `getCloudflareContext().env.ASSETS.fetch()` on Cloudflare, falls back to `node:fs` for local Node. Refactored `app/lib/wiki/fs-wiki-repository.ts` (E-042) and `app/lib/coach/graph-walk.ts` + `guided-walk.ts` + `retriever.ts` (E-043) to use it. The graph-walk refactor required making `loadCompendium()` and `readFileBody()` async — `guided-walk.ts`'s two callers were already in async tool-execute contexts so the cascade was small.

**Phase B — build pipeline**

4. Create `wrangler.jsonc` with the canonical OpenNext shape (main + compat_date + flags + ASSETS binding + WORKER_SELF_REFERENCE + IMAGES + vars + observability). `name: joe-hudson-coach`.
5. Create `open-next.config.ts` with the minimal `defineCloudflareConfig({})` — no R2 incremental cache binding needed since the app has no static-or-ISR pages.
6. Create `scripts/prebuild-cf.ts` that copies `content/wiki/` → `public/content/wiki/` before each build so OpenNext packs it into `.open-next/assets/`. 93ms for the full 38 MiB / 2,379 files. Idempotent: removes `public/content/wiki/` first then re-copies.
7. Create `scripts/cf-build.ts` — a security-critical build wrapper. Without it, `next build` reads `.env.local`, finds `OPENROUTER_API_KEY` + `ANTHROPIC_API_KEY` + `HF_TOKEN`, and bakes them as JS literals into `.open-next/cloudflare/next-env.mjs` AND the server handler. The wrapper: (a) strips known-secret env vars from the spawned env via an allowlist, (b) renames `.env.local` to `.env.local.cf-build-stash` for the build duration and restores after, (c) post-build greps the entire `.open-next/` tree for `sk-or-v1-*`, `sk-ant-*`, `hf_*` patterns and FAILS LOUDLY if any match. Verified: 0 leakages across 1,085 JS files after the wrapper. Restoration is wired through `process.on('exit'|'SIGINT'|'SIGTERM')` so an aborted build doesn't lose the developer's `.env.local`.
8. Create `scripts/verify-e046-assets.ts` — 13-check pre-flight that validates wrangler.jsonc structure, compatibility_date threshold, both required flags, the ASSETS binding, the workflow file shape, `server-only` guard on the runtime reader, and a Node-mode read of `_index.md` + `.ingest-manifest.json`. Runs as a step in the GitHub workflow before the wrangler-action deploy; catches misconfig in <1s.
9. Lazy-construct OpenRouter provider (`app/lib/openrouter.ts`). The previous fail-fast-at-module-load check rejected `next build` because the build deliberately runs without the secret. Refactored to lazy construction via a Proxy that delegates `apply` and `get` to the real provider, constructed on first call. Verified: local dev still works (Proxy is transparent), build succeeds without the secret, runtime fails with a descriptive error if the secret is genuinely missing post-deploy.
10. Create `next.config.js` adjustment to call `initOpenNextCloudflareForDev()` (try/catch wrapped so a fresh checkout without `bun install` doesn't crash dev).
11. Create `public/_headers` for static-asset cache control (immutable for `_next/static/*`, stale-while-revalidate for `content/wiki/*`, `noindex` for the manifest).
12. Create `.github/workflows/deploy.yml`. Triggers on push-to-main when `web-app/**`, `coach/**`, or the workflow itself changes. Concurrency-grouped (`cancel-in-progress: true`) so two rapid pushes don't race. Steps: checkout → setup-bun → install → ingest (E-039) → prebuild:cf → typecheck → verify:e046:assets → cloudflare/wrangler-action@v3 `command: deploy:cf` → smoke (curl + grep). Secrets needed: `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`.

**Phase C — live deploy** (blocked on [[REQ-003]])

13. User provisions Cloudflare account, API token (scoped: `Cloudflare Workers Scripts:Edit`, `Account Settings:Read`, `User Details:Read`), and the GitHub repo secrets `CLOUDFLARE_API_TOKEN` + `CLOUDFLARE_ACCOUNT_ID` + `OPENROUTER_API_KEY`.
14. First push to main triggers the workflow. Cloudflare's side creates the Worker `joe-hudson-coach` at `https://joe-hudson-coach.<account>.workers.dev`.
15. Run `wrangler secret put OPENROUTER_API_KEY` once (locally with the API token in env) to bind the runtime secret to the Worker.
16. Re-trigger via `gh workflow run deploy.yml` so the next build picks up the secret.
17. End-to-end verification:
    - Public URL serves the welcome flow (E-045) on first visit
    - `/api/chat` streams a coach response (graph-walk + Sonnet walker + Sonnet coach via OpenRouter)
    - Wiki view (E-042) renders entries from the ingested compendium (via ASSETS binding)
    - SQLite persistence (E-041) survives a reload (OPFS works in the deployed environment — COOP/COEP headers are emitted by next.config.js so they reach the browser even from the Worker)
    - Resource modal (E-044) opens and switches focus correctly
    - No secrets in any client payload (already verified at build time by the cf-build wrapper; re-verify against the live deploy)
    - Cost cap (E-043) observable in Cloudflare logs (process-level circuit breaker still works since OpenNext's worker runs as a single long-lived Worker instance per request burst)
18. Capture the pattern as a playbook: `meta/wiki/playbooks/cloudflare-deploy.md`. **Done — landed during Phase B**; the playbook is shape-complete and ready for the next consumer. Refinement based on the live deploy may add gotchas to "Common pitfalls."

## Result

### What landed

Cloudflare backend: `getCloudflareContext().env.ASSETS.fetch('https://assets.local/content/wiki/<relPath>')`. Node backend: `fs.promises.readFile(path.join(WIKI_ROOT, relPath), 'utf-8')`. Defensive: refuses paths containing `..` or starting with `/`. Both exported APIs (`readWikiAsset` async, `readWikiAssetSyncOrNull` Node-only) memoize the chosen backend.

The `getWikiRoot()` and `existsSync/readFileSync` imports are gone — all reads route through `readWikiAsset`. `parseIndex()` stays sync (string parsing only).

## Analysis

**The deploy is fully prepared and dry-run-validated. The blocker is purely credential-side.** Everything that this experiment can validate without an actual Cloudflare account has been validated. The 7,134-KiB / 1,444-KiB-gzip worker bundle would push successfully today if a token were available.

**Security finding worth its own line item**: Next.js's standalone build bundles values from `.env.local` into the server function bundle by default. When OpenNext consumes that standalone build for a Cloudflare Worker, those values become Worker bundle literals — uploaded to Cloudflare's edge as plain JS strings. Without the `cf-build.ts` wrapper that strips the env AND temporarily moves `.env.local` aside AND post-scans the bundle, a developer running `bun run deploy:cf` locally would silently leak their `OPENROUTER_API_KEY` into the public Worker JS. This finding is captured in the playbook as "the most important pitfall" because it would be invisible without the post-scan step. The 5-second cost of the scan is well worth catching a key-leak before deploy.

**Adapter choice: `@opennextjs/cloudflare` was the only viable path.** `next-on-pages` is deprecated. Vercel-on-Cloudflare doesn't exist. Hosting on Vercel and proxying through Cloudflare would defeat the "push-to-main rebuilds on Cloudflare" goal. Running raw `wrangler` against a hand-rolled Worker would require reimplementing Next's route handler runtime. OpenNext is the official, supported, current path — and it works.

**The 10 MiB bundle limit is a real binding constraint, not theoretical.** Our content tarball is 10.01 MiB compressed — exactly on the line. Without the ASSETS-binding pattern, we'd have been forced to either (a) trim the corpus (would break E-039's exit criterion), (b) move to R2 (would add operational + cost complexity for v1), or (c) pay for Workers Paid (would barely fit and provide no headroom). The ASSETS pattern is the right answer and is documented in the playbook as the default for any app that reads markdown / JSON / image content at runtime.

**Compatibility date matters more than it looks.** `2025-09-01` was chosen because: pre-2025-04-01, `process.env` isn't auto-populated from wrangler `vars` (manual `globalThis.<var>` plumbing needed); pre-2025-09-01, `node:fs` isn't available against the bundled-modules VFS. The cost of bumping to `2025-09-01` is zero (Cloudflare's policy: newer compat dates strictly add capabilities); the cost of choosing `2024-12-30` (OpenNext's example) is several hours of post-deploy debugging of "why isn't this env var visible."

**On the playbook quality**: the playbook documents the pattern, the canonical files, the three biggest gotchas (bundle ceiling, env-bake-in, COOP/COEP through next.config not _headers), and a step-by-step recipe. It would unblock a next consumer (e.g., G-011's web app) without that consumer needing to re-derive any of these decisions. The author's confidence the playbook is correct is "high for the pattern, medium for the live-deploy mechanics" — until Phase C actually runs end-to-end, the recipe steps for the post-deploy `wrangler secret put` are tested only by Cloudflare's own docs, not by us.

## Dead Ends

**`fs.readFileSync` for runtime data files on Cloudflare** — initially considered, since `compatibility_date >= 2025-09-01` gives node:fs against `/bundle`. Doesn't scale: the bundle ceiling cap means 38 MiB of content can't ride along. Replaced with the ASSETS-binding pattern.

**Per-file ASSETS fetch from sync code** — initially attempted to keep `readFileBody()` sync by pre-loading all bodies at boot. Doesn't scale either: 2,379 files × ~16 KiB average = 38 MiB resident in Worker memory per instance, AND the boot-time cost would be hundreds of fetch calls. Replaced with async + lazy + in-memory cache (the cache is hot once a graph-walk starts touching the same 4-7 files repeatedly).

**Letting Next bake env-vars into the bundle and trusting "no public URL means no exposure"** — considered as a "ship now, fix later" path. Rejected. The deploy artifact (wrangler's dry-run output) is the same one that Cloudflare's edge serves; if it has a literal key, that key is on Cloudflare's CDN edge and `curl https://.../foo.js | grep sk-or-` returns it. There is no safe "no one knows the URL" posture for a Worker.

**Storing `OPENROUTER_API_KEY` in a GitHub Actions `env:` block and passing it through wrangler-action** — considered. Rejected. The wrangler-action can set secrets on the Worker on every deploy, but doing it via a GitHub Actions env var means the secret value transits the Actions log on every run. The cleaner pattern (Cloudflare's recommendation) is one-time `wrangler secret put` from a developer machine, with subsequent rotations going through the dashboard.
